# Communication in Engineering

## 1. Quick Summary
| Area | Details |
|---|---|
| Topic | How to write, speak, and present effectively as a software engineer |
| Difficulty | Beginner |
| Used For | PR reviews, incident reports, technical specs, and stakeholder updates |
| Common Mistake | Using highly technical jargon to explain a delay to a business stakeholder |
| Performance | Time: Writing good docs takes time upfront but saves hundreds of hours later. |

## 2. Engineering Story
A critical payment service went down for 45 minutes on a Friday afternoon. You fixed it quickly by reverting a bad database migration. When the Director of Engineering asked what happened in the incident channel, you typed: *"The v4 migration script had a missing foreign key constraint on the `user_uuid` which caused a cascade failure in the Postgres query planner, resulting in a table lock."* The Director stared at the screen, completely lost, and asked, "So... are customer payments working or not?" Bro, you answered the technical question perfectly, but you failed the communication test. You provided deep implementation details to an audience that only cared about business impact.

## 3. Real-World Analogy
Think of technical communication like giving directions in a busy Indian city.

| Scenario | Engineering Equivalent |
|---|---|
| Auto driver asking for directions | Explaining a code change to another engineer (needs exact street names and landmarks) |
| Relatives asking where you live | Explaining a feature to a PM (needs the general area and travel time) |
| Explaining a traffic jam to your boss | Explaining an outage to a stakeholder (needs to know the delay time, not the model of the broken car) |

## 4. Concept Explanation
Communication in engineering is about **audience adaptation**. The way you talk to a compiler is not the way you talk to a Tech Lead, which is different from how you talk to a Product Manager. Good communication is lossy compression; you must strip away irrelevant details while preserving the core message. In engineering, written communication (PR descriptions, design docs, post-mortems) is actually more important than verbal communication because it scales. A well-written Slack message can unblock a teammate in a different timezone while you sleep. Bro, if you write terrible PR descriptions that just say "Fixed bug," your teammates will hate reviewing your code, and your promotions will stall.

## 5. Syntax Table
Here is the "syntax" for writing a good Slack update.

| Bad Update | Good Update (The BLUF Format - Bottom Line Up Front) |
|---|---|
| "I spent all day looking at the auth service. I found a bug in the token parser. I think I fixed it. Going to write a test now." | **Status:** In Progress. <br/> **Impact:** Found the root cause of the login bug. <br/> **Next Steps:** Writing tests, PR will be ready by 3 PM. |
| "Database is down." | **SEV-1 Alert:** Production Postgres is rejecting connections. <br/> **Impact:** Users cannot log in. <br/> **Action:** I am restarting the connection pooler. |

## 6. Beginner Example
Here is a very common beginner mistake when writing a Pull Request description.

```markdown
# PR Title: Update login

## Description
Fixed the thing we talked about in standup.
```
*(No context. Reviewers have no idea what "the thing" was, or what files to look at closely).*

## 7. Real-World Engineering Example
Here is how a senior engineer writes a Pull Request description.

```markdown
# PR: Fix race condition in Auth Token Refresh (Ticket-123)

## Why are we doing this?
Users on slow connections were getting logged out randomly because the refresh token API was firing twice concurrently.

## What changed?
1. Added a debounce wrapper to the `refreshToken` network call.
2. Added a loading state to the UI to prevent double-clicks.

## How to test?
1. Throttle network to "Slow 3G" in Chrome DevTools.
2. Click "Login" twice rapidly.
3. Verify only one network request is sent.
```

## 9. Performance Table
Impact of communication mediums.

| Medium | Synchronicity | Best Used For |
|---|---|---|
| Slack Message | Near-Sync | Quick questions, status updates |
| Video Call | Sync | Complex debugging, architectural debates |
| Written Doc (RFC) | Async | Major design decisions, cross-team alignment |
| Jira/Ticket | Async | Tracking work history, defining requirements |

## 10. Top Interview Questions
| Question | Answer |
|---|---|
| How would you explain an API to a non-technical stakeholder? | I would use the restaurant analogy. The user is the customer, the frontend is the waiter taking the order, the API is the ticket sent to the kitchen, and the database is the kitchen cooking the food. |
| Describe a time you had a miscommunication with a teammate. | (Focus on resolution): "We misaligned on the data format for a microservice. I assumed JSON, they built XML. We resolved it by writing a strict Swagger doc, and I learned to always write the contract down before coding." |
| Why is documentation important if the code is clean? | Clean code explains *how* the system works. Documentation explains *why* the system was built that way, capturing business decisions and edge cases that aren't obvious from syntax alone. |
| What do you do if you disagree with a code review comment? | I assume positive intent. I ask for clarification on the review. If we still disagree, I suggest a quick 5-minute call to resolve it verbally, as tone is often lost in text. |
| How do you run an effective technical meeting? | Have a clear agenda sent beforehand, timebox the discussions, take notes, and end with explicitly assigned action items. If there is no agenda, the meeting should be an email. |

## 11. Tricky Questions & Edge Cases
What if a senior engineer leaves aggressive, nitpicky comments on your PRs? Bro, do not react defensively in text. Text communication strips away tone, making neutral statements seem hostile. Reply politely, address the technical merits, and if it continues, take it to a 1-on-1 video call to humanize the interaction.

## 12. Real-World Usage
At Amazon, PowerPoint is banned. Instead, engineers must write 6-page narrative memos for major decisions. The meeting starts with everyone silently reading the memo for 20 minutes before discussion begins. This forces deep, clear, written communication over charismatic presenting.

## 13. Best Practices
| DO | DON'T |
|---|---|
| Use "BLUF" (Bottom Line Up Front) when updating managers. | Bury the lead. State the impact first, details second. |
| Use thread replies in Slack to keep channels clean. | @channel for non-emergencies. |
| Write documentation assuming the reader knows nothing about your project. | Assume "it's obvious" because you just wrote it. |

## 14. Production Notes
<Admonition type="info" title="The Post-Mortem Rule">
During an incident post-mortem document review, communication must be strictly blameless. Do not write "Rahul deployed bad code." Write "The deployment pipeline allowed an untested configuration to reach production." Focus on the system failure, not the human error.
</Admonition>

## 15. Common Mistakes
| Mistake | Correction |
|---|---|
| Answering "Why?" with "How?" | If asked why a feature is late, don't explain the code bug. Explain the timeline impact. |
| Writing massive blocks of text | Use bullet points, bold text, and line breaks for readability. |
| Asking "Are you there?" on Slack | Ask the question immediately with all context ("Hello! Can you help me with X?") |
