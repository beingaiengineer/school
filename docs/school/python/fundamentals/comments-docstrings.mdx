| Area | Details |
|---|---|
| Topic | Python Comments & Docstrings |
| Difficulty | Beginner |
| Used For | Explaining *why* code exists and documenting APIs |

Bro, writing code that a computer can understand is easy. Writing code that another human engineer can understand six months later is the true mark of a senior developer.

## 2. Engineering Story

You inherit a massive, 5,000-line script from an engineer who quit the company. On line 402, you see: `time.sleep(3)`. Why is there a 3-second delay? Is it waiting for a database to boot? Is it a rate limit for an API? Is it just a leftover hack from debugging?

You don't know. So you remove it to speed up the code. The next day, the entire production pipeline collapses because the downstream API was silently dropping requests that came in too fast. If the previous engineer had just written `# Wait 3 seconds to avoid Shopify API rate limits`, you wouldn't have brought down production. Comments are the connective tissue of engineering teams.

## 4. Concept Explanation

In Python, there are two distinct ways to leave notes:

**1. Comments (`#`)**: Used for inline explanations. Anything after the hash symbol on that specific line is completely ignored by the Python compiler. They explain the *why* behind a tricky piece of logic.

**2. Docstrings (`"""`)**: Short for "Documentation Strings". These are multi-line strings placed immediately inside a function, class, or module. Unlike `#` comments, docstrings are actually saved in memory as an attribute of the object (`obj.__doc__`). Automated tools use them to generate official API documentation (like the manual you read online for libraries like Pandas).

## 5. Syntax Table

| Type | Syntax | Use Case |
|---|---|---|
| Inline Comment | `# text` | Explaining a specific line of code or edge case |
| Block Comment | `# line 1`<br/>`# line 2` | Explaining a larger block of logic |
| Function Docstring | `"""Description"""` | Placed immediately under a `def` to explain the function |
| Module Docstring | `"""Description"""` | Placed at the very top of a `.py` file |
| Type Hints in Docstrings | `:param name: str` | Sphinx/Sphinx-style documentation for arguments |

Here is the difference between a bad comment, a good comment, and a docstring.

```python
x = x + 1 # Adds 1 to x

# GOOD COMMENT: Explains the 'why'
x = x + 1 # Adjusting for zero-based index offset from the frontend

def calculate_taxes(income):
    """
    Calculates federal taxes based on 2024 brackets.

    Returns:
        float: The total tax owed.
    """
    # We use a hardcoded 0.20 rate for this prototype.

    return income * 0.20
```

## 7. Real-World Engineering Example

In production, docstrings are not optional. If you write a utility function for your team, you must document what it takes and what it returns, or nobody will use it.

```python
import time

def fetch_user_data(user_id: int, retries: int = 3) -> dict:
    """
    Fetches user profile data from the legacy Auth0 system.

    This function includes exponential backoff because the legacy
    system is known to randomly drop connections during peak hours.

    Args:
        user_id (int): The internal database ID of the user.
        retries (int, optional): Max connection attempts. Defaults to 3.

    Returns:
        dict: A JSON payload containing 'name', 'email', and 'role'.

    Raises:
        ConnectionError: If the legacy system fails after all retries.
    """
    for attempt in range(retries):
        # Simulate network call
        success = False

        if success:
            return {"name": "Jules", "role": "admin"}

        # Wait longer on each subsequent failure (exponential backoff)
        time.sleep(2 ** attempt)

    raise ConnectionError(f"Auth0 is completely down for user {user_id}.")
```

## 8. Internal Working

Bro, what happens to comments when Python runs?
During the Lexing phase, the Python compiler completely strips out `#` comments. They do not exist in the `.pyc` bytecode.
However, Docstrings (`"""`) are recognized as actual string objects. The compiler attaches them to the `__doc__` attribute of the function or class.

## 9. Performance Table

| Documentation Type | Overhead | Impact |
|---|---|---|
| `# Inline Comments` | Zero | Stripped before bytecode. Write as many as you need. |
| `"""Docstrings"""` | Near Zero | Stored in memory as a string reference. Completely negligible. |
| Type Hints | Zero at runtime | Evaluated statically by IDEs, ignored by the execution engine. |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Does Python have multi-line comments? | No. Python does not have a specific multi-line comment syntax like `/* */` in C++. Engineers either use multiple `#` lines or use a multi-line string `"""` (though technically the latter is an unassigned string, not a true comment). |
| What is the difference between a comment and a docstring? | Comments (`#`) are ignored by the compiler and are meant for inline explanations. Docstrings (`"""`) are retained in memory, attached to the `__doc__` attribute, and used to generate official API documentation. |
| How do you view a function's docstring at runtime? | You can call `print(my_function.__doc__)` or use the built-in `help(my_function)` command in the REPL. |
| Should you comment every line of code? | Absolutely not. Clean code should be self-documenting through good variable names. Comments should only explain *why* something is done, not *what* is being done. |
| What is a TODO comment? | It is a standardized comment format (`# TODO: Fix this later`) that IDEs scan and aggregate into a task list for the engineering team. |

## 11. Tricky Questions & Edge Cases

Bro, can a docstring be placed anywhere?
No. For Python to officially recognize a string as a docstring (and attach it to the `__doc__` attribute), it must be the **very first statement** inside the module, function, class, or method definition. If you put a `print()` statement before the `"""` string, Python just treats it as a random, floating string in memory and ignores it.

## 12. Real-World Usage

Every major open-source library (like NumPy, Pandas, or Django) relies heavily on Docstrings. When you hover over `pd.read_csv()` in VS Code and a massive helpful popup appears explaining the arguments, your IDE is literally reading the `__doc__` attribute written by the Pandas engineers. Without docstrings, the modern Python ecosystem would collapse into chaos.

## 13. Best Practices

| DO | DON'T |
|---|---|
| DO write a Docstring for every public function and class. | DON'T write a Docstring for a 2-line internal helper function. |
| DO explain the *Why* in inline comments. | DON'T explain the *What* (e.g., `# This is a loop`). |
| DO update comments when you change the code. | DON'T leave outdated comments. A wrong comment is worse than no comment at all. |

## 14. Production Notes

> ⚠️ **The Danger of Outdated Comments**
> In production, code rots. An engineer updates a function's logic but forgets to update the Docstring. Six months later, another engineer reads the Docstring, assumes the function behaves a certain way, and introduces a critical bug. Always review comments during Pull Request reviews just as rigorously as the code itself.

## 15. Common Mistakes

| Mistake | Bad Code | Good Code |
|---|---|---|
| Redundant commenting | `# Set x to 5`<br/>`x = 5` | `x = 5  # Default retry limit` |
| Misplaced Docstring | `def func():`<br/>`  x = 1`<br/>`  """Docstring here"""` | `def func():`<br/>`  """Docstring first!"""`<br/>`  x = 1` |
| Leaving dead code | `# old_logic_that_didnt_work()` | Just delete it. Git remembers history. |
