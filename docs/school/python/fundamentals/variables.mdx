# Variables

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Python Variables |
| Difficulty | Beginner |
| Used For | Naming values so code is readable and reusable |
| Core Idea | A variable name points to an object |
| Common Mistake | Thinking the variable is the object itself |
| Performance | Assignment is O(1); object size matters |

Bro, a variable is a name you use to refer to a value. The name helps humans understand what the value means.

## 2. Real-World Analogy

Think of a variable like a label on a tiffin box in an office pantry.

The label says `rahul_lunch`, but the label is not the food. It points to the box.

## 3. Concept Explanation

Variables make code readable.

Use variables when:

- a value has meaning
- a value is reused
- an expression needs a clear name
- code would be confusing without a label

Avoid variables when the name adds no value, like `x = 1` in business code.

## 4. Syntax Table

| Operation | Example |
|---|---|
| Assign | `name = "Asha"` |
| Reassign | `name = "Ravi"` |
| Number | `count = 10` |
| Boolean | `is_active = True` |
| Multiple assignment | `x, y = 10, 20` |
| Constant style | `MAX_RETRIES = 3` |

## 5. Beginner Example

```python
student_name = "Asha"
score = 92

print(student_name, score)
```

Output:

```text
Asha 92
```

The variables make the data readable.

## 6. Real-World Engineering Example

```python
payment_status = "failed"
retry_count = 0
max_retries = 3

should_retry = payment_status == "failed" and retry_count < max_retries

print(should_retry)
```

Bro, names like `should_retry` explain the business decision better than a raw condition.

Python variables are names bound to objects. Reassigning a name changes what it points to. Mutating an object changes the object itself.

## 8. Performance Table

| Operation | Complexity | Note |
|---|---|---|
| Assign name | O(1) | Binds name to object |
| Reassign name | O(1) | Old object may be garbage collected |
| Mutate list | Depends | Example: `append` is usually O(1) |
| Copy large object | O(n) | Size matters |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is a variable in Python? | A name bound to an object. |
| Is Python pass-by-value? | Python passes object references by assignment. |
| What is aliasing? | Two names pointing to the same object. |
| Are constants enforced? | No, uppercase names are convention only. |
| What is dynamic typing? | Names can point to objects of different types over time. |

## 10. Tricky Questions & Edge Cases

```python
a = [1, 2]
b = a
b.append(3)

print(a)
```

Output:

```text
[1, 2, 3]
```

`a` and `b` point to the same list. Bro, this is not copying.

## 11. Real-World Usage

Variables show up everywhere:

| Area | Example |
|---|---|
| APIs | `user_id`, `request_id`, `status_code` |
| Payments | `amount`, `currency`, `retry_count` |
| AI pipelines | `prompt`, `embedding`, `model_response` |
| Monitoring | `error_count`, `latency_ms` |

## 12. Best Practices

| DO | DON'T |
|---|---|
| Use meaningful names | Use `x`, `y`, `z` for business data |
| Use booleans like `is_active` | Use unclear flags like `flag1` |
| Keep names consistent | Rename the same concept everywhere |
| Use uppercase for constants | Assume constants are enforced |

## 13. Production Notes

> Production Note:
> Clear variable names reduce debugging time during incidents.

When logs, traces, and code use names like `payment_id` and `retry_count`, the system becomes easier to reason about.

## 14. Common Mistakes

- Confusing reassignment with mutation.
- Using vague names.
- Shadowing built-ins like `list` or `str`.
- Treating uppercase constants as protected.
