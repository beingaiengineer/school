# File Best Practices

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | File Best Practices |
| Difficulty | Intermediate |
| Used For | Writing robust, production-safe file handling code |
| Main Idea | Avoid data loss, memory crashes, and cross-platform bugs |
| Common Mistake | Leaving files open or using hardcoded string paths |
| Production Angle | File operations are the #1 cause of flaky tests and missing data |

Bro, writing files locally on your laptop is easy. Writing files on a production server serving 10,000 users requires discipline.

## 2. Real-World Analogy

Think of managing files like managing a shared Google Doc in a busy office.

| Office Step | Python Step |
|---|---|
| Ask if someone else is editing | Handle file locks and permissions |
| Read one page at a time | Use generators/iterators to save RAM |
| Save a backup before major edits | Use temporary files for safe writes |
| Close the tab when done | Always use `with open()` context managers |

## 3. Concept Explanation

File best practices revolve around three core themes:
1. **Safety:** Preventing data loss if the program crashes mid-write.
2. **Efficiency:** Preventing Out-Of-Memory (OOM) errors when processing gigabytes of data.
3. **Portability:** Ensuring paths and text encodings work on Windows, Mac, and Linux seamlessly.

## 4. Syntax Table

| Bad Practice | Best Practice |
|---|---|
| `f = open('data.txt')` | `with open('data.txt', 'r') as f:` |
| `path = dir + "\\" + file` | `path = Path(dir) / file` |
| `data = f.read()` | `for line in f:` |
| `open('log.txt', 'w')` | `open('log.txt', 'a')` |
| `open('file.txt', 'w')` | `open('file.txt', 'w', encoding='utf-8')` |

## 5. Beginner Example

```python
from pathlib import Path

# Good: Using pathlib
data_dir = Path("app_data")
# Good: Ensuring directory exists without crashing
data_dir.mkdir(parents=True, exist_ok=True)

file_path = data_dir / "user_stats.txt"

# Good: Explicit encoding and context manager
with open(file_path, 'a', encoding='utf-8') as f:
    f.write("User 101 logged in\n")
```

## 6. Real-World Engineering Example

Bro, how do you prevent a corrupted file if your server crashes exactly while writing? You write to a temporary file, then rename it! Renaming is an "atomic" (instant) operation on most OSs.

```python
import os
from pathlib import Path

def safe_write(filepath, content):
    target = Path(filepath)
    temp_path = target.with_suffix('.tmp')

    # 1. Write to a temporary file
    with open(temp_path, 'w', encoding='utf-8') as f:
        f.write(content)
        f.flush()
        os.fsync(f.fileno()) # Force OS to write to physical disk

    # 2. Instantly swap the temp file to the real target
    temp_path.replace(target)
    print("Bro, file saved safely!")

safe_write("critical_data.json", '{"status": "ok"}')
```

## 8. Performance Table

| Strategy | Performance / Safety | Use Case |
|---|---|---|
| `file.read()` | Bad for large files (OOM) | Small configs |
| Line-by-line iteration | High performance, low RAM | Parsing 10GB logs |
| Atomic Replace | Highly Safe, slightly slower | Critical configs, databases |
| `f.flush()` + `os.fsync()` | Slow (forces disk spin) | Financial records |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is an atomic write? | Writing to a temp file and renaming it so the file is never observed in a half-written state. |
| Why specify `encoding='utf-8'`? | To prevent `UnicodeDecodeError`. Windows defaults to `cp1252`, Linux to `utf-8`. |
| How do you prevent Out Of Memory errors with files? | Never use `.read()` or `.readlines()` on large files. Iterate over the file object directly. |
| What does `f.flush()` do? | It pushes data from Python's memory buffer to the OS buffer, but does not guarantee it reached the physical disk. |
| What happens if you don't close a file? | Memory leaks, file locks preventing other apps from reading it, and eventual "Too many open files" OS crash. |

## 10. Tricky Questions & Edge Cases

```python
with open('data.txt', 'r') as f:
    for line in f:
        print(line)
        break
```

Bro, what happens to the file after the `break`?
It closes safely! The `with` block guarantees that the file is closed the moment execution leaves the block, whether it finished, broke, or threw an exception.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Use `with open(...)` | Call `f.close()` manually |
| Use `pathlib` | Concatenate paths with `+ "/"` |
| Specify encodings | Assume ASCII/Default |
| Iterate line by line | Call `.read()` on unknown sizes |

## 13. Production Notes

> ⚠ Production Note:
> Always assume file systems are fragile. Network drives (NFS, EFS) can drop connections. Disks can fill up (`No space left on device`). Files can be locked by antivirus software. Always wrap critical file operations in robust `try/except` blocks.

## 14. Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| Hardcoding paths | `C:\Users\Bro\app` will crash when deployed to AWS Linux. |
| `.readlines()` | Creates a list of strings in RAM. A 2GB file will consume 2GB+ of RAM instantly. |
| Silently ignoring errors | `except OSError: pass` masks disk failures until the whole system goes down. |
| Overwriting instead of appending | Opening a log file in `'w'` mode deletes all history. Use `'a'`. |
