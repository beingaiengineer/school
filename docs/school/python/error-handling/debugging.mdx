# Debugging

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Debugging |
| Difficulty | Beginner / Intermediate |
| Used For | Finding out *why* your code is broken and fixing it |
| Main Strength | Saves hours of guessing by letting you pause code execution |
| Common Mistake | Guessing the bug instead of actually inspecting the variables |
| Performance | Running code in a debugger is much slower, but debugging is an active developer task, not a production task |

Bro, debugging is the most important skill in software engineering. Writing code is easy; figuring out why it doesn't work is the real job.

While `print()` statements are the beginner's way to debug, using an interactive debugger (like Python's built-in `pdb` or an IDE debugger) is how professionals find bugs fast.

## 2. Real-World Analogy

Bro, think of your code like a water pipe system.

Water is supposed to come out of the tap, but it's not.
- **The Junior Way (`print`):** You drill a hole at the start of the pipe, look if water is there, patch it. Then drill another hole 5 feet down. It's messy and takes forever.
- **The Senior Way (Debugger):** You install clear, transparent valves (breakpoints) at key locations. You turn on the water (run the code), pause time exactly when the water reaches the valve, and inspect the pressure and flow inside the pipe without breaking anything.

## 3. Concept Explanation

Debugging usually involves:
1. Identifying that a bug exists (often via an Exception traceback or a failed test).
2. Isolating the code that causes it.
3. Pausing execution right before the crash using a **Breakpoint**.
4. Stepping through the code line-by-line.
5. Inspecting the state of variables.

Python provides a built-in debugger module called `pdb` (Python Debugger). In Python 3.7+, you can trigger it simply by calling the built-in `breakpoint()` function.

Use debuggers when:
- You have a complex logic bug (e.g., a loop calculating the wrong total).
- You are dealing with nested dictionaries or JSON and need to see exactly what data looks like.
- You don't want to restart your heavy script 50 times just to add more `print` statements.

## 4. Syntax Table (PDB Commands)

When you hit a `breakpoint()`, your terminal pauses and waits for your command.

| Command | Shortcut | What it does |
|---|---|---|
| `list` | `l` | Shows the 11 lines of code around where you are paused |
| `next` | `n` | Executes the current line and goes to the next line |
| `step` | `s` | Steps *into* a function call (goes deeper) |
| `continue` | `c` | Unpauses and runs until the next breakpoint |
| `return` | `r` | Continues execution until the current function returns |
| `print(var)` | `p var` | Prints the value of a variable |
| `quit` | `q` | Aborts the program entirely |

## 5. Beginner Example

Let's look at a script with a math bug.

```python
def calculate_discount(price, discount_percent):
    # Bug: We are dividing by 10 instead of 100
    discount_amount = price * (discount_percent / 10)
    final_price = price - discount_amount
    return final_price

def checkout(cart_total):
    print(f"Cart total is ${cart_total}")

    # We drop a breakpoint here to pause execution
    breakpoint()

    final = calculate_discount(cart_total, 20)
    print(f"Final price is ${final}")

# checkout(100) # This would run the code and pause in the terminal
```

When you run this in your terminal, the code stops at `breakpoint()`.
You can type `n` to go to the next line.
You can type `s` to step *inside* `calculate_discount`.
Inside that function, you can type `p discount_percent / 10` and see it equals `2.0` instead of `0.2`. You found the bug instantly without adding any print statements!

## 6. Real-World Engineering Example

In the real world, you often use your IDE's visual debugger (like VS Code or PyCharm). It's just a UI wrapped around `pdb`.

Imagine processing API data:

```python
def process_user_data(api_response):
    users = api_response.get("data", [])

    active_users = []
    for user in users:
        # BUG: The API changed! "status" is now nested under "account_info"
        # But this code still looks for user["status"]

        try:
            if user["status"] == "active":
                active_users.append(user)
        except KeyError:
            # We don't know why it failed, so we drop a breakpoint!
            breakpoint()

    return active_users

mock_api_data = {
    "data": [
        {"id": 1, "account_info": {"status": "active"}},
        {"id": 2, "account_info": {"status": "inactive"}}
    ]
}

# process_user_data(mock_api_data)
```

Bro, when the `KeyError` hits, the debugger opens. You type `p user` and instantly see `{'id': 1, 'account_info': {'status': 'active'}}`. You immediately realize the JSON structure changed. Bug solved in 10 seconds.

Under the hood, Python has a debugging hook called `sys.settrace()`. When you trigger a debugger, it hooks into the interpreter loop, stopping execution right before the next line of bytecode is run.

It keeps the entire memory state (local and global variables) alive and hands control over to an interactive prompt so you can poke around.

## 8. Performance Table

| Environment | Performance Impact | Note |
|---|---|---|
| Normal Execution | None | Code runs at full CPython speed. |
| With `sys.settrace` active | Extremely Slow | The debugger intercepts every single line of code. |
| Leaving `breakpoint()` in Prod | Catastrophic | If your prod server hits a `breakpoint()`, it will pause forever waiting for terminal input, freezing your app. |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between `print` debugging and using a debugger? | `print` requires modifying code, rerunning, and guessing what variables to check. A debugger pauses state, letting you inspect any variable dynamically and step through logic line-by-line. |
| What does `breakpoint()` do in Python? | Introduced in Python 3.7, it drops you into the `pdb` interactive debugger at the exact line it is called. |
| In `pdb`, what is the difference between `step` and `next`? | `next` executes the current line and goes to the next line in the current function. `step` goes *inside* any function call on the current line. |
| How do you read an Exception Traceback? | Read from the bottom up. The bottom tells you the exact error. Above it is the exact line that crashed. Above that is the function that called it, up to the entry point. |

## 10. Tricky Questions & Edge Cases

### Variable Name Collision in `pdb`
What if you have a variable named `c` and you try to print it in the debugger?

```python
c = 10
breakpoint()
```
If you type `c` in `pdb`, it executes the `continue` command!
To print a variable that shares a name with a pdb command, you must type `p c` (print c) or `!c` (execute the Python statement `c`).

### Debugging List Comprehensions
You cannot step "inside" a standard list comprehension `[x for x in data]` easily with `pdb` because it executes as a single C-level loop block.
If a comprehension is crashing, rewrite it as a standard `for` loop temporarily to debug it line-by-line.

## 11. Real-World Usage

- **VS Code / PyCharm Debugger:** 99% of professional engineers use the visual debuggers built into their IDEs. You click the gutter to add a red dot (breakpoint) and hit F5.
- **Jupyter Notebooks:** Using `%debug` magic command after an error occurs to instantly drop into the state where the crash happened.
- **Web Frameworks:** Dropping a `breakpoint()` inside a Django view or FastAPI route to inspect exactly what the incoming HTTP Request object contains.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Use `breakpoint()` conditionally. `if user_id == 99: breakpoint()`. | Put a breakpoint inside a loop of 10,000 items without a condition. You'll be typing `c` forever. |
| Learn to read tracebacks from the bottom up. | Panic when you see 50 lines of red error text. Only the bottom 3 lines usually matter. |
| Remove `breakpoint()` before pushing code to git! | Leave `breakpoint()` in your code. It will freeze your CI/CD pipeline or production server. |

## 13. Production Notes

> ⚠ Production Note:
> Never debug in Production using `pdb`.

If a bug only happens in production, you cannot SSH into the server and drop a `breakpoint()`. You will freeze the thread, and actual customers will sit staring at loading spinners.

Production debugging relies entirely on **Logging**, **Metrics**, **Exception Trackers (like Sentry)**, and **APM tools (like Datadog)**.

## 14. Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| "Guessing" the bug | Staring at code for 2 hours guessing why it's broken is a waste of time. Run it in a debugger and *know* why it's broken in 2 minutes. |
| Ignoring the Exception Traceback | The traceback literally tells you the exact file, line number, and reason for the crash. Read it carefully. |
| Printing massive dictionaries | `print(huge_api_response)` fills your terminal with unreadable text. Use the debugger to explore the dict cleanly. |
