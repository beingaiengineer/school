# Production Debugging

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Production Debugging |
| Difficulty | Advanced |
| Used For | Finding and fixing bugs in live, running systems without taking them down |
| Main Strength | Resolves critical incidents quickly using observability tools |
| Common Mistake | Guessing the bug, restarting the server blindly, or adding `print` statements to live code |
| Performance | Production debugging tools are designed to have near-zero overhead |

Bro, debugging on your laptop with an IDE is easy. You can pause time.

Debugging in production is like trying to fix an airplane's engine while it is flying. You cannot pause the code (customers are waiting), you cannot easily replicate the exact state, and you are usually under high pressure.

You need a completely different set of tools: Logging, Metrics, Exception Trackers, and Tracing.

## 2. Real-World Analogy

Bro, think of a massive factory floor.

- **Local Debugging:** You are building a prototype machine on your workbench. It jams. You turn it off, take it apart, look inside, and fix the gear.
- **Production Debugging:** The factory is running at full speed producing 1,000 items a minute. A machine starts making weird noises. You cannot shut down the whole factory. You look at the **pressure gauges** (Metrics), you check the **security cameras** (Logs), and you look at the **incident reports** (Exception Trackers) to figure out what is wrong without stopping production.

## 3. Concept Explanation

Production Debugging relies on **Observability**. Observability means you can understand the internal state of your system just by looking at its external outputs (logs, metrics, traces).

If a user says "I click checkout and get a 500 error", you don't guess. You:
1. Open your Exception Tracker (e.g., Sentry) to see the exact stack trace and variables at the time of the crash.
2. Search your centralized Logs (e.g., Datadog, ELK) for that specific User ID to see what they did right before the crash.
3. Check Metrics (e.g., Prometheus, Grafana) to see if the database CPU spiked at the same time.

Use Production Debugging techniques when:
- Your code is deployed to real users.
- A bug only happens under heavy load or with real-world, dirty data.

## 4. Syntax Table (Concepts, not just Python syntax)

| Tool/Concept | What it provides | Popular Tools |
|---|---|---|
| Centralized Logging | Searchable text records of events | ELK Stack, Datadog, AWS CloudWatch |

## 5. Beginner Example: The Logging Correlation ID

In production, thousands of logs are printing at once. If you don't link them together, you can't debug.

**Bad (Uncorrelated Logs):**
```text
INFO: Checking out
INFO: Payment processing...
ERROR: Database timeout
INFO: Checkout successful
```
Wait, whose checkout succeeded and whose failed?

**Good (Correlated Logs via Request ID):**
```python
import logging
import uuid

# Simulate a web request context
def handle_request(user_action):
    # Generate a unique ID for this specific web request
    req_id = uuid.uuid4().hex[:8]

    logging.info(f"[{req_id}] Started action: {user_action}")

    try:
        if user_action == "checkout":
            logging.info(f"[{req_id}] Payment processing...")
            raise ConnectionError("DB Timeout")
    except Exception as e:
        logging.error(f"[{req_id}] Failed: {e}")

# If we run two requests simultaneously...
handle_request("checkout")
handle_request("view_cart")
```

Output:
```text
INFO: [a1b2c3d4] Started action: checkout
INFO: [e5f6g7h8] Started action: view_cart
INFO: [a1b2c3d4] Payment processing...
ERROR: [a1b2c3d4] Failed: DB Timeout
```

Bro, now you can easily search your log viewer for `a1b2c3d4` and see the exact lifecycle of the failed request, completely ignoring the noise from `e5f6g7h8`.

## 6. Real-World Engineering Example: Sentry Integration

In a real Python backend (like FastAPI or Django), you don't manually catch every single error to log it. You install an Exception Tracker like Sentry.

```python
import sentry_sdk

# This happens once when your app starts
sentry_sdk.init(
    dsn="https://your-api-key@sentry.io/12345",
    traces_sample_rate=1.0,
)

def divide_numbers(a, b):
    # Sentry automatically intercepts unhandled exceptions!
    return a / b

# Imagine a user submits a form causing this:
# divide_numbers(10, 0)
```

If that code runs in production and crashes, Sentry automatically catches the `ZeroDivisionError`, sends it to a dashboard, groups it with similar errors, and tells you:
1. The exact line of code.
2. The values of `a` (10) and `b` (0) at the time of the crash.
3. The user's browser, OS, and IP address.
4. The exact time it happened.

Bro, this turns a stressful "The site is broken!" complaint into a 5-minute targeted fix.

In production, your Python app shouldn't write logs to local `.log` files, because servers can be destroyed or restarted randomly (like in Kubernetes).

Instead, apps print to `stdout` (standard output), and an infrastructure tool captures those prints and ships them to a centralized database (like Datadog).

## 8. Performance Table

| Tool | Production Overhead | Advice |
|---|---|---|
| Metrics (Prometheus) | Extremely low | Safe to expose thousands of metrics. |
| Logging | Low to Moderate | High volume string formatting is slow. Don't log huge JSON payloads on every request. |
| Exception Trackers | Low | Only triggers when a crash actually happens. |
| APM Tracing | Moderate | If you trace 100% of requests, it slows down the app. Usually, you "sample" (e.g., only trace 5% of requests). |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| How do you debug an issue that only happens in production? | I rely on Observability. I check Exception Trackers (like Sentry) for stack traces, search centralized logs for the specific Request ID/User ID, and check metrics dashboards for resource spikes. |
| What is a Request ID / Trace ID? | A unique identifier generated when a request enters the system. It is attached to every log message generated by that request, allowing you to trace the flow across multiple services. |
| Why shouldn't you log to local files in modern deployments? | In containerized environments (Docker/Kubernetes), containers are ephemeral. If the container restarts, local log files are deleted. Logs must be streamed to a centralized external system. |
| What is the difference between Logging and Exception Tracking? | Logging records the expected lifecycle of an application (INFO, WARNING). Exception Tracking specifically captures unhandled application crashes, aggregates them, and captures the memory state/stack trace for debugging. |

## 10. Tricky Questions & Edge Cases

### The Heisenberg Bug (Heisenbug)
A bug that disappears or alters its behavior when you try to study it.

For example, a race condition in multi-threading. If you add `logging.info()` to the thread, the extra milliseconds it takes to write the log might change the timing of the threads, making the bug disappear!

Production fix: Rely on high-level APM traces and metrics rather than injecting code that changes execution timing.

### Logging PII (Personally Identifiable Information)
Bro, if a user's payment fails and you log:
`logger.error(f"Payment failed for Card: {credit_card_number}, CVV: {cvv}")`
You just committed a massive security violation (PCI-DSS). Logs are often readable by many developers. Never log plain-text passwords, tokens, or credit cards.

## 11. Real-World Usage

- **E-Commerce:** Using Datadog APM to see that the `POST /checkout` endpoint is slow specifically because the 3rd-party inventory API is taking 4 seconds to respond.
- **Fintech:** Setting up a PagerDuty alert that pages the on-call engineer at 2 AM if the Prometheus metric for `payment_failure_rate` goes above 5% in a 5-minute window.
- **AI Engineering:** Logging the exact prompt sent to the LLM and the token usage, so if a user reports a hallucinated output, developers can replay the exact prompt locally.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Output logs in JSON format in production. Machines parse JSON logs perfectly. | SSH into a production server to run `tail -f app.log`. |
| Set up alerts for when error rates spike. | Wait for customers to email support before realizing the site is broken. |
| Attach User IDs and Request IDs to all logs. | Restart the server to "fix" a bug without checking the logs first. You destroy the evidence! |

## 13. Production Notes

> ⚠ Production Note:
> The "Restart Fix" is a junior move.

If a server is locked up and you restart it, yes, it fixes the site for 5 minutes. But you destroyed the memory state.

Before you restart, grab a core dump, check the metrics, or run `py-spy` (a production profiler) to see *why* it locked up. Otherwise, it will lock up again tomorrow.

## 14. Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| Blindly deploying untested code to fix a prod bug | You usually introduce a new, worse bug because you are stressed. Replicate the issue locally or in staging first. |
| Not grouping alerts | If your database goes down, your app might generate 10,000 errors a minute. If your alert system emails you 10,000 times, you will ignore it. Use tools that group identical errors. |
