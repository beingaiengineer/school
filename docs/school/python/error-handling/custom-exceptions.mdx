# Custom Exceptions

## 1. Quick Summary

| Area | Details |

| Topic | Custom Exceptions |

| Used For | Creating specific, meaningful errors tailored to your application's domain |
| Common Mistake | Inheriting from `BaseException` instead of `Exception` |

## 2. Engineering Story

In my last startup, our payment gateway would sometimes fail. But instead of returning a specific error, the backend just threw a generic 'ValueError'. We spent hours tracing through logs because a 'ValueError' could mean anything from bad JSON to a missing credit card. That's when I realized we needed custom, domain-specific exceptions to instantly know what broke.

## 4. Concept Explanation

Custom Exceptions are user-defined classes that inherit from Python's built-in `Exception` class.

Instead of raising a generic `ValueError` when business logic fails, you define a specific exception. This makes debugging much easier because the error name itself tells you exactly what went wrong in your domain logic. It also allows other parts of your code to catch *only* that specific business error without catching unrelated `ValueErrors`.

| Operation | Syntax |
|---|---|

| Raise Exception | `raise MyError("Something broke")` |
| Catch Exception | `except MyError as e:` |

## 6. Beginner Example

Let's build that bank example.

```python
# 1. Define the custom exception
class InsufficientFundsError(Exception):
    pass

def withdraw(balance, amount):
    if amount > balance:
        # 2. Raise it when business logic fails
        raise InsufficientFundsError(f"Cannot withdraw {amount}. Balance is only {balance}.")
    return balance - amount

try:
    my_balance = 50
    new_balance = withdraw(my_balance, 10000)
except InsufficientFundsError as e:
    # 3. Catch our specific error
    print(f"Transaction failed, bro: {e}")
```

## 7. Real-World Engineering Example

In production, custom exceptions often hold extra metadata (like HTTP status codes or internal error codes) so that a central error handler can format the response automatically.

```python
class APIError(Exception):
    """Base class for all API errors in our app."""
    def __init__(self, message: str, status_code: int = 500, error_code: str = "INTERNAL_ERROR"):
        super().__init__(message)
        self.message = message
        self.status_code = status_code
        self.error_code = error_code

class UserNotFoundError(APIError):
    def __init__(self, user_id: int):
        super().__init__(
            message=f"User with ID {user_id} was not found.",
            status_code=404,
            error_code="USER_NOT_FOUND"
        )

# Fast-API/Flask style global error handler
def handle_request(user_id):
    try:
        if user_id != 1:
            raise UserNotFoundError(user_id)
        return {"status": 200, "data": "John Doe"}

    except APIError as e:
        # We catch ANY of our custom API errors here and format them!
        return {
            "error": True,
            "code": e.error_code,
            "message": e.message,
            "http_status": e.status_code
        }

print(handle_request(99))
# Output: {'error': True, 'code': 'USER_NOT_FOUND', 'message': 'User with ID 99 was not found.', 'http_status': 404}
```

## 8. Internal Working

When you inherit from `Exception`, your custom class inherits the `__init__` and `__str__` methods of the base class. The base `Exception` class stores any arguments passed to it in a tuple called `self.args`. When Python prints the traceback, it automatically calls `__str__` on your exception object, which by default prints whatever is inside `self.args`.

By overriding `__init__`, you can attach custom attributes (like `status_code` above) to the exception object before it gets thrown up the call stack.

## 9. Performance Table

| Operation | Complexity | Note |
|---|---|---|
| Class Definition | O(1) | Just defining the class costs nothing |
| Raising Custom Error | Slow | Same cost as raising a built-in exception (stack unwinding) |
| Adding Metadata | O(1) | Setting attributes inside `__init__` is extremely fast |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why not just use `ValueError` everywhere? | Because `ValueError` is too generic. If you use it for a missing user AND a bad password, the caller can't distinguish between the two when writing the `except` block. |
| Which base class should you inherit from? | Always inherit from `Exception`, never from `BaseException`. `BaseException` includes system-level events like `KeyboardInterrupt`, which you usually don't want to catch by accident. |
| Can you have a hierarchy of custom exceptions? | Yes! It's best practice to define a `BaseAppError(Exception)` and have all your specific errors (`UserError`, `DatabaseError`) inherit from it. |

## 11. Tricky Questions & Edge Cases

Bro, what happens if you forget to call `super().__init__()` in your custom exception?

```python
class BadError(Exception):
    def __init__(self, message):
        self.message = message
        # Forgot super().__init__(message)!

try:
    raise BadError("Something went wrong")
except BadError as e:
    print(f"Caught: {e}")
```

**Trick:** It prints `Caught: ` (an empty string)! The base `Exception` class relies on `super().__init__()` to populate `self.args` and format the default `__str__`. If you don't call it, your exception will print completely blank in the terminal traceback, making debugging a nightmare.

## 12. Real-World Usage

Companies use custom exceptions for:
- **Boto3 (AWS SDK):** Has specific exceptions like `botocore.exceptions.ClientError`.
- **SQLAlchemy:** Raises custom `sqlalchemy.exc.IntegrityError` when a database unique constraint is violated.
- **Microservices:** Defining a shared package of `DomainExceptions` so all services raise the exact same error types.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Name exceptions ending with `Error` (e.g., `TimeoutError`). | Don't name them `Exception` (e.g., `TimeoutException` is mostly a Java convention). |
| Create a Base Exception class for your specific module/app. | Don't create a custom exception if a built-in one perfectly describes the exact programmatic error (e.g., `TypeError` for wrong variable types). |

## 14. Production Notes

> ⚠ Production Note:
In large codebases, never raise raw `Exception("message")`. Always use custom exceptions or specific built-ins. Raw `Exceptions` force the caller to use a bare `except Exception:` block, which will accidentally catch unrelated errors (like `NameError` typos) and swallow them, hiding critical bugs.

## 15. Common Mistakes

- **Reinventing the Wheel:** Creating a `FileNotFoundErrorCustom` when Python already has `FileNotFoundError`. Only create custom exceptions for your *business domain logic*.
- **Empty Custom Exceptions Everywhere:** Creating 50 different custom exception files that just say `pass` and never actually using them to differentiate error handling logic.
