# 2. Real-World Analogy

Think of string formatting like a fill-in-the-blank Mad Libs game, bro.

You have a template: `"Hello, my name is ____ and I like ____."`
Instead of trying to manually glue strings together with `+` signs and tape, string formatting lets you declare the placeholders upfront and perfectly slide your variables into the slots when the program runs.

---

# 3. Concept Explanation

String formatting is how we inject dynamic variables (like integers, lists, or other strings) into a base string.

Python has evolved three major ways to do this:
1. **The Old Way (`%` formatting):** Borrowed from C. Hard to read, prone to type errors.
2. **The Better Way (`.format()`):** Introduced in Python 3. Safer, more flexible, but verbose.
3. **The Best Way (F-Strings):** Introduced in Python 3.6. Extremely fast, readable, and evaluated at runtime. (We have a whole separate chapter just for F-Strings!)

We still need to know `.format()` because you will see it constantly in older codebases, enterprise systems, and logging setups.

---

# 4. Syntax Table

| Method | Example | Note |
|---|---|---|
| `%` Operator | `"User %s is %d" % ("Alice", 25)` | Legacy. Avoid in new code. |
| `.format()` | `"User {} is {}".format("Alice", 25)` | Standard. Very common. |
| Positional `.format()` | `"{1} ate {0}".format("apple", "Bob")` | Order matters. |
| Named `.format()` | `"{name} is {age}".format(name="Alice", age=25)`| Best for readability in templates. |

---

# 5. Beginner Example

Let's look at `.format()`, which is still heavily used for template strings.

```python
template = "Welcome, {name}! Your balance is ${balance}."

# Injecting variables
message = template.format(name="Rahul", balance=150.50)

print(message)
# Output: Welcome, Rahul! Your balance is $150.50.
```

Bro, notice how clean that is compared to `message = "Welcome, " + name + "! Your balance is $" + str(balance) + "."`!

# 6. Real-World Engineering Example

In backend systems, we often keep our templates separate from our logic. For example, generating automated emails or SQL queries (though use SQL parameters for real DBs to prevent injection!).

```python
def send_alert_email(server_name: str, cpu_usage: float):
    # The template is defined once, maybe loaded from a file
    EMAIL_TEMPLATE = """
    ALERT: High CPU Usage
    Server: {server}
    Current Load: {load:.2f}%
    Please investigate immediately.
    """

    # We format it dynamically
    body = EMAIL_TEMPLATE.format(server=server_name, load=cpu_usage)

    # Simulating email send
    print(body)

send_alert_email("prod-db-01", 98.7654321)

# Output:
# ALERT: High CPU Usage
# Server: prod-db-01
# Current Load: 98.77%
# Please investigate immediately.
```
*Note the `:.2f` — that tells `.format()` to round the float to 2 decimal places!*

---

# 7. Internal Working

When you use `.format()`, Python parses the string, finds the `{}` brackets, and replaces them. Internally, `.format()` automatically calls the `__str__()` method on whatever object you pass it. This is why you don't have to manually cast integers to strings like `str(25)` when formatting.

# 8. Performance Table

| Formatting Method | Relative Speed | Note |
|---|---|---|
| F-Strings | 🚀 Fastest | Evaluated directly at runtime in C. |
| `.format()` | 🐢 Slower | Requires parsing the string and mapping arguments. |
| `%` Operator | 🐢 Slower | Legacy parsing overhead. |
| Concatenation `+` | 🐌 Slowest | Allocates new memory repeatedly. |

---

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| Why should we avoid `+` for string concatenation? | It creates new string objects in memory for every `+`, which is slow and memory-heavy. |
| What does `{:0>3}` do in `.format()`? | It pads the string with zeros on the left to ensure it's at least 3 characters long (e.g., `5` becomes `005`). |
| Why might someone use `.format()` instead of f-strings today? | If the template string needs to be loaded dynamically from a file or database, you must use `.format()`. F-strings are evaluated immediately in code. |

---

# 10. Tricky Questions & Edge Cases

What happens if you provide too many or too few arguments to `.format()`?

```python
# Too many arguments? Python ignores the extra ones.
print("Hello {}".format("Alice", "Bob"))
# Output: 'Hello Alice'

# Too few arguments? Python throws an IndexError!
# print("Hello {} {}".format("Alice"))
# IndexError: Replacement index 1 out of range for positional args tuple
```

If you are using named arguments, a missing key will throw a `KeyError`.

# 11. Real-World Usage

- **Dynamic SQL/HTML Templates:** Loading an email template from a database and injecting user data using `.format()`.
- **Log Formatting:** Standard `logging` modules often use lazy `%s` formatting under the hood so they don't waste CPU cycles formatting logs that won't be printed.
- **CLI Outputs:** Formatting tabular data in the terminal with specific widths and alignments.

---

# 12. Best Practices

| DO | DON'T |
|---|---|
| Use `.format()` when your template string comes from outside the source code (like a text file). | Use `.format()` for simple, inline variable injection (use f-strings instead). |
| Use named arguments `"{name}".format(name=x)` for templates with many variables. | Use `%s` syntax unless you are specifically working with the standard `logging` module. |

---

# 13. Production Notes

> ⚠ Production Note:
Never use `.format()` or f-strings to inject user input directly into a database query. This leads directly to **SQL Injection**, one of the most critical security vulnerabilities.
> `query = "SELECT * FROM users WHERE name = '{}'".format(user_input)` ❌ **DANGEROUS!**
> Always use your database driver's built-in parameter substitution (like `execute("SELECT * FROM users WHERE name = %s", (user_input,))`).

---

# 14. Common Mistakes

- **Forgetting the `f` in f-strings:** Writing `"User is {name}"` instead of `f"User is {name}"`. The variables won't inject!
- **Overcomplicating `.format()`:** Doing `"{} {}".format(a, b)` instead of just using an f-string `f"{a} {b}"`. Keep it modern.
