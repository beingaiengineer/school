# 4. Concept Explanation

Performance best practices in Python involve a combination of choosing the right data structures, minimizing I/O operations, and leveraging the standard library effectively.

Because Python is an interpreted language, it is inherently slower than C or Go. To make Python fast in production, you have to write code that avoids the interpreter's weaknesses (like heavy loop overhead) and leans into its strengths (like C-backed standard libraries and async I/O). Bro, the goal isn't to write perfect code; it's to write code that meets your Service Level Agreements (SLAs).

|---|---|---|
| String Concatenation | `s = ""; for word in words: s += word` | `s = "".join(words)` |
| Lookups | `if item in my_list:` (O(N)) | `if item in my_set:` (O(1)) |
| Loop Variables | `for i in range(len(arr)): do(arr[i])` | `for item in arr: do(item)` |
| Object Creation | Instantiating classes inside loops | Reusing objects or using `__slots__` |

# 6. Beginner Example

Let's look at the classic mistake of using the wrong data structure for lookups.

```python
import time

# A database of 1 million registered user IDs
registered_users_list = list(range(1_000_000))
# Convert it to a set for O(1) lookups
registered_users_set = set(registered_users_list)

# We want to check if user ID 999,999 is registered
user_to_check = 999_999

# BAD: List Lookup (O(N))
start = time.time()
is_registered = user_to_check in registered_users_list
print(f"List lookup took: {time.time() - start:.6f} seconds")  # Takes ~0.015000s

# GOOD: Set Lookup (O(1))
start = time.time()
is_registered = user_to_check in registered_users_set
print(f"Set lookup took: {time.time() - start:.6f} seconds")   # Takes ~0.000001s
```

Bro, the set lookup is over 10,000 times faster because it uses a hash table. Choosing the right data structure is the easiest and most powerful performance optimization.

# 7. Real-World Engineering Example

In production, database queries (I/O) are almost always your biggest bottleneck. The N+1 query problem is the enemy of performance.

```python
# Assume we have an ORM like Django or SQLAlchemy

# BAD: The N+1 Query Problem
def get_user_cities_slow(users):
    cities = []
    # Query 1: Getting the users (already done)
    for user in users:
        # Query 2 through N+1: Hitting the database for EVERY SINGLE USER
        address = db.query("SELECT city FROM addresses WHERE user_id = ?", user.id)
        cities.append(address.city)
    return cities

# GOOD: Batching I/O
def get_user_cities_fast(users):
    user_ids = [user.id for user in users]
    # One single query using an IN clause
    # Query 1: SELECT city FROM addresses WHERE user_id IN (1, 2, 3...)
    addresses = db.query("SELECT user_id, city FROM addresses WHERE user_id IN ?", user_ids)

    # Map them in memory (Memory is fast, Databases are slow)
    address_map = {a.user_id: a.city for a in addresses}
    return [address_map.get(user.id) for user in users]
```

Bro, if you have 1,000 users, the bad way hits the database 1,001 times over the network. The good way hits it exactly 1 time.

Performance optimization requires identifying if you are I/O bound (waiting on disks or network) or CPU bound (doing heavy math). You fix I/O by batching, caching, or using `asyncio`. You fix CPU bounds by using better algorithms, C-extensions (NumPy), or `multiprocessing`.

# 10. Top Interview Questions

| Question | Answer | Explanation & Tricks |
|---|---|---|
| What is Premature Optimization? | Spending time optimizing code that isn't actually a bottleneck. | "Premature optimization is the root of all evil." You should write clean, readable code first, then use a profiler to find what actually needs fixing. |
| How do you optimize string concatenation in Python? | Use `"".join(list_of_strings)`. | Python strings are immutable. Doing `a += b` in a loop creates a brand new string object every time, which is O(N^2). |
| What is the difference between CPU bound and I/O bound? | CPU bound means the processor is doing heavy calculations. I/O bound means the program is waiting for network/disk responses. | Threading/AsyncIO fixes I/O bound problems. Multiprocessing fixes CPU bound problems in Python. |

# 11. Tricky Questions & Edge Cases

**Tricky Scenario:** Global variables in loops.
Python looks up global variables slower than local variables because they are stored in a dictionary, whereas locals are stored in a fast C array.
**Solution:** If you are using a global function inside a massive loop, assign it to a local variable first.
```python
# Slower
for i in range(1_000_000): math.sqrt(i)

# Faster
local_sqrt = math.sqrt
for i in range(1_000_000): local_sqrt(i)
```

**Edge Case:** The cost of function calls.
In Python, calling a function has significant overhead. If you have a loop of 10 million items, and inside that loop you call a tiny function `def add(a, b): return a + b`, the overhead of making 10 million function calls will dwarf the actual math. For extreme performance, inline the code.

# 13. Best Practices

| DO | DON'T |
|---|---|
| Use built-in functions (`map`, `sum`) | Write custom `for` loops to do basic math |
| Use List Comprehensions | Use `.append()` in a loop if you can avoid it |
| Profile with `cProfile` before changing code | Guess what is slow based on gut feeling |

# 14. Production Notes

> ⚠ Production Note:
Bro, the fastest code is the code that never runs. Before you optimize a function to be 10x faster, ask the product manager if you even need to run that function. Can it be cached? Can it be run once a day in a cron job instead of on every user click? Architectural changes always beat micro-optimizations.

---

# 15. Common Mistakes

- **Ignoring the Database:** Spending 3 days optimizing a Python loop that takes 0.05 seconds, while ignoring the SQL query that takes 3.0 seconds.
- **Misusing AsyncIO:** Using `async`/`await` and then accidentally calling a blocking synchronous library (like `requests`) inside the event loop, freezing the entire async server.
