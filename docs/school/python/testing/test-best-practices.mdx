# 4. Concept Explanation

Test best practices in Python involve writing tests that are **Fast, Isolated, Repeatable, Self-Validating, and Timely (FIRST)**.

When you write tests, you want them to be a safety net, not an anchor. This means avoiding flaky tests, mocking external dependencies appropriately, and structuring your test code so it's as clean (or cleaner) than your production code. Bro, if your tests are a mess, nobody will want to write them.

# 5. Syntax Table

| Concept | Bad Practice | Best Practice |
|---|---|---|
| **Naming** | `test_function()` | `test_user_auth_fails_with_invalid_token()` |
| **Structure** | Mixed setup, action, and assertions | Arrange, Act, Assert (AAA) pattern |
| **Dependencies** | Hitting a real API | Mocking the API or using VCR.py |
| **State** | Sharing global variables between tests | Using Pytest fixtures for fresh state |
| **Assertions** | `assert result != None` | `assert result == expected_dict` |

# 6. Beginner Example

Let's look at a bad test and how to fix it using the Arrange-Act-Assert pattern.

```python
# The Code
def calculate_discount(price, discount_percent):
    return price - (price * (discount_percent / 100))

# BAD TEST: Hard to read, mixed logic
def test_calc():
    assert calculate_discount(100, 20) == 80
    assert calculate_discount(50, 10) == 45

# GOOD TEST: Clear AAA structure and descriptive name
def test_calculate_discount_applies_correct_percentage():
    # Arrange
    original_price = 100
    discount = 20
    expected_price = 80

    # Act
    actual_price = calculate_discount(original_price, discount)

    # Assert
    assert actual_price == expected_price
```

Bro, the good test is self-documenting. Anyone reading it instantly knows what is being tested and what the expected outcome is.

# 7. Real-World Engineering Example

In production, you often need to test code that interacts with databases. A common best practice is using Pytest fixtures to guarantee a clean state.

```python
import pytest
from my_app.db import User, create_user

# 1. Provide a clean database for every test
@pytest.fixture
def db_session():
    # Setup: Create tables
    setup_database()
    yield
    # Teardown: Drop tables so the next test is clean
    teardown_database()

# 2. Test the behavior, relying on the clean state
def test_create_user_saves_to_database_successfully(db_session):
    # Arrange
    email = "test@example.com"

    # Act
    user = create_user(email=email)

    # Assert
    assert user.id is not None
    assert user.email == email
```

Notice the descriptive test name and the isolated state. If this test fails, we know exactly why, and it won't affect `test_delete_user()`.

# 9. Performance Table

| Practice | Impact on Test Suite Speed | Why? |
|---|---|---|
| Hitting Real DBs | Slow | Network I/O and disk writes are heavy |
| Mocking External APIs | Fast | Memory-only operations |
| `session` scoped fixtures | Very Fast | Runs heavy setup (like starting a Docker DB) only once |
| `function` scoped fixtures | Slower | Runs setup before *every single test* |

# 10. Top Interview Questions

| Question | Answer | Explanation & Tricks |
|---|---|---|
| What does AAA stand for in testing? | Arrange, Act, Assert. | It's a pattern to structure tests. Arrange sets up the data, Act executes the code, Assert verifies the outcome. |
| What is a "flaky" test? | A test that sometimes passes and sometimes fails without code changes. | Often caused by shared state, relying on exact timestamps, or network latency. They destroy trust in the CI/CD pipeline. |
| Why should tests be deterministic? | So they yield the same result every time they run. | If a test fails, you need to know it's because the code broke, not because the wind was blowing the wrong way. |
| How do you prevent tests from interacting with each other? | By tearing down state (e.g., clearing the DB or resetting mocks) after each test. | Pytest fixtures using `yield` are the standard Pythonic way to handle this. |

# 11. Tricky Questions & Edge Cases

**Tricky Scenario:** Testing code that uses `datetime.now()`.
If you hardcode an expected date in your test, it will fail tomorrow.
**Solution:** Bro, use a library like `freezegun` or mock the `datetime` module to return a static time during the test.

**Edge Case:** Testing exceptions.
Don't just use a `try/except` block in your test. Use `pytest.raises()`.
```python
def test_divide_by_zero_raises_error():
    with pytest.raises(ZeroDivisionError):
        divide(10, 0)
```

# 13. Best Practices

| DO | DON'T |
|---|---|
| Test public interfaces (behavior) | Test private, internal helper functions (`_my_helper()`) |
| Use descriptive, long test names | Use generic names like `test_auth_1` |
| Keep tests independent of each other | Rely on Test A passing for Test B to work |
| Keep assertions strictly focused | Write one giant test with 50 assertions |

# 14. Production Notes

> ⚠ Production Note:
Bro, treat your test code with the same respect as your production code. Don't let a mess of commented-out assertions and duplicated mock setups rot in your repository. Refactor your tests. Extract common setups into Pytest fixtures or helper functions. A clean test suite is the only thing standing between you and a 2:00 AM production outage page.

---

# 15. Common Mistakes

- **Testing the Mock:** Writing a test where you mock a function to return X, and then assert that the function returned X. You just tested your mocking library, not your code.
- **Ignoring Flaky Tests:** Retrying a failed test until it passes and ignoring the root cause. This *will* blow up later.
- **Low Coverage / Fake Coverage:** Having 100% test coverage but only checking that functions execute without crashing, rather than asserting the outputs are actually correct.
