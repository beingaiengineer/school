# Load Testing

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Load Testing |
| Difficulty | Advanced |
| Used For | Proving your system won't crash when 10,000 users log in at once |
| Common Mistake | Running a load test from your local laptop against a production server |
| Performance | Evaluates the maximum Requests Per Second (RPS) of an application |

## 2. Real-World Analogy

Bro, imagine you open a new coffee shop.

You tested making 1 cup of coffee, and it takes 30 seconds. Perfect.
But what happens when a bus full of 50 people arrives at exactly 8:00 AM? Can the barista handle the pressure? Do you run out of cups? Does the espresso machine overheat?

**Load Testing** is simulating that bus arriving *before* opening day, so you know exactly where your system breaks and how to fix it before real customers get angry.

## 3. Concept Explanation

Load testing involves writing scripts that generate massive amounts of concurrent HTTP requests (traffic) against your application to see how it performs under stress.

**Key Metrics Monitored:**
- **RPS (Requests Per Second):** How much traffic is hitting you.
- **Latency / Response Time:** Does a 50ms request jump to 5000ms under load?
- **Error Rate:** Do 200 OK responses suddenly become 502 Bad Gateways?
- **Resource Utilization:** Are CPU and RAM maxing out?

In Python, the industry standard tool for this is **Locust**. It allows you to write user behavior in pure Python, and it handles the concurrency.

## 5. Beginner Example

Let's write a simple Locust test to simulate users hitting our API.

```python
# load_test.py
from locust import HttpUser, task, between

class APIUser(HttpUser):
    # Simulate a user pausing between 1 and 3 seconds between clicks
    wait_time = between(1, 3)

    @task
    def view_homepage(self):
        self.client.get("/")

    @task(3) # This task runs 3 times more often than the homepage
    def view_product(self):
        self.client.get("/products/1")
```

To run it, type `locust -f load_test.py` in your terminal, and open `http://localhost:8089` to see a beautiful web UI where you can spawn 500 fake users.

## 6. Real-World Engineering Example

In production, load tests must simulate real authentication flows, otherwise you are just testing your server's ability to reject unauthorized requests (which is fast).

```python
from locust import HttpUser, task, between

class AuthenticatedUser(HttpUser):
    wait_time = between(1, 2)
    token = None

    def on_start(self):
        # This runs ONCE when the fake user spawns
        response = self.client.post("/login", json={
            "username": "testuser",
            "password": "password123"
        })
        self.token = response.json().get("access_token")

    @task
    def fetch_dashboard(self):
        # Use the token for all subsequent requests
        headers = {"Authorization": f"Bearer {self.token}"}
        self.client.get("/api/dashboard", headers=headers)
```

Locust works using asynchronous coroutines (gevent under the hood). Instead of using a heavy OS thread for every fake user, it uses lightweight greenlets. This allows a single Python process to simulate thousands of concurrent users. For massive scale (e.g., 100,000 users), Locust can run in distributed mode, where a Master node commands dozens of Worker nodes spread across multiple servers to fire at the target simultaneously.

## 8. Performance Table

| System | Bottleneck | Symptoms under Load |
|---|---|---|
| API Application (Python) | CPU Bound | 100% CPU usage, slow response times |
| Database | I/O or Locks | "Too Many Connections" errors, Deadlocks |
| Network | Bandwidth | Packets dropped, 504 Gateway Timeouts |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between Load Testing and Stress Testing? | Load testing verifies normal peak capacity. Stress testing pushes the system *past* its limits until it breaks, to see how it recovers. |
| Why is `wait_time` important in a load test script? | Real users don't click a button 50 times a second. `wait_time` simulates human "think time" reading a page. |
| If CPU is at 20% but API responses are taking 5 seconds, what is wrong? | It's likely an I/O bottleneck. The DB queries are slow, or the DB connection pool is exhausted, so the API is just waiting, not calculating. |

## 10. Tricky Questions & Edge Cases

Bro, what happens if you run a massive load test from your MacBook?

**The Trap:**
You spin up 50,000 users in Locust on your laptop hitting your AWS server. Your laptop's CPU hits 100%, its network card maxes out, and it reports that your AWS server can only handle 500 RPS.

**The Fix:**
You bottlenecked your *tester*, not the *target*. Your laptop cannot generate enough traffic. You must run heavy load tests from cloud servers (often in the same region) using distributed workers so the traffic generation doesn't become the limiting factor.

## 11. Real-World Usage

- **E-Commerce (Flash Sales):** Load testing the checkout flow 2 weeks before Black Friday to ensure the payment gateway doesn't time out.
- **Ticketing Systems:** Testing what happens when 1 million Taylor Swift fans try to view a seating chart at the exact same second.
- **Memory Leak Detection:** Running a low-level load test for 24 hours (a "Soak Test") to see if RAM usage slowly creeps up over time.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Load test against a Staging environment identical to Production | Load test Production during peak business hours |
| Monitor CPU, RAM, and Database logs *during* the test | Just look at the Locust RPS graph and ignore server health |
| Populate the test database with realistic amounts of data (millions of rows) | Load test a database that only has 10 rows in it |

## 13. Production Notes

> ⚠ Production Note:
Always notify your DevOps or Cloud team before running a load test! Generating massive traffic looks exactly like a DDoS attack. If you don't tell anyone, AWS WAF or Cloudflare will auto-block your IP, or worse, the on-call engineer will be paged at 2 AM for a fake emergency.

## 14. Common Mistakes

- **Caching Illusions:** Hitting the exact same URL `/api/data/1` repeatedly. Your CDN or Redis cache intercepts it, returning responses in 1ms. You think your API is incredibly fast, but in reality, your Python code was never even executed. Always randomize test data!
- **Ignoring 3rd Party Quotas:** Running a load test that sends 10,000 real emails via SendGrid, costing the company money and getting the account banned. Mock external integrations during load tests.
