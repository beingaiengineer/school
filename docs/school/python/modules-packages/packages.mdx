# Packages

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Python Packages |
| Difficulty | Intermediate |
| Used For | Organizing multiple modules into a folder hierarchy |
| Main Idea | A package is a folder containing Python modules and an `__init__.py` file |
| Common Mistake | Forgetting `__init__.py` or messing up relative imports |
| Production Angle | Large apps are structured as packages so they can be distributed or tested easily |

Bro, if a module is a single toolbox, a package is the entire mechanic's workshop.

## 2. Real-World Analogy

Think of managing code like managing a computer's file system.

| File System | Python Equivalent |
|---|---|
| A text file (`.txt`) | A Module (`.py` file) |
| A folder holding text files | A Package (directory with `.py` files) |
| A folder holding other folders | A Sub-package |
| Double-clicking a folder to open it | The `__init__.py` file executing |

## 3. Concept Explanation

When you have 50 modules, dumping them all in one folder is chaos. A **Package** is simply a directory containing multiple Python modules.

Historically, Python required a special file named `__init__.py` inside the folder to recognize it as a package. While Python 3.3+ allows "namespace packages" without it, using `__init__.py` is still the standard engineering practice to initialize package-level variables and control imports.

## 4. Syntax Table

| Action | Example |
|---|---|
| Import module from package | `import myapp.database.queries` |
| Import specific function | `from myapp.database.queries import get_user` |
| Relative import (same dir) | `from . import models` |
| Relative import (parent dir)| `from ..utils import helpers` |

## 5. Beginner Example

Folder Structure:
```text
my_game/
│-- __init__.py
│-- player.py
│-- enemies.py
```

Code in `main.py` (outside the `my_game` folder):
```python
# Import the player module from the my_game package
from my_game import player
from my_game.enemies import spawn_dragon

player.walk()
spawn_dragon()
```

## 6. Real-World Engineering Example

```text
ecommerce/
│-- __init__.py
│-- billing/
│   │-- __init__.py
│   │-- stripe_api.py
│   │-- invoices.py
│-- shipping/
│   │-- __init__.py
│   │-- fedex_api.py
```

In `ecommerce/billing/__init__.py`, you can clean up the imports for the user:

```python
# Expose only the important functions at the package level
from .stripe_api import charge_credit_card
from .invoices import generate_pdf
```

Now, the user code is super clean:
```python
from ecommerce.billing import charge_credit_card
```

## 8. Performance Table

| Strategy | Performance / Memory |
|---|---|
| Empty `__init__.py` | Zero overhead |
| Heavy imports in `__init__.py` | Slow startup time for the whole app |
| Importing deeply nested modules | Fast (same as normal module) |
| Relative imports | Fast (resolved at compile time) |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is the purpose of `__init__.py`? | It marks a directory as a Python package and runs initialization code when the package is imported. |
| What is a relative import? | Importing a module relative to the current file's location using dots (e.g., `from . import config`). |
| Can you run a script that uses relative imports directly? | No. If you run `python child.py`, Python sets `__name__` to `__main__` and loses the package context, causing a `ImportError: attempted relative import with no known parent package`. |
| What is `__all__`? | A list inside `__init__.py` defining exactly what gets imported when someone uses `from package import *`. |

## 10. Tricky Questions & Edge Cases

```python
# Inside my_package/__init__.py
print("Initializing package!")

# Inside main.py
import my_package.module_a
import my_package.module_b
```

Bro, how many times does "Initializing package!" print?
Exactly **once**. Python caches packages in `sys.modules` just like modules. The `__init__.py` runs only the very first time the package is accessed.

## 11. Real-World Usage

| Scenario | Usage |
|---|---|
| Web Frameworks | Django and FastAPI apps are structured as large packages. |
| Open Source | `numpy`, `pandas`, and `requests` are massive packages distributed via PyPI. |
| Microservices | Sharing a common `utils` package across multiple company repositories. |

## 12. Best Practices

| DO | DON'T |
|---|---|
| Keep `__init__.py` mostly empty | Put heavy logic or database connections in `__init__.py` |
| Use absolute imports (`from app.db import session`) | Overuse relative imports (`from ...db import session`) |
| Structure packages by feature (e.g., `billing`, `auth`) | Put 50 files in one folder |

## 13. Production Notes

> ⚠ Production Note:
> Circular imports often happen when you structure packages poorly. If `billing` needs `users`, and `users` needs `billing`, your architecture is wrong. Extract the shared logic into a third package (e.g., `core_models`).

## 14. Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| Running nested files directly | Running `python myapp/api/views.py` breaks all relative imports. Run the app from the root directory using `python -m myapp.api.views`. |
| Dashes in folder names | You cannot `import my-package` because `-` is a minus sign in Python syntax. Use underscores `my_package`. |
| Bloated `__init__.py` | If `__init__.py` imports massive AI libraries, just importing your package takes 10 seconds. |
