# Modules

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Modules |
| Difficulty | Beginner |
| Used For | Organizing Python code into separate files |
| Core Idea | Any `.py` file is a module. You `import` it to use its code in another file |
| Common Mistake | Naming a module the same as a built-in Python module (like `math.py`) |
| Performance | Modules are loaded into memory only once, even if imported multiple times |

Bro, writing a 5,000-line Python script is a nightmare to read and debug. Modules allow you to split your code into multiple `.py` files. It keeps your codebase clean, organized, and reusable.

## 2. Engineering Story

Bro, imagine you're deploying a large-scale system at a tech giant like Zomato. You have complex logic for handling high concurrency and massive data flows. Without understanding modules, you might end up with severe performance bottlenecks or unmaintainable code that breaks under pressure. Mastering this concept is crucial for building robust, production-ready systems that can handle real-world scale smoothly.

You have a separate factory for the engine, a separate factory for the tires, and a separate factory for the seats. These are **Modules**. The final assembly plant just "imports" the engine and tires to build the car.

## 4. Concept Explanation

In Python, a module is literally just a file containing Python code (functions, classes, and variables) with a `.py` extension.

If you have a file named `calculator.py`, you have created a module named `calculator`.
To use the functions from `calculator.py` inside `main.py`, you use the `import` statement.

Python has three types of modules:
1. **Built-in Modules:** Come with Python (`math`, `sys`, `os`).
2. **Third-Party Modules:** Downloaded from the internet using Pip (`requests`, `django`).
3. **Custom Modules:** The `.py` files you write yourself.

## 5. Syntax Table

| Operation | Example | Note |
|---|---|---|
| Import whole module | `import math` | You must use `math.sqrt()` |
| Import specific item | `from math import sqrt` | You can use `sqrt()` directly |
| Import with alias | `import pandas as pd` | Renames the module locally |
| Import all (Avoid!) | `from math import *` | Pollutes your global namespace |

## 6. Beginner Example

Let's create our own module.

**File 1: `greetings.py` (The Module)**
```python
def say_hello(name):
    return f"Hello, {name}!"

def say_bye(name):
    return f"Goodbye, {name}!"
```

**File 2: `main.py` (The Script)**
```python
# Import the module
import greetings

# Use the functions
message = greetings.say_hello("Bro")
print(message)
```

**Output:**
```
Hello, Bro!
```

## 7. Real-World Engineering Example

In a real web backend (like Django or FastAPI), you never put database logic and API routes in the same file. You split them into modules.

**`database.py`**
```python
def get_user_from_db(user_id):
    # Imagine this connects to Postgres
    return {"id": user_id, "name": "Rahul"}
```

**`api.py`**
```python
from database import get_user_from_db

def user_profile_route(user_id):
    user = get_user_from_db(user_id)
    return {"status": 200, "data": user}
```

## 8. Internal Working

When you write `import math`, what actually happens?

1. Python checks `sys.modules` to see if `math` is already loaded in memory. If yes, it just uses it.
2. If not, it searches for `math.py` in the directories listed in `sys.path` (which includes standard libraries, your current folder, and pip packages).
3. Once found, it compiles the `.py` file into bytecode (a `.pyc` file usually hidden in `__pycache__`).
4. It executes the module code from top to bottom and loads the functions into a new namespace.

## 9. Performance Table

| Operation | Speed | Note |
|---|---|---|
| First `import module` | Slow | Has to search disk, compile, and execute the file. |
| Subsequent `import module` | Instant | Fetched directly from RAM (`sys.modules`). |
| `from module import func` | Same as above | The *entire* module is still executed and cached! |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| What happens if you import a module twice? | Python only loads and executes the module once. The second import just references the already loaded module in `sys.modules`. |
| Why is `from module import *` bad? | It pollutes your global namespace. If two modules have a function with the same name, they will silently overwrite each other, causing bugs. |
| What does `if __name__ == "__main__":` do? | It ensures that specific code only runs when the file is executed directly as a script, NOT when it is imported as a module by another file. |
| How does Python find modules? | It searches the directories listed in `sys.path`. |

## 11. Tricky Questions & Edge Cases

**Edge Case: Circular Imports**

Bro, this is the most common error for intermediate developers.

**`a.py`**
```python
from b import func_b

def func_a(): return "A"
```

**`b.py`**
```python
from a import func_a

def func_b(): return "B"
```
If you run this, Python will crash with an `ImportError`. `a` tries to load `b`, but `b` needs `a` to finish loading first.

**The Fix:** Move the shared logic into a third module (`c.py`), or put the `import` statement *inside* the function so it only runs when called, not when the file is parsed.

## 12. Real-World Usage

- **Microservices:** Splitting routing logic, database models, and business logic into different files.
- **Data Science:** Importing `pandas` and `numpy` into a Jupyter notebook.
- **Scripting:** Creating a `utils.py` file to hold helper functions (like `format_date()`) that you use across 10 different automation scripts.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Group imports at the top of the file. | Hide `import` statements randomly in the middle of your code. |
| Use `import x` or `from x import y`. | Use `from x import *`. |
| Name your modules using short, lowercase words. | Name a module `email.py` or `math.py` (it shadows built-in modules!). |

## 14. Production Notes

> ⚠ Production Note:
Shadowing is a massive headache. If you create a file named `json.py` in your project folder, and then your code does `import json`, Python will load YOUR file instead of the built-in JSON library. Everything will immediately break with weird "AttributeError" crashes. Never name your files the same as standard Python libraries.

## 15. Common Mistakes

- **Importing everything:** `from utils import *`. Later, someone adds a variable `user = "admin"` to `utils.py`, and it accidentally overwrites the `user` variable in your main script.
- **Running module code on import:** If you put `print("Hello")` at the root level of `math_helpers.py`, that print statement will run the moment *any* other file imports it. Always hide script execution behind `if __name__ == "__main__":`.
