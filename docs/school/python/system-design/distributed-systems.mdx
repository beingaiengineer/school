## 2. Engineering Story

Bro, I remember when we first split our app across multiple regions for high availability. It sounded great until a network partition happened between our US and EU data centers. Think of a massive college group project, bro.

If you distribute the work among 10 students (Distributed System), you can finish 10 times faster! BUT, now you have new problems: communicating, merging work, what if one student gets sick (fails), and how do you ensure everyone is working on the latest version?

## 4. Concept Explanation

A Distributed System is a collection of independent computers that appear to the end-user as a single coherent system.

Why use it?
- **High Availability:** If one server dies, another takes over. No single point of failure.
- **Horizontal Scaling:** You can't buy an infinitely large CPU, but you can rent 10,000 small ones.
- **Geographic Locality:** Serve Asian users from Tokyo servers and US users from New York servers to reduce latency.

When NOT to use it:
- When you have a small user base. The complexity of distribution will slow your team down more than the performance gains will help.

## 6. Beginner Example

A simple conceptual demonstration of why we distribute state using a shared cache (Redis) instead of local memory.

```python
# BAD: Single Node State (Breaks when scaled to 3 servers)
local_cache = {}

def get_user(user_id):
    if user_id in local_cache:
        return local_cache[user_id] # Other servers don't have this!
    # Fetch from DB and store...

# GOOD: Distributed State
import redis
redis_client = redis.Redis(host='shared-cache.local')

def get_user_distributed(user_id):
    cached_user = redis_client.get(user_id) # All servers see the same cache
    if cached_user:
        return cached_user
    # Fetch from DB, set in Redis...
```

## 7. Real-World Engineering Example

In a distributed system, handling failures gracefully is critical. We use the **Circuit Breaker** pattern to prevent cascading failures.

```python
import requests
import time

class CircuitBreaker:
    def __init__(self, failure_threshold=3):
        self.failure_threshold = failure_threshold
        self.failures = 0
        self.state = "CLOSED" # Closed means traffic flows normally

    def call_external_service(self, url):
        if self.state == "OPEN":
            print("Circuit is OPEN. Fast-failing!")
            return {"error": "Service unavailable"}

        try:
            response = requests.get(url, timeout=2)
            response.raise_for_status()
            self.failures = 0 # Reset on success
            return response.json()
        except Exception as e:
            self.failures += 1
            print(f"Request failed. Failure count: {self.failures}")
            if self.failures >= self.failure_threshold:
                self.state = "OPEN"
                print("Tripping Circuit Breaker to OPEN state.")
            return {"error": "Service failed"}

# Usage
breaker = CircuitBreaker()
breaker.call_external_service("http://flaky-service.internal/api")
```

## 8. Internal Working

Bro, how do distributed systems agree on anything?

They use **Consensus Algorithms** (like Paxos or Raft).
Imagine a distributed database cluster needing to elect a "Leader" node to handle writes.
1. Nodes send "votes" over the network.
2. A node must receive a majority (quorum) of votes to become the leader.
3. If the leader goes offline, a timeout occurs, and a new election happens automatically.

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is the CAP Theorem? | A distributed data store can only provide two of three: Consistency, Availability, Partition Tolerance. |
| What is a Split-Brain scenario? | When a network partition happens and a cluster splits into two, both electing their own "Leader", leading to data corruption. |
| Consistent Hashing? | A technique used in load balancing and databases to distribute data evenly across servers while minimizing data movement when servers are added or removed. |

## 11. Tricky Questions & Edge Cases

Bro, what happens if server A sends a message to server B, but gets no response?

Did server B crash before processing it?
Did server B process it, but the network crashed on the reply?
Did the network delay the message, and server B is still working on it?

**You cannot know.** This is why distributed transactions are incredibly hard. You must design APIs to be retriable (idempotent) and handle duplicate requests.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Design for failure. Assume nodes will crash randomly. | Assume network latency is zero. |
| Centralize logging and monitoring (Datadog/Splunk). | Store local state on web servers. |
| Use Correlation IDs to trace requests across nodes. | Share a single relational DB if you have massive scale. |

## 14. Production Notes

> ⚠ Production Note:
The "Fallacies of Distributed Computing" is required reading for senior engineers. The most common mistake juniors make is assuming the network is reliable. Always configure aggressive timeouts, retries with exponential backoff, and circuit breakers for any network call between nodes.

---

## 15. Common Mistakes

- **Ignoring clock drift:** Clocks on different servers are rarely exactly synced. Relying on `time.time()` to order events across a distributed system will lead to subtle, impossible-to-reproduce bugs. Use logical clocks (Vector Clocks) or central sequencing.
- **Over-distributing:** Breaking a small app into 20 microservices before you have the team to support it.
