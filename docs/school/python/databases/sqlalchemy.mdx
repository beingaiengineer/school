# SQLAlchemy

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | SQLAlchemy |
| Difficulty | Intermediate |
| Used For | Connecting Python objects to relational databases (ORM) |
| Main Strength | Database-agnostic, powerful querying, prevents SQL injection automatically |
| Common Mistake | Triggering the N+1 query problem by lazy loading too much |
| Performance | Slower than raw SQL due to Python overhead, but fast enough for 95% of use cases |

Bro, SQLAlchemy is the industry standard tool for talking to databases in Python.

It acts as a translator between your Python code (which uses objects and classes) and your database (which uses tables and rows). Instead of writing raw SQL strings, you write Python code, and SQLAlchemy generates the SQL for you.

## 2. Real-World Analogy

Bro, think of SQLAlchemy like a highly skilled bilingual legal assistant.

You (the Python programmer) speak English. The judge (the Database) speaks Legal Jargon.

You tell your assistant: "Get me the file for John Doe." (Python code: `session.query(User).filter_by(name="John Doe").first()`)

The assistant immediately translates this into perfect Legal Jargon: `SELECT * FROM users WHERE name = 'John Doe' LIMIT 1;`, talks to the judge, gets the paper, and hands it back to you as an English summary document (a Python object).

You don't need to learn the judge's exact dialect, and if you move to a new state with a different judge (switching from PostgreSQL to MySQL), the assistant handles the new dialect automatically.

## 3. Concept Explanation

SQLAlchemy has two main layers:

1. **Core:** A SQL toolkit that lets you write SQL using Python expressions. It's closer to the metal.
2. **ORM (Object Relational Mapper):** A higher-level system that maps database tables to Python classes, and rows to Python objects. This is what most people mean when they say "I'm using SQLAlchemy."

Use SQLAlchemy when you are building a backend API, a web app (like Flask or FastAPI), or any system that needs complex, structured database interactions.

Do not use it for ultra-high-performance bulk data pipelines (like inserting 50 million rows a minute); use raw SQL or specialized batch tools for that.

## 4. Syntax Table

| Concept | SQLAlchemy ORM Example |
|---|---|
| Define a Table | `class User(Base): __tablename__ = 'users'` |
| Create a Record | `user = User(name="Alex") ; session.add(user)` |
| Read Records | `users = session.query(User).all()` |
| Update Record | `user.name = "Bro" ; session.commit()` |
| Delete Record | `session.delete(user) ; session.commit()` |

## 5. Beginner Example

Here is how you define a user, save them, and retrieve them using an in-memory SQLite database.

```python
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker

# 1. Connect to DB (SQLite in memory for testing)
engine = create_engine('sqlite:///:memory:', echo=False)
Base = declarative_base()

# 2. Define the Model
class User(Base):
    __tablename__ = 'users'
    id = Column(Integer, primary_key=True)
    name = Column(String)

# Create tables
Base.metadata.create_all(engine)

# 3. Create a Session
Session = sessionmaker(bind=engine)
session = Session()

# 4. Insert Data
new_user = User(name="Engineer Bro")
session.add(new_user)
session.commit()

# 5. Query Data
found_user = session.query(User).filter_by(name="Engineer Bro").first()
print(f"Found: {found_user.name}")
```

## 6. Real-World Engineering Example

In production systems (like FastAPI), you usually don't create a new engine every request. You manage sessions carefully. Here is how a user registration service might look.

```python
from sqlalchemy.exc import IntegrityError

def register_user(db_session, username: str, email: str):
    """
    Production-grade user creation.
    """
    try:
        new_user = User(username=username, email=email)
        db_session.add(new_user)
        db_session.commit()
        db_session.refresh(new_user) # Get the DB-generated ID back
        return {"status": "success", "user_id": new_user.id}

    except IntegrityError:
        # Happens if email violates a UNIQUE constraint in the DB
        db_session.rollback() # CRITICAL: Undo the failed transaction
        return {"status": "error", "message": "Email already exists."}
    except Exception as e:
        db_session.rollback()
        # Log the real error to Datadog/Sentry here
        return {"status": "error", "message": "Internal server error."}
```

Notice the `db_session.rollback()`. If an error happens and you don't roll back, the session becomes "poisoned" and all future queries in that session will crash.

When you call `session.query(User)`, it doesn't execute immediately. SQLAlchemy builds an Abstract Syntax Tree (AST) representing your query.

It only converts it to raw SQL and hits the database when you evaluate it (e.g., by calling `.all()`, `.first()`, or iterating over it). This "lazy" evaluation lets you chain filters dynamically before executing.

## 8. Performance Table

| Operation | Performance Impact | Note |
|---|---|---|
| Single Row Query | Very Fast | Negligible overhead vs raw SQL. |
| Bulk Insert (ORM) | Slow | ORM creates Python objects for every row. |
| Bulk Insert (Core) | Very Fast | Bypasses ORM, sends raw parameterized SQL. |
| Nested Relationships | Warning | Can trigger N+1 queries if not eager loaded. |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between SQLAlchemy Core and ORM? | Core uses Python expressions to write SQL; ORM maps tables to Python classes and rows to objects. |
| What does `session.commit()` actually do? | It flushes all pending changes to the database and permanently saves the transaction. |
| How do you prevent the N+1 query problem in SQLAlchemy? | By using `joinedload()` or `selectinload()` to fetch related tables in a single query. |
| Why must you call `session.rollback()` in an exception block? | Because if a DB error occurs, the transaction is aborted at the DB level, but SQLAlchemy's session needs to be reset to become usable again. |
| Is SQLAlchemy thread-safe? | The Engine is thread-safe, but the `Session` is NOT thread-safe. You must create a new session per thread/request. |

## 10. Tricky Questions & Edge Cases

Look at this code, bro. What happens?

```python
user = session.query(User).filter_by(id=1).first()
user.name = "New Name"
# Note: I forgot to call session.add(user)
session.commit()
```

**Answer:** The name *will* be updated in the database!

Why? Because SQLAlchemy's session implements the **Unit of Work** pattern. It tracks the `user` object. When you change `.name`, SQLAlchemy marks the object as "dirty". When you call `.commit()`, it automatically scans for dirty objects and generates the `UPDATE` SQL. You only need `.add()` for completely *new* objects.

## 11. Real-World Usage

Every major Python web backend uses SQLAlchemy:
- **Reddit** heavily uses SQLAlchemy Core for performance.
- **Yelp** built custom abstractions over it.
- **FastAPI** applications almost exclusively use SQLAlchemy (often with `asyncpg` for async support) to handle their relational database needs.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Define models clearly using `declarative_base`. | Hardcode raw SQL queries if an ORM method exists. |
| Use Alembic (SQLAlchemy's sister project) for database migrations. | Manually run `ALTER TABLE` scripts in production. |
| Scope one `Session` per web request. | Share a single global `Session` across your whole app. |

## 13. Production Notes

> ⚠ Production Note:
> **Session Lifecycle is critical.** In web frameworks, always use dependency injection or middleware to open a session at the start of a request, and absolutely ensure you `.close()` it in a `finally` block at the end of the request. Leaking sessions will quickly exhaust your database connection pool and take down your entire API.

## 14. Common Mistakes

- **The N+1 Problem:** Looping over a list of `Users` and printing their `user.address.city`. If you have 100 users, SQLAlchemy will silently run 1 query for the users, and 100 separate queries for the addresses. Always use `joinedload` for relationships you know you need.
- **Detached Instances:** Trying to access a related property (like `user.address`) *after* the session has been closed. It will throw a `DetachedInstanceError`.
- **Using `session.query` in SQLAlchemy 2.0:** While still supported, SQLAlchemy 2.0 pushes towards the newer `session.execute(select(User))` syntax which unifies Core and ORM and fully supports Async.
