# 3. Concept Explanation

SQLAlchemy is the most popular SQL toolkit and Object-Relational Mapper (ORM) for Python.

It exists to map Python classes to database tables. This means you can interact with your database using normal Python objects instead of writing raw string queries. It has two main parts:
1. **Core**: A SQL abstraction toolkit (writing SQL with Python methods).
2. **ORM**: The Object-Relational Mapper (mapping tables to Python classes).

**When to use it:**
- Almost any serious Python backend project connecting to SQL (PostgreSQL, MySQL, SQLite).
- When you want code maintainability over writing raw SQL strings everywhere.

**When NOT to use it:**
- If you are doing extremely heavy data engineering/analytics where Pandas + raw SQL is faster.
- For simple one-off scripts where a quick `sqlite3` raw query is enough.

# 5. Beginner Example

Let's look at how clean it is to create a user and fetch them.

```python
from sqlalchemy import create_engine, String, select
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, Session

# 1. Setup Engine (SQLite in memory for this example)
engine = create_engine("sqlite:///:memory:")

class Base(DeclarativeBase):
    pass

# 2. Define the Model
class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(50))

# Create tables
Base.metadata.create_all(engine)

# 3. Use it!
with Session(engine) as session:
    # Insert
    new_user = User(name="Amit")
    session.add(new_user)
    session.commit()

    # Query
    stmt = select(User).where(User.name == "Amit")
    user = session.scalars(stmt).first()

    print(f"Found user: {user.name} with ID: {user.id}")
```

# 6. Real-World Engineering Example

In a production app (like FastAPI or Flask), you manage database sessions carefully, often per request, and handle updates and relationships.

```python
from sqlalchemy import select, update
from sqlalchemy.orm import Session

def deactivate_inactive_users(session: Session, days_inactive: int):
    """
    Production-style function to bulk update users.
    Notice we use the Core update() for performance instead of loading ORM objects!
    """
    # Find inactive users
    stmt = (
        update(User)
        .where(User.last_login_days_ago > days_inactive)
        .values(is_active=False)
    )

    result = session.execute(stmt)
    session.commit()

    print(f"Bro, deactivated {result.rowcount} users.")
```

Bro, SQLAlchemy works in layers.
Your Python app talks to the ORM (Objects). The ORM translates that into the Core (Expressions). The Core passes it to the Engine. The Engine uses a Dialect (like `psycopg2` for Postgres) to talk raw SQL to the actual Database.

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between SQLAlchemy Core and ORM? | Core is the SQL abstraction toolkit (writing SQL statements). ORM maps those tables to Python classes. |
| What is the N+1 problem in SQLAlchemy? | When you loop through a list of objects and access a related child object, triggering a new DB query for every single child. |
| How do you fix the N+1 problem? | By using Eager Loading strategies like `joinedload` or `selectinload` during the initial query. |
| What does `session.commit()` actually do? | It flushes all pending changes to the database and issues a COMMIT command to conclude the transaction. |
| Should you pass `Session` objects between threads? | No, SQLAlchemy Sessions are not thread-safe. Create a new session per thread/request. |

# 10. Tricky Questions & Edge Cases

Look at this tricky situation:

```python
user = session.scalars(select(User).where(User.id == 1)).first()
user.name = "New Name"

# Wait, I didn't call session.add(user). Will it save?
session.commit()
```

**Will it save?**
Yes, bro! SQLAlchemy tracks changes to objects attached to the session via the "Identity Map". When you call `commit()`, it sees the object changed and automatically generates the `UPDATE` statement.

# 12. Best Practices

| DO | DON'T |
|---|---|
| Use `selectinload` for 1-to-Many relationships to avoid N+1 | Don't default to lazy loading if you know you need the related data |
| Close sessions in a `finally` block or context manager | Don't leave sessions open indefinitely |
| Use async SQLAlchemy if using an async framework (FastAPI) | Don't use blocking sync calls in an async web worker |

---

# 13. Production Notes

> ⚠ Production Note: Avoid `session.query()` in modern codebases.
SQLAlchemy 2.0 changed the standard syntax. The old `session.query(User)` is legacy. You should always use the new `select()` construct: `session.execute(select(User)).scalars().all()`. This prepares you for Asyncio compatibility and makes type checking work perfectly.

---

# 14. Common Mistakes

- **Memory Leaks**: Keeping references to ORM objects for a long time without expiring or closing the session.
- **The N+1 Query**: This is the #1 killer of application performance. Always check your SQL query logs to see if your ORM is firing 100 small queries instead of 1 big JOIN.
- **Putting Business Logic in Models**: Keep your models clean. Put complex logic in service layers, not directly inside the SQLAlchemy class methods.
