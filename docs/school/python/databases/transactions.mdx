# Transactions

# 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Database Transactions |
| Difficulty | Intermediate |
| Used For | Ensuring data integrity during multiple related database operations |
| Common Mistake | Forgetting to rollback on failure, leaving partial data |
| Performance | Long-running transactions lock rows and degrade performance |

---

# 2. Real-World Analogy

Bro, think of a database transaction like transferring money between bank accounts.

If you send ₹1000 to your friend:
1. ₹1000 is deducted from your account.
2. The bank's server crashes.
3. ₹1000 is NOT added to your friend's account.

Without a transaction, your money is gone into the void.
With a transaction, the database groups these two steps. If step 2 fails, it **rolls back** (undoes) step 1. It guarantees that either *both* steps succeed (commit), or *neither* happens. All or nothing.

---

# 3. Concept Explanation

A database transaction is a single unit of work that consists of one or more database operations.

Transactions follow the **ACID** properties:
- **Atomicity:** All or nothing.
- **Consistency:** The database moves from one valid state to another.
- **Isolation:** Concurrent transactions don't interfere with each other.
- **Durability:** Once committed, changes are permanent, even if the power fails.

In Python (whether using raw DBAPI or an ORM like SQLAlchemy), you manage transactions by executing queries, and then explicitly calling `.commit()` to save or `.rollback()` to abort.

# 5. Beginner Example

Here is a basic money transfer using raw SQLite transactions.

```python
import sqlite3

conn = sqlite3.connect('bank.db')
cursor = conn.cursor()

try:
    # 1. Deduct from Alice
    cursor.execute("UPDATE accounts SET balance = balance - 100 WHERE name = 'Alice'")

    # Simulate an error (e.g., Bob's account doesn't exist)
    raise ValueError("Bob's account is invalid!")

    # 2. Add to Bob (This won't execute)
    cursor.execute("UPDATE accounts SET balance = balance + 100 WHERE name = 'Bob'")

    # Save changes (This won't execute either)
    conn.commit()
    print("Transfer successful!")

except Exception as e:
    # UNDO step 1
    conn.rollback()
    print(f"Transfer failed. Rolled back. Error: {e}")

finally:
    conn.close()
```

# 6. Real-World Engineering Example

In production web apps, you don't write `try/except/rollback` everywhere. You use context managers or decorators to guarantee transaction safety automatically.

```python
from contextlib import contextmanager

@contextmanager
def transaction(session):
    """Automatically commits on success, rolls back on error."""
    try:
        yield session
        session.commit()
    except Exception:
        session.rollback()
        raise
    finally:
        session.close()

# Usage in a business logic function
def create_order(user_id, item_id, session):
    with transaction(session) as db:
        # 1. Create order record
        order = Order(user_id=user_id, item_id=item_id)
        db.add(order)

        # 2. Deduct inventory
        item = db.query(Item).filter_by(id=item_id).with_for_update().first() # Lock the row
        if item.stock < 1:
            raise ValueError("Out of stock!")
        item.stock -= 1

        # If no exceptions are raised, it auto-commits here!
```

# 7. Internal Working

When a transaction starts, the database creates a temporary workspace.
As you issue `UPDATE` or `INSERT` commands, the database records these changes in a Write-Ahead Log (WAL) or transaction journal, and locks the affected rows so no one else can change them.

If you `COMMIT`, the changes are flushed to the main data files.
If you `ROLLBACK`, the database discards the temporary workspace and releases the locks.

# 8. Performance Table

| Action | Performance Impact |
|---|---|
| Short Transaction | Low. Quick lock, quick release. |
| Long Transaction | High. Holds locks, blocking other users. |
| Missing Indexes in Txn | High. Causes table locks instead of row locks. |

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| What does ACID stand for? | Atomicity, Consistency, Isolation, Durability. |
| What happens if you don't commit a transaction? | The database holds locks indefinitely (or until timeout/connection drop), and changes are not saved. |
| What is a deadlock? | When Transaction A locks Row 1 and waits for Row 2, while Transaction B locks Row 2 and waits for Row 1. They wait forever. |
| What is `SELECT ... FOR UPDATE`? | It locks the selected rows until the transaction ends, preventing other transactions from modifying them. |

# 10. Tricky Questions & Edge Cases

Bro, what happens if an external API call is inside your transaction?

```python
with transaction(session) as db:
    user = User(name="Alice")
    db.add(user)

    # DANGEROUS!
    response = requests.post("https://slow-payment-gateway.com/charge")

    if response.status_code == 200:
        db.commit()
```

**The Trap:**
If the payment gateway takes 30 seconds to respond, your database transaction stays open for 30 seconds. This holds database connection pool slots and row locks, bringing your entire application to a halt.

**The Fix:**
Never put slow external network calls inside a database transaction. Do the network call *before* or *after* the transaction.

# 12. Best Practices

| DO | DON'T |
|---|---|
| Keep transactions as short as possible. | Put API calls or heavy computations inside transactions. |
| Use context managers to auto-rollback on errors. | Rely on manual `try/except` logic everywhere. |
| Catch exceptions and explicitly call `rollback()`. | Swallow exceptions and leave transactions hanging. |

---

# 13. Production Notes

> ⚠ **Production Note:**
> In high-concurrency systems, be careful of "Lost Updates." If two users load the same product, see 1 in stock, and both click "Buy", standard transactions might let both succeed. You must use **Pessimistic Locking** (`with_for_update()`) or **Optimistic Locking** (using a `version` column) to prevent race conditions.

---

# 14. Common Mistakes

- **Swallowing Errors:** Catching an error and not rolling back, leading to a polluted SQLAlchemy session state. The next query will fail with "PendingRollbackError".
- **Nested Transactions:** Trying to start a transaction inside a transaction. Some databases don't support true nested transactions, requiring complex "Savepoints".
- **Over-locking:** Locking entire tables when only row locks are needed (often caused by missing indexes).
