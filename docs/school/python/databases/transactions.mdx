# Transactions

# 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Database Transactions |
| Difficulty | Intermediate |
| Used For | Ensuring data integrity during multiple related operations |
| Common Mistake | Forgetting to rollback on failure, leaving partial data |
| Performance | Holding transactions open too long blocks other queries |

# 2. Real-World Analogy

Bro, think of a transaction like an ATM money transfer. You want to send ₹10,000 to your friend.
Two things must happen:
1. Deduct ₹10,000 from your account.
2. Add ₹10,000 to your friend's account.

If step 1 succeeds but the power goes out before step 2, your money is gone and your friend didn't get it. A transaction ensures both steps succeed together, or if *anything* fails, both steps are completely canceled (rolled back). It's an "all or nothing" deal.

# 3. Concept Explanation

A database transaction is a sequence of operations performed as a single logical unit of work. It follows the **ACID** properties:
- **Atomicity:** All operations succeed or all fail (no partial updates).
- **Consistency:** The database moves from one valid state to another.
- **Isolation:** Concurrent transactions don't interfere with each other.
- **Durability:** Once committed, the changes are permanent, even if the system crashes.

In Python, whether using Raw SQL or an ORM, you start a transaction, perform your logic, and then explicitly call `commit()` to save or `rollback()` to undo.

# 4. Syntax Table

| Operation | SQLAlchemy Example | Raw SQL Example (`sqlite3`) |
|---|---|---|
| Start | (Implicit on first query) | `conn.execute("BEGIN TRANSACTION")` |
| Execute | `session.add(user)` | `cursor.execute("INSERT...")` |
| Save Changes | `session.commit()` | `conn.commit()` |
| Undo Changes | `session.rollback()` | `conn.rollback()` |

# 5. Beginner Example

Here is how you handle a transaction manually using standard Python exception handling.

```python
import sqlite3

conn = sqlite3.connect(':memory:')
cursor = conn.cursor()
cursor.execute('''CREATE TABLE accounts (id INTEGER, balance INTEGER)''')
cursor.execute("INSERT INTO accounts VALUES (1, 1000), (2, 500)")
conn.commit()

try:
    # Deduct from Account 1
    cursor.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1")

    # Simulate a crash or an error happening in the middle
    raise ValueError("System crashed during transfer!")

    # Add to Account 2 (This won't run)
    cursor.execute("UPDATE accounts SET balance = balance + 100 WHERE id = 2")

    # Commit if everything worked
    conn.commit()
    print("Transfer successful")

except Exception as e:
    # If anything failed, undo the deduction!
    conn.rollback()
    print(f"Transfer failed, changes rolled back. Error: {e}")

# Check final state - Account 1 still has 1000.
cursor.execute("SELECT balance FROM accounts WHERE id = 1")
print(f"Account 1 Balance: {cursor.fetchone()[0]}")
```

# 6. Real-World Engineering Example

In modern production apps, you rarely write manual `try/except/rollback` blocks. Instead, you use Context Managers (`with` blocks) provided by your ORM. This guarantees that if the block completes, it commits, and if an exception is raised, it rolls back automatically.

```python
from sqlalchemy import create_engine, Column, Integer
from sqlalchemy.orm import declarative_base, sessionmaker

engine = create_engine('sqlite:///:memory:')
Base = declarative_base()

class Account(Base):
    __tablename__ = 'accounts'
    id = Column(Integer, primary_key=True)
    balance = Column(Integer)

Base.metadata.create_all(engine)
SessionLocal = sessionmaker(bind=engine)

def transfer_funds(from_id: int, to_id: int, amount: int):
    # Using a context manager for the transaction
    with SessionLocal() as session:
        with session.begin(): # This starts the transaction block
            sender = session.query(Account).filter_by(id=from_id).first()
            receiver = session.query(Account).filter_by(id=to_id).first()

            if sender.balance < amount:
                raise ValueError("Insufficient funds")

            sender.balance -= amount
            receiver.balance += amount

        # The session.begin() context manager automatically commits here!
        # If any exception was raised, it automatically rolls back!
```

When you start a transaction, the database creates an isolated snapshot or workspace in memory/logs. As you make changes, they are written to a Write-Ahead Log (WAL) but are not yet visible to other connections. When you `COMMIT`, the database marks those changes as permanent. If you `ROLLBACK`, it simply discards the workspace and ignores the log entries.

# 8. Performance Table

| Action | Performance Impact |
|---|---|
| Keeping a transaction open | Locks rows/tables, blocking other users. Keep them fast. |
| Large bulk inserts in one transaction | Uses lots of memory, may cause timeouts. Batch them. |
| Calling external APIs inside a transaction | DANGEROUS. Network delays will hold database locks open. |

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| What does ACID stand for? | Atomicity, Consistency, Isolation, Durability. |
| What happens if you don't commit or rollback? | The transaction stays open (dangling), holding locks and eventually causing a timeout or deadlock. |
| What is a Deadlock? | When Transaction A holds a lock on Row 1 and wants Row 2, while Transaction B holds Row 2 and wants Row 1. They block forever. |
| Can you nest transactions? | Most databases support "Savepoints" which act like nested transactions, allowing partial rollbacks within a larger transaction. |

# 10. Tricky Questions & Edge Cases

Bro, look at this API endpoint. What is the catastrophic mistake here?

```python
@app.post("/checkout")
def checkout(cart_id):
    with db.session.begin():
        # 1. Deduct inventory
        deduct_inventory(cart_id)

        # 2. Charge credit card via Stripe API
        response = stripe.Charge.create(amount=100)

        # 3. Mark as paid
        mark_cart_paid(cart_id)
```

**Answer:** You are calling an external network API (Stripe) *inside* the database transaction!
**Why it's bad:** If Stripe takes 5 seconds to respond, your database transaction stays open for 5 seconds. If 100 users checkout at once, you lock up your entire database waiting for Stripe.
**The Fix:** Do the API call *before* or *after* the transaction, or use a background job/two-phase commit.

# 11. Real-World Usage

- **Banking/Fintech:** Every single ledger entry uses strict transactions.
- **E-commerce:** Deducting inventory and confirming an order must be atomic so you don't sell an item twice.
- **Data Migrations:** When updating production schemas, wrap the changes in a transaction so if one table fails to migrate, the whole database isn't left corrupted.

# 12. Best Practices

| DO | DON'T |
|---|---|
| Use context managers (`with session.begin():`) to guarantee cleanup. | Rely on manual `commit()` and `rollback()` calls scattered in code. |
| Keep transactions as short and fast as possible. | Put slow operations (file I/O, API calls) inside a transaction. |
| Catch specific database exceptions. | Swallow generic `Exception` and silently rollback without logging. |

# 13. Production Notes

> ⚠ Production Note:
Isolation levels matter in production. By default, Postgres uses `Read Committed`. This means two concurrent transactions might see different data if one commits midway. For high-stakes financial operations, you might need to use `Serializable` isolation or explicit `SELECT ... FOR UPDATE` locks to prevent race conditions during the transaction.

# 14. Common Mistakes

- **Silent Failures:** Catching an exception, rolling back, but not returning an error to the user, making them think it succeeded.
- **Auto-commit Confusion:** Forgetting that some databases (like SQLite in certain modes) auto-commit every statement unless you explicitly type `BEGIN`.
- **Dangling Transactions:** Leaving a connection open in an error state, which breaks the connection pool for the next user.
