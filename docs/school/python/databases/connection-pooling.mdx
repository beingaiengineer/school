# Connection Pooling

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Database Connection Pooling |
| Difficulty | Advanced |
| Used For | Scaling database connections in high-traffic apps |
| Common Mistake | Leaking connections by not closing them |
| Performance | Dramatically reduces connection overhead |

---

## 2. Real-World Analogy

Bro, imagine a call center for a bank.

**Without a pool:** Every time a customer calls, the bank hires a new employee, trains them on the spot, has them answer the question, and then immediately fires them. It takes 10 minutes to train them for a 30-second call!

**With a pool:** The bank has 50 trained employees sitting at desks waiting. When a call comes in, an employee takes it. When they finish, they don't get fired; they just wait for the next call.

Connection Pooling is keeping a set of database connections open and ready to use, so you don't waste time "hiring and firing" connections!

---

## 3. Concept Explanation

Connecting to a database (like PostgreSQL or MySQL) is an expensive operation over the network. It requires a TCP handshake, authentication, and memory allocation.

If your web server gets 1,000 requests per second, creating 1,000 new database connections per second will crush both the web server and the database.

A **Connection Pool** solves this by:
1. Creating a set of connections at startup (e.g., 20 connections).
2. "Lending" a connection to a request when needed.
3. Returning the connection to the pool when the request finishes, instead of closing it.

## 5. Beginner Example

Most ORMs like SQLAlchemy handle pooling automatically. You just need to configure it when creating the engine.

```python
from sqlalchemy import create_engine

# Create an engine with Connection Pooling
engine = create_engine(
    'postgresql://user:pass@localhost/db',
    pool_size=5,          # Keep 5 connections permanently open
    max_overflow=10,      # Allow up to 10 extra temporary connections during spikes
    pool_timeout=30,      # Wait 30s for a connection before crashing
    pool_recycle=1800     # Refresh connections every 30 mins to avoid stale drops
)

# When you do this, it borrows from the pool
with engine.connect() as connection:
    result = connection.execute("SELECT 1")
# When the 'with' block ends, the connection goes BACK to the pool, it is NOT closed!
```

## 6. Real-World Engineering Example

Bro, in a real production FastAPI app, connection pooling is critical. FastAPI handles thousands of async requests, and if you don't pool properly, you'll hit the dreaded `Too many connections` error from Postgres.

```python
from fastapi import FastAPI, Depends
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import sessionmaker

# 1. Async engine with pooling
engine = create_async_engine(
    "postgresql+asyncpg://user:pass@localhost/db",
    pool_size=20,
    max_overflow=0 # In async apps, strict pool limits are often safer
)

# 2. Session factory
async_session = sessionmaker(
    engine, class_=AsyncSession, expire_on_commit=False
)

# 3. Dependency to get a session
async def get_session() -> AsyncSession:
    async with async_session() as session:
        yield session # Lends connection to the endpoint
        # Automatically returns to pool when endpoint finishes

app = FastAPI()

@app.get("/users")
async def get_users(session: AsyncSession = Depends(get_session)):
    # Uses the borrowed connection
    return await session.execute("SELECT * FROM users")
```

When a request asks for a connection, the pool checks if one is idle. If yes, it lends it out. If all connections are busy, the request goes into a queue and waits (up to `pool_timeout`). When another request finishes, it returns its connection, and the waiting request grabs it.

## 8. Performance Table

| Configuration | Impact |
|---|---|
| Too Small Pool | Requests queue up and time out (Slow API) |
| Too Large Pool | Database crashes from OOM (Out Of Memory) |
| No Pooling | Huge latency on every API call (TCP handshake) |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is Connection Pooling? | A cache of database connections kept open so they can be reused. |
| Why is creating a new DB connection slow? | It requires network routing, TCP handshakes, and database authentication. |
| What happens if the pool is full? | New requests wait in a queue until a connection is returned or they hit a timeout. |
| What is PgBouncer? | An external connection pooler often used with PostgreSQL to manage connections across multiple application servers. |

## 10. Tricky Questions & Edge Cases

**The Connection Leak:**
Bro, this is a massive interview red flag. If you take a connection and crash without returning it, the pool loses that connection forever.

```python
# 🚨 DISASTER: Connection Leak!
conn = engine.connect()
# If this crashes, the connection is NEVER returned to the pool!
1 / 0
conn.close()

# ✅ SAFE: Context Managers guarantee return to the pool
with engine.connect() as conn:
    1 / 0 # Even if it crashes, 'with' block returns the conn
```

## 11. Real-World Usage

- **Microservices Architecture:** 10 microservices might each need a pool of 10 connections. That's 100 connections hitting the DB!
- **Serverless (AWS Lambda):** Lambdas spin up and down constantly. Traditional pooling fails here. Companies use tools like AWS RDS Proxy or PgBouncer to manage pools for serverless apps.

---

## 12. Best Practices

| DO | DON'T |
|---|---|
| Use external poolers (PgBouncer) for massive scale. | Don't set `pool_size` to 1000. Databases perform worse with too many connections! |
| Size your pool based on CPU cores. Formula: `(cores * 2) + effective_spindle_count`. | Don't forget `pool_recycle` to prevent MySQL/Postgres from dropping stale connections. |

---

## 13. Production Notes

> ⚠ Production Note:
Bro, setting `pool_size=1000` is a classic junior mistake. PostgreSQL uses a separate process for every connection, meaning 1,000 connections will eat all your RAM and crash the DB. A standard rule of thumb is keeping pool sizes surprisingly low (e.g., 20-50 per server). If you need more concurrency, your queries are probably too slow and need indexing, not more connections!

---

## 14. Common Mistakes

1. **Forgetting `pool_recycle`:** Databases often kill idle connections after a few hours. If the app tries to use a killed connection, it throws an error. `pool_recycle` forces the app to refresh old connections gracefully.
2. **Global Variables in Lambdas:** Trying to use in-memory connection pools in AWS Lambda functions often leads to connection spikes and database death.
3. **Leaving Transactions Open:** If you start a transaction but don't commit/rollback, that connection is locked and useless to the pool!
