# Connection Pooling

# 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Connection Pooling |
| Difficulty | Intermediate |
| Used For | Reusing database connections to handle high traffic efficiently |
| Common Mistake | Creating a new DB connection inside every API request |
| Performance | Drastically reduces latency and CPU load on the database server |

---

# 2. Real-World Analogy

Bro, think of a database connection like hiring a taxi.

Imagine if every time you wanted to go 1 km, you had to call a taxi company, wait 10 minutes for them to dispatch a car, get in, drive 1 km, and then the taxi gets crushed in a junkyard. That's extremely wasteful!

Instead, imagine a **Taxi Stand (Connection Pool)** outside your office. There are always 5 taxis waiting with their engines running. You jump in, do your trip, and when you're done, the taxi goes back to the stand to wait for the next person. That is connection pooling!

---

# 3. Concept Explanation

Opening a TCP connection to a database (like PostgreSQL) is slow and expensive. It requires network handshakes, authentication, and allocating memory on the database server.

If your web server gets 1,000 requests per second, and each request opens a *new* DB connection, your database will crash from the CPU/memory load of just managing connections.

A **Connection Pool** is a cache of database connections maintained so that the connections can be reused when future requests to the database are required.

**When to use it:**
- Always, for any production web application or API.

**When NOT to use it:**
- A simple, single-threaded scheduled cron job that runs once a night and connects, does work, and exits.

---

# 4. Syntax Table

| Concept | SQLAlchemy Syntax |
|---|---|
| Default Pool | `create_engine("postgresql://...")` (Defaults to `QueuePool` with 5 connections) |
| Configure Pool | `create_engine(..., pool_size=10, max_overflow=20)` |
| Disable Pool | `create_engine(..., poolclass=NullPool)` (Creates new conn every time) |

# 5. Beginner Example

If you use SQLAlchemy, bro, you are already using connection pooling! It happens automatically in the background when you create the engine.

```python
from sqlalchemy import create_engine
from sqlalchemy.orm import Session

# This creates a QueuePool with 5 connections by default.
# It DOES NOT connect yet (lazy initialization).
engine = create_engine("postgresql://user:pass@localhost/db")

# User 1 hits the API
with Session(engine) as session:
    # 1. Grabs a connection from the pool (or creates one if empty)
    session.execute("SELECT 1")
    # 2. At the end of the 'with' block, the connection is NOT closed.
    # It is RETURNED to the pool!

# User 2 hits the API
with Session(engine) as session:
    # 1. Grabs the EXACT SAME connection User 1 just used. Lightning fast!
    session.execute("SELECT 2")
```

# 6. Real-World Engineering Example

In a highly concurrent FastAPI app, you need to configure the pool carefully to handle spikes in traffic without overwhelming the database.

```python
from sqlalchemy.ext.asyncio import create_async_engine

# Production Engine Setup
engine = create_async_engine(
    "postgresql+asyncpg://user:pass@db_host/prod_db",
    pool_size=20,          # Keep 20 connections open at all times
    max_overflow=10,       # Allow 10 extra temporary connections during traffic spikes
    pool_timeout=30,       # If all 30 are busy, wait 30 seconds before throwing an error
    pool_recycle=1800,     # Reconnect connections older than 30 mins to avoid DB firewalls dropping them
)

# Now pass this engine to your async session maker!
```

When a request asks for a connection, the Pool hands over an existing one. If the pool is empty and `pool_size` is reached, it creates `max_overflow` temporary connections. If those are full too, the request **blocks and waits** until another request finishes and returns its connection.

# 8. Performance Table

| Strategy | Request Latency | DB Server CPU Load | Risk |
|---|---|---|---|
| New Connection Per Request | +50ms to +200ms | Extremely High | DB crashes under load (Too many connections error) |
| Connection Pooling | +0ms (Instant) | Low | App deadlock if connections aren't returned to pool |

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is a connection pool? | A cache of database connections kept open so they can be reused by future requests, avoiding the overhead of establishing a new TCP connection every time. |
| What happens if the connection pool is full? | New requests will block and wait (up to `pool_timeout`). If no connection is returned in time, an error (TimeoutError) is thrown. |
| What is PgBouncer? | An external connection pooler often used with PostgreSQL to manage connections across multiple application servers. |
| Why use `pool_recycle`? | Firewalls or database servers often silently drop TCP connections that have been idle for too long. `pool_recycle` forces the app to recreate old connections proactively. |

# 10. Tricky Questions & Edge Cases

Look at this code:

```python
conn = engine.connect()
result = conn.execute("SELECT * FROM users")
# ... processing result ...

# Bro, where is the conn.close() ???
```

**The Trap:**
If you forget to call `conn.close()` (or fail to use a Context Manager `with engine.connect() as conn:`), the connection is NEVER returned to the pool.

If your pool size is 5, after 5 requests, your entire application will freeze because all 5 connections are "checked out" and lost in memory. This is called **Connection Leak**.

---

# 11. Real-World Usage

- **Microservices**: If you have 10 Kubernetes pods running FastAPI, and each has a pool size of 20, your database sees 200 open connections. You must calculate this so you don't exceed PostgreSQL's `max_connections` setting!
- **Serverless (AWS Lambda)**: Lambdas spin up and down constantly. They cannot maintain a persistent connection pool easily. This is why tools like RDS Proxy or PgBouncer are required for serverless apps connecting to relational databases.

---

# 12. Best Practices

| DO | DON'T |
|---|---|
| Use Context Managers (`with Session:`) to guarantee connections return to the pool | Manually open connections and forget to close them in error blocks |
| Keep your pool size relatively small (10-20 is often plenty for a web worker) | Set pool size to 1000 thinking "bigger is better" |
| Use external poolers (PgBouncer) if you have hundreds of web servers | Rely only on app-level pooling in massive distributed systems |

---

# 13. Production Notes

> ⚠ Production Note: The "Too many connections" error.
PostgreSQL typically defaults to a maximum of 100 open connections. If you set your SQLAlchemy `pool_size` to 50, and start 3 Gunicorn worker processes... `50 * 3 = 150`. Your app will crash on startup because it immediately exhausts the database connection limit. Size your pools based on `Max DB Connections / Total App Workers`.

---

# 14. Common Mistakes

- **Giant Pool Sizes**: Databases are actually faster when handling fewer concurrent active connections. A pool of 20 connections processing queries rapidly is faster than a pool of 500 connections causing the DB CPU to thrash context-switching.
- **Leaking Connections**: Doing network requests (calling external APIs) while holding a database connection from the pool.
- **Ignoring Dead Connections**: Not setting `pool_pre_ping=True` in SQLAlchemy, resulting in random "MySQL server has gone away" or connection reset errors.
