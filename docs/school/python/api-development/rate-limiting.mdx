# Rate Limiting

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | API Rate Limiting |
| Difficulty | Intermediate |
| Used For | Controlling how many requests a user can make in a given time |
| Main Strength | Prevents DDOS attacks, brute force, and protects database scale |
| Common Mistake | Storing rate limit counters in a standard relational DB (too slow) |
| Performance | Requires fast, in-memory datastores (like Redis) |

Bro, Rate Limiting is how you stop a single user (or an attacker) from crashing your server. If your server can handle 1,000 requests a second, and a bot starts sending 5,000, your server dies. Rate limiting blocks the bot after 100 requests and returns a `429 Too Many Requests` status.

## 3. Real-World Analogy

Think of rate limiting like an all-you-can-eat buffet.

You paid for the buffet (authenticated). You can eat as much as you want. BUT, the chef only lets you take 2 slices of pizza every 5 minutes.
If you try to take 10 slices at once, the chef slaps your hand (HTTP 429).
If you wait 5 minutes, you can get 2 more slices. This ensures there is enough pizza for everyone in the restaurant, and the kitchen doesn't collapse under your greed.

## 4. Concept Explanation

Rate Limiting defines a threshold for API access.

Common algorithms:
1. **Fixed Window:** You get 100 requests from 1:00 PM to 2:00 PM. At 2:00 PM, it resets to 100.
2. **Sliding Window:** You get 100 requests in the *last 60 minutes*.
3. **Token Bucket:** You have a bucket of 10 tokens. Every request costs 1 token. A background process adds 1 token back into the bucket every second.

To track this, the server usually looks at your **IP Address** or your **API Key / User ID**, and increments a counter in an in-memory database like Redis.

## 5. Syntax Table

| HTTP Header | Description |
|---|---|
| `HTTP/1.1 429 Too Many Requests` | The standard status code when a user is blocked. |
| `X-RateLimit-Limit` | How many total requests the user is allowed (e.g., 100). |
| `X-RateLimit-Remaining` | How many requests they have left in the window (e.g., 4). |
| `X-RateLimit-Reset` | Unix timestamp of when the limit resets. |

*Note: These headers are industry standard, though not strict RFC rules.*

## 6. Beginner Example

Let's look at a simple rate limiter in FastAPI using a Python dictionary (Not for production!).

```python
from fastapi import FastAPI, Request, HTTPException
import time

app = FastAPI()

# Store requests in memory (BAD FOR PROD)
user_requests = {}

@app.middleware("http")
async def rate_limit_middleware(request: Request, call_next):
    client_ip = request.client.host
    current_time = time.time()

    # Init user if new
    if client_ip not in user_requests:
        user_requests[client_ip] = []

    # Filter out requests older than 60 seconds (Sliding Window)
    recent_requests = [t for t in user_requests[client_ip] if current_time - t < 60]
    user_requests[client_ip] = recent_requests

    # Check limit (5 requests per minute)
    if len(recent_requests) >= 5:
        return HTTPException(status_code=429, detail="Bro, chill. Too many requests.")

    # Log this request
    user_requests[client_ip].append(current_time)

    return await call_next(request)

@app.get("/")
def home():
    return {"message": "Success"}
```

Bro, if you refresh this page 6 times fast, you get blocked.

## 7. Real-World Engineering Example

In production, you NEVER use a Python dictionary for rate limiting, because your app will run on multiple workers/servers. The dictionary isn't shared! You must use Redis.

```python
# Pseudo-code for production Redis rate limiting
import redis
from fastapi import HTTPException

# Connect to Redis
r = redis.Redis(host='localhost', port=6379, db=0)

def check_rate_limit(user_id: str):
    # Create a unique key for this user for this minute
    import time
    current_minute = int(time.time() / 60)
    key = f"rate_limit:{user_id}:{current_minute}"

    # Atomically increment the counter
    requests = r.incr(key)

    # If this is the first request, set the key to expire in 60s
    if requests == 1:
        r.expire(key, 60)

    if requests > 100:
        raise HTTPException(status_code=429, detail="Rate limit exceeded")
```

The bucket holds tokens. When you request, you take a token. If the bucket is empty, you get a 429. Because the refill process adds tokens slowly, it smooths out traffic bursts (you can handle a spike of 5 requests instantly, but then you must wait).

## 9. Performance Table

| Storage | Lookup Speed | Recommended For |
|---|---|---|
| Python Dict | Extremely Fast | Prototyping (Fails in multi-worker setups) |
| PostgreSQL / SQL | Very Slow | NEVER use SQL for API rate limiting |
| Redis | Very Fast | Standard production. Atomic increments are perfect. |
| API Gateway (Kong/Nginx) | Instant | Enterprise. Blocks requests before they even hit Python. |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| What HTTP code is for Rate Limiting? | `429 Too Many Requests`. |
| Why use Redis instead of a DB for rate limits? | Redis stores data in RAM, making it incredibly fast. Standard DBs hit the disk, causing bottlenecks. |
| How do you rate limit an unauthenticated user? | By their IP Address. |
| What is the issue with IP-based rate limiting? | Many users in a corporate office or coffee shop might share the same public IP (NAT). You might accidentally block everyone. |
| Fixed Window vs Token Bucket? | Fixed window resets completely at a specific time (e.g., top of the hour). Token bucket continuously trickles tokens, smoothing out traffic. |

## 11. Tricky Questions & Edge Cases

### The Thundering Herd at Reset Time

If you use a Fixed Window limit (e.g., 1000 requests per hour, resetting at 12:00), attackers will fire 1000 requests at 11:59:59, and another 1000 requests at 12:00:01.
Your server gets hit with 2000 requests in 2 seconds.
**Fix:** Use a Sliding Window or Token Bucket.

### The Headers

If you don't return `X-RateLimit-Remaining` headers, developers using your API have to guess when they will be blocked. Always provide headers so client applications can intentionally slow themselves down before hitting the 429 error.

| Domain | Real-World Usage |
|---|---|
| Login Endpoints | Limit to 5 attempts per minute per IP to prevent Brute Force password guessing. |
| SaaS Pricing | Free tiers get 100 req/day. Pro tiers get 10,000 req/day. Rate limits enforce business models. |

## 13. Best Practices

| DO | DON'T |
|---|---|
| Implement rate limiting at the API Gateway level (Nginx/Cloudflare) if possible | Rely solely on Python to stop massive DDOS attacks |
| Use Redis `INCR` for atomic, race-condition-free counting | Read a value, add 1 in Python, and write it back (race condition!) |
| Separate limits by endpoint (Login is strict, Search is loose) | Apply a blanket 100 req/sec rule to every single route |

## 14. Production Notes

> ⚠ Production Note:
> If Redis goes down, what happens to your API? Does it block everyone (Fail-closed) or allow everyone (Fail-open)? In production, you usually want Rate Limiting to *Fail-Open*. It's better to risk heavy traffic for a few minutes than to take your entire API offline because the Redis cache restarted.

## 15. Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| Rate limiting inside the route | If you check the limit *after* you parse the JSON body, the attacker still burns your CPU. Check limits in Middleware or API Gateways. |
| Not handling the 429 in the client | If your Python script uses `requests` and hits an API that returns 429, and you don't catch it, your script crashes. Always implement exponential backoff retries. |
