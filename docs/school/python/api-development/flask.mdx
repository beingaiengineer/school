## 4. Concept Explanation

Flask is a **WSGI** (Web Server Gateway Interface) micro-framework.

"Micro" does not mean it's only for small applications. It means the core framework is kept simple and extensible. By default, Flask does not include:
- A database ORM
- Form validation
- Authentication

It gives you routing, request/response handling, and a templating engine (Jinja2). Everything else is up to you.

Bro, this is amazing for small APIs or highly custom services, but it means you must establish your own folder structure and coding standards if the project grows.

## 6. Beginner Example

Here is a complete web server in 7 lines of code.

```python
from flask import Flask, jsonify

app = Flask(__name__)

@app.route('/api/ping', methods=['GET'])
def ping():
    return jsonify({"message": "pong!"})

if __name__ == '__main__':
    app.run(debug=True)
```

Run this file, and boom, you have a web server running on `localhost:5000`. Bro, this simplicity is why millions of developers learn Flask first.

## 7. Real-World Engineering Example

In the real world, you need to read JSON, validate it slightly, and return specific HTTP status codes.

```python
from flask import Flask, request, jsonify, abort

app = Flask(__name__)

users_db = []

@app.route('/api/users', methods=['POST'])
def create_user():
    data = request.get_json()

    if not data or 'username' not in data:
        # Abort returns a 400 Bad Request
        abort(400, description="Missing username")

    new_user = {
        "id": len(users_db) + 1,
        "username": data['username']
    }
    users_db.append(new_user)

    # Return data with a 201 Created status code
    return jsonify(new_user), 201
```

Flask uses **Thread-Local objects** (powered by Werkzeug). When a request comes in, Flask pushes a "Request Context" onto a stack for that specific thread.

This means `request` looks like a global variable you just import, but it magically only contains the data for the specific user currently hitting your server. Bro, it's elegant but sometimes confusing to test.

## 11. Tricky Questions & Edge Cases

### The App Context Error

```python
from flask import Flask, current_app
app = Flask(__name__)

# This will crash!
print(current_app.name)
```

Bro, `current_app` and `request` only exist *during* a request. If you try to use them outside of a route function (like at the top of the file), Flask throws a `RuntimeError: Working outside of application context.`

To fix it:
```python
with app.app_context():
    print(current_app.name)
```

### Global Variables as State

```python
# Bad: Global state in a web server
active_users = 0

@app.route('/login')
def login():
    global active_users
    active_users += 1
    return f"Users: {active_users}"
```

In production, you run Flask with multiple workers (processes). Global variables do NOT share data across processes. User A might see `1` and User B might see `0`. Always use a database or Redis for state.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Use **Blueprints** to split your app into modules (`auth/`, `users/`) | Keep 50 routes in `app.py` |
| Use environment variables for secrets | Hardcode API keys in the app config |
| Return proper HTTP status codes | Return string messages for errors |
| Validate input using `marshmallow` or `pydantic` | Blindly trust `request.json` |

## 14. Production Notes

> ⚠ Production Note:
> Never use `app.run()` in production. The built-in Flask server cannot handle concurrent traffic safely. You MUST use a production WSGI server like **Gunicorn** or **uWSGI** sitting behind a reverse proxy like **Nginx**.

Command to run in prod: `gunicorn -w 4 'app:app'` (4 worker processes).

## 15. Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| Not setting `FLASK_ENV=production` | Exposes interactive debuggers to users if an error happens (huge security risk). |
| Ignoring Blueprints | The app becomes an unmaintainable 2,000-line monolith. |
| Forgetting to return a response | A route function must return a string, dict, or Response object, or Flask throws an error. |
