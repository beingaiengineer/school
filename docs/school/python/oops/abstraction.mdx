# Abstraction

# 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Abstraction in Python |
| Difficulty | Intermediate |
| Used For | Hiding complex implementation details and defining strict contracts |
| Common Mistake | Relying on `raise NotImplementedError` instead of using the `abc` module |
| Performance | Minimal overhead from `abc` module checks during instantiation |

# 2. Real-World Analogy

Bro, think of a coffee machine in your office. You just press the "Cappuccino" button, and coffee comes out. You don't need to know the water temperature, the grinding speed, or how the milk is frothed.

The machine **abstracts** all the complex internal mechanics and gives you a simple interface. In Python, abstraction works the same way: it hides the complex logic behind a simple interface so other developers can use your code easily and strictly defines what features must be present.

# 3. Concept Explanation

Abstraction is an OOP principle that hides the internal implementation details of a class and only exposes the necessary features to the outside world.

In Python, we implement abstraction using the `abc` (Abstract Base Classes) module. An abstract class cannot be instantiated directly—it serves as a blueprint for other classes. It defines methods that the child classes **must** implement.

Why use it?
- It forces a standard structure across multiple classes.
- It hides complex details behind a clean interface.
- It prevents developers from using incomplete parent classes.

# 5. Beginner Example

Let's look at a basic example using animal sounds. We create an abstract blueprint that every animal must follow.

```python
from abc import ABC, abstractmethod

class Animal(ABC):
    @abstractmethod
    def make_sound(self):
        pass

class Dog(Animal):
    def make_sound(self):
        return "Woof!"

class Cat(Animal):
    def make_sound(self):
        return "Meow!"

# This would cause an error:
# animal = Animal()  # TypeError: Can't instantiate abstract class

dog = Dog()
print(dog.make_sound())  # Output: Woof!
```

# 6. Real-World Engineering Example

Bro, imagine you're building an e-commerce platform. You support multiple payment gateways like Stripe and Razorpay. You want to make sure every new payment gateway implements a standard `pay` method.

```python
from abc import ABC, abstractmethod

class PaymentProvider(ABC):
    @abstractmethod
    def authenticate(self):
        pass

    @abstractmethod
    def pay(self, amount: float):
        pass

class StripePayment(PaymentProvider):
    def authenticate(self):
        print("Authenticating with Stripe API...")

    def pay(self, amount: float):
        print(f"Processing ${amount} via Stripe.")

class RazorpayPayment(PaymentProvider):
    def authenticate(self):
        print("Authenticating with Razorpay API...")

    def pay(self, amount: float):
        print(f"Processing ${amount} via Razorpay.")

# If you forget to implement authenticate() in a new child class, Python throws an error immediately upon instantiation!
processor = StripePayment()
processor.authenticate()
processor.pay(150.0)
```

Under the hood, Python's `abc` module uses a metaclass (`ABCMeta`). When you try to instantiate a class, the metaclass checks if all methods decorated with `@abstractmethod` have been overridden. If even one is missing, it raises a `TypeError` preventing the object creation.

| Operation | Performance Impact |
|---|---|
| ABC Inheritance | Negligible overhead |
| Instantiation Check | Checked at runtime (O(1) dictionary lookup) |
| Method Calling | Same as standard method calls |

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| Can we instantiate an abstract class? | No, attempting to do so raises a `TypeError`. |
| What happens if a child class doesn't implement an abstract method? | The child class also becomes abstract and cannot be instantiated. |
| Does Python support interfaces? | Not directly, but abstract classes with only abstract methods act exactly like interfaces. |
| Can an abstract class have concrete methods? | Yes, you can mix abstract and normal (concrete) methods in an ABC. |
| Why use `@abstractmethod` over `raise NotImplementedError`? | `@abstractmethod` fails at instantiation. `NotImplementedError` fails later at runtime when the method is called. |

# 10. Tricky Questions & Edge Cases

Bro, watch out for this trap in interviews: using `NotImplementedError` instead of the `abc` module.

```python
class BaseDatabase:
    def connect(self):
        raise NotImplementedError("Subclasses must implement this!")

class MySQLDB(BaseDatabase):
    pass

# Interviewer asks: Will this fail?
db = MySQLDB()
# Answer: It succeeds! It only fails when db.connect() is called.
```

If we used `abc` instead, `db = MySQLDB()` would fail immediately. Failing fast is much safer in production.

# 11. Real-World Usage

Tech companies use abstraction heavily in:
- **Cloud Infrastructure Plugins:** Creating standard interfaces for AWS, GCP, and Azure services.
- **Database Adapters:** Django and SQLAlchemy use abstract classes to define what a database backend must support.
- **Logging Systems:** Defining standard formats for writing logs to files, databases, or third-party services like Datadog.

# 12. Best Practices

| DO | DON'T |
|---|---|
| Use `abc` module to enforce strict contracts | Don't rely on `raise NotImplementedError` alone for large systems |
| Keep abstract base classes focused (Single Responsibility) | Don't pack too many unrelated methods into one abstract class |
| Provide concrete helper methods if they are universally shared | Don't force child classes to write boilerplate logic that is always identical |

# 13. Production Notes

> ⚠ Production Note:
Bro, in large systems, abstraction prevents partial implementations from going to production. By failing at object creation (instantiation) rather than during execution, you avoid hidden bugs where a missing method crashes a script halfway through a critical data pipeline.

# 14. Common Mistakes

- **Forgetting to import `abstractmethod`**: Just inheriting from `ABC` does nothing to enforce rules if you don't use the decorator.
- **Instantiating the Abstract Class**: Trying to create `x = PaymentProvider()` will crash.
- **Partial Implementation**: If an interface requires 5 methods and you only implement 4, your class remains abstract and will crash on instantiation.
