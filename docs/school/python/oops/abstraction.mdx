# 2. Real-World Analogy

Think of Abstraction like driving a car, bro.

When you press the accelerator pedal, the car speeds up. You just interact with a simple interface: the pedal. You don't need to know how the fuel injector works, how the spark plugs fire, or how the transmission shifts gears.

The complex engine mechanics are **abstracted** away from you. You only care about the essential action (accelerating), not the internal implementation details.

---

# 3. Concept Explanation

Abstraction in Python is about hiding the complex implementation details of a class and exposing only the necessary parts. It acts as a blueprint or a contract.

You use **Abstract Base Classes (ABCs)** to define methods that *must* be implemented by child classes, but you don't implement them in the base class itself. It ensures a consistent interface across different implementations.

We use the `abc` module to create abstract classes. You cannot create an object (instantiate) directly from an abstract class.

# 5. Beginner Example

Let's build a simple abstract shape system.

```python
from abc import ABC, abstractmethod

# 1. Create the Abstract Base Class
class Shape(ABC):
    @abstractmethod
    def area(self):
        pass

# 2. Create a concrete class that implements the abstract method
class Square(Shape):
    def __init__(self, side):
        self.side = side

    def area(self):
        return self.side * self.side

# shape = Shape()  # ERROR! Cannot instantiate abstract class
my_square = Square(5)
print(f"Area: {my_square.area()}")  # Output: Area: 25
```

# 6. Real-World Engineering Example

Bro, in a real production system, abstraction is heavily used for things like payment gateways. You write your checkout code to talk to an abstract `PaymentProvider`, so switching between Stripe and PayPal is easy.

```python
from abc import ABC, abstractmethod

class PaymentGateway(ABC):
    @abstractmethod
    def process_payment(self, amount: float, user_id: str) -> bool:
        pass

    @abstractmethod
    def refund(self, transaction_id: str) -> bool:
        pass

class StripeGateway(PaymentGateway):
    def process_payment(self, amount: float, user_id: str) -> bool:
        print(f"Calling Stripe API for ${amount} to user {user_id}")
        return True

    def refund(self, transaction_id: str) -> bool:
        print(f"Stripe refunding {transaction_id}")
        return True

class PayPalGateway(PaymentGateway):
    def process_payment(self, amount: float, user_id: str) -> bool:
        print(f"Calling PayPal API for ${amount} to user {user_id}")
        return True

    def refund(self, transaction_id: str) -> bool:
        print(f"PayPal refunding {transaction_id}")
        return True

# Our checkout logic doesn't care which gateway is used!
def checkout(gateway: PaymentGateway, amount: float):
    gateway.process_payment(amount, "user_123")

# Usage
stripe = StripeGateway()
checkout(stripe, 50.0)
```

# 7. Internal Working

When you inherit from `ABC` and use `@abstractmethod`, Python's `ABCMeta` metaclass kicks in during class creation.

Before allowing you to instantiate an object, Python checks an internal set called `__abstractmethods__`. If this set is not empty (meaning some abstract methods were not overridden by the child class), Python raises a `TypeError`.

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between Encapsulation and Abstraction? | Encapsulation hides **data** (internal state). Abstraction hides **complexity/implementation details** (how things work). |
| Can you instantiate an abstract class? | No. Trying to do so raises a `TypeError`. |
| Do abstract classes have to be completely empty? | No, abstract classes can contain concrete (implemented) methods alongside abstract methods. |
| What happens if a child class doesn't implement all abstract methods? | It also becomes an abstract class and cannot be instantiated. |

# 10. Tricky Questions & Edge Cases

Look at this tricky setup, bro. What happens here?

```python
from abc import ABC, abstractmethod

class Animal(ABC):
    @abstractmethod
    def make_sound(self):
        print("Generic animal noise")

class Dog(Animal):
    def make_sound(self):
        super().make_sound()
        print("Woof!")

d = Dog()
d.make_sound()
```

**Output:**
```text
Generic animal noise
Woof!
```
**Explanation:** Abstract methods *can* have an implementation! You use `super()` in the child class to call the base logic. The `@abstractmethod` decorator just forces the child class to override the method, but it doesn't mean the parent's method has to be empty `pass`.

# 12. Best Practices

| DO | DON'T |
|---|---|
| Use `abc` module for explicit contracts | Rely on `raise NotImplementedError` in base classes (it only fails at runtime, not instantiation) |
| Keep abstract base classes small (Interface Segregation) | Make massive "God interfaces" that force children to implement useless methods |

# 13. Production Notes

> ⚠ **Production Note:**
Bro, explicitly using `ABC` prevents partial instantiations early. If a junior developer adds a new abstract method to `DatabaseAdapter` but forgets to update `PostgresAdapter`, Python will catch it immediately when the service starts up and tries to initialize Postgres, rather than waiting for the method to be called in production and throwing an error.

---

# 14. Common Mistakes

A major mistake is using `NotImplementedError` instead of `@abstractmethod`.

**Bad Practice:**
```python
class Logger:
    def log(self, msg):
        raise NotImplementedError("Subclasses must implement log!")

# You can instantiate Logger perfectly fine, the error only happens when calling log()
bad_logger = Logger()
```

**Good Practice:**
```python
from abc import ABC, abstractmethod

class Logger(ABC):
    @abstractmethod
    def log(self, msg):
        pass

# Fails immediately at instantiation, much safer!
good_logger = Logger() # TypeError!
```
