## 2. Real-World Analogy

Bro, think about driving a car. You know how to use the steering wheel, the accelerator, and the brakes. You don't need to know how the fuel injection system works, or how the spark plugs ignite the fuel, or how the transmission gears shift.

The car's dashboard and pedals are an **interface** that hides the complex internal **implementation details**. That's abstraction! You expose only what's necessary to use the system and hide the messy stuff underneath.

## 3. Concept Explanation

Abstraction in Python is about creating a blueprint for other classes. It allows you to define methods that *must* be created within any child classes built from your abstract class.

It serves two main purposes:
1. **Hiding complexity:** Users of the class only see the high-level methods they need.
2. **Enforcing structure:** It guarantees that subclasses implement certain methods, establishing a common interface.

Python doesn't have abstract classes built into its core syntax like Java or C#. Instead, we use the `abc` (Abstract Base Classes) module. A class that contains one or more abstract methods (decorated with `@abstractmethod`) and inherits from `ABC` is considered abstract.

## 4. Syntax Table

| Operation | Example |
|---|---|
| Import Module | `from abc import ABC, abstractmethod` |
| Define Abstract Class | `class MyAbstractClass(ABC):` |
| Define Abstract Method | `@abstractmethod`<br/>`def my_method(self): pass` |

## 5. Beginner Example

Let's look at a simple example involving shapes.

```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self):
        pass

class Square(Shape):
    def __init__(self, side):
        self.side = side

    def area(self):
        return self.side * self.side

# Trying to instantiate Shape directly will fail:
# s = Shape() # TypeError: Can't instantiate abstract class Shape

sq = Square(5)
print(f"Area of square: {sq.area()}")
```

Here, `Shape` is abstract. It forces any shape we create (like `Square`) to have an `area()` method.

## 6. Real-World Engineering Example

Bro, in a real system, you often have multiple ways to do the same thing, like storing data or sending messages. Abstraction ensures all these different ways behave identically from the outside.

Imagine a payment processing system.

```python
from abc import ABC, abstractmethod

# The Interface
class PaymentProcessor(ABC):
    @abstractmethod
    def process_payment(self, amount: float) -> bool:
        """Process a payment of the given amount."""
        pass

    @abstractmethod
    def refund_payment(self, transaction_id: str) -> bool:
        """Refund a specific transaction."""
        pass

# Implementation 1
class StripeProcessor(PaymentProcessor):
    def process_payment(self, amount: float) -> bool:
        print(f"Connecting to Stripe API... charging ${amount}")
        # Real Stripe logic goes here
        return True

    def refund_payment(self, transaction_id: str) -> bool:
        print(f"Refunding {transaction_id} via Stripe")
        return True

# Implementation 2
class PayPalProcessor(PaymentProcessor):
    def process_payment(self, amount: float) -> bool:
        print(f"Redirecting to PayPal... charging ${amount}")
        # Real PayPal logic goes here
        return True

    def refund_payment(self, transaction_id: str) -> bool:
        print(f"Refunding {transaction_id} via PayPal")
        return True

# The core system doesn't care WHICH processor it is!
def checkout(processor: PaymentProcessor, amount: float):
    print("Starting checkout...")
    success = processor.process_payment(amount)
    if success:
        print("Checkout complete!")

# Usage
stripe = StripeProcessor()
checkout(stripe, 100.0)
```

The `checkout` function just needs something that acts like a `PaymentProcessor`. It doesn't care if it's Stripe, PayPal, or Crypto.

## 7. Internal Working

Under the hood, Python's `abc` module uses a metaclass called `ABCMeta`.

When you try to instantiate a class, Python checks if its metaclass is `ABCMeta`. If it is, Python looks at the `__abstractmethods__` attribute of the class. This attribute contains a set of method names that were decorated with `@abstractmethod`.

If `__abstractmethods__` is not empty, Python throws a `TypeError`. When a subclass overrides an abstract method, it removes that method name from its own `__abstractmethods__` set.

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| How do you create an abstract class in Python? | By importing `ABC` and `abstractmethod` from the `abc` module. The class must inherit from `ABC` and have at least one `@abstractmethod`. |
| Can you instantiate an abstract class? | No. Trying to do so raises a `TypeError`. |
| What happens if a subclass doesn't implement an abstract method? | The subclass itself becomes abstract and cannot be instantiated. It will raise a `TypeError` upon instantiation. |
| Can an abstract class have concrete (normal) methods? | Yes, abstract classes can have fully implemented methods that subclasses can inherit or override. |
| Is it possible to add properties as abstract? | Yes, using `@property` along with `@abstractmethod` (e.g., `@property` above `@abstractmethod`). |

## 10. Tricky Questions & Edge Cases

Bro, watch out for this one. What if an abstract class has an `__init__` method?

```python
from abc import ABC, abstractmethod

class Database(ABC):
    def __init__(self, connection_string):
        self.connection_string = connection_string

    @abstractmethod
    def connect(self):
        pass

class PostgresDB(Database):
    def connect(self):
        print(f"Connecting to {self.connection_string}")

db = PostgresDB("postgresql://localhost:5432")
db.connect()
```

Yes, abstract classes *can* have constructors (`__init__`). Subclasses should call `super().__init__(...)` to initialize those base attributes. The abstract class itself still can't be instantiated, but its constructor logic is valid and useful for subclasses.

## 11. Real-World Usage

- **Cloud Providers:** Libraries like `libcloud` or `boto3` use abstractions to provide a common interface over different cloud providers (AWS, GCP, Azure).
- **ORMs (Object-Relational Mappers):** SQLAlchemy uses abstraction to define database dialects (Postgres, MySQL, SQLite) so the core ORM logic remains consistent.
- **Plugins/Extensions:** Applications that support plugins define abstract "Plugin" classes that third-party developers must implement.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Use abstraction to define strict interfaces for teams. | Use abstraction for simple scripts where duck typing is enough. |
| Inherit from `ABC` to make it clear the class is abstract. | Just raise `NotImplementedError` in base classes without using `ABC`. |
| Keep abstract base classes focused on a single responsibility. | Create massive "God" abstract classes with dozens of methods. |

## 13. Production Notes

> ⚠ Production Note:
While Python's duck typing ("if it walks like a duck and quacks like a duck...") is often enough for small projects, large enterprise codebases heavily rely on the `abc` module. It catches missing implementations at module import/instantiation time rather than blowing up randomly at runtime when a missing method is finally called. This fail-fast behavior is crucial for stable production systems.

## 14. Common Mistakes

- **Forgetting `ABC`:** Just decorating a method with `@abstractmethod` without inheriting from `ABC` doesn't prevent instantiation in older Python versions, and is generally incorrect usage.
- **Forgetting to implement:** A common error is a subclass forgetting to implement *one* of the many abstract methods from a base class, leading to a confusing `TypeError` when you try to create an object of the subclass.
- **Using `NotImplementedError` alone:** Before `abc` was introduced, people used to write `def my_method(self): raise NotImplementedError()`. This works, but it only fails when the method is *called*, not when the class is *instantiated*. `abc` is safer.
