# Abstraction

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Abstraction |
| Difficulty | Intermediate |
| Used For | Hiding complex implementation details and showing only necessary features |
| Main Strength | Reduces code complexity and forces a contract on child classes |
| Common Mistake | Confusing abstraction with encapsulation |
| Performance | Minimal overhead, highly beneficial for design |

Bro, abstraction is all about hiding the messy internal details and only exposing what the user actually needs to interact with.

It lets you focus on *what* an object does instead of *how* it does it.

## 2. Real-World Analogy

Bro, think of driving a car.

You have a steering wheel, an accelerator, and brakes. You press the accelerator to go faster. You don't need to know how the fuel pump injects gas into the engine, how the spark plugs fire, or how the transmission shifts gears.

| Car Feature (Visible) | Internal Mechanism (Hidden) |
|---|---|
| Accelerator pedal | Fuel injection, engine combustion |
| Steering wheel | Rack and pinion, power steering fluid |
| Brake pedal | Brake lines, calipers, brake pads |

The car exposes a simple interface (pedals and wheel) and abstracts away the complex engine mechanics.

## 3. Concept Explanation

In Python, abstraction is usually achieved using abstract base classes (ABCs) and abstract methods.

An abstract class is a blueprint for other classes. It defines a common interface for its subclasses. You cannot create an object directly from an abstract class.

Instead, you create subclasses that inherit from the abstract class, and these subclasses **must** provide concrete implementations for all the abstract methods.

Why use abstraction?
- It forces a standard contract across multiple classes.
- It hides background details, making large systems easier to work with.

Use abstraction when:
- You have multiple classes that share the same behavior signature but implement it differently.
- You are designing plugins, APIs, or large enterprise systems.

## 4. Syntax Table

| Operation | Python Example |
|---|---|
| Import ABC | `from abc import ABC, abstractmethod` |
| Define Abstract Class | `class Vehicle(ABC):` |
| Define Abstract Method | `@abstractmethod` <br/> `def start(self): pass` |
| Inherit and Implement | `class Car(Vehicle):` <br/> `def start(self): return "Vroom"` |

Bro, remember you have to import `ABC` and `abstractmethod` from the `abc` module. Python doesn't enforce abstraction strictly by default like Java does.

## 5. Beginner Example

```python
from abc import ABC, abstractmethod

# Abstract Class
class Shape(ABC):
    @abstractmethod
    def area(self):
        pass

# Concrete Class
class Square(Shape):
    def __init__(self, side):
        self.side = side

    # Must implement the abstract method
    def area(self):
        return self.side * self.side

# shape = Shape() # Bro, this will throw an error! You can't instantiate ABC.
sq = Square(4)
print(f"Area of square: {sq.area()}")
```

Output:
```text
Area of square: 16
```

Here, `Shape` says: "Bro, if you want to be a shape, you MUST have an `area` method." `Square` follows the rule.

## 6. Real-World Engineering Example

Imagine you are building a payment gateway for an e-commerce platform. You want to support Stripe, PayPal, and Razorpay.

You don't want your main checkout code to worry about the specific API calls for each provider. You just want to call `.process_payment()`.

```python
from abc import ABC, abstractmethod

class PaymentProcessor(ABC):
    @abstractmethod
    def process_payment(self, amount: float) -> bool:
        pass

class StripeProcessor(PaymentProcessor):
    def process_payment(self, amount: float) -> bool:
        print(f"Connecting to Stripe API... Charging ${amount}")
        return True

class PayPalProcessor(PaymentProcessor):
    def process_payment(self, amount: float) -> bool:
        print(f"Redirecting to PayPal... Charging ${amount}")
        return True

# Main Business Logic
def checkout(processor: PaymentProcessor, amount: float):
    # The checkout logic is abstracted! It doesn't care WHICH processor it is.
    success = processor.process_payment(amount)
    if success:
        print("Order confirmed!")

stripe = StripeProcessor()
checkout(stripe, 100.0)
```

Output:
```text
Connecting to Stripe API... Charging $100.0
Order confirmed!
```

Bro, if you ever need to add Google Pay later, you just create a new class. You don't have to rewrite the `checkout` function!

## 7. Internal Working

When you create a class inheriting from `ABC`, Python's metaclass machinery checks if all methods marked with `@abstractmethod` have been overridden when you call `__new__` to create an object.

If they aren't, Python raises a `TypeError` immediately, preventing buggy, incomplete objects from entering your system.

## 8. Performance Table

| Area | Python Behavior | Practical Impact |
|---|---|---|
| Method Lookup Time | Negligible overhead | Resolving the overridden method takes standard Python attribute lookup time. |
| Object Instantiation | Slightly slower | Python checks for abstract methods during instantiation, but it's completely unnoticeable in web/app contexts. |
| Code Maintenance | Very Fast | Saves hours of debugging because errors are caught when objects are created, not deeply inside running logic. |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is abstraction in Python? | Hiding complex implementation details and exposing only the essential features, usually via Abstract Base Classes (ABCs). |
| How is abstraction different from encapsulation? | Abstraction is hiding *complexity* (design level). Encapsulation is hiding *state/data* (implementation level, using private variables). |
| Can you instantiate an abstract class? | No, Python will raise a `TypeError`. |
| What module is used for abstraction? | The `abc` module, specifically `ABC` and `@abstractmethod`. |
| What happens if a subclass doesn't implement an abstract method? | You won't get an error when defining the class, but you will get a `TypeError` when you try to instantiate an object of that subclass. |

## 10. Tricky Questions & Edge Cases

### The Instantiation Trap

```python
from abc import ABC, abstractmethod

class Base(ABC):
    @abstractmethod
    def do_work(self):
        pass

class Child(Base):
    def other_method(self):
        pass

# When does this fail?
# A) When parsing the file
# B) When defining Child
# C) When creating Child()
```

Bro, it fails at **C**. Python lets you define `Child` just fine. It only crashes when you do `c = Child()`.

### Abstract Properties

You can abstract properties too, not just methods!

```python
from abc import ABC, abstractproperty # or @property + @abstractmethod in modern Python

class Animal(ABC):
    @property
    @abstractmethod
    def legs(self):
        pass

class Dog(Animal):
    @property
    def legs(self):
        return 4
```

## 11. Real-World Usage

Abstraction is heavily used in production:
- **Cloud Infrastructure SDKs:** `BaseStorage` abstract class, with `S3Storage` and `AzureBlobStorage` subclasses.
- **Database ORMs:** Abstract `DatabaseDialect` class, with `PostgresDialect` and `MySQLDialect` implementations.
- **Machine Learning:** `BaseModel` abstract class in libraries like Scikit-Learn or PyTorch, forcing standard `.fit()` and `.predict()` methods.
- **Notifications:** `NotificationService` abstraction, with `EmailNotification`, `SMSNotification`, and `PushNotification`.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Use `ABC` when creating systems that expect multiple plugins or drivers. | Create abstract classes for a single simple script where inheritance isn't needed. |
| Make abstract method signatures clear (use type hints). | Put complex business logic inside the abstract base class itself. |
| Enforce standard naming conventions across implementations. | Rely on `NotImplementedError` instead of `ABC` if you want strict enforcement. |

## 13. Production Notes

> ⚠ Production Note:
> While you can achieve "soft abstraction" by just raising `NotImplementedError` in a base class method, using the `abc` module is much safer for production.

Using `NotImplementedError` fails *when the method is called*. Using `ABC` fails *when the object is created*. Bro, failing earlier during object creation is always better because it prevents invalid objects from moving through your application pipeline.

## 14. Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| Forgetting to inherit from `ABC` | The `@abstractmethod` decorator does nothing to stop instantiation if the class doesn't inherit from `ABC` or use the `ABCMeta` metaclass. |
| Confusing abstraction with encapsulation | You sound like a junior engineer in interviews if you mix them up. Abstraction = hiding complexity. Encapsulation = hiding data. |
| Creating giant abstract classes | If your abstract class has 20 methods, subclasses are burdened. Keep interfaces small and focused. |
