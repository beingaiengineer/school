# Dataclasses

# 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Dataclasses |
| Difficulty | Beginner / Intermediate |
| Used For | Generating boilerplate OOP code automatically (like `__init__`, `__repr__`, `__eq__`) |
| Common Mistake | Using mutable defaults like `[]` or `{}` directly instead of `field(default_factory=list)` |
| Performance | Very fast, almost identical to writing classes manually |

---

# 2. Real-World Analogy

Think of Dataclasses like buying a pre-furnished apartment, bro.

If you build a normal Python class, you have to write the constructor `__init__`, set up the string representation `__repr__`, and figure out how to compare them with `__eq__`. It's like moving into an empty apartment and having to build your own bed and buy your own stove.

With `@dataclass`, Python pre-furnishes the class for you. You just declare the variables you want, and Python automatically writes all the boring underlying methods instantly.

---

# 3. Concept Explanation

Introduced in Python 3.7, `dataclasses` provide a decorator and functions for automatically adding generated special methods such as `__init__()` and `__repr__()` to user-defined classes.

They are primarily used when you need a class that is just a container for data, without complex logic. They make your code incredibly clean and short.

---

# 4. Syntax Table

| Concept | Code | Description |
|---|---|---|
| Import | `from dataclasses import dataclass` | Standard library import. |
| Decorator | `@dataclass` | Put this above your class. |
| Type Hints | `name: str` | Variables *must* have type hints. |
| Default Val | `age: int = 18` | Easily set default values. |

# 5. Beginner Example

Let's look at the old way versus the new way to create a `User` profile.

**The Old Way (Boilerplate Hell):**
```python
class OldUser:
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    def __repr__(self):
        return f"OldUser(name='{self.name}', age={self.age})"

    def __eq__(self, other):
        if not isinstance(other, OldUser):
            return False
        return self.name == other.name and self.age == other.age
```

**The Dataclass Way (Clean & Elegant):**
```python
from dataclasses import dataclass

@dataclass
class User:
    name: str
    age: int

# It works exactly the same, but with 90% less code!
user1 = User("Alice", 25)
user2 = User("Alice", 25)

print(user1) # Output: User(name='Alice', age=25)
print(user1 == user2) # True (Automatically compares attributes!)
```
Bro, look how clean that is.

# 6. Real-World Engineering Example

In production, we often need data structures that hold configuration or API responses, and we need default values, especially default lists.

```python
from dataclasses import dataclass, field

@dataclass
class ServerConfig:
    host: str
    port: int = 8080

    # We CANNOT do `tags: list = []`. That shares the list across all instances!
    # We MUST use `default_factory` for mutable defaults.
    tags: list = field(default_factory=list)

    # We can hide sensitive data from the __repr__ print output!
    secret_key: str = field(repr=False, default="HIDDEN")

config = ServerConfig(host="192.168.1.1")
config.tags.append("production")

print(config)
# Output: ServerConfig(host='192.168.1.1', port=8080, tags=['production'])
# Notice `secret_key` is not printed!
```

# 7. Internal Working

How does the `@dataclass` decorator work?

It's literally a code generator! When Python reads the class definition at runtime, the `@dataclass` function inspects the class annotations (the `__annotations__` dict created by your type hints).

It then dynamically generates the text for the `__init__`, `__repr__`, etc., compiles it, and attaches it to the class object before passing it back. It happens instantly when the module is loaded.

# 8. Performance Table

| Mechanism | Speed/Memory | Note |
|---|---|---|
| Auto `__init__` | Fast | Identical to manually writing `__init__`. |
| Memory Size | Normal | Uses standard dictionary `__dict__` under the hood. |
| `slots=True` | Faster | In Python 3.10+, adding `@dataclass(slots=True)` makes objects much smaller in memory! |

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| What happens if you don't provide type hints in a dataclass? | The `@dataclass` generator will ignore the variable completely! Type hints are strictly required for it to work. |
| How do you make a dataclass immutable (read-only)? | Use `@dataclass(frozen=True)`. Any attempt to change an attribute will raise a `FrozenInstanceError`. |
| Why must you use `default_factory=list` instead of `[]`? | Because `[]` is evaluated once at class definition time. All instances would share the exact same list in memory! |

# 10. Tricky Questions & Edge Cases

What if you need to run some logic *immediately after* the auto-generated `__init__` runs?

```python
from dataclasses import dataclass

@dataclass
class Rectangle:
    width: float
    height: float

    # We want this to calculate automatically
    # area: float

    def __post_init__(self):
        # This hook runs automatically right after the generated __init__ finishes!
        self.area = self.width * self.height

rect = Rectangle(10, 5)
print(rect.area) # 50.0
```
The `__post_init__` hook is critical for validation or calculating dependent fields.

---

# 12. Best Practices

| DO | DON'T |
|---|---|
| Use dataclasses when a class is mostly data and has little logic. | Use dataclasses for massive complex controllers with heavy inheritance. |
| Use `@dataclass(frozen=True)` when using the object as a dictionary key or in a set. | Use mutable default arguments directly (e.g., `data: dict = {}`). |

---

# 13. Production Notes

> ⚠ Production Note:
If you are processing millions of objects (e.g., streaming IoT data), standard dataclasses use a lot of memory because of their internal `__dict__`. Always use `@dataclass(slots=True)` on Python 3.10+. This tells Python not to use dynamic dictionaries, reducing memory usage by over 40% and making attribute access slightly faster!

---

# 14. Common Mistakes

- **Ordering attributes poorly:** If you put a field with a default value *before* a field without a default value, Python will crash with a `TypeError`.
  ```python
  @dataclass
  class Bad:
      age: int = 18   # Default
      name: str       # NO default! ERROR!
  ```
- **Ignoring type hints:** `name = "Alice"` (without `: str`) will completely bypass the dataclass engine.
