# 2. Real-World Analogy

Bro, imagine building an airplane in your garage.

It starts up, the propeller spins, and the lights turn on. "It works on my machine!" you say, and you invite 100 passengers to fly across the ocean.

Without a pre-flight checklist, you don't know if you have enough fuel, if the cabin is pressurized, or if the backup radio works. A Production Checklist is the pre-flight inspection for your code. It ensures that when you launch, your app survives the real world (traffic spikes, network failures, and hackers).

---

# 3. Concept Explanation

A Production Checklist is a standardized set of rules and verifications a team must pass before deploying code to a live environment.

In Python, this involves checking deployment configurations (like WSGI/ASGI servers), logging setups, security measures, dependency locking, and database migrations.

Use it when:
- You are deploying a system that users actually care about.

Don't use it when:
- Prototyping a quick idea on `localhost` to show a friend.

| Concept | Local Development | Production Standard |
|---|---|---|

| Dependencies| `pip install foo` | `requirements.txt` with hashes / `poetry.lock` |

# 5. Beginner Example

Let's look at the classic web server mistake.

**Local (Do not do this in prod):**
```python
# main.py
from flask import Flask
app = Flask(__name__)

@app.route("/")
def hello():
    return "Hello World"

if __name__ == "__main__":
    # The built-in Werkzeug server is strictly for development.
    # It cannot handle concurrent production traffic securely.
    app.run(host="0.0.0.0", port=80, debug=True)
```

**Production (WSGI/ASGI Server):**
You don't run the script directly. You use a production-grade server like Gunicorn to manage multiple worker processes.

```bash
# Run this in your Dockerfile or systemd service
gunicorn -w 4 -b 0.0.0.0:80 main:app
```

Bro, leaving `debug=True` in production is a critical security vulnerability. It can expose interactive Python consoles on the web.

# 6. Real-World Engineering Example

Here is a condensed, realistic **Python Production Checklist**:

1. **Security & Config**
   - [ ] `DEBUG` mode is completely disabled.
   - [ ] All secrets (API keys, DB passwords) are loaded via environment variables, not hardcoded.
   - [ ] CORS (Cross-Origin Resource Sharing) is restricted to your frontend domains.
2. **Server & Concurrency**
   - [ ] Application is running behind a production server (Gunicorn for sync, Uvicorn/Hypercorn for async).
   - [ ] Number of workers is tuned to the CPU count (usually `2 * CPU + 1`).
   - [ ] A reverse proxy (Nginx, Traefik, ALB) is handling HTTPS and static files.
3. **Database**
   - [ ] Using a production database (Postgres), not SQLite.
   - [ ] Connection pooling is enabled (e.g., PgBouncer or SQLAlchemy pooling) to prevent connection limits.
   - [ ] Migrations (Alembic/Django) are tested and ready.
4. **Observability**
   - [ ] Logs are structured (JSON) and streaming to a central aggregator (Datadog, CloudWatch).
   - [ ] Error tracking (Sentry, Rollbar) is installed to capture unhandled exceptions.

# 7. Internal Working

Why do we need Gunicorn instead of `app.run()`?

Python's built-in development servers are usually single-threaded and block on every request. If User A uploads a large file (taking 5 seconds), User B cannot even load the homepage until User A finishes.

Gunicorn is a **Pre-fork Worker Model**. It spins up a Master process, which forks multiple Worker processes. The Master just watches the workers. If a worker gets stuck or crashes handling User A, the Master kills it and spawns a new one, while the other workers continue serving User B seamlessly.

# 9. Top Interview Questions

| Question | Answer |
|---|---|
| What happens if you leave `DEBUG=True` in production? | It leaks stack traces, source code, and environment variables to users. In Flask/Django, it can even allow Remote Code Execution via the debugger UI. |
| What is the difference between a Web Server and an App Server? | Nginx (Web Server) handles HTTP connections, SSL, and static files. Gunicorn (App Server) translates HTTP into Python dictionary environments (WSGI) so your code can run. |
| Why is SQLite bad for web production? | SQLite locks the entire database file during a write operation, causing high-concurrency web apps to queue up and timeout. |

# 10. Tricky Questions & Edge Cases

Look at this logging trap:

```python
import logging

def process_data(data):
    try:
        # Business logic here
        1 / 0
    except Exception as e:
        # Trap: You swallowed the exception!
        logging.error("Something went wrong.")
        return False
```

**The Trap:**
In production, you check your logs and see 10,000 lines saying "Something went wrong." But you have *no idea* what the error was or where it happened because you didn't log the stack trace.

**The Fix:**
Use `logging.exception()` inside except blocks. It automatically attaches the full stack trace to the log.

```python
    except Exception as e:
        logging.exception("Data processing failed")
```

# 12. Best Practices

| DO | DON'T |
|---|---|
| Pin all dependency versions precisely (e.g., `Flask==3.0.0`). | Use `pip install Flask` which grabs the latest version and might break your app tomorrow. |
| Setup health check endpoints (e.g., `/health`) that check the DB connection. | Rely on pinging the root URL to see if the app is alive. |
| Enable automated database backups before launch. | Wait for the first data loss incident to learn about pg_dump. |

# 13. Production Notes

> ⚠ Production Note:
> **Graceful Shutdowns are mandatory.** When you deploy a new version of your app, the old server shuts down. If it dies instantly, active user requests are dropped (HTTP 502). You must configure your server (Gunicorn/Docker/K8s) to intercept the `SIGTERM` signal, stop accepting *new* requests, finish the *current* requests, and then exit cleanly.

---

# 14. Common Mistakes

- **Timezone Ignorance:** Storing local server time in the database instead of UTC. When you scale your servers globally or Daylight Saving Time hits, your data becomes a tangled mess. Always use UTC.
- **Missing Timeouts:** Calling an external API (like a payment gateway) without a `timeout=10` parameter in the `requests` library. If the gateway hangs forever, your Python workers will hang forever, taking down your entire app.
- **Ignoring Static Files:** Serving images/CSS through Django or Flask in production. Python is slow at serving files. Nginx, Apache, or a CDN (Cloudflare) should serve static assets.
