# Hierarchical Agent Teams

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Hierarchical Agent Teams |
| Difficulty | Advanced |
| Used For | Massive, enterprise-level agentic applications requiring distinct departments. |
| Common Mistake | Putting 50 agents under a single supervisor, causing context collapse. |
| Performance | Evaluated based on token usage and latency. |

## 2. Engineering Story

Bro, imagine a company with 500 engineers. You don't have 500 engineers reporting directly to the CEO. The CEO talks to VPs, VPs talk to Directors, and Directors manage the engineers. In agentic frameworks, when you have dozens of specialized agents, a single Supervisor will fail because its context window gets overwhelmed. You need Hierarchical Teams.

## 4. Concept Explanation

Hierarchical Agent Teams involve nesting graphs within graphs (subgraphs). A top-level supervisor routes a complex query to a mid-level supervisor, which then routes it to specific workers. This isolates context windows—the top supervisor only sees the summarized output of the sub-teams, not the gritty details of every tool call.

## 5. Syntax Table

| Concept | Implementation Pattern |
|---|---|
| Top Graph | Routes to Sub-graphs as if they were nodes. |
| Sub-graph | A self-contained StateGraph with its own local state. |

## 6. Beginner Example

```typescript
// Conceptual Hierarchy
const devTeam = createSubGraph(["Coder", "Reviewer"]);

```

## 7. Real-World Engineering Example

```typescript
import { StateGraph } from "@langchain/langgraph";

// 1. Create the Research Sub-team
const researchTeam = new StateGraph({ /* local state */ });
researchTeam.addNode("WebSearch", searchAgent);
researchTeam.addNode("Summarizer", summarizeAgent);
// ... compile research sub-graph

// 2. Create the Writing Sub-team
const writingTeam = new StateGraph({ /* local state */ });
writingTeam.addNode("Draft", draftAgent);
writingTeam.addNode("Edit", editAgent);
// ... compile writing sub-graph

// 3. Top-Level Supervisor
const enterpriseGraph = new StateGraph({ /* global state */ });
enterpriseGraph.addNode("ResearchDept", researchTeam.compile());
enterpriseGraph.addNode("WritingDept", writingTeam.compile());
enterpriseGraph.addEdge("ResearchDept", "WritingDept");

export const system = enterpriseGraph.compile();
```

## 9. Performance Table

| Benefit | Trade-off |
|---|---|
| Context Isolation | Harder to debug state across boundaries |
| High Scalability | Increased latency due to nested LLM routing |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why use hierarchical teams? | To prevent context window exhaustion and maintain focus in specialized agent groups. |
| What is context isolation? | Ensuring the top supervisor doesn't see the raw tool outputs of bottom-tier workers. |

## 11. Tricky Questions & Edge Cases

Bro, passing state between nested graphs is a nightmare if you don't define clean schemas. The sub-graph must accept the global state, do its thing, and return a clean patch to the global state.

## 12. Real-World Usage

Large-scale AI software engineering systems (like SWE-agent or enterprise internal tools).

## 13. Best Practices

| Do | Don't |
|---|---|
| Summarize sub-graph outputs | Leak 100 lines of logs to the top supervisor |

## 14. Production Notes

<Admonition type="warning" title="Production Scaling Warning">
  When implementing Hierarchical Agent Teams in production, always monitor system boundaries—whether it's token limits, database connections, or API rate limits.
</Admonition>
