# Supervisor Pattern

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Supervisor Pattern |
| Difficulty | Intermediate |
| Used For | Dynamic routing of conversation or tasks based on the current state. |
| Common Mistake | Allowing the supervisor to do the actual work instead of just picking the next node. |
| Performance | Evaluated based on token usage and latency. |

## 2. Engineering Story

Bro, imagine a call center router. You call in, and it asks, 'Press 1 for Sales, 2 for Support.' The router doesn't solve your issue; it just looks at your intent and transfers you. In an agent graph, a Supervisor is an LLM whose only job is to look at the conversation history and output a single word: the name of the next agent to take over.

## 4. Concept Explanation

The Supervisor Pattern is a specific implementation of a routing node in a State Graph (like LangGraph). The Supervisor is given a list of available agents and their descriptions. At each step, it evaluates the shared state and outputs a structured response indicating `NEXT_AGENT` or `FINISH`.

## 5. Syntax Table

| Component | Description |
|---|---|
| Workers | Standard agents that perform tasks and update state. |
| Supervisor | LLM forced to output a constrained enum of agent names. |

## 7. Real-World Engineering Example

```typescript
import { z } from "zod";
import { ChatOpenAI } from "@langchain/openai";

const members = ["Researcher", "Coder", "Reviewer"];
const systemPrompt = `You are a supervisor. Given the conversation, who should act next?
Choose from: ${members.join(", ")}. If the task is fully complete, choose FINISH.`;

const routingSchema = z.object({
  next: z.enum(["Researcher", "Coder", "Reviewer", "FINISH"]),
});

const supervisorAgent = new ChatOpenAI({ modelName: "gpt-4" }).withStructuredOutput(routingSchema);

// Usage inside a graph node
async function supervisorNode(state) {
  const result = await supervisorAgent.invoke([
    { role: "system", content: systemPrompt },
    ...state.messages
  ]);
  return { nextAgent: result.next };
}
```

### Supervisor Edge Cases

The Supervisor pattern excels at quality control, but it introduces unique failure modes that must be carefully managed in production environments.

A common anti-pattern is the "Infinite Revision Loop." If the Supervisor demands a level of quality that the Worker is fundamentally incapable of achieving (due to LLM limitations or missing tools), the two agents will endlessly pass the task back and forth, consuming massive amounts of tokens and time.

To prevent infinite loops, production Supervisor implementations require:

1. **Max Iteration Caps:** A hard limit on the number of revision cycles. If the limit is reached, the Supervisor must either accept the best-effort result, flag the task for human intervention, or intentionally fail the workflow.
2. **Progressive Degradation:** If the Worker fails to improve after two revisions, the Supervisor might dynamically simplify its prompt or lower its quality threshold to ensure the system can eventually produce an output.
3. **Clear Rubrics:** The Supervisor should not rely on vague criteria like "make it better." It must evaluate the Worker's output against a strict, objective rubric (e.g., "Does the code compile?", "Are there exactly 5 paragraphs?").

By establishing strict boundaries, you ensure the Supervisor pattern remains a productive quality gate rather than a resource drain.
