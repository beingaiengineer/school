## 4. Concept Explanation
In the OpenAI SDK ecosystem, "Tools" are simply standard Python functions. When you pass a list of functions to an Agent, the SDK automatically parses the function signatures (type hints) and docstrings, converts them into a JSON schema, and sends that schema to the OpenAI API.

When the model decides it needs data, it halts generation and says, "Call function X with arguments Y". The SDK executes your Python code and feeds the result back to the LLM.

## 5. Syntax Table
| Concept | Syntax | Description |
|---|---|---|
| Function Definition | `def my_tool(arg: str) -> str:` | Standard Python function with type hints. |
| Docstring | `"""Fetches the user data."""` | Crucial! This tells the LLM what the tool does. |
| Agent Setup | `Agent(functions=[my_tool])` | Attaching the tool to the agent. |

## 6. Beginner Example
Bro, let's create a tool that gets the weather.

```python
from swarm import Swarm, Agent

def get_weather(location: str) -> str:
    """Get the current weather in a given location."""
    # Hardcoded for example, usually an API call
    if location.lower() == "mumbai":
        return "It's 32°C and humid, bro."
    return "Weather unknown."

weather_agent = Agent(
    name="WeatherBot",
    instructions="You provide weather updates.",
    functions=[get_weather]
)

client = Swarm()
response = client.run(
    agent=weather_agent,
    messages=[{"role": "user", "content": "What's the weather in Mumbai?"}]
)

print(response.messages[-1]["content"])
```

## 7. Real-World Engineering Example
In production, your tools will interact with real databases or APIs. Let's look at a tool that updates a database record.

```python
from swarm import Swarm, Agent
import json

def update_user_email(user_id: int, new_email: str) -> str:
    """Updates the user's email address in the database.

    Args:
        user_id (int): The unique ID of the user.
        new_email (str): The new email address to set.
    """
    try:
        # FAKE DB CALL: db.execute("UPDATE users SET email=? WHERE id=?", (new_email, user_id))
        print(f"DEBUG: Executing DB update for user {user_id}")
        return json.dumps({"status": "success", "message": f"Email updated to {new_email}"})
    except Exception as e:
        return json.dumps({"status": "error", "message": str(e)})

support_agent = Agent(
    name="SupportBot",
    instructions="You are customer support. You can update user emails if requested.",
    functions=[update_user_email]
)

client = Swarm()
# User asks to update email
response = client.run(
    agent=support_agent,
    messages=[{"role": "user", "content": "Update the email for user ID 405 to new@example.com."}]
)
print(response.messages[-1]["content"])
```

## 9. Performance Table
| Operation | Latency Impact |
|---|---|
| Simple Python Tool | ~1ms (negligible) |
| API Call Tool | +100ms to 2000ms (depends on external API) |
| Tool-Calling Loop | +1 full LLM roundtrip (high latency impact) |

## 10. Top Interview Questions
| Question | Answer |
|---|---|
| How does the LLM know how to use your Python function? | The SDK converts the function's docstring and type hints into a JSON schema that the LLM understands natively. |
| What happens if a tool function throws an error? | If unhandled, it crashes the orchestrator. You should catch errors in the tool and return an error string. |
| Can a tool return complex objects? | Tools should ideally return strings or JSON strings, as that is what the LLM can read back into its context window. |
| Why are type hints mandatory? | Without them, the SDK cannot generate the correct JSON schema, and the LLM might hallucinate argument types. |
| Can an agent call multiple tools in parallel? | Yes, modern OpenAI models support parallel function calling automatically. |

## 11. Tricky Questions & Edge Cases
Bro, what happens if the LLM hallucinates an argument that isn't in your function signature? (e.g., passing `date` when your function only takes `location`).
**The Fix:** The SDK handles basic validation based on the schema, but you should always write defensive Python code. Use `**kwargs` if you want to silently catch and ignore hallucinated arguments, or let it fail gracefully and return "Invalid arguments provided" to the LLM so it can try again.

## 12. Real-World Usage
Production support agents use tools to check Jira tickets, query PostgreSQL databases for user states, and trigger PagerDuty alerts.

## 13. Best Practices
| DO | DON'T |
|---|---|
| Use explicit type hints (`arg: int`). | Use `arg: Any` or omit type hints. |
| Write descriptive docstrings explaining the tool's purpose. | Write tools with no documentation. |
| Return stringified JSON for complex data. | Return raw Python class objects. |

## 14. Production Notes
:::warning
Never give an agent a destructive tool (like `DROP TABLE` or `DELETE USER`) without a Human-in-the-Loop approval step. LLMs will hallucinate and confidently delete production data if given the chance.
:::

## 15. Common Mistakes
| Mistake | Fix |
|---|---|
| Forgetting to import dependencies in the tool | Ensure your tool function has access to the libraries it needs (e.g., `import requests`). |
| Returning massive datasets | Don't return 10,000 rows of a DB query. Summarize or paginate it, or it will blow up the context window. |
| Relying on global variables | Keep tool functions pure and self-contained if possible. |
