# OpenAI Agents SDK Production Deployment

## 1. Quick Summary
| Area | Details |
|---|---|
| Topic | Production Deployment for OpenAI SDK |
| Difficulty | Advanced |
| Used For | Running your OpenAI agents at scale in a reliable, secure environment. |
| Common Mistake | Running the synchronous SDK loop inside the main thread of a web server. |
| Performance | Requires asynchronous architecture to prevent HTTP timeouts. |

## 3. Concept Explanation
The OpenAI Agents SDK is synchronous by default, meaning `client.run()` blocks the thread until the agent finishes thinking and calling tools. In production, web servers like FastAPI or Express handle many users at once. If one user's agent takes 15 seconds to run, a synchronous server will freeze and ignore other users.

To deploy safely, you must wrap the agent execution in asynchronous task queues (like Celery) or use async endpoints, stream the results via Server-Sent Events (SSE) or WebSockets, and handle API rate limits gracefully.

## 5. Beginner Example
Here is the naive way people try to deploy it. **Don't do this for heavy agents**, but it shows the basic API wrapping.

```python
from fastapi import FastAPI
from pydantic import BaseModel
from swarm import Swarm, Agent

app = FastAPI()
client = Swarm()

my_agent = Agent(name="Support", instructions="You are helpful.")

class ChatRequest(BaseModel):
    message: str

@app.post("/chat")
async def chat_endpoint(request: ChatRequest):
    # DANGER: This blocks the event loop while waiting for OpenAI
    response = client.run(
        agent=my_agent,
        messages=[{"role": "user", "content": request.message}]
    )
    return {"reply": response.messages[-1]["content"]}
```

## 6. Real-World Engineering Example
Bro, the real way to do this is using the asynchronous OpenAI client and streaming the response back using Server-Sent Events (SSE) so the user sees the text appearing instantly.

```python
import asyncio
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from openai import AsyncOpenAI

app = FastAPI()
client = AsyncOpenAI() # Use the async client!

class ChatRequest(BaseModel):
    message: str

async def agent_stream(prompt: str):
    # This is a simplified async streaming loop
    stream = await client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        stream=True,
    )
    async for chunk in stream:
        if chunk.choices[0].delta.content is not None:
            yield chunk.choices[0].delta.content
            # Small sleep to yield control to event loop
            await asyncio.sleep(0.01)

@app.post("/chat/stream")
async def chat_stream_endpoint(request: ChatRequest):
    # Returns immediately, streams data as it arrives
    return StreamingResponse(agent_stream(request.message), media_type="text/event-stream")
```

## 7. Internal Working
In a proper production architecture, the HTTP request simply adds a job to a queue. A pool of worker servers picks up the jobs, runs the heavy OpenAI SDK loop, and pushes updates to a database or directly to the client via WebSockets.

## 8. Performance Table
| Bottleneck | Risk | Solution |
|---|---|---|
| OpenAI API Latency | High (2-10s) | Stream responses to UI immediately. |
| Concurrent Connections | Medium | Use Asyncio/FastAPI, avoid synchronous blocking. |
| Context Window Size | High (cost & speed) | Implement a sliding window to forget old messages. |

## 9. Top Interview Questions
| Question | Answer |
|---|---|
| Why use WebSockets or SSE for agents? | Because LLM agents are slow. HTTP requests will timeout, leaving the user staring at a loading spinner. Streaming provides instant UX feedback. |
| How do you handle OpenAI rate limits? | Implement retry logic with exponential backoff in your worker code using libraries like `tenacity`. |
| What happens to memory if the server restarts? | If you store `messages` in a Python list in RAM, it's gone. You must persist conversation history to a database like PostgreSQL or Redis. |
| How do you scale this? | Because agents are stateless between runs, you can horizontally scale your worker nodes (e.g., in Kubernetes) as long as state is in a shared database. |

## 10. Tricky Questions & Edge Cases
Bro, what if your agent's tool needs to run a database query that takes 30 seconds?
**The Fix:** The OpenAI API has a strict timeout. If your local tool takes too long, the OpenAI connection drops. Your tools must be fast, or you must decouple the long-running task (e.g., the agent says "I started the query, check back later") instead of waiting inline.

## 11. Real-World Usage
Top startups deploy their agents behind API gateways (like Kong or Cloudflare) to manage rate limiting and DDOS protection, before routing the request to a Kubernetes cluster running FastAPI and async Celery workers.

## 12. Best Practices
| DO | DON'T |
|---|---|
| Use `.env` and Secrets Managers for API keys. | Hardcode `sk-...` in your git repo. |
| Implement strict timeouts on your tool functions. | Let a tool hang forever, freezing the agent. |
| Persist conversation state to a database. | Store state in global Python variables. |

## 13. Production Notes
:::warning
Security is your biggest threat in production. Ensure your agent cannot be prompt-injected into calling a destructive tool. Apply principle of least privilege to the database credentials your tools use.
:::

## 14. Common Mistakes
| Mistake | Fix |
|---|---|
| Synchronous blocking | Use `AsyncOpenAI` or background task queues. |
| Infinite retry loops | Set a hard `max_retries` on tool failures so you don't burn tokens on a broken API. |
| Forgetting Observability | You must integrate LangSmith/Helicone before deploying, or you will be blind to errors. |
