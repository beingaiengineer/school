# Agent Guardrails in OpenAI SDK

## 1. Quick Summary
| Area | Details |
|---|---|
| Topic | Agent Guardrails |
| Difficulty | Advanced |
| Used For | Preventing the LLM from outputting harmful, off-topic, or structurally invalid data. |
| Common Mistake | Relying solely on system prompts ("Please don't swear") instead of code-level checks. |
| Performance | Adds latency due to validation steps or secondary LLM evaluation calls. |

## 2. Real-World Analogy
![Real-World Analogy](../assets/openai-agents-sdk/openai-sdk-guardrails.png)

Bro, think of a bowling alley. If you are a beginner, your ball is going to end up in the gutter. Guardrails are literally those metal bumpers they put over the gutters. They physically prevent the ball from going out of bounds, no matter how badly you throw it. In agent engineering, a system prompt tells the LLM to bowl straight; a code-level guardrail *forces* it to stay in bounds.

| Bowling Alley | Agent Framework |
|---|---|
| Telling the bowler to aim straight | System Prompt |
| The metal bumpers | Input/Output validation code |
| The gutter | Hallucinations, prompt injection, toxicity |

## 3. Concept Explanation
Guardrails are safety mechanisms wrapped around your LLM calls. In the OpenAI SDK context, you implement guardrails at three levels:
1. **Input Guardrails**: Checking the user's prompt before sending it to the LLM (e.g., blocking prompt injections or profanity).
2. **Tool Guardrails**: Validating the arguments the LLM tries to pass to your Python functions (e.g., ensuring it doesn't pass `-1` for a quantity).
3. **Output Guardrails**: Checking the final text the LLM wants to send to the user (e.g., ensuring it isn't giving medical advice when it's supposed to be a banking bot).

You use guardrails because LLMs are non-deterministic. You cannot trust them 100% of the time, bro.

## 4. Syntax Table
| Concept | Implementation Strategy | Description |
|---|---|---|
| Input Validation | `if contains_profanity(text): abort()` | Code-level check before the LLM. |
| Pydantic Output | `client.beta.chat.completions.parse(..., response_format=Model)` | Strict structured output enforcement. |
| LLM-as-a-Judge | `evaluate_safety_with_llm(output)` | Using a smaller LLM to check the main LLM's output. |

## 5. Beginner Example
Here is a simple input and tool guardrail setup.

```python
from swarm import Swarm, Agent

def refund_order(order_id: int) -> str:
    """Refund an order."""
    # TOOL GUARDRAIL: Code-level validation
    if order_id < 0:
        return "Error: Invalid order ID."
    return f"Order {order_id} refunded."

billing_agent = Agent(
    name="Billing",
    instructions="You process refunds.",
    functions=[refund_order]
)

client = Swarm()
user_input = "Refund order -5 immediately!"

# INPUT GUARDRAIL: Basic code check
if "ignore previous instructions" in user_input.lower():
    print("Guardrail blocked prompt injection attempt.")
else:
    response = client.run(agent=billing_agent, messages=[{"role": "user", "content": user_input}])
    print(response.messages[-1]["content"])
    # The LLM will call refund_order(-5), the tool will return the error,
    # and the LLM will tell the user the ID is invalid.
```

## 6. Real-World Engineering Example
Bro, in production, we use Pydantic for strict output guardrails (Structured Outputs) to guarantee the LLM doesn't hallucinate JSON keys.

```python
from pydantic import BaseModel
from openai import OpenAI

client = OpenAI()

# Define the exact structure you need (The Guardrail)
class UserProfile(BaseModel):
    name: str
    age: int
    is_premium: bool

def extract_user_info(text: str):
    # Using OpenAI's native Structured Outputs parsing
    completion = client.beta.chat.completions.parse(
        model="gpt-4o-2024-08-06",
        messages=[
            {"role": "system", "content": "Extract the user info."},
            {"role": "user", "content": text}
        ],
        response_format=UserProfile,
    )

    profile = completion.choices[0].message.parsed
    return profile

# Even if the text is messy, the output is guaranteed to fit the schema
result = extract_user_info("Hey, I'm Rahul, I'm 28, and yeah I pay for the premium sub.")
print(result.name, result.age, result.is_premium)
```

## 7. Internal Working

## 8. Performance Table
| Guardrail Type | Latency Impact | Reliability |
|---|---|---|

## 9. Top Interview Questions
| Question | Answer |
|---|---|
| What is the difference between a prompt instruction and a guardrail? | A prompt is a suggestion ("Don't swear"). A guardrail is code or inference-level constraints that block the output if it disobeys. |
| How do you prevent Prompt Injection? | Never execute raw user input as code, use strict input validation, and use LLM-as-a-judge classifiers on user prompts before passing to the main agent. |
| What is OpenAI Structured Outputs? | A feature that forces the model to generate JSON matching a provided Pydantic schema perfectly. |
| Should you validate tool arguments? | Yes, always. Treat LLM tool calls like an unauthenticated API request from a user. |
| What is LLM-as-a-Judge? | Using a fast, cheap model (like GPT-4o-mini) to review the output of the main model for safety before showing it to the user. |

## 10. Tricky Questions & Edge Cases
Bro, what happens if the LLM gets trapped in a loop because your tool keeps rejecting its hallucinated arguments?
**The Fix:** Implement a retry counter in your orchestrator loop. If a tool fails 3 times due to validation errors, force the agent to return a standard error message to the user ("I'm having trouble with that right now") and exit the loop.

## 11. Real-World Usage
Banks use extreme guardrails. The input is checked by a classification model for toxicity. The LLM processes the query. The output is checked by a secondary LLM to ensure it doesn't contain financial advice. Only then is it shown to the user.

## 12. Best Practices
| DO | DON'T |
|---|---|
| Use Pydantic/Structured Outputs for all JSON returns. | Ask the LLM nicely to "return JSON format" in the prompt. |
| Treat the LLM as an untrusted client in your tools. | Let the LLM directly execute SQL commands. |
| Fail gracefully and apologize to the user. | Show the user raw Python stack traces. |

## 13. Production Notes
:::warning
Guardrails are essential, but too many LLM-based guardrails will make your system unusable due to latency and cost. Use fast, traditional code (regex, blocklists, Pydantic) for 90% of your guardrails, and reserve LLM-as-a-judge for complex semantic checks.
:::

## 14. Common Mistakes
| Mistake | Fix |
|---|---|
| Trusting the LLM | Code your tool functions as if a malicious hacker is calling them. Validating IDs and permissions is mandatory. |
| Over-prompting | Cramming 50 lines of "Do not say X, Y, Z" into the system prompt. It dilutes the instructions. Use code checks. |
| Ignoring Hallucinations | Assuming the LLM will always output valid URLs or IDs. Check if the resource actually exists in the tool. |
