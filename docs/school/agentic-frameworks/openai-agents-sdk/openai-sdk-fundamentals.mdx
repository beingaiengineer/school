# OpenAI Agents SDK Fundamentals

| Area | Details |

## 3. Concept Explanation
The OpenAI Agents SDK (formerly Swarm, now evolved into official patterns) provides a lightweight, native way to build agents using the OpenAI API. It abstracts away the heavy lifting of function calling loops and context management.

You should use it when you are exclusively using OpenAI models (GPT-4o) and want the fastest path to a working agent. You should NOT use it if you need model agnosticism (e.g., you want to switch to Claude tomorrow) or complex state machine logic.

## 4. Syntax Table
| Concept | Code | Description |
|---|---|---|
| Initialize Client | `client = OpenAI()` | Standard OpenAI client setup. |
| Define Agent | `Agent(name="...", instructions="...")` | Create a core persona. |
| Define Tools | `def get_weather(): ...` | Standard python functions. |
| Execution | `client.run(agent, messages)` | Run the loop. |

## 5. Beginner Example
Here is the most basic example of an agent that can say hello.

```python
from openai import OpenAI
from swarm import Swarm, Agent

client = Swarm()

my_agent = Agent(
    name="Greeting Agent",
    instructions="You are a friendly agent. Always greet the user with 'Bro!'",
)

messages = [{"role": "user", "content": "Hello there."}]

response = client.run(agent=my_agent, messages=messages)
print(response.messages[-1]["content"])
# Output: "Bro! How's it going?"
```

## 6. Real-World Engineering Example
Bro, let's look at an agent with a real tool. We'll give it a function to fetch a user's account balance.

```python
from swarm import Swarm, Agent

def get_account_balance(user_id: str) -> str:
    # In reality, this queries a database
    if user_id == "123":
        return "$500"
    return "Account not found"

finance_agent = Agent(
    name="FinanceBot",
    instructions="Help users with their finances. You have access to their balance.",
    functions=[get_account_balance]
)

client = Swarm()
response = client.run(
    agent=finance_agent,
    messages=[{"role": "user", "content": "What is my balance? My id is 123."}]
)

print(response.messages[-1]["content"])
# Output: "Your current account balance is $500, bro."
```

## 7. Internal Working

## 9. Top Interview Questions
| Question | Answer |
|---|---|
| What is the main advantage of the OpenAI SDK for agents? | It provides the most native, reliable interface to OpenAI's function calling without external abstraction layers. |
| Does the SDK support Anthropic models? | No, it is exclusively built for the OpenAI API ecosystem. |
| How does the SDK handle the function calling loop? | It internally manages a while loop, executing requested tools and appending results until the model provides a final text response. |
| Is it suitable for highly complex, stateful multi-agent systems? | No, for complex state management, frameworks like LangGraph are preferred. |
| Can agents transfer control in this SDK? | Yes, "handoffs" are a core pattern where an agent returns another agent to transfer the conversation. |

## 10. Tricky Questions & Edge Cases
Bro, what happens if your Python tool function raises an exception?
**The Fix:** By default, it might crash your script. You must implement `try/except` blocks inside your tool functions and return a string like "Error fetching data" back to the LLM so the LLM knows it failed and can apologize to the user.

## 11. Real-World Usage
Companies heavily invested in the Microsoft/OpenAI ecosystem use this SDK for lightweight internal tools, like Slack bots that query internal documentation or basic customer service triaging bots.

## 12. Best Practices
| DO | DON'T |
|---|---|
| Use clear, concise docstrings on your tool functions. | Leave tool functions undocumented. The LLM reads the docstring! |
| Handle exceptions inside the tool function. | Let exceptions crash the orchestration loop. |
| Use this SDK for fast prototyping. | Try to force it to do complex DAG state routing. |

## 13. Production Notes
:::warning
Because this SDK is deeply tied to OpenAI, if OpenAI goes down, your application goes down. If you require multi-provider failover (e.g., falling back to Claude or Gemini), you must use a more agnostic framework like LangChain or LiteLLM.
:::

## 14. Common Mistakes
| Mistake | Fix |
|---|---|
| Missing type hints | Ensure all tool function arguments have Python type hints (e.g., `def query(date: str):`). |
| Poor docstrings | Write detailed docstrings. The LLM uses this to understand what the tool does. |
| Overcomplicating the logic | Keep the agent logic simple. If it's too complex, switch to a graph framework. |
