# The ReAct Pattern

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Core Agent Cognitive Architecture |
| Difficulty | Intermediate |
| Used For | Forcing an LLM to think step-by-step and interleave reasoning with taking actions |
| Common Mistake | Assuming ReAct means the React.js frontend framework. It means Reason + Act. |
| Performance | High token cost and high latency, as it requires multiple LLM round-trips to complete a single task |

## 2. Engineering Story

A healthcare provider deployed a customer service bot to help patients navigate their insurance benefits. The initial version relied entirely on a massive prompt containing the entire policy manual, leading to frequent hallucinations and excessive token costs. When patients asked about specific copays, the model just guessed. The engineering team replaced this static approach with the ReAct (Reasoning and Acting) pattern. Instead of guessing, the agent learned to interleave thought processes with API actions. It would output a thought, execute a tool to fetch user data, observe the result, and iterate to query the exact copay API. This grounded the agent's responses in real-time backend data, virtually eliminating hallucinations and restoring trust in the automated support system.

Bro, ReAct is exactly how a detective works. You don't just blindly arrest someone based on a footprint. You observe a clue, you *reason* about what it means, you take an *action* to get more info, and then you observe the result. ReAct forces the LLM to be a detective, not just a guesser.

## 4. Concept Explanation

ReAct (Reasoning and Acting) is a foundational prompt engineering pattern introduced in a 2022 Princeton/Google paper.

Before ReAct, we either had:
1. **Chain of Thought (CoT):** The model reasons step-by-step but has no tools. It's stuck in a box.
2. **Action-Only (Function Calling):** The model immediately picks a tool without explaining why. If it guesses wrong, it fails.

ReAct combines them. The system prompt forces the LLM into a strict output format:
- `Thought:` (The reasoning phase)
- `Action:` (The tool invocation)
- `Observation:` (Provided by the external system)

By writing out its `Thought` first, the LLM creates a scratchpad for itself. It processes the logic *before* committing to an `Action`, drastically reducing errors and hallucinations.

## 5. Syntax Table

| Phase | Description | Example |
|---|---|---|
| **Thought** | The LLM's internal monologue | `Thought: I need to find the user's IP address.` |
| **Action** | The tool name and arguments | `Action: lookup_ip {"user_id": "123"}` |
| **Observation** | The result returned by the tool | `Observation: {"ip": "192.168.1.1"}` |
| **Final Answer**| The end of the ReAct loop | `Final Answer: The user is on a local network.` |

## 6. Beginner Example

Here is a classic LangChain implementation of the ReAct pattern. Notice how we explicitly use the `create_react_agent` function, which injects the ReAct formatting instructions into the system prompt under the hood.

```python
from langchain.agents import create_react_agent, AgentExecutor
from langchain.prompts import PromptTemplate
from langchain_openai import ChatOpenAI
from my_tools import web_search, calculator

llm = ChatOpenAI(temperature=0)
tools = [web_search, calculator]

# The ReAct prompt MUST include specific instructions on the format
prompt_template = """
Answer the following questions as best you can. You have access to the following tools:
{tools}

Use the following format:
Question: the input question you must answer
Thought: you should always think about what to do
Action: the action to take, should be one of [{tool_names}]
Action Input: the input to the action
Observation: the result of the action
... (this Thought/Action/Action Input/Observation can repeat N times)
Thought: I now know the final answer
Final Answer: the final answer to the original input question

Question: {input}
{agent_scratchpad}
"""

prompt = PromptTemplate.from_template(prompt_template)

# Create the agent
agent = create_react_agent(llm, tools, prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# Run the ReAct loop
agent_executor.invoke({"input": "What is 20% of the current stock price of Tesla?"})
```

## 7. Real-World Engineering Example

Bro, in modern production systems (2024+), we often don't use raw text-based ReAct parsing anymore because it's brittle (the LLM might misspell `Action:`). Instead, we use **Tool Calling ReAct**.

The LLM still outputs a "Thought" (as text), but the "Action" is a native JSON tool call. Here is how you build a robust Tool-Calling ReAct loop using LangGraph.

```python
from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def get_weather(city: str):
    """Returns the weather for a city."""
    return f"The weather in {city} is sunny."

@tool
def send_email(to: str, subject: str):
    """Sends an email."""
    return "Email sent successfully."

tools = [get_weather, send_email]
model = ChatOpenAI(model="gpt-4o")

# LangGraph has a pre-built React Agent that handles the state graph perfectly
app = create_react_agent(model, tools)

# The agent will:
# 1. Think: "I need to get the weather for NYC"
# 2. Act: get_weather("NYC")
# 3. Observe: "Sunny"
# 4. Think: "Now I need to email Bob"
# 5. Act: send_email("bob@example.com", "Weather is sunny")
# 6. Output Final Answer

messages = app.invoke({"messages": [("user", "Check NYC weather and email it to bob@example.com")]})

for m in messages['messages']:
    m.pretty_print()
```

## 9. Performance Table

| Phase | Time Latency | Token Consumption |
|---|---|---|
| **Thought Generation** | High (Streaming output) | Medium (Usually 20-50 tokens) |
| **Action Generation** | Low | Low (JSON object) |
| **Observation** | Varies by Tool (API speed) | High (Data is appended to context) |
| **Total ReAct Loop** | **High** (3-15 seconds) | **High** (Context grows linearly) |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why is ReAct better than zero-shot tool calling? | Zero-shot forces the LLM to guess the tool immediately. ReAct forces it to write out its logic first. This "scratchpad" acts as active working memory, drastically improving accuracy for complex, multi-step problems. |
| What is the "agent scratchpad"? | It's the string (or message array) containing all previous Thoughts, Actions, and Observations from the current loop. The LLM needs this so it doesn't get stuck in an infinite loop doing the same thing. |
| How do you prevent a ReAct agent from looping forever? | You must set a `max_iterations` limit in your orchestration code (e.g., limit to 5 loops). If it hits the limit, force it to return a `Final Answer` stating it failed. |
| Why is text-based ReAct deprecated in favor of Tool-Calling ReAct? | Text-based ReAct relies on regex parsing (e.g., looking for the word "Action:"). If the LLM formats it slightly wrong, the parser crashes. Native Tool Calling guarantees valid JSON. |

## 11. Tricky Questions & Edge Cases

Bro, what happens when an Observation contradicts the agent's initial Thought?
Imagine the agent thinks: "I need to get Bob's ID, which is 1."
Action: `get_user(1)`.
Observation: `User Not Found`.
If the LLM isn't smart enough, it might just retry `get_user(1)` forever.
**The Fix:** You need a high-quality model (like GPT-4o) that can *reason about failures*. The next Thought should be: "Ah, ID 1 failed. Let me try searching by name instead."

## 12. Real-World Usage

- **Customer Support Triage:** Thought: "User is asking about a refund." Action: `check_policy()`. Observation: "Refunds allowed within 30 days." Thought: "I need to check when they bought it." Action: `check_order()`.
- **Data Engineering:** Thought: "I need to join table A and B." Action: `execute_sql()`. Observation: "Error: Column 'user_id' is ambiguous." Thought: "I need to specify the table alias." Action: `execute_sql_fixed()`.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Use native Tool Calling APIs (like OpenAI's) alongside the ReAct methodology for maximum stability. | Don't rely on raw Regex parsing for `Action: ` unless you are using an older, open-source model that lacks tool support. |
| Stream the "Thoughts" to the frontend UI so the user knows the bot isn't frozen. | Don't hide the reasoning process from the user if the task takes more than 3 seconds. |

## 14. Production Notes

:::caution Production Warning
Bro, the ReAct scratchpad grows with every loop. If your agent executes a DB query and the Observation is 10,000 rows of JSON, the scratchpad will blow past the token limit instantly, and the next API call will fail. You MUST implement observation truncation (e.g., only return the first 500 characters of a tool result).
:::

## 15. Common Mistakes

| Mistake | Why it's bad | The Fix |
|---|---|---|
| Forgetting to append the Observation to the memory array | The LLM will think its tool call was ignored, and will just issue the exact same tool call again. | Ensure your orchestration loop properly saves state. |
| Prompting an agent without explicitly asking for a "Thought" | The LLM will skip reasoning, act impulsively, and make logic errors. | Ensure your system prompt mandates step-by-step thinking before tool usage. |
