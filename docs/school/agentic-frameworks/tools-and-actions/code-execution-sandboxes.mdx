## 2. Engineering Story

Bro, your new coding assistant agent generated a Python script to process some files and immediately ran it on your host server. The script had a subtle bug (or was maliciously prompted by a user) and ended up deleting critical system configurations, causing a major outage. You can never, ever run AI-generated code directly on your host machine. This is a fundamental security requirement. As an engineer, you must implement secure code execution sandboxes. Whether using Docker containers, specialized micro-VMs like Firecracker, or managed services like E2B, you have to isolate the execution environment, restrict network access, and set hard limits on memory and CPU to safely harness the power of code-generating agents.

## 4. Concept Explanation

Code Execution Sandboxes are isolated, secure environments (like Docker containers, WebAssembly, or firecracker microVMs) where an agent can run generated code to solve math problems, analyze data, or test software. Why does it exist? To give agents computational power while protecting the host system from malicious or buggy code. When to use it? When building coding assistants, data science agents, or advanced math solvers (like OpenAI's Advanced Data Analysis). When NOT to use it? Never run agent code directly on bare metal.

## 7. Real-World Engineering Example

Show production-style code representing actual system scenarios, like this production-grade implementation.

```python
import docker
import logging
import tempfile
import os

logger = logging.getLogger(__name__)

class DockerSandboxTool:
    def __init__(self):
        self.client = docker.from_env()

    def run_python_code(self, code: str) -> str:
        """
        Executes Python code in a secure Docker sandbox with no network access.
        """
        with tempfile.TemporaryDirectory() as temp_dir:
            script_path = os.path.join(temp_dir, 'script.py')
            with open(script_path, 'w') as f:
                f.write(code)

            try:
                # Run an ephemeral container, completely isolated
                container = self.client.containers.run(
                    "python:3.9-slim",
                    command=["python", "/workspace/script.py"],
                    volumes={temp_dir: {'bind': '/workspace', 'mode': 'ro'}},
                    working_dir="/workspace",
                    network_disabled=True,      # Block internet access
                    mem_limit="128m",           # Prevent memory bombs
                    cpu_quota=50000,            # Limit CPU
                    remove=True,
                    timeout=5
                )
                return container.decode('utf-8')
            except docker.errors.ContainerError as e:
                logger.error("Sandbox execution failed")
                return f"Execution Error: {e.stderr.decode('utf-8')}"
```

## 10. Top Interview Questions

| Why can't we use Python's `eval()` or `exec()`? | It runs in the same process as your app. An agent could read environment variables, steal API keys, or format your hard drive. | Beginner |
| What is eBPF and how does it relate to sandboxing? | It's a Linux kernel technology used to monitor and restrict system calls, providing deep security for sandboxes. | Advanced |
| What is the easiest way to add sandboxing today? | Using hosted services like E2B, Daytona, or Docker containers with strict resource limits. | Intermediate |
| What are the alternatives? | Depending on the exact need, standard APIs or simpler prompt engineering might suffice. | Intermediate |
| How do you debug it? | By using tracing tools like LangSmith or Arize Phoenix to inspect the exact payloads and intermediate steps. | Advanced |

## 14. Production Notes

<Admonition type="caution" title="Production Warning">
When scaling this in production, watch out for context window token limits and API rate limits. Memory leaks can happen if you don't clear conversational state for long-running agents. Security boundaries are critical when executing generated code or DB queries.
</Admonition>

## 15. Common Mistakes

| Giving sandbox containers network access | Agent gets hacked or acts as a botnet | Set `network_disabled=True` in Docker |
