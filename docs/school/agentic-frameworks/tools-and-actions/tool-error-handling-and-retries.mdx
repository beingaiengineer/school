# Tool Error Handling And Retries

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Tool Error Handling And Retries |
| Difficulty | Intermediate |
| Used For | Preventing agent loops and crashes during external API/tool calls. |
| Common Mistake | Allowing the LLM to hallucinate arguments indefinitely on failure. |
| Performance | Evaluated based on token usage and latency. |

## 2. Engineering Story

Bro, imagine you're running a high-scale customer support agent. It tries to refund a user via a payment API, but the API times out. Without proper error handling, the agent might either crash, tell the user the refund failed permanently, or worse, try to blindly call it 100 times, burning through your API rate limits. We need a systematic way to handle tool failures.

## 4. Concept Explanation

When an LLM decides to call a tool, it's stepping out of its deterministic reasoning environment into the messy real world of network latency, bad arguments, and API outages. Tool error handling involves catching these failures, preventing infinite retry loops, and giving the LLM enough context (e.g., 'API rate limited, try again in 60s') so it can decide the next best action.

## 5. Syntax Table

| Concept | Implementation Pattern |
|---|---|
| Standard Try/Catch | Wrap tool execution in try-catch blocks. |
| Exponential Backoff | Use libraries to retry with increasing delays. |
| LLM Fallback | Return the exact error string back to the LLM as observation. |

## 6. Beginner Example

```typescript
async function safeToolCall(args: any) {
  try {
    return await executeTool(args);
  } catch (error) {
    // Return error to LLM instead of throwing
    return `Error: ${error.message}. Please adjust your arguments and try again.`;
  }
}
```

## 7. Real-World Engineering Example

```typescript
import { retry } from 'async-retry';

async function robustToolExecution(toolName: string, args: Record<string, any>) {
  return await retry(async (bail) => {
    try {
      const result = await runExternalTool(toolName, args);
      return { success: true, data: result };
    } catch (e) {
      if (e.status === 400) {
        // Bad request from LLM, do not retry, just return error
        bail(new Error(`Bad Arguments provided: ${e.message}`));
      }
      throw e; // Retry on 5xx or network errors
    }
  }, {
    retries: 3,
    factor: 2,
    minTimeout: 1000,
  });
}
```

## 9. Performance Table

| Operation | Impact |
|---|---|
| Excessive Retries | Latency spikes, Token waste if LLM is in the loop |
| Graceful Degradation | High reliability, low cost |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why not just throw an error on tool failure? | Because it crashes the agent. You want to pass the error to the LLM so it can reason and recover. |
| What is an LLM retry loop? | When an LLM repeatedly calls a tool with the same bad arguments, expecting a different result. |
| How do you prevent retry loops? | Set a strict `max_iterations` counter on the agent framework. |

## 11. Tricky Questions & Edge Cases

Bro, if you return a raw stack trace to an LLM, it might get confused or hallucinate a fix that doesn't exist. Always sanitize your error messages so they are actionable for the LLM.

## 12. Real-World Usage

Used extensively in systems like AutoGPT or Devin, where agents must recover from failed compilation steps or 404 API errors gracefully.

## 13. Best Practices

| Do | Don't |
|---|---|
| Return structured error messages to the LLM | Throw raw exceptions that crash the process |
| Implement exponential backoff for network errors | Allow infinite tool retries |
