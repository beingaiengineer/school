## 2. Engineering Story

Bro, you want your agent to answer questions about user analytics, so you give it a tool that can execute raw SQL queries against your production database. Within an hour, the agent hallucinates a poorly optimized `JOIN` that locks up a critical table, bringing down the entire application. Later, it accidentally drops a view. Giving an AI direct, unrestricted SQL access is an engineering disaster waiting to happen. When building database tools for agents, you must implement strict read-only roles, query timeouts, and schema constraints. Better yet, you should abstract the database behind a semantic layer or specific analytical tools, ensuring the agent can retrieve insights safely without risking the integrity of your production infrastructure.

## 4. Concept Explanation

Database Tools allow agents to query structured databases (SQL, MongoDB) to answer analytical questions. They typically involve a pipeline: the LLM sees the schema, generates a query, the tool executes it safely, and returns the result. Why does it exist? To build Text-to-SQL agents and data analysts. When to use it? For internal BI tools or answering questions based on live relational data. When NOT to use it? NEVER use these on production write-replicas without extreme guardrails.

## 7. Real-World Engineering Example

Show production-style code representing actual system scenarios, like this production-grade implementation.

```python
from sqlalchemy import create_engine, text
import logging

logger = logging.getLogger(__name__)

class SafeSQLTool:
    def __init__(self, connection_string: str):
        # Connect to a READ-ONLY replica in production
        self.engine = create_engine(connection_string)

    def execute_query(self, sql_query: str) -> str:
        """Executes a SQL query and returns the first 100 rows."""
        if not sql_query.strip().upper().startswith("SELECT"):
            return "Security Error: Only SELECT queries are permitted."

        try:
            with self.engine.connect() as conn:
                # Enforce limits
                safe_query = f"SELECT * FROM ({sql_query}) AS sub LIMIT 100"
                result = conn.execute(text(safe_query))
                rows = [dict(row._mapping) for row in result]
                return str(rows)
        except Exception as e:
            logger.error(f"SQL Execution failed: {e}")
            return f"Query Failed: {str(e)}. Please fix the syntax and try again."
```

## 10. Top Interview Questions

| What is the biggest risk of DB tools? | SQL Injection or destructive operations (DROP/DELETE). | Beginner |
| How do you mitigate this risk? | Connect the tool to a read-only database replica and strictly validate that the query is a SELECT statement. | Intermediate |
| What if the schema is too large to fit in the prompt? | Use a vector store to retrieve only the relevant table schemas based on the user's question, rather than dumping the whole DB schema. | Advanced |
| What are the alternatives? | Depending on the exact need, standard APIs or simpler prompt engineering might suffice. | Intermediate |
| How do you debug it? | By using tracing tools like LangSmith or Arize Phoenix to inspect the exact payloads and intermediate steps. | Advanced |

## 13. Best Practices

| DO | DON'T |
|---|---|
| Do use guardrails | Don't let agents execute raw code unchecked |
| Do limit context size | Don't stuff everything into the prompt |
| Do monitor errors | Don't ignore agent hallucinations |

## 14. Production Notes

<Admonition type="caution" title="Production Warning">
When scaling this in production, watch out for context window token limits and API rate limits. Memory leaks can happen if you don't clear conversational state for long-running agents. Security boundaries are critical when executing generated code or DB queries.
</Admonition>

## 15. Common Mistakes

| Connecting LLMs to primary write databases | Production data loss | Use read-only replicas and strict IAM roles |
