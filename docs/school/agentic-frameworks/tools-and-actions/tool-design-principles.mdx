## 2. Engineering Story

Bro, you just gave your LLM agent access to an API tool that deletes a user account. The tool requires a user ID, but the agent hallucinates a string instead of an integer, and the API crashes. Then, the agent tries again, passing the string 'all', and somehow bypasses a check, deleting multiple records. This is a catastrophic production incident caused by poor tool design. When exposing tools to AI agents, you cannot design them the same way you design APIs for human developers. Agents are unpredictable and prone to hallucinations. You must engineer your tools with strict type validation, comprehensive error messaging that tells the agent how to fix its mistake, and bulletproof guardrails to prevent destructive actions.

## 4. Concept Explanation

Tool Design Principles dictate how we build functions that LLMs will execute via function calling. It involves applying the Single Responsibility Principle, using strong typing (Pydantic), writing highly descriptive docstrings, and failing gracefully. Why does it exist? LLMs are probabilistic. They will hallucinate arguments if you don't constrain them. Good tool design drastically reduces errors. When to use it? Whenever you expose external actions to an agent. When NOT to use it? Never skip good design.

## 7. Real-World Engineering Example

Show production-style code representing actual system scenarios, like this production-grade implementation.

```python
from pydantic import BaseModel, Field
import logging

logger = logging.getLogger(__name__)

class RefundInput(BaseModel):
    order_id: str = Field(description="The unique 8-character alphanumeric order ID")
    amount: float = Field(description="The amount to refund in USD, must be greater than 0")
    reason: str = Field(description="A short explanation of why the refund is issued")

def process_refund(inputs: RefundInput) -> str:
    """
    Processes a refund for a customer.
    Only use this tool when the customer explicitly asks for their money back.
    """
    if inputs.amount <= 0:
        return "Error: Refund amount must be positive."

    logger.info(f"Processing refund of {inputs.amount} for {inputs.order_id}")
    # API call to payment gateway goes here
    return f"Successfully refunded ${inputs.amount}"
```

## 10. Top Interview Questions

| Why are docstrings so important for agent tools? | The docstring is literally injected into the prompt. It's how the LLM decides if and when to use the tool. | Beginner |
| Why use Pydantic for tool inputs? | It enforces strict typing and validation, rejecting hallucinations before they hit your execution logic. | Intermediate |
| What happens if a tool throws an unhandled exception? | The agentic loop crashes. Tools must catch exceptions and return them as strings so the agent can retry. | Advanced |
| What are the alternatives? | Depending on the exact need, standard APIs or simpler prompt engineering might suffice. | Intermediate |
| How do you debug it? | By using tracing tools like LangSmith or Arize Phoenix to inspect the exact payloads and intermediate steps. | Advanced |

## 13. Best Practices

| DO | DON'T |
|---|---|
| Do use guardrails | Don't let agents execute raw code unchecked |
| Do limit context size | Don't stuff everything into the prompt |
| Do monitor errors | Don't ignore agent hallucinations |

## 14. Production Notes

<Admonition type="caution" title="Production Warning">
When scaling this in production, watch out for context window token limits and API rate limits. Memory leaks can happen if you don't clear conversational state for long-running agents. Security boundaries are critical when executing generated code or DB queries.
</Admonition>
