## 2. Engineering Story

Bro, your agent is smart, but it's isolated. You want it to create Jira tickets, send Slack messages, and check Stripe payments. To do this, it needs to talk to the outside world. The production problem is securely and reliably wrapping HTTP APIs so an LLM can use them. If you just give an LLM raw `requests.post()` access, it might construct terrible payloads or leak API keys. You need dedicated API Tools for REST and GraphQL that handle authentication, pagination, and schema mapping safely.

## 4. Concept Explanation

API Tools are wrappers around external REST or GraphQL endpoints designed specifically for agent consumption. They abstract away headers, authentication (API keys/OAuth), and complex payload construction. Why does it exist? To connect agents to the SaaS ecosystem safely. When to use it? When your agent needs to interact with GitHub, Jira, Slack, Salesforce, etc. When NOT to use it? Don't use raw HTTP tools if a specialized SDK tool (like the official Stripe Python library) is available.

## 7. Real-World Engineering Example

Show production-style code representing actual system scenarios, like this production-grade implementation.

```python
import requests
import logging
from pydantic import BaseModel

logger = logging.getLogger(__name__)

class JiraTicketInput(BaseModel):
    summary: str
    description: str

class JiraTool:
    def __init__(self, domain: str, email: str, api_token: str):
        self.base_url = f"https://{domain}.atlassian.net/rest/api/3"
        self.auth = (email, api_token)

    def create_ticket(self, inputs: JiraTicketInput) -> str:
        """Creates a new bug ticket in Jira."""
        payload = {
            "fields": {
                "project": {"key": "ENG"},
                "summary": inputs.summary,
                "description": inputs.description,
                "issuetype": {"name": "Bug"}
            }
        }
        try:
            res = requests.post(f"{self.base_url}/issue", json=payload, auth=self.auth)
            res.raise_for_status()
            return f"Success: Created ticket {res.json().get('key')}"
        except requests.exceptions.RequestException as e:
            logger.error(f"Jira API failed: {e}")
            return f"API Error: {str(e)}"
```

## 10. Top Interview Questions

| How should API keys be handled in agent tools? | Pass them via environment variables or secure secret managers during initialization. Never expose them to the LLM. | Beginner |
| Why shouldn't you give agents raw HTTP capabilities? | Security. They might send malicious requests (SSRF) or hallucinate payloads. Always use constrained wrappers. | Intermediate |
| How do you handle large JSON API responses? | Filter the response in your Python code to only return the essential fields to the LLM to save tokens. | Advanced |
| What are the alternatives? | Depending on the exact need, standard APIs or simpler prompt engineering might suffice. | Intermediate |
| How do you debug it? | By using tracing tools like LangSmith or Arize Phoenix to inspect the exact payloads and intermediate steps. | Advanced |

| DO | DON'T |
|---|---|
| Do use guardrails | Don't let agents execute raw code unchecked |
| Do limit context size | Don't stuff everything into the prompt |
| Do monitor errors | Don't ignore agent hallucinations |

## 14. Production Notes

<Admonition type="caution" title="Production Warning">
When scaling this in production, watch out for context window token limits and API rate limits. Memory leaks can happen if you don't clear conversational state for long-running agents. Security boundaries are critical when executing generated code or DB queries.
</Admonition>
