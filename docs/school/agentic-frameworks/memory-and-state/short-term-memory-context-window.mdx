## 3. Concept Explanation

Short-term memory refers to the immediate conversation history and state maintained within the LLM's active context window. It is fast and highly accurate but strictly limited in size and cost. It solves the problem of "one giant prompt" failing on complex tasks by breaking them down and distributing them. Bro, the key is knowing when to use which pattern and tool.

## 4. Syntax Table

| Concept | Description | Typical Use Case |
|---|---|---|
| Hand-off | Transferring control to another agent | Linear workflows |
| Broadcast | Sending a message to all agents | Event-driven systems |
| Supervisor | One agent delegates and reviews | Complex tasks needing oversight |

## 5. Beginner Example

```python
# A simple conceptual example of agent delegation
def main_agent(task):
    if "research" in task:
        return research_agent(task)
    elif "write" in task:
        return writer_agent(task)
    return "Task completed by main agent"
```

## 6. Real-World Engineering Example

```python
# Real-world scenario: routing incoming queries based on Short-Term Memory (Context Window)
class CustomerSupportRouter:
    def route_ticket(self, ticket):
        intent = self.analyze_intent(ticket)
        if intent == "billing":
            return self.billing_agent.handle(ticket)
        elif intent == "technical":
            return self.tech_support_agent.handle(ticket)
        else:
            return self.general_agent.handle(ticket)
```

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| How do you prevent infinite loops in multi-agent systems? | Implement a max-turns counter or state-based termination checks. |
| When would you use a supervisor vs a router? | Router just delegates. Supervisor delegates, reviews, and can send back for rework. |
| How do agents share memory? | Through a shared state dictionary, external database, or message passing. |
| What's the main challenge with multi-agent setups? | Error propagation; if one agent hallucinates, it affects downstream agents. |
| How do you test a multi-agent system? | Unit test agents individually with mocked inputs, then integration test the flows. |

## 10. Tricky Questions & Edge Cases

Bro, what happens if two agents need to update the same shared state simultaneously? You get race conditions! Always ensure atomic updates or use a single state-manager agent. Also, watch out for context window overflow when agents pass large messages back and forth.

## 11. Real-World Usage

Companies like Swiggy use multi-agent patterns for complex support tickets. One agent classifies, another fetches order data, and a third formulates the response. This compartmentalization keeps individual prompts focused and reliable.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Keep agent responsibilities narrow and focused. | Create "god agents" that try to do everything. |
| Use structured outputs (JSON) for inter-agent communication. | Rely on raw text parsing between agents. |
| Implement strict timeouts and max-iteration limits. | Let agents talk indefinitely without boundaries. |

## 13. Production Notes

> **Warning:** Multi-agent systems multiply your API costs and latency. If a single LLM call takes 2 seconds, a 5-agent sequence might take 10+ seconds. Always stream intermediate progress to the user to manage expectations.

## 14. Common Mistakes

| Mistake | Correction |
|---|---|
| Allowing the conversation history to grow infinitely until it crashes the context limit. | Implement sliding windows or conversation summarization when history exceeds a certain threshold. |
| Hardcoding agent handoffs. | Use a dynamic router or state machine framework (like LangGraph). |
