# LangGraph Human in the Loop (HITL)

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Human in the Loop (HITL) in LangGraph |
| Difficulty | Advanced |
| Used For | Pausing agent execution to ask for human approval, input, or edits before continuing |
| Common Mistake | Trying to build HITL without a persistent checkpointer |
| Performance | Graph execution freezes and yields to the event loop, so minimal server overhead while waiting |

## 2. Engineering Story

A quantitative trading firm built a LangGraph agent to autonomously rebalance their portfolio. In staging it was impressive — it analyzed market data, called broker APIs, and executed trades without human intervention. Then they promoted it to production with live funds.

On its third run, the agent misread a currency conversion and placed a sell order 10x larger than intended, liquidating a position worth $280,000. By the time a human noticed, the trade was done. The fix wasn't to shut down the agent — it was to add a Human-in-the-Loop checkpoint. Now, before any `execute_trade()` tool call, the graph pauses, serializes its full state to a Postgres checkpointer, and sends a Slack message to the risk desk for approval. The agent waits — indefinitely — until a human types `/approve` or `/reject`. The same autonomous intelligence, now with a human safety valve at every irreversible step.

## 3. Real-World Analogy

| Office Task | LangGraph HITL Equivalent |
|---|---|
| A junior dev writes a PR and submits it | The agent drafts an email and prepares to send it |
| The junior waits for the senior's approval | Graph hits a `interrupt_before` node and pauses |
| The senior reviews and says "LGTM" | The human calls `invoke(None, config)` to resume |
| The senior says "Fix this bug first" | The human updates the graph State, then resumes |

Bro, think of Human in the Loop like a train conductor checking tickets. The train (the agent) drives itself most of the way, but right before it crosses the border (doing something dangerous like deleting a database or spending money), the conductor (the human) has to turn the key to let it proceed.

## 4. Concept Explanation

Sometimes, you do not want your AI agent to act completely autonomously. If your agent's job is to refund a customer, you probably want a human manager to click "Approve" before the Stripe API is called.

LangGraph implements HITL natively using **breakpoints**. Because LangGraph relies on state checkpointers (memory), you can configure a compiled graph to pause *before* or *after* a specific node.

When the graph hits a breakpoint, it saves its state to the database and stops executing. The Python process literally yields. Hours or days later, the human can view the state, optionally edit the state, and then call `graph.invoke` again with the exact same `thread_id` to resume execution exactly where it left off.

## 5. Syntax Table

| Action | Syntax Example | Use Case |
|---|---|---|
| Pause before a node | `builder.compile(checkpointer=mem, interrupt_before=["refund_node"])` | Wait for approval before executing an action |
| Pause after a node | `builder.compile(checkpointer=mem, interrupt_after=["draft_email"])` | Let agent draft work, pause to let human review the draft |
| Get current state | `graph.get_state(config)` | Read the state while paused to show the human |
| Update state | `graph.update_state(config, \{"email": "Edited email"\})` | Human edits the agent's draft before resuming |
| Resume graph | `graph.invoke(None, config)` | Tell the paused graph to continue |

## 6. Beginner Example

Here's how you pause before the agent executes a dangerous tool.

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from typing import TypedDict

class State(TypedDict):
    input_text: str
    is_approved: bool

def generate_plan(state: State):
    return {"input_text": "Plan: Delete all tables."}

def execute_plan(state: State):
    if state.get("is_approved"):
        return {"input_text": "Tables deleted."}
    return {"input_text": "Execution rejected."}

builder = StateGraph(State)
builder.add_node("plan", generate_plan)
builder.add_node("execute", execute_plan)
builder.add_edge(START, "plan")
builder.add_edge("plan", "execute")
builder.add_edge("execute", END)

# Must use a checkpointer for HITL!
memory = MemorySaver()

# Pause BEFORE the execute node
graph = builder.compile(checkpointer=memory, interrupt_before=["execute"])

config = {"configurable": {"thread_id": "1"}}

# 1. Run the graph. It will pause after "plan".
print("First run:", graph.invoke({"input_text": "Do it", "is_approved": False}, config))

# 2. Check the state. It will show next node is 'execute'
state = graph.get_state(config)
print("Next node:", state.next) # Output: ('execute',)

# 3. Human updates the state to approve it
graph.update_state(config, {"is_approved": True})

# 4. Resume execution by passing None
print("Second run:", graph.invoke(None, config))
```

## 7. Real-World Engineering Example

Bro, in a real FastAPI application, the human clicking "Approve" is a completely different HTTP request than the one that started the agent. Here's how that API structure looks.

```python
from fastapi import FastAPI, BackgroundTasks
from pydantic import BaseModel
# ... assumes `graph` is compiled with Postgres checkpointer
# and interrupt_before=["send_email_tool"]

app = FastAPI()

class StartRequest(BaseModel):
    thread_id: str
    prompt: str

class ApproveRequest(BaseModel):
    thread_id: str
    human_edits: str | None = None

@app.post("/start-agent")
async def start_agent(req: StartRequest, bg_tasks: BackgroundTasks):
    config = {"configurable": {"thread_id": req.thread_id}}

    # Run the graph until it hits the breakpoint
    state = await graph.ainvoke({"user_input": req.prompt}, config)

    current_state = await graph.aget_state(config)
    if "send_email_tool" in current_state.next:
        return {"status": "waiting_for_approval", "draft": state["email_draft"]}

    return {"status": "completed"}

@app.post("/approve-agent")
async def approve_agent(req: ApproveRequest):
    config = {"configurable": {"thread_id": req.thread_id}}

    # Check if graph is actually paused
    current_state = await graph.aget_state(config)
    if not current_state.next:
        return {"error": "Graph is not waiting for approval."}

    # Human edits the draft if provided
    if req.human_edits:
        await graph.aupdate_state(config, {"email_draft": req.human_edits})

    # Resume the graph
    await graph.ainvoke(None, config)
    return {"status": "resumed and email sent"}
```

When LangGraph encounters an `interrupt_before` node during its loop, it throws an internal `GraphInterrupt` exception. This is caught by the runner, which immediately saves the current checkpoint, sets the `next` tuple in the state to indicate which node is queued, and exits the loop.

## 9. Performance Table

| Operation | Overhead | Note |
|---|---|---|
| Pausing Graph | Minimal (1 DB write) | The process yields, freeing up the CPU/thread |
| Waiting | Zero | The state just sits in Postgres/SQLite safely |
| Resuming Graph | Minimal (1 DB read) | Instant resumption based on `thread_id` |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why does Human-in-the-Loop require a checkpointer? | Without a checkpointer, the graph has nowhere to save its state when it pauses. When the human tries to resume, the graph wouldn't know what the previous variables were. |
| How do you know if a graph is currently paused waiting for human input? | You call `graph.get_state(config)`. If the `.next` property of the returned state is not empty, it means the graph is paused waiting to execute that node. |
| How do you resume a paused graph? | You call `graph.invoke(None, config)`. Passing `None` as the input tells LangGraph "I have no new input, just continue from where you paused." |
| Can a human edit the state before resuming? | Yes, using `graph.update_state(config, {"key": "new value"})`. This creates a new checkpoint overriding the previous values. |
| What is the difference between `interrupt_before` and `interrupt_after`? | `interrupt_before=["A"]` pauses the graph *right before* node A runs. `interrupt_after=["A"]` lets node A finish, saves its state, and then pauses before whatever comes next. |

## 11. Tricky Questions & Edge Cases

Bro, what if you have a routing function (conditional edge) pointing to a node that has a breakpoint on it?

**Answer:** It works perfectly. The conditional edge evaluates, determines the next node is `dangerous_node`, sees that `dangerous_node` is in the `interrupt_before` list, and pauses the graph. The `.next` state will show `dangerous_node`.

## 12. Real-World Usage

Companies use HITL for:
- **Financial bots:** Agent researches stock, suggests trade, waits for portfolio manager to click "Buy".
- **Social media bots:** Agent drafts 5 tweets, pauses. Human deletes 2, edits 1, and approves. Agent then posts the 3 approved tweets.
- **Code reviewers:** Agent reviews a PR, suggests a fix, and pauses to let the original author review the fix before committing it.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Do wrap your HITL logic in a separate frontend dashboard so non-engineers can approve actions. | Don't build HITL loops that block the main thread waiting for an `input()` in production. |
| Do use `update_state` if the human needs to provide feedback like "Try again, but make it shorter". | Don't put breakpoints on every single tool; only on actions that have irreversible real-world impact. |

## 14. Production Notes

> **Production Warning:**
> If an agent is paused and waiting for human approval, that state will sit in your Postgres database forever if the human never responds. You MUST implement a cron job or background worker that finds stale checkpoints (e.g., waiting > 48 hours) and either auto-rejects them or alerts a manager.

## 15. Common Mistakes

| Mistake | Fix | Code Example |
|---|---|---|
| Passing the original input again when resuming | Pass `None` to resume | `graph.invoke(None, config)` |
| Trying to resume without the exact `thread_id` | Keep track of the `thread_id` | `config = {"configurable": {"thread_id": "user_123"}}` |
| Calling `update_state` without a specific node target (advanced) | Sometimes you need to specify which node authored the update | `graph.update_state(config, updates, as_node="human")` |
