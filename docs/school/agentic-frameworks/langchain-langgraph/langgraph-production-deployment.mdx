# LangGraph Production Deployment

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Deploying LangGraph Agents to Production |
| Difficulty | Advanced |
| Used For | Taking local agent scripts and running them as highly available, scalable APIs |
| Common Mistake | Using `MemorySaver` in a FastAPI app behind Gunicorn, losing state on every worker rotation |
| Performance | Heavily dependent on checkpointer latency, LLM API latency, and async concurrency |

## 3. Concept Explanation

Deploying a LangGraph agent means exposing your compiled `StateGraph` as an API so frontends, Slack bots, or cron jobs can interact with it.

There are three primary ways to deploy LangGraph:
1. **Custom FastAPI / Litestar:** You write your own web server, handle the async execution, manage the Postgres checkpointer pool, and stream server-sent events (SSE) manually.
2. **LangGraph Server (Open Source):** A pre-built server provided by LangChain that automatically wraps your graph in a REST API with streaming, background tasks, and checkpointer support out of the box.
3. **LangGraph Cloud (Managed SaaS):** You push your code, and LangChain hosts it, scales it, and provides a beautiful studio UI.

For serious engineering teams who want to control their infrastructure, deploying the open-source **LangGraph Server** via Docker is usually the best move.

## 5. Beginner Example

To use LangGraph Server, you don't write FastAPI code. You write a config file.

First, your agent code (`agent.py`):
```python
from langgraph.graph import StateGraph, START, END
from typing import TypedDict

class State(TypedDict):
    input: str
    output: str

def echo(state: State):
    return {"output": f"Echo: {state['input']}"}

builder = StateGraph(State)
builder.add_node("echo", echo)
builder.add_edge(START, "echo")
builder.add_edge("echo", END)

# IMPORTANT: You must expose the compiled graph
graph = builder.compile()
```

Second, your config (`langgraph.json`):
```json
{
  "dependencies": ["."],
  "graphs": {
    "my_agent": "./agent.py:graph"
  },
  "env": ".env"
}
```

Then you run `langgraph dev` and it automatically spins up a REST API on port 8123 and a Studio UI on port 2024.

## 6. Real-World Engineering Example

Bro, if you want to deploy a custom FastAPI server instead of using LangGraph Server (maybe because you have massive custom authentication logic), you must use async execution and SSE (Server-Sent Events) for streaming.

```python
import asyncio
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver
from psycopg_pool import AsyncConnectionPool
from agent import builder # Your StateGraph builder

app = FastAPI()
DB_URI = "postgresql://user:pass@db:5432/agents"
pool = AsyncConnectionPool(DB_URI)
checkpointer = AsyncPostgresSaver(pool)
graph = builder.compile(checkpointer=checkpointer)

class ChatRequest(BaseModel):
    thread_id: str
    message: str

async def stream_generator(thread_id: str, message: str):
    config = {"configurable": {"thread_id": thread_id}}
    input_state = {"messages": [("user", message)]}

    # Use astream_events for granular streaming
    async for event in graph.astream_events(input_state, config, version="v1"):
        kind = event["event"]

        # Stream LLM tokens as they arrive
        if kind == "on_chat_model_stream":
            content = event["data"]["chunk"].content
            if content:
                yield f"data: {content}\n\n"

        # Stream tool executions
        elif kind == "on_tool_start":
            tool_name = event["name"]
            yield f"data: [Calling tool: {tool_name}]\n\n"

@app.post("/chat/stream")
async def chat_stream(req: ChatRequest):
    return StreamingResponse(
        stream_generator(req.thread_id, req.message),
        media_type="text/event-stream"
    )

@app.on_event("startup")
async def startup():
    await checkpointer.setup()

@app.on_event("shutdown")
async def shutdown():
    await pool.close()
```

Because FastAPI and Uvicorn use async Python (`asyncio`), a single worker process can handle hundreds of concurrent agent executions. When an agent is waiting for OpenAI to respond (which takes seconds), the worker yields and processes another agent's request.

The checkpointer is critical here. It uses row-level locking in Postgres to ensure that if the user clicks a button twice quickly, Worker 1 and Worker 2 don't try to update the exact same graph state at the exact same time, which would corrupt the memory.

## 8. Performance Table

| Component | Bottleneck Risk | Mitigation Strategy |
|---|---|---|
| Postgres Checkpointer | High (Connection limits) | Use `AsyncConnectionPool` / PgBouncer |
| LLM API Calls | Very High (Rate limits, latency) | Stream responses, implement fallback models |
| Tool Executions | Medium (Network I/O) | Make tools completely `async def` |
| Python Event Loop | Low | Do not use `time.sleep()`, use `asyncio.sleep()` |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| Why must you use an asynchronous database checkpointer in production? | If you use an in-memory saver, state is lost when the pod restarts. If you use a synchronous DB checkpointer, it will block the entire Python event loop while writing to the DB, destroying your server's concurrency. |
| What is the difference between LangGraph Server and a custom FastAPI app? | LangGraph Server is a pre-built Docker image that automatically exposes endpoints for streaming, background runs, and state management based on a `langgraph.json`. Custom FastAPI requires writing all that boilerplate manually. |
| How do you handle long-running agents that take 5 minutes to finish? | Do not hold the HTTP request open. Use background tasks or LangGraph Server's async API to return a `run_id` immediately, then let the frontend poll for the status or stream events via a separate WebSocket/SSE connection. |
| What happens if an agent crashes mid-execution in production? | The checkpointer has saved the state of the *last successfully completed node*. You can resume the graph from that exact state without losing prior work. |
| How do you update a LangGraph agent's code without breaking ongoing sessions? | You must version your graph logic or use LangGraph Server's built-in deployment revision system to ensure old threads finish on the old code while new threads use the new code. |

## 10. Tricky Questions & Edge Cases

Bro, what happens if you deploy an agent and use `requests.get()` inside one of your tools?

**Answer:** You will bottleneck your entire server. `requests` is a synchronous library. If it takes 2 seconds to fetch a URL, the entire Uvicorn worker process freezes for 2 seconds. No other users can be served by that worker. You **MUST** use async HTTP libraries like `httpx.AsyncClient` or `aiohttp` inside tools for production deployments.

## 11. Real-World Usage

Companies deploying LangGraph to production usually use Kubernetes (EKS/GKE):
- **Image:** They use `langgraph build` to generate a Dockerfile.
- **Database:** They connect it to Amazon RDS (Postgres).
- **Caching:** They put Redis in front for semantic caching (to avoid paying OpenAI for identical questions).
- **Scale:** They run 5-10 replicas of the LangGraph server, behind an AWS Application Load Balancer.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Do write all tools as `async def` functions so they don't block the event loop. | Don't put heavy CPU tasks (like video encoding) inside an agent node without offloading it to a queue like Celery. |
| Do implement strict API timeouts on LLM calls so a degraded provider doesn't lock up your agent. | Don't expose your LangGraph endpoints directly to the internet; put them behind an API Gateway for auth and rate limiting. |

## 13. Production Notes

> **Production Warning:**
> Model fallbacks are mandatory. OpenAI and Anthropic go down. If your graph relies entirely on `ChatOpenAI(model="gpt-4o")`, your whole app crashes when they have an outage. You must use `ChatOpenAI(...).with_fallbacks([ChatAnthropic(...)])` so your agent seamlessly routes to a backup LLM provider during degraded performance.

## 14. Common Mistakes

| Mistake | Fix | Code Example |
|---|---|---|
| Blocking the event loop | Use async tools | `async def my_tool(): await httpx.get()` |
| Losing state on restart | Use Postgres Saver | `checkpointer = AsyncPostgresSaver(pool)` |
| Storing unpicklable objects | Keep state pure | Don't put DB connection objects inside `State` |
