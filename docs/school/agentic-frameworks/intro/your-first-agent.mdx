## 2. Engineering Story

A bootcamp graduate joined an AI startup and was immediately handed a task: "Build an agent that answers weather questions." He opened up LangChain's docs, saw fifteen abstraction layers, and spent three days confused. His tech lead suggested a different approach: "Before you touch any framework, build the loop yourself."

He wrote 40 lines of raw Python — a `while True` loop that called the OpenAI API, checked if the response contained a tool call, executed the `get_weather()` function manually if it did, and fed the result back. It was ugly. It was verbose. But when it finally worked, something clicked. He understood exactly what an "agent" was: an LLM in a loop with tools. Every framework he touched after that felt transparent instead of magical. That 40-line script was worth more than any tutorial he'd read.

## 4. Concept Explanation

To build your first agent, we need to assemble the core anatomy we discussed previously:
1. **The Brain:** The OpenAI API.
2. **The Tool:** A simple Python function that returns the weather.
3. **The Memory:** A Python list containing the conversation history.
4. **The Loop:** A `while` loop that passes the memory to the brain, checks if it wants to use the tool, executes it, and repeats until finished.

By doing this without abstractions, you will understand exactly what "Function Calling" actually looks like under the hood.

## 5. Syntax Table

| Step | Code |
|---|---|
| **1. Define Tool Schema** | `tools = [{"type": "function", "function": {"name": "get_weather", ...}}]` |
| **2. Pass to LLM** | `client.chat.completions.create(..., tools=tools)` |
| **3. Check for Action** | `if response.tool_calls: ...` |
| **4. Append Result** | `messages.append({"role": "tool", "content": result})` |

## 6. Beginner Example

Before the full engineering example, here is the tool we will give the agent. It's just a dummy python function.

```python
def get_weather(location: str):
    """Returns the weather for a given location."""
    # In a real app, this hits the OpenWeather API
    if "chicago" in location.lower():
        return "Windy and 40 degrees."
    return "Sunny and 75 degrees."
```

## 7. Real-World Engineering Example

Bro, here is the complete, runnable code for an agent from scratch. Copy this, put in your OpenAI API key, and run it.

```python
import json
import os
from openai import OpenAI

# 1. Setup
client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))

# 2. Define the actual Python function (The Actuator)
def get_weather(location: str):
    print(f"\n[TOOL EXECUTED] Fetching weather for {location}...")
    if "chicago" in location.lower():
        return "Windy and 40 degrees."
    return "Sunny and 75 degrees."

# 3. Define the JSON Schema so the LLM knows how to call it
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get the current weather in a given location",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {
                        "type": "string",
                        "description": "The city and state, e.g. Chicago, IL"
                    }
                },
                "required": ["location"]
            }
        }
    }
]

# 4. Initialize Memory
messages = [
    {"role": "system", "content": "You are a helpful weather assistant."},
    {"role": "user", "content": "Should I wear a jacket in Chicago today?"}
]

# 5. The Orchestration Loop
print("Agent: Thinking...")
while True:
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=messages,
        tools=tools
    )

    response_msg = response.choices[0].message
    messages.append(response_msg) # Save the thought to memory

    # Did the LLM decide to call a tool?
    if not response_msg.tool_calls:
        # No tool called, we have our final answer!
        print(f"\nAgent Final Answer: {response_msg.content}")
        break

    # Yes, it called a tool. Execute it.
    for tool_call in response_msg.tool_calls:
        # Parse the JSON arguments the LLM generated
        args = json.loads(tool_call.function.arguments)

        # Route to our python function
        if tool_call.function.name == "get_weather":
            result = get_weather(args["location"])

            # Feed the observation back to the LLM
            messages.append({
                "tool_call_id": tool_call.id,
                "role": "tool",
                "name": "get_weather",
                "content": result
            })

    print("Agent: Thinking about the tool result...")
```

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why do we append the `response_msg` to `messages` before executing the tool? | The LLM needs the full context. If you don't save its decision to use the tool, when you pass the tool result back, the LLM won't remember asking for it and will throw an error. |
| What happens if `json.loads(tool_call.function.arguments)` fails? | The agent crashes. In a robust system, you would wrap that in a `try/except` block, catch the `JSONDecodeError`, and pass the error string back to the LLM as a tool result so it can try again. |
| Why is `tool_call.id` required when passing the observation back? | OpenAI allows parallel tool calling (calling 3 tools at once). The ID maps the specific tool result to the specific tool request so the LLM doesn't get confused. |

## 11. Tricky Questions & Edge Cases

Bro, what if the user says: "What's the weather in Chicago and Miami?"
With standard function calling, the LLM might generate *two* `tool_calls` in a single response array.
Our `for tool_call in response_msg.tool_calls:` loop handles this beautifully! It will execute both, append two tool results to the memory, and only then loop back to the LLM.
**Edge case warning:** If your loop just executes `tool_calls[0]` and ignores the rest, your agent will silently fail to check Miami.

## 12. Real-World Usage

This exact raw SDK loop is used in production by teams who are allergic to framework bloat.
- Vercel AI SDK uses a very similar raw loop under the hood for their TS/JS implementations.
- Teams building high-throughput data extraction agents often stick to the raw SDK to minimize latency.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Print `[TOOL EXECUTED]` logs to your terminal so you can see exactly when the agent is acting vs thinking. | Don't assume the LLM will always return valid JSON in the arguments. Always handle parse errors. |
| Give your tools highly descriptive `description` fields. This is how the LLM knows when to use it. | Don't write descriptions like "Does a thing." Write "Calculates the exact weather for a given US city." |

## 14. Production Notes

:::caution Production Warning
Bro, this script loops `while True`. If you give the agent a bad tool that always returns an error, the LLM will just keep trying to call it forever, draining your API credits. In production, always implement a `loop_count` variable and break if `loop_count > 5`.
:::

## 15. Common Mistakes

| Mistake | Why it's bad | The Fix |
|---|---|---|
| Forgetting the `"role": "tool"` message format | The OpenAI API will reject your request with a 400 error because the message history is malformed. | Strictly follow the `{"tool_call_id": id, "role": "tool", "name": name, "content": result}` schema. |
| Returning a Python dictionary directly as `content` | The API expects the tool `content` to be a string. | Always cast your tool results using `str(result)` or `json.dumps(result)` before appending. |
