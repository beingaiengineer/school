# What Are AI Agents?

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | AI Agents |
| Difficulty | Beginner |
| Used For | Building autonomous systems that can reason and take action to achieve goals |
| Common Mistake | Confusing a simple LLM chat prompt with a true agentic workflow |
| Performance | Varies wildly; dominated by network latency of LLM API calls and tool executions |

## 2. Real-World Analogy

![Illustration of Bae as a new IT employee](../assets/intro/what-are-ai-agents.png)

| Human World (Corporate IT Team) | AI Agent Equivalent |
|---|---|
| The IT Manager receiving the ticket | The Orchestrator LLM |
| Thinking: "I need to check the DB, then email the user" | Reasoning & Planning (Chain of Thought) |
| Running SQL queries or Bash scripts | Tool Execution (Function Calling) |
| Reading the SQL output and deciding the next step | Observation & Next Action |
| Sending the final email resolution | Final Generation & Handoff |

Bro, think about when a new employee joins. You don’t write a script that does everything perfectly in one go. You give the IT guy a goal ("Onboard Bob") and a set of tools (Active Directory, Google Workspace admin, Slack API). The IT guy *reasons* about what to do first, *uses a tool*, *observes* the result, and then decides what to do next. That’s exactly what an AI agent is.

## 3. Concept Explanation

An AI Agent is a system where a Large Language Model (LLM) acts as the central reasoning engine, directing a loop of thought, action, and observation. It doesn't just generate text; it makes decisions about *which* tools to use, *when* to use them, and *how* to interpret their output to progress toward a goal.

Why do we need this? Because LLMs are frozen in time and trapped in a box. They don't know today's weather, they can't query your production database, and they can't restart a server. By wrapping the LLM in an agentic loop and giving it tools (APIs, calculators, code interpreters), you give it hands and eyes in the real world.

## 4. Syntax Table

*Note: Since "Agents" are a conceptual pattern, the syntax depends heavily on the framework. Here's a high-level pseudo-code pattern.*

| Concept | Typical Implementation |
|---|---|
| **System Prompt** | `You are a helpful assistant. You have access to these tools...` |
| **Tool Definition** | `@tool def get_weather(location: str): ...` |
| **Agent Loop** | `while not task_complete: action = llm.think(); result = execute(action)` |

## 5. Beginner Example

Here is a conceptual look at how an agent might be invoked using a typical framework like LangChain.

```python
from langchain.agents import initialize_agent, AgentType
from langchain.chat_models import ChatOpenAI
from my_tools import web_search, calculator

llm = ChatOpenAI(temperature=0)
tools = [web_search, calculator]

# We give the LLM tools and a specific reasoning strategy
agent = initialize_agent(
    tools,
    llm,
    agent=AgentType.ZERO_SHOT_REACT_DESCRIPTION,
    verbose=True
)

# The agent figures out it needs to search first, then calculate
agent.run("What is the current stock price of Apple divided by 2?")
```

## 6. Real-World Engineering Example

Bro, let's talk production. Imagine an automated Incident Response agent. It shouldn't just guess; it needs to check Datadog, read logs, and maybe page someone.

```python
import os
from openai import OpenAI
import json

client = OpenAI()

def query_datadog_logs(service: str):
    """Simulated tool to fetch recent error logs."""
    # In reality, this hits the Datadog API
    return f"Found 500 errors in {service}. DB connection timed out."

def page_oncall(service: str, message: str):
    """Simulated tool to page PagerDuty."""
    return f"Paged on-call for {service} with message: {message}"

# Define the tools for the LLM
tools = [
    {
        "type": "function",
        "function": {
            "name": "query_datadog_logs",
            "description": "Get recent logs for a service",
            "parameters": {
                "type": "object",
                "properties": {"service": {"type": "string"}},
                "required": ["service"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "page_oncall",
            "description": "Page the on-call engineer if things are critical",
            "parameters": {
                "type": "object",
                "properties": {
                    "service": {"type": "string"},
                    "message": {"type": "string"}
                },
                "required": ["service", "message"]
            }
        }
    }
]

# The agent loop
messages = [{"role": "user", "content": "The payment API is down. Investigate and page if needed."}]

response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    tools=tools
)

# The LLM decides what tool to call based on its reasoning!
if response.choices[0].message.tool_calls:
    print("Agent decided to take an action:", response.choices[0].message.tool_calls[0].function.name)
```

## 8. Performance Table

| Operation | Typical Latency | Notes |
|---|---|---|
| Simple LLM Call | 500ms - 2s | Just getting the next reasoning step |
| Web Search Tool | 1s - 3s | External API dependency |
| Full Agent Loop | 5s - 20s | Multiple LLM calls + multiple tool executions |
| Complex Multi-Agent | 30s+ | Handoffs between specialized agents take serious time |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between an LLM and an AI Agent? | An LLM is just a model that predicts text. An agent uses the LLM to decide on actions, execute tools, and observe results in a loop. |
| What is "function calling" in the context of agents? | It's the ability of the LLM to output structured data (like JSON) matching a predefined signature, which your code then uses to execute a real function. |
| How do you prevent an agent from getting stuck in an infinite loop? | You implement a `max_iterations` limit in your orchestration loop, or use a framework that tracks execution graphs and halts after a certain depth. |
| What is the ReAct pattern? | Reason + Act. The prompt forces the LLM to output its "Thought", then its "Action", then observe the result before continuing. |
| Why might an agent fail even if the LLM is smart? | Poor tool descriptions, bad system prompts, external API failures, or context window limits being exceeded during the loop. |

## 10. Tricky Questions & Edge Cases

Bro, what happens when a tool returns an error?
A naive implementation just crashes. A proper agent takes the error string (e.g., `TimeoutException from DB`), feeds it *back* into the LLM as an observation, and the LLM reasons: *"Ah, the DB timed out. Let me try a different query or retry."*

Another tricky case: The context window filling up. If an agent loops 20 times, the conversation history grows massively. Eventually, the LLM forgets the original goal, or you hit token limits.

## 11. Real-World Usage

Companies use agents to automate complex backend workflows.
- **DevOps:** Agents that trigger on PagerDuty alerts, read logs, summarize the blast radius, and propose rollback commands.
- **Customer Support:** Resolving tickets by actually calling refund APIs in Stripe, not just generating polite apologies.
- **Data Engineering:** Agents that take natural language questions, generate SQL, run it against Snowflake, fix any syntax errors, and generate a chart.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Give tools extremely clear, distinct descriptions. | Don't give overlapping tools (e.g., `search_google` and `search_bing`). |
| Implement timeouts and hard limits on loop iterations. | Don't let an agent loop indefinitely if a tool keeps failing. |
| Use structured output (JSON/Pydantic) for tool arguments. | Don't parse raw text with regex to trigger functions. |

## 13. Production Notes

:::caution Production Warning
Bro, agents are inherently non-deterministic. A simple prompt tweak can completely change the path an agent takes. You MUST have robust observability. Log every thought, every tool input, and every tool output. If your agent goes rogue and deletes a user record, you need the exact trace of *why* it decided to do that.
:::

## 14. Common Mistakes

| Mistake | Why it's bad | The Fix |
|---|---|---|
| Assuming the LLM will format JSON perfectly every time | It will eventually hallucinate a missing comma or quote. | Use a framework (like Pydantic AI or OpenAI Structured Outputs) that enforces schema. |
| Giving the agent destructive tools (like `DROP TABLE`) without humans in the loop | The agent might misunderstand a prompt and execute a catastrophic action. | Implement "Human-in-the-loop" (HITL) approval steps before critical tool execution. |
