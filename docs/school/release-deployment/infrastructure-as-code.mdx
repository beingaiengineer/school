# Infrastructure as Code (IaC)

### 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Infrastructure as Code (IaC) |
| Difficulty | Intermediate |
| Used For | Managing cloud resources through code instead of manual clicks |
| Common Mistake | Clicking around in the AWS console to fix a broken server |
| Performance | Massive time savings when scaling from 1 to 100 servers |

### 2. Engineering Story

Bro, you just joined a startup. The lead engineer tells you, "We need a Staging environment. Just log into AWS, create an EC2 instance, set up a VPC, attach a Postgres RDS, configure the security groups, open port 443 on the load balancer, and make sure IAM roles are set." You spend three days clicking through 40 different AWS screens. Six months later, the database is mysteriously accessible to the public internet because someone misclicked a checkbox during a late-night debugging session.

Worse, nobody knows *who* clicked it, *when* they clicked it, or *what* the previous setting was. Your infrastructure is a fragile house of cards built on undocumented manual actions. Infrastructure as Code (IaC) fixes this by forcing you to define every server, network, and database in code, review it in Git, and deploy it automatically.

### 3. Real-World Analogy

Think of Infrastructure as Code like ordering a custom house, bro.

| Manual Infrastructure (ClickOps) | Infrastructure as Code |
|---|---|
| Telling the builder verbally where every brick goes | Handing the builder a detailed architectural blueprint |
| If the house burns down, you have to remember how to rebuild it | If the house burns down, you just hand the blueprint to a new builder |
| Can't review changes before they happen | The city approves the blueprint before construction begins |
| Difficult to build 10 identical houses | Just photocopy the blueprint 10 times |

### 4. Concept Explanation

Infrastructure as Code (IaC) is the management of infrastructure (networks, virtual machines, load balancers, and connection topology) in a descriptive model, using the same versioning as DevOps team uses for source code.

We care about this because it eliminates "Configuration Drift" (where environments slowly become mismatched). It allows infrastructure to be reviewed via Pull Requests, tested automatically, and destroyed to save money when not needed. Instead of clicking buttons in a UI, you write a configuration file, and a tool provisions exactly what you described.

### 5. Syntax Table

| IaC Type | Focus | Example Tools |
|---|---|---|
| Declarative Provisioning | "I want 3 servers that look like X" | Terraform, AWS CloudFormation |
| Configuration Management | "Install Nginx and Python on these existing servers" | Ansible, Chef, Puppet |
| Programmatic | Write standard code (Python/TS) to generate infra | AWS CDK, Pulumi |

### 6. Beginner Example

Instead of logging into a console to create a server, you define it in a file. Here is a simple Terraform example to create an AWS EC2 instance.

```hcl
# This tells Terraform we are using AWS
provider "aws" {
  region = "us-east-1"
}

# This is the "blueprint" for our server
resource "aws_instance" "my_web_server" {
  ami           = "ami-0c55b159cbfafe1f0" # Ubuntu Linux
  instance_type = "t2.micro"              # Small, cheap server

  tags = {
    Name = "Production-Web-01"
    Team = "Engineering"
  }
}
```

### 7. Real-World Engineering Example

In the real world, you combine resources into reusable modules. Here, we define a Load Balancer that routes traffic to a specific security group.

```hcl
resource "aws_security_group" "web_sg" {
  name        = "allow_web_traffic"
  description = "Allow inbound web traffic"

  ingress {
    description = "HTTP from anywhere"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

resource "aws_elb" "web_elb" {
  name            = "production-elb"
  security_groups = [aws_security_group.web_sg.id]
  # ... other configurations
}
```

### 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between Declarative and Imperative IaC? | Imperative means writing scripts (`do X, then do Y`). Declarative (like Terraform) means stating the desired end state (`I want 3 servers`), and the tool figures out how to make it happen. |
| What is "Configuration Drift"? | When the actual state of the cloud infrastructure deviates from the code definition (e.g., someone manually changes a setting in the AWS console). |
| What is an IaC "State File"? | A file that maps the resources defined in your code to the actual real-world resources in the cloud provider. It is the single source of truth. |
| Can you test Infrastructure as Code? | Yes. Tools like `terratest` or `tfsec` allow you to write unit tests and security scans against your infrastructure code before it is deployed. |
| Why use Pulumi or CDK instead of Terraform? | If your team hates learning domain-specific languages (like HCL) and prefers writing infrastructure using Python or TypeScript, Pulumi/CDK provides that programmatic flexibility. |

### 11. Tricky Questions & Edge Cases

**Bro, you delete the state file by accident. Your infrastructure code is safe in Git, and the servers are still running in AWS. What happens when you run `apply`?**
Chaos. Because the state file is gone, the IaC tool forgets that it built those servers. It reads your code (which says "create 3 servers") and attempts to create 3 brand *new* servers, potentially causing name collisions or double billing. The state file is critical and must be stored securely (e.g., in an S3 bucket with locking).

### 12. Real-World Usage

When scaling up for a massive event (like a Black Friday sale), companies don't manually launch 100 new servers. They simply change a variable `server_count = 100` in their IaC repository, merge the PR, and the pipeline automatically provisions the exact infrastructure needed. After the sale, they change it back to `10`, and the extra servers are cleanly destroyed.

### 13. Best Practices

| DO | DON'T |
|---|---|
| Revoke human write access to the Cloud Console. All changes must go through the IaC pipeline. | Allow "just one quick manual fix" in production. |
| Store state files remotely (S3, Terraform Cloud) with state locking enabled so two pipelines don't run at once. | Commit state files (which contain plain-text database passwords) into Git. |

### 14. Production Notes

<Admonition type="caution" title="The Blast Radius">
Because IaC makes it easy to build 1,000 servers in a minute, it also makes it easy to delete 1,000 servers in a minute with a single typo. Always run a "plan" (dry-run) and review what resources will be destroyed before executing the "apply".
</Admonition>

### 15. Common Mistakes

| Mistake | Correction |
|---|---|
| Hardcoding secrets in IaC | Use a secrets manager (AWS Secrets Manager, HashiCorp Vault) and reference them dynamically in your code. |
| Putting the entire company's infrastructure in one file | Break infrastructure down into smaller modules (e.g., Networking, Databases, App Servers) to limit the blast radius. |
