# Incident Communication

Bro, fixing the database is only half the job. If the CEO thinks the company is still burning down because you forgot to tell them you put the fire out, you have failed the incident. Communication is mitigation.

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Incident Communication |
| Difficulty | Intermediate |
| Used For | Keeping stakeholders informed during an outage |
| Common Mistake | Going dark (silent) while trying to debug the problem |
| Performance | Reduces inbound tickets and executive panic |

## 2. Engineering Story

A team of engineers recently faced a critical challenge related to this concept. Their existing processes were failing under the load of thousands of concurrent users, and manual workarounds were causing major delays in deployment. By deeply understanding and correctly implementing this concept, the lead engineer was able to architect a solution that not only resolved the immediate bottleneck but also paved the way for massive scalability. This transformation turned a chaotic, error-prone system into a resilient, automated powerhouse.

## 3. Real-World Analogy

| Delayed Train | Incident Equivalent |
|---|---|
| The train breaks down | The production database goes down |
| Train driver fixes the engine | Engineer fixes the database |
| **Passengers are sitting in the dark wondering if they will die** | **Customers are submitting angry support tickets** |
| The conductor announces: "We have an engine fault, working on it, 10 min delay" | The Status Page is updated |

Bro, a 10-minute delay with an announcement is mildly annoying. A 10-minute delay in total silence causes a riot. You must be the conductor.

## 4. Concept Explanation

Incident communication is the structured process of updating internal stakeholders (executives, customer support, other engineering teams) and external stakeholders (customers, users) about the status of an ongoing incident.

The goal is to provide **predictable, accurate, and concise** updates. You don't need to explain the technical details of a memory leak to the marketing team; you just need to tell them that the checkout page is broken and you expect an update in 15 minutes.

## 5. Syntax Table

| Audience | Medium | Content Focus |
|---|---|---|
| **War Room (Internal)** | Slack / Zoom | Deeply technical. Logs, metrics, hypotheses. |
| **Execs/Support (Internal)** | Email / Dedicated Slack Channel | Business impact, current status, next update time. |
| **Customers (External)** | Public Status Page | Apologetic, transparent, non-technical impact summary. |

## 6. Beginner Example

A typical internal status update template used by the Communications Lead:

```markdown
**[UPDATE 1] SEV-1: Checkout API Failing**
- **Status**: Investigating
- **Impact**: Users cannot complete purchases. Approx 500 errors/min.
- **Current Action**: Rolling back the recent deployment to v1.4.2.
- **Next Update**: In 15 minutes (or upon resolution).
```

## 7. Real-World Engineering Example

Many companies automate the communication process using chatbots to ensure formatting is consistent and sent to the right channels.

```bash
# Example usage of a custom Slack bot
/incident update
> Status: Mitigating
> Message: We have identified the bad DB query. Reverting the PR now.
> Next Update: 10m

# The bot then posts this to #engineering-general, emails the execs,
# and updates the internal Jira ticket automatically.
```

## 9. Performance Table

| Action | Target Timeframe |
|---|---|
| Initial acknowledgement (Internal) | < 5 mins from incident start |
| First public status page update | < 15 mins from incident start |
| Regular internal updates (SEV-1) | Every 15-30 mins |
| Resolution announcement | Immediately upon confirmed fix |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Who is responsible for sending updates during an incident? | The Communications Lead, who is instructed by the Incident Commander. |
| Why do we promise a "Next Update" time? | It stops executives and support teams from constantly pinging the war room asking "Is it fixed yet?". |
| Should you lie on the public status page to save face? | Never. Trust is hard to build and easy to lose. Be transparent about the outage. |
| How technical should an executive update be? | Not very. Focus on what is broken for the user, what you are doing to fix it, and when you will update them next. |
| What happens if you miss your promised update time? | You lose trust. If you said 15 minutes, send an update in 15 minutes even if the update is "We are still looking." |

## 11. Tricky Questions & Edge Cases

- **The Security Breach**: If the incident is a data breach, do you update the public status page immediately? Usually NO. Security incidents require Legal and PR approval before public disclosure. The communication playbook for security is completely different from reliability.
- **The Empty Update**: You promised an update in 15 minutes, but the team has made zero progress. Do you stay silent until you have news? No, bro. Send the empty update: "Still investigating. No new findings. Next update in 15 mins." Consistency > Content.

## 12. Real-World Usage

Companies like GitHub and Cloudflare are famous for their excellent incident communication. When Cloudflare has an outage, they update their status page within minutes, clearly explaining which regions are affected. This stops thousands of engineers from debugging their own code and instead waiting for Cloudflare to resolve it.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Always include a "Next Update Time". | Leave people wondering when they will hear from you next. |
| Translate technical jargon into business impact. | Tell the CEO "The Redis OOM killed the pod." (Say: "Cache failure causing slow logins"). |
| Keep the public status page updated until the issue is fully resolved. | Fix the issue and forget to mark the status page as "Resolved." |

## 14. Production Notes

> **Warning**: A flooded support queue is an unmitigated incident. If your support team is receiving 1,000 tickets a minute because the site is down, your first communication priority is putting a banner on the website saying "We know it's broken." This immediately stops the ticket flood.

## 15. Common Mistakes

| Mistake | Correction |
|---|---|
| The IC trying to write the status updates themselves. | Delegate this to a Communications Lead. The IC must focus on the technical response. |
| Over-promising a fix time ("It will be fixed in 5 mins"). | Never promise a fix time. Promise an *update* time. |
| Using overly technical language externally. | Users don't care about your Kubernetes cluster. They care that they can't pay. |
