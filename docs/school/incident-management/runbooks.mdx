# Runbooks

Bro, at 3 AM your brain doesn't work. If you have to invent a solution while the database is on fire, you are going to make it worse. You need a recipe to follow. You need a runbook.

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Runbooks (or Playbooks) |
| Difficulty | Beginner |
| Used For | Providing step-by-step instructions to resolve known issues |
| Common Mistake | Writing runbooks that are out of date or require deep domain knowledge to understand |
| Performance | Drastically reduces Time to Mitigate (TTM) |

## 2. Engineering Story

A junior SRE joined a team on a Thursday. On Friday night, she was added to the on-call rotation as backup. On Saturday at 2am, her first page fired — high memory usage on a Kafka consumer causing message lag. She had never touched Kafka before. She spent 40 minutes reading Kafka documentation, trying random `kafka-consumer-groups.sh` commands, and pinging her senior colleague at 2am begging for help.

The senior engineer fixed it in 3 minutes: restart the consumer group and check for memory leak in the config. He stayed up for the next two hours writing a runbook for this exact scenario. Next month, when the same alert fired at 3am, the junior SRE opened the runbook: "Step 1: Confirm which consumer group is lagging using `kafka-consumer-groups.sh --describe`. Step 2: Check memory graph in Grafana. Step 3: If memory >85%, restart with `kubectl rollout restart deployment/kafka-consumer`." She resolved it in 4 minutes, alone, at 3am. A runbook is what happens when an expert's knowledge survives being an expert.

## 3. Real-World Analogy

| Medical Emergency | Incident Equivalent |
|---|---|
| Paramedic arrives at the scene | Engineer receives a page at 3 AM |
| Open the CPR instruction manual | Open the Alert Runbook |
| Check airway, do 30 compressions | Check metrics, run restart script |
| Patient stabilizes | Service recovers |

Bro, a paramedic doesn't invent a new way to do CPR on the spot. They follow the manual exactly. Your runbook is that manual.

## 4. Concept Explanation

A runbook (sometimes called a playbook) is a document containing step-by-step instructions on how to handle a specific alert or operational task.

When an alert fires, there should *always* be a link to a runbook attached to it. The runbook should be written so clearly that an engineer from a completely different team can follow the steps to mitigate the issue without understanding the underlying code.

If an alert doesn't have a runbook, it shouldn't be an alert.

## 5. Syntax Table

| Section | Description |
|---|---|
| **Context** | What does this service do, and why did this alert fire? |
| **Impact** | What are the users experiencing right now? |
| **Investigation** | Dashboards to look at, queries to run to confirm the issue. |
| **Mitigation** | Exact commands to run to stop the bleeding (e.g., rollback, restart). |
| **Escalation** | Who to page if the mitigation doesn't work. |

## 6. Beginner Example

A typical markdown runbook for a "High CPU" alert.

```markdown
# Runbook: API Gateway High CPU

## Context
The API Gateway is experiencing >90% CPU utilization.

## Impact
Requests may be dropping or experiencing high latency (>2 seconds).

## Investigation
1. Open the [API Gateway Dashboard](#link).
2. Check if the incoming request rate has spiked.

## Mitigation
If it is a traffic spike, auto-scaling should handle it. If auto-scaling failed:
1. Manually scale the deployment:
   `kubectl scale deployment api-gateway --replicas=10`
2. Wait 2 minutes and check the dashboard again.

## Escalation
If CPU does not drop, page the `@network-platform-team`.
```

## 7. Real-World Engineering Example

Modern systems try to automate runbooks. Instead of a human reading markdown and running a bash script, a tool runs the script automatically.

```yaml
# GitHub Actions Automated Runbook
name: "Auto-Mitigate Memory Leak"
on:
  repository_dispatch:
    types: [trigger-restart]

jobs:
  restart-service:
    runs-on: ubuntu-latest
    steps:
      - name: Restart Kubernetes Pods
        run: |
          kubectl rollout restart deployment/memory-hungry-service
          echo "Service restarted automatically due to memory alert."
```

## 9. Performance Table

| Scenario | Time to Mitigate |
|---|---|
| No Runbook (Inventing solution) | 30 - 120+ minutes |
| Manual Runbook (Reading markdown) | 5 - 15 minutes |
| Automated Runbook (Scripted response) | < 1 minute |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is the most important part of an alert? | The link to the runbook. Without it, the alert is just noise. |
| Who should write the runbook? | The engineer who built the service or configured the alert. |
| Can a runbook contain instructions to debug code? | No. Runbooks are for mitigation (stopping the bleeding). Debugging is for the root cause analysis later. |
| How do you ensure runbooks stay up to date? | Practice "Game Days" or chaos engineering where you follow the runbook in a controlled environment to verify it still works. |
| What if the runbook fails? | The runbook must have an "Escalation" section telling the engineer exactly who to page next. |

## 11. Tricky Questions & Edge Cases

- **The Stale Runbook**: An alert fires, the engineer runs the command in the runbook, and it destroys the database because the architecture changed 6 months ago but the runbook didn't. Solution: Treat runbooks as code. They must be reviewed and tested when the system changes.
- **The "Automate Everything" Trap**: If a runbook is just "Run script X", why is a human doing it? The script should run automatically when the alert fires. Only page a human if the automated runbook fails.

## 12. Real-World Usage

Companies like Netflix and Amazon rely heavily on automated runbooks. They use tools like AWS Systems Manager Run Command or Jenkins to execute predefined scripts when alerts fire, often resolving the issue before a human even sees the page.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Make commands copy-pasteable. | Write vague instructions ("Restart the server"). |
| Include links directly to the relevant dashboards. | Assume the engineer knows where to find the metrics. |
| Review runbooks during postmortems to see if they need updates. | Write a runbook once and never look at it again. |

## 14. Production Notes

> **Warning**: Never put sensitive credentials or passwords inside a runbook document. Runbooks should reference secure secret managers or rely on the engineer's existing IAM permissions.

## 15. Common Mistakes

| Mistake | Correction |
|---|---|
| Writing runbooks like a novel. | Use bullet points, bold text, and code blocks. Make it skimmable. |
| Assuming domain knowledge. | Write the runbook so a new hire on their first day could execute it. |
| Not linking the runbook in the alert. | Every single alerting rule configuration must have a `runbook_url` field. |
