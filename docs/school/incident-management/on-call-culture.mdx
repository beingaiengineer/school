# On-Call Culture

Bro, carrying the pager is a rite of passage. But it shouldn't feel like a punishment. A healthy on-call culture is the difference between an engineering team that thrives and one that burns out and quits.

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | On-Call Culture |
| Difficulty | Intermediate |
| Used For | Managing human response to production systems |
| Common Mistake | Leaving on-call to a single "hero" engineer |
| Performance | Reduces time-to-mitigation and prevents burnout |

## 2. Real-World Analogy

| Night Watch | On-Call Equivalent |
|---|---|
| Guarding the fort while everyone sleeps | Being on-call after hours |
| Ringing the bell if enemies attack | Responding to a PagerDuty alert |
| Waking up the commander | Escalating a SEV-1 incident |
| Handing over the shift in the morning | Passing the pager to the next engineer |

Bro, you don't want the same guard working 24/7. They will fall asleep on duty. You need a rotation, clear instructions, and backup.

## 3. Concept Explanation

On-call culture defines how an organization manages the responsibility of supporting production systems outside of (and during) normal business hours.

The modern standard is "You build it, you run it." The developers who write the code are the ones on call for it. This creates a powerful feedback loop: if you write brittle code, you are the one who gets woken up at 3 AM.

A healthy culture ensures on-call is a shared, supported, and compensated responsibility, not a burden dumped on the junior dev.

## 4. Syntax Table

| Term | Description |
|---|---|
| **Primary On-Call** | The first person paged when an alert fires. |
| **Secondary On-Call** | The backup person paged if the Primary doesn't acknowledge within a set time. |
| **Rotation** | The schedule dictating who is on-call and when (e.g., weekly shifts). |
| **Shift Handoff** | A meeting/sync where the outgoing on-call updates the incoming on-call. |

## 5. Beginner Example

A typical weekly on-call schedule configuration in a tool like PagerDuty:

```json
{
  "schedule": {
    "name": "Backend Services Primary",
    "rotation_type": "weekly",
    "handoff_time": "Monday 10:00 AM",
    "users": [
      "alice@company.com",
      "bob@company.com",
      "charlie@company.com"
    ]
  }
}
```

## 6. Real-World Engineering Example

In production, you don't want alerts going into a black hole if someone sleeps through their alarm. You build escalation policies.

```yaml
# Escalation Policy Example
name: "Critical Backend Incidents Escalation"
rules:
  - step: 1
    target: "Primary On-Call Engineer"
    timeout: "5 minutes"
  - step: 2
    target: "Secondary On-Call Engineer"
    timeout: "10 minutes"
  - step: 3
    target: "Backend Engineering Manager"
    timeout: "15 minutes"
  - step: 4
    target: "VP of Engineering"
    timeout: "30 minutes"
```

## 8. Performance Table

| Metric | Healthy Culture | Toxic Culture |
|---|---|---|
| Alert Volume | Few, highly actionable | Constant noise, alert fatigue |
| Handoffs | Documented, structured meeting | "Good luck bro, it's yours now" |
| Sleep Interruption | Rare (1-2 times a month) | Frequent (multiple times a week) |
| Post-Incident Action | Blameless review, fix root cause | Blame the engineer, fix nothing |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What does "You build it, you run it" mean? | Developers are responsible for operating their own code in production, including being on-call for it. |
| How do you prevent alert fatigue? | Regularly audit alerts. If an alert fires and requires no human action, delete it or downgrade it to an email. |
| What is an escalation policy? | A defined sequence of who to contact next if the primary person on-call does not respond. |
| What is the purpose of an on-call handoff? | To pass context to the next person: ongoing issues, disabled alerts, and pending deployments. |
| Should managers be in the on-call rotation? | They should be the final escalation point to handle coordination, but usually shouldn't be the primary technical responder. |

## 10. Tricky Questions & Edge Cases

- **The Hero Developer**: One engineer knows everything and always fixes things fast. Is this good? No. That's a single point of failure. If they quit or go on vacation, the team is doomed. Share the knowledge.
- **Off-Hours Deployments**: If a team in India deploys code at 2 PM IST, but it breaks a service owned by a US team at 3 AM EST, who gets paged? The US team. To fix this, teams must coordinate global rollouts or own services geographically.

## 11. Real-World Usage

Companies like Amazon mandate that every software engineer goes on call. It forces developers to care about operability, logging, and metrics. If your code is hard to debug, you will be the one suffering at 2 AM trying to figure it out.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Give engineers time off (TOIL relief) after a heavy on-call shift. | Expect an engineer to work a full day after being up all night. |
| Have a clear, structured handoff meeting. | Put new hires on call without a shadowing period. |
| Ensure there is always a secondary on-call as backup. | Rely on one "hero" to solve every incident. |

## 13. Production Notes

> **Warning**: Burnout is real. If your on-call engineers are getting paged more than 2-3 times a week outside of working hours, your system is too unstable. Stop building new features and fix the reliability.

## 14. Common Mistakes

| Mistake | Correction |
|---|---|
| Paging the on-call for disk usage at 80%. | Unless it will hit 100% in the next hour, make it a ticket for the morning. |
| Silent handoffs with no context. | Implement a 15-minute handoff sync every week. |
| Blaming the engineer who broke production. | Blame the system that *allowed* the engineer to break production. |
