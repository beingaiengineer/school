## 2. Engineering Story

In a heavy support or Site Reliability Engineering (SRE) environment, issues come in unpredictably. Prod might go down on Tuesday, and a critical security patch might be needed on Thursday. You can't plan a rigid 2-week Scrum Sprint if your priorities change daily.

This is where Kanban shines. It's all about continuous flow. Instead of timeboxing work, you limit Work In Progress (WIP) so your engineers don't start 50 things and finish none. Kanban is how mature ops, infrastructure, and continuous delivery teams manage a never-ending stream of tasks without burning out. It focuses on getting a single task from 'To Do' to 'Done' as quickly as possible.

## 4. Concept Explanation

Kanban (Japanese for "visual signal" or "card") is a lean method to manage and improve work across human systems. It does not use fixed timeboxes (like Sprints).

The core principles are:
1. **Visualize the workflow:** Use a board with columns (e.g., Backlog, Doing, Review, Done).
2. **Limit Work in Progress (WIP):** A column can only hold a maximum number of cards. This forces the team to finish old tasks before starting new ones.
3. **Manage flow:** Identify bottlenecks (e.g., if the 'Review' column is always full, you need more reviewers).
4. **Make policies explicit:** Define exactly what it means to move a card from one column to the next.

It operates on a "pull" system. When an engineer finishes a task, they pull the next one from the top of the prioritized backlog.

## 5. Syntax Table

| Concept | Description |
|---|---|
| WIP Limit | The maximum number of active tasks allowed in a stage |
| Lead Time | Time from when a request is made until it is delivered |
| Cycle Time | Time from when work *starts* on a request until it is delivered |

## 6. Beginner Example

A simple simulation of a pull system with WIP limits:

```python
class KanbanBoard:
    def __init__(self, wip_limit):
        self.wip_limit = wip_limit
        self.in_progress = []

    def pull_task(self, task):
        if len(self.in_progress) < self.wip_limit:
            self.in_progress.append(task)
            print(f"Started working on {task}")
        else:
            print(f"WIP Limit reached! Cannot start {task} until something is finished.")

    def finish_task(self, task):
        self.in_progress.remove(task)
        print(f"Finished {task}")
```

## 7. Real-World Engineering Example

How an SRE team might handle incoming incident alerts using a Kanban queue:

```python
import queue

class IncidentQueue:
    def __init__(self):
        # The WIP limit prevents the on-call engineer from getting overwhelmed
        self.active_incidents = queue.Queue(maxsize=3)

    def route_alert(self, alert_id):
        try:
            self.active_incidents.put_nowait(alert_id)
            print(f"Alert {alert_id} assigned to engineer.")
        except queue.Full:
            # Buffer is full, escalate to secondary on-call
            print(f"WIP limit hit! Escalating Alert {alert_id} to secondary on-call.")
```

## 10. Top Interview Questions

| Question | Answer Summary |
|---|---|
| What is the primary difference between Scrum and Kanban? | Scrum uses fixed timeboxes (Sprints); Kanban is a continuous flow system without timeboxes. |
| Why are WIP limits important? | They prevent context switching, expose bottlenecks, and ensure tasks actually get finished. |
| What is a pull system? | Work is pulled into a stage only when there is capacity, rather than being pushed onto overwhelmed workers. |
| How do you measure performance in Kanban? | Using Cycle Time and Lead Time. |
| When would you choose Kanban over Scrum? | When work is highly unpredictable, like support queues or DevOps tasks. |

## 11. Tricky Questions & Edge Cases

- **Can you change priorities in Kanban?** Yes, constantly. Because there are no sprints, the Product Owner can rearrange the Backlog daily. The next pulled task is always the most important one.
- **What happens if a column hits its WIP limit?** The team swarms the bottleneck. Developers might stop coding and help review PRs to clear the 'Review' column so work can flow again.

## 12. Real-World Usage

Toyota invented Kanban for manufacturing. Today, AWS infrastructure teams, GitHub's internal ops, and support desks worldwide use it. CI/CD pipelines are essentially automated Kanban boards moving code from 'Build' to 'Test' to 'Prod' based on capacity.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Respect the WIP limits | Ignore limits and start 10 tasks just to look busy |
| Swarm bottlenecks when they occur | Ignore a backlog of PRs because "reviewing isn't your job" |
| Make the board visible to everyone | Hide the board in a private management tool |

## 14. Production Notes

<Admonition type="warning">
A board with columns is NOT Kanban if it lacks WIP limits. Without limits, it's just a visual list of tasks that will eventually turn into a massive traffic jam in the 'In Progress' column.
</Admonition>

## 15. Common Mistakes

| Mistake | Correction |
|---|---|
| High WIP limits | Set limits low enough to force collaboration (e.g., 3 tasks for 4 devs). |
| Pushing work | Do not assign tickets to devs; let them pull when they have capacity. |
| Ignoring blocked items | Blocked items must be highly visible and prioritized for unblocking. |
