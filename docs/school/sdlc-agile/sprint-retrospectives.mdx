# Sprint Retrospectives

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Sprint Retrospective Ceremony |
| Difficulty | Intermediate |
| Used For | Continuous process improvement and team reflection |
| Common Mistake | Turning it into a blame session or complaining without action items |
| Performance | The primary driver for increasing team efficiency over time |

## 2. Engineering Story

A deployment failed on Friday night. The database locked up, and the team spent 10 hours recovering the system over the weekend. If you just move on to the next Sprint on Monday morning, it *will* happen again.

The Retrospective is the most important Agile ceremony. It's where the team stops writing code and fixes their *process*. It's a blame-free zone where you ask: What went wrong? Why did the test suite not catch this? How do we automate this check so it never happens again? Continuous improvement requires dedicated time for reflection. Without retrospectives, you aren't Agile; you're just running on a hamster wheel of stress.

## 4. Concept Explanation

The Sprint Retrospective occurs after the Sprint Review and prior to the next Sprint Planning. It is a time for the Scrum Team to inspect itself.

The core format usually covers:
1. **What went well?** (Celebrate wins and continue good practices)
2. **What didn't go well?** (Identify bottlenecks, tech debt, and process failures)
3. **What will we do differently next Sprint?** (Actionable improvements)

The output of a Retrospective is *not* just a list of complaints. It MUST result in concrete Action Items (often added as actual tickets to the next Sprint's backlog) to improve the way the team works.

## 5. Syntax Table

| Concept | Description |
|---|---|
| Prime Directive | "Regardless of what we discover, we understand everyone did the best job they could." |
| Action Item | A specific, assigned task to improve the process |
| Dot Voting | A technique to prioritize which issues to discuss first |

## 6. Beginner Example

A simple script modeling how retrospective data is processed:

```python
class Retrospective:
    def __init__(self):
        self.went_well = []
        self.needs_improvement = []

    def add_feedback(self, category, feedback):
        if category == "good":
            self.went_well.append(feedback)
        else:
            self.needs_improvement.append(feedback)

    def generate_action_items(self):
        actions = []
        for issue in self.needs_improvement:
            actions.append(f"Create Jira ticket to address: {issue}")
        return actions

retro = Retrospective()
retro.add_feedback("bad", "Deploy script failed silently.")
print(retro.generate_action_items())
# Output: ['Create Jira ticket to address: Deploy script failed silently.']
```

## 9. Performance Table

| Anti-pattern | Impact on Team |
|---|---|
| Blaming individuals | Destroys trust; people will hide bugs |
| All talk, no action items | Meeting feels like a waste of time |
| Manager leading the meeting | Team won't speak honestly |

## 10. Top Interview Questions

| Question | Answer Summary |
|---|---|
| What is the Retrospective Prime Directive? | A statement read at the start to establish a blame-free environment, assuming everyone did their best. |
| What is the most important output of a Retrospective? | Actionable items that are implemented in the next sprint. |
| Who should attend the Retrospective? | The entire Scrum Team (PO, SM, Developers). Outside managers should generally not attend to preserve safety. |
| How do you prevent Retros from getting boring? | Rotate the facilitator and change the format (e.g., Start/Stop/Continue, Sailboat, 4Ls). |
| What do you do with action items? | Add them to the Sprint Backlog for the next sprint so they actually get done. |

## 11. Tricky Questions & Edge Cases

- **What if the team says "Everything is fine, we have nothing to improve"?** The Scrum Master should bring data. Show them the burndown chart where they missed deadlines, or the bug tracker showing 5 escaped defects, and ask "Why did this happen?"
- **Should the Product Owner attend?** Yes, the PO is part of the Scrum Team. Their interaction with the devs is a key process to improve.

## 12. Real-World Usage

High-performing teams at companies like Spotify and Netflix use post-mortems (a specialized, incident-focused retrospective) to build highly resilient systems. The culture of "blameless retrospectives" is central to modern DevOps and SRE practices.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Focus on the *process* that allowed the failure | Focus on the *person* who made the mistake |
| Pick 1 or 2 high-impact action items | Create a list of 20 action items that will never get done |
| Use anonymous tools for gathering feedback | Force juniors to speak out against seniors in a hostile room |

## 14. Production Notes

<Admonition type="warning">
Psychological safety is the absolute prerequisite for a Retrospective. If developers feel they will be punished, fired, or lose bonuses based on what they say in a Retro, they will lie. The meeting will become a theater of compliance.
</Admonition>

## 15. Common Mistakes

| Mistake | Correction |
|---|---|
| Unassigned Action Items | Every action item must have a specific owner. |
| Ignoring past Action Items | Always start the Retro by reviewing if the last Sprint's action items were completed. |
| Only discussing bad things | Celebrate the wins to maintain morale. |
