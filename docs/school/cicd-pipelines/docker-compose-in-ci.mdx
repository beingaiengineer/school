# Docker Compose in CI

### 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Using Docker Compose in CI/CD Pipelines |
| Difficulty | Intermediate |
| Used For | Spinning up multi-container environments (app + database + cache) for integration and E2E testing. |
| Common Mistake | Hardcoding sleep timers (`sleep 30`) instead of health checks to wait for databases to boot. |
| Performance | Heavy. Pulling multiple images and starting them takes 1-3 minutes before tests even begin. |

### 2. Engineering Story

A team of engineers recently faced a critical challenge related to this concept. Their existing processes were failing under the load of thousands of concurrent users, and manual workarounds were causing major delays in deployment. By deeply understanding and correctly implementing this concept, the lead engineer was able to architect a solution that not only resolved the immediate bottleneck but also paved the way for massive scalability. This transformation turned a chaotic, error-prone system into a resilient, automated powerhouse.

You don't want to manually start a Postgres database, a Redis cache, and a backend API one by one in your pipeline. Docker Compose lets you define the entire "stadium" in one YAML file and bring it up with a single command.

### 4. Concept Explanation

While a `Dockerfile` builds a single application image, **Docker Compose** is a tool for defining and running multi-container Docker applications. You use a `docker-compose.yml` file to configure your app's services.

In a CI pipeline, your unit tests are fast because they mock everything. But for **Integration** and **End-to-End (E2E) tests**, you need a real database and a real cache.

Instead of configuring a complex Postgres server directly on the CI runner, you simply run `docker-compose up -d`. This spins up your entire production-like architecture as interconnected containers on the CI runner. You run your tests against this temporary environment, and when the pipeline finishes, you run `docker-compose down` to destroy it cleanly.

### 5. Syntax Table

Here are the key directives in a `docker-compose.yml` file used for CI testing.

| Directive | Description | Example |
|---|---|---|
| `version` | (Deprecated in newer versions, but common) Compose file format version. | `version: '3.8'` |
| `services` | Defines the different containers to run. | `services: db: api:` |
| `image` | The Docker image to pull for the service. | `image: postgres:15-alpine` |
| `environment` | Environment variables for that container. | `POSTGRES_PASSWORD: secret` |
| `ports` | Maps host runner ports to container ports. | `ports: - "5432:5432"` |
| `depends_on` | Boot order and readiness checks. | `depends_on: db: condition: service_healthy` |

### 6. Beginner Example

A simple Compose file spinning up a Postgres database and a Redis cache for testing.

```yaml
# docker-compose.test.yml
services:
  postgres-test-db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: testuser
      POSTGRES_PASSWORD: testpassword
      POSTGRES_DB: testdb
    ports:
      - "5432:5432"

  redis-cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"
```

In your CI pipeline, you would just run:
```bash
docker-compose -f docker-compose.test.yml up -d
npm run test:integration
docker-compose -f docker-compose.test.yml down
```

### 7. Real-World Engineering Example

Bro, in reality, databases take a few seconds to boot. If your tests start immediately after `docker-compose up`, they will crash with a "Connection Refused" error. You must use **Health Checks**.

```yaml
# docker-compose.ci.yml
services:
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: ci_user
      POSTGRES_PASSWORD: ci_password
      POSTGRES_DB: ci_test_db
    ports:
      - "5432:5432"
    # Senior move: The healthcheck ensures the DB is actually ready to accept connections
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ci_user -d ci_test_db"]
      interval: 5s
      timeout: 5s
      retries: 5

  backend-api:
    build:
      context: .
      dockerfile: Dockerfile
    environment:
      DATABASE_URL: postgres://ci_user:ci_password@db:5432/ci_test_db
    ports:
      - "8080:8080"
    depends_on:
      db:
        # The API will wait until the DB healthcheck passes before booting
        condition: service_healthy
```

In your GitHub Actions YAML:
```yaml
steps:
  - uses: actions/checkout@v4
  - name: Boot full stack
    run: docker-compose -f docker-compose.ci.yml up -d
  - name: Run E2E Tests
    run: npx cypress run
  - name: Teardown
    if: always() # Ensure teardown happens even if tests fail!
    run: docker-compose -f docker-compose.ci.yml down -v
```

### 8. Internal Working

How does Compose orchestrate this? When you run `docker-compose up`, the Docker daemon creates a private internal Virtual Network. It then boots the containers and attaches them to this network. By default, containers can talk to each other using their service names as hostnames (e.g., the API connects to `http://db:5432`).

### 9. Performance Table

| Strategy | Speed | Reliability |
|---|---|---|
| `sleep 30` to wait for DB | Slow (Always wastes exactly 30s) | Flaky (What if it takes 31s?) |
| Docker Healthchecks | Fast (Starts exact second it's ready) | Extremely reliable |
| Using `alpine` images for DBs | Fast to pull over network | Standard |
| Retaining volumes between runs | Faster boot (no schema creation) | Bad (State leaks between CI runs) |

### 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why use Docker Compose in CI instead of installing Postgres directly on the runner? | Compose isolates the environment. It guarantees the exact same Postgres version and configuration is used by all developers locally and by the CI server, avoiding "it works on my machine" issues. |
| What does `docker-compose down -v` do? | It stops and removes the containers, the network, AND the named volumes (`-v`). You must use `-v` in CI to ensure the database data is completely wiped, guaranteeing a clean slate for the next pipeline run. |
| How do containers in a Compose file communicate with each other? | Through an internal Docker network. They use their service names defined in the YAML file as hostnames (DNS resolution). |
| What happens if `depends_on` is used without `condition: service_healthy`? | Compose will wait for the DB container to *start*, but it won't wait for the DB software inside to be *ready to accept connections*. Your API will likely crash trying to connect to a booting DB. |
| Can you use multiple Compose files? | Yes. You can have a `docker-compose.yml` for local dev, and an override file `docker-compose.ci.yml` that overrides ports or adds testing tools specifically for the pipeline. |

### 11. Tricky Questions & Edge Cases

**The Port Conflict Trap:**
Bro, if your CI runner is executing multiple jobs concurrently on the same machine (like self-hosted runners), and two jobs run `docker-compose up` mapping Postgres to `5432:5432`, the second job will crash with `address already in use`.
*The Fix:* In CI, don't map ports statically. Let Docker assign random ports (`ports: - "5432"`) or use fully isolated runners (like default GitHub Actions).

**The Zombie Container Problem:**
If your E2E tests fail, the pipeline step exits with code `1`. If the next step is `docker-compose down`, it might be skipped because the previous step failed! The containers are left running forever.
*The Fix:* Always put the teardown step under an `if: always()` block to guarantee cleanup, or run ephemeral CI runners that get destroyed entirely.

### 12. Real-World Usage

When you build a microservices architecture, you can't test one service in isolation. If you work on the "Cart Service," your integration tests might require the "User Service" and a "Redis Cache" to be running. Teams use Docker Compose in CI to pull the `latest` images of adjacent microservices, spin them all up, and run tests against the whole cluster.

### 13. Best Practices

| DO | DON'T |
|---|---|
| Use `healthcheck` to ensure services are fully booted. | Use `sleep 15` in your CI scripts and pray the DB boots in time. |
| Run `docker-compose down -v` in an `always()` block. | Leave containers running and corrupt the next pipeline run. |
| Use separate Compose files for dev (`docker-compose.yml`) and CI (`docker-compose.ci.yml`). | Clutter your dev environment with CI-specific testing tools. |

### 14. Production Notes

> **Warning: Do NOT use Compose for Production Deployments**
> Docker Compose is amazing for local development and CI testing. It is **terrible** for production at scale. It has no self-healing, no auto-scaling, and no rolling updates. For actual production deployments, you must graduate from Compose to orchestration systems like Kubernetes (K8s) or Amazon ECS.

### 15. Common Mistakes

| Mistake | Consequence | The Fix |
|---|---|---|
| Not waiting for DB migrations | E2E tests query tables that don't exist yet. | Add a container that runs the migration script and use `depends_on` to ensure it finishes before tests start. |
| Using `.env` files blindly | Dev passwords leak into CI or Prod. | Pass environment variables explicitly in the CI configuration, not via committed `.env` files. |
| Forgetting `-d` flag | `docker-compose up` blocks the terminal; the next pipeline step never runs. | Always use `docker-compose up -d` (detached mode) in CI pipelines. |
