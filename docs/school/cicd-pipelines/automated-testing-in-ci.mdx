# Automated Testing in CI

### 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Automated Testing in CI/CD |
| Difficulty | Intermediate |
| Used For | Catching logic bugs, regressions, and broken features before they reach production. |
| Common Mistake | Running slow End-to-End (E2E) tests on every single commit, causing pipeline bottlenecks. |
| Performance | Fast unit tests take seconds; heavy database integration tests take minutes. |

### 2. Real-World Analogy

Bro, imagine a massive Royal Enfield motorcycle factory.

| Factory Floor | CI Testing Equivalent |
|---|---|
| Checking if a single spark plug fires on a desk. | Unit Tests |
| Connecting the spark plug to the engine block and starting it. | Integration Tests |
| Putting the whole bike on a rolling road and driving it. | End-to-End (E2E) Tests |
| Sending a broken bike back to the mechanic. | Failing the CI Pipeline |

If you wait until the whole bike is built to test the spark plug, you waste hours taking the bike apart again. The factory line (CI pipeline) runs quick checks first, and only runs the full road test at the end.

### 3. Concept Explanation

Automated testing in CI is the process of executing a suite of scripts against your code to verify it behaves exactly as expected. Without tests, your CI pipeline is just a glorified code downloader.

A strong CI pipeline implements the **Testing Pyramid**:
1. **Unit Tests (The Base):** Thousands of tests. They check isolated functions (e.g., `calculateTotal()`). They run in milliseconds because they don't touch the database or network.
2. **Integration Tests (The Middle):** Hundreds of tests. They check if two components talk to each other correctly (e.g., your API saving data to a real test database). They take seconds.
3. **E2E Tests (The Peak):** Dozens of tests. They simulate a real user clicking buttons in a browser (e.g., Cypress or Playwright). They take minutes.

Bro, if your pipeline doesn't have tests, you are deploying on hope and prayers.

### 4. Syntax Table

Here is how you typically invoke different testing phases in a CI YAML file.

| Test Type | Example Command | Purpose |
|---|---|---|
| Unit Test | `npm run test:unit` | Fast execution, mock everything. |
| Integration | `pytest tests/integration` | Requires a test database/redis spun up. |
| E2E | `npx playwright test` | Requires a browser engine and full app running. |
| Coverage | `go test -coverprofile=c.out` | Measures what percentage of code was tested. |

### 5. Beginner Example

A simple Node.js pipeline running Jest unit tests.

```yaml
name: Unit Tests

on: [pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run Jest Unit Tests
        run: npm test
```

### 6. Real-World Engineering Example

Bro, in a real system, you often need a database to run integration tests. In GitHub Actions, you can use `services` to spin up temporary Docker containers right next to your code.

```yaml
name: API Integration Tests

on:
  pull_request:
    branches: [ "main" ]

jobs:
  integration-tests:
    runs-on: ubuntu-latest

    # The senior move: Spin up a real database for the tests
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_USER: testuser
          POSTGRES_PASSWORD: testpassword
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        # Wait until the database is ready to accept connections
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - run: npm ci

      - name: Run Database Migrations
        env:
          DB_URL: postgres://testuser:testpassword@localhost:5432/testdb
        run: npm run migrate

      - name: Run Integration Tests
        env:
          DB_URL: postgres://testuser:testpassword@localhost:5432/testdb
        run: npm run test:integration
```

### 7. Internal Working

How does the runner handle a failing test? Testing frameworks (Jest, PyTest, JUnit) exit the Node/Python process with a specific code. If all tests pass, they exit with code `0`. If even one test fails, they exit with code `1` (or another non-zero code). The runner intercepts this exit code and instantly marks the step as failed.

### 8. Performance Table

| Test Type | Target Execution Time | Optimization Strategy |
|---|---|---|
| Unit Tests | < 1 minute | Mock everything. No network calls. |
| Integration | 1 - 5 minutes | Use an in-memory DB or a fresh Docker DB container. |
| E2E | 5 - 15 minutes | Run them in parallel using matrix strategies or Playwright sharding. |

### 9. Top Interview Questions

| Question | Answer |
|---|---|
| Why do my tests pass locally but fail in CI? | Usually due to environment differences. Local might have environment variables set, a different OS file system (case sensitivity), or lingering database state. CI is a clean slate. |
| What is "Mocking" and why is it important in CI? | Replacing external dependencies (like Stripe API) with fake objects that return predictable data. Important because the CI server shouldn't make real network calls that cost money or rely on third-party uptime. |
| How do you speed up a test suite that takes 30 minutes? | Run tests in parallel across multiple CI runners using matrix builds. Split the test suite into chunks (sharding). |
| What is a "Flaky" test? | A test that sometimes passes and sometimes fails without changing any code. Usually caused by race conditions, relying on network speed, or bad state cleanup between tests. |
| Should you test the database in CI? | Yes, via Integration tests. Use CI service containers (like a Postgres Docker image) to spin up a clean database strictly for the pipeline run. |

### 10. Tricky Questions & Edge Cases

**The Timezone Trap:**
Bro, your local machine is in India (IST). The GitHub Actions runner defaults to UTC time. If you have a test that says `expect(date.getHours()).toBe(10)`, it will pass on your machine and fail in CI because UTC is 5.5 hours behind.
*Fix:* Always use UTC or explicitly set the timezone in your CI environment variables (`env: TZ=Asia/Kolkata`).

**The Shared DB State Trap:**
If Test A inserts a user, and Test B expects the database to be empty, Test B will fail if it runs after Test A. Tests in CI must be totally isolated. Run database transactions that rollback after every single test, or truncate tables in a `beforeEach` block.

### 11. Real-World Usage

At companies like **Zomato**, a pull request triggers thousands of unit tests immediately. If those pass, integration tests run against a staging database. If those pass, Playwright boots up headless browsers to simulate a user adding a biryani to the cart and checking out. Merging is physically impossible until the robot proves the biryani can be ordered.

### 12. Best Practices

| DO | DON'T |
|---|---|
| Run unit tests on every commit. | Run 30-minute UI tests on a typo fix in a README. |
| Fail the pipeline if code coverage drops significantly. | Ignore coverage metrics entirely. |
| Generate XML/HTML test reports and upload them as pipeline artifacts. | Force developers to read 10,000 lines of raw terminal logs to find the error. |

### 13. Production Notes

> **Warning: API Rate Limiting**
> If your integration tests hit a real external API (like a mapping service or weather API), and your team opens 50 PRs a day, you will get rate-limited or billed heavily. Always mock external APIs in CI, or use dedicated, isolated sandbox environments.

### 14. Common Mistakes

| Mistake | Consequence | The Fix |
|---|---|---|
| Forgetting to wait for the DB | "Connection refused" errors on integration tests. | Use health checks to delay test execution until the DB port is open. |
| Not archiving test results | When tests fail, you have no visual report to debug. | Use `actions/upload-artifact` to save the `/coverage` or `/playwright-report` folder. |
| Hardcoding localhost | E2E tests fail because the frontend tries to reach `localhost:3000` but it's not running. | Ensure a step actually starts the development server before running E2E tests. |
