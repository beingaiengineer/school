# Product Requirement Documents (PRD)

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Product Requirement Documents (PRD) |
| Difficulty | Intermediate |
| Used For | Documenting the "what" and "why" of a product before building it. |
| Common Mistake | Making it a 50-page thesis no one reads. |
| Performance | Speeds up engineering alignment and reduces meetings. |

## 2. Engineering Story

Bro, I once joined a team where the PM just threw JIRA tickets over the wall with titles like "Make login faster". No context, no metrics, no user journey. Half the team was optimizing database queries, while the frontend guy was rewriting the UI in React. We wasted two sprints. Then, a senior PM joined and introduced Product Requirement Documents (PRDs). Suddenly, we had a single source of truth. The PRD explicitly stated the goal ("Reduce time to interactive on login to &lt;1 second"), the target user, and the out-of-scope items. It was like someone turned the lights on. A PRD is the contract between Product and Engineering.

## 4. Concept Explanation

A Product Requirement Document (PRD) is a living document written mostly by Product Managers, but reviewed heavily by Engineering and Design. Bro, it answers *what* we are building, *who* it is for, and *why* it matters. It rarely dictates *how* to build it—that's the engineer's job.

The PRD acts as the single source of truth. When the frontend engineer and the backend engineer argue over whether the API needs pagination, they look at the PRD. If the PRD says the UI will show 10,000 items, then yes, you need pagination.

A good PRD is concise. It contains the objective, success metrics, user stories, out-of-scope features, and dependencies. If it's 50 pages long, it's a bad PRD because nobody will read it.

## 5. Syntax Table

| PRD Section | Purpose |
|---|---|
| Objective | The main goal of the feature (e.g., increase user retention). |
| Target Audience | Who exactly is using this feature. |
| In Scope / Out of Scope | Clear boundaries of what will NOT be built. |
| Success Metrics | How we know the feature worked (e.g., 5% increase in DAU). |

## 6. Beginner Example

```python
# A simple representation of reading a vague ticket
def build_feature(ticket):
    if ticket.title == "Add dark mode":
        # We don't know if we need a toggle, or auto-detect OS settings
        implement_css_changes()
    return "Done, maybe?"
```

## 7. Real-World Engineering Example

```python
# Translating a strict PRD into technical architecture
def build_feature_from_prd(prd_spec):
    # The PRD specifies exactly what to do
    if prd_spec.feature == "Dark Mode":
        # Requirement: Respect OS settings first
        os_preference = get_os_theme()

        # Requirement: Allow user to override
        user_override = db.get_user_setting("theme_override")

        # Requirement: Fallback to light mode
        final_theme = user_override or os_preference or "light"

        apply_theme(final_theme)

        # Requirement: Analytics tracking for success metrics
        analytics.track("theme_applied", {"theme": final_theme})
```

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| As an engineer, what do you look for in a PRD? | I look for clear edge cases, out-of-scope definitions, and measurable success metrics so I know what to optimize for. |
| What if the PRD dictates the technical implementation? | I gently push back. The PRD should define the 'what' and 'why'. Engineering owns the 'how'. |
| How do you handle a PRD that keeps changing? | If it's in the middle of a sprint, I flag it as scope creep and ask the PM to move the changes to the next iteration. |
| What is the most important section of a PRD? | The "Out of Scope" section. It actively prevents engineers from over-engineering features nobody asked for. |
| Have you ever worked without a PRD? | Yes, and it led to misaligned expectations. I now advocate for at least a 1-page PRD for any major feature. |

## 11. Tricky Questions & Edge Cases

Bro, what if the PRD is perfect but the technical reality makes it impossible within the deadline? You do not just stay silent. You bring it up immediately during the PRD review. You offer alternatives: "We can't do real-time updates in 2 weeks, but we can do polling every 5 minutes." The PRD is a negotiation, not a dictatorship.

## 12. Real-World Usage

At companies like Stripe, a PRD is often paired with an API design document. The PM writes the business requirements (the PRD), and the engineering team writes the technical response. This dual-document system ensures the business gets what it wants and the technical architecture remains pristine.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Keep it concise and readable | Write a 40-page essay |
| Clearly define success metrics | Use vague terms like "make it fast" |
| Include UX wireframes or links to Figma | Describe UI layouts purely in text |

## 14. Production Notes

<Admonition type="warning" title="Watch Out in Prod!">
Never start coding until the PRD is "frozen" or approved. If you start coding while the PM is still changing the core requirements, you will write throwaway code. Wait for the ink to dry.
</Admonition>

## 15. Common Mistakes

| Mistake | Correction |
|---|---|
| Engineers ignoring the PRD | Read it thoroughly and leave comments before the kickoff meeting. |
| Forgetting analytics | Ensure the PRD explicitly states what data needs tracking. |
| No 'Out of Scope' section | Always define what you are explicitly choosing NOT to build. |
