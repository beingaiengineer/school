# Rate Limiting: Protecting Your API

### 1. Quick Summary

| Area | Details |
|---|---|
| Topic | API Rate Limiting |
| Difficulty | Intermediate |
| Used For | Preventing DDoS attacks, brute force login attempts, and controlling resource usage |
| Common Mistake | Using in-memory rate limiting when running multiple server instances behind a load balancer |
| Performance | Minimal overhead (O(1) lookups) when using fast distributed caches like Redis |

### 2. Engineering Story

You are maintaining the backend for a popular e-commerce platform during the massive Diwali Diwali festival sale. At midnight, the flash sales go live. Suddenly, database CPU usage spikes to 100%, and the site crashes. You scramble to check the logs and discover that a competitor has deployed an army of automated bots to scrape your discounted prices, hitting your `/api/products` endpoint 5,000 times a second.

Your system was trying to fulfill every single request fairly. You realize that without boundaries, an API is easily abused. You immediately implement Rate Limiting. Now, when a single IP address or user account tries to make more than 100 requests per minute, the API instantly rejects the excess requests with a `429 Too Many Requests` status, protecting the database. The bots are blocked, legitimate users can browse the sales, and the servers stabilize.

### 4. Concept Explanation

Rate Limiting is the process of controlling the amount of traffic sent to or received by a network interface. In API design, it restricts the number of API calls a client can make within a specified time window (e.g., 100 requests per minute).

You should use rate limiting to protect your infrastructure from denial-of-service (DoS) attacks, to prevent brute-force attacks on login endpoints, and to enforce usage quotas in monetized SaaS products (e.g., Free tier: 100 req/day, Pro tier: 10,000 req/day). You should NOT apply overly aggressive rate limits to critical public webhook endpoints where dropped requests mean lost transactional data (unless a retry mechanism is in place).

### 5. Syntax Table

*Note: FastAPI does not have built-in rate limiting, so we typically use the excellent `slowapi` library.*

| Concept | Syntax (using slowapi) | Explanation |
|---|---|---|
| Limiter Object | `limiter = Limiter(key_func=get_remote_address)` | Initializes the rate limiter, tracking by IP address |
| Decorator | `@limiter.limit("5/minute")` | Enforces a limit of 5 requests per minute on a specific endpoint |
| Exception Handler | `app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)` | Catches the violation and returns a standard 429 response |

### 6. Beginner Example

```python
from fastapi import FastAPI, Request
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded

# Track users by their IP address
limiter = Limiter(key_func=get_remote_address)
app = FastAPI()

# Register the custom exception handler for 429 errors
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

@app.get("/login")
@limiter.limit("3/minute") # Strict limit for security
async def login(request: Request):
    return {"message": "Please provide credentials"}
```

### 7. Real-World Engineering Example

In production, relying on the client's IP address (`get_remote_address`) is dangerous because many users might share the same NAT/IP (like a corporate office), and you might accidentally block an entire company. Instead, you rate limit based on authenticated user IDs, and store the hit counts in Redis so the limits are enforced globally across all your API servers.

```python
from fastapi import FastAPI, Request, Depends, HTTPException
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.errors import RateLimitExceeded
from redis import asyncio as aioredis # Use async redis
import os

# Assume we have a function to extract the current user from a JWT token
async def get_current_user_id(request: Request):
    user_id = request.headers.get("X-User-ID")
    if not user_id:
        raise HTTPException(status_code=401, detail="Unauthorized")
    return user_id

# Connect to a centralized Redis cache
redis_pool = aioredis.from_url("redis://localhost", encoding="utf-8", decode_responses=True)

# Initialize limiter to use Redis instead of local memory
# In a real app, you would pass the redis connection to the Limiter's storage backend
# For demonstration, we use a custom key function based on the authenticated user
limiter = Limiter(key_func=lambda req: req.headers.get("X-User-ID", "anonymous"))

app = FastAPI()
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

@app.post("/api/v1/generate-report")
@limiter.limit("10/hour") # Expensive operation, strict limits
async def generate_heavy_report(request: Request, user_id: str = Depends(get_current_user_id)):
    return {"message": f"Report generation started for user {user_id}"}
```

### 8. Internal Working

When a request arrives, the rate limiter intercepts it:
1. It extracts a unique identifier (the key) from the request (e.g., API key, user ID, or IP).
2. It queries a fast datastore (like Redis) for the current request count associated with that key within the current time window.
3. If the count is below the threshold, it increments the counter and allows the request to proceed to your FastAPI route.
4. If the count exceeds the threshold, it aborts the request and immediately returns an HTTP 429 status code.

### 9. Performance Table

| Strategy | Speed | Scalability | Notes |
|---|---|---|---|
| In-Memory (Dict) | Extremely Fast | Poor | Fails when using multiple server instances; limits are tracked per server. |
| Redis (Centralized) | Fast (< 1ms) | Excellent | The industry standard for distributed rate limiting. |
| Database (SQL) | Very Slow | Poor | Never use a relational database for rate limiting counters; it will crash under load. |

### 10. Top Interview Questions

| Question | Answer |
|---|---|
| What HTTP status code is used for Rate Limiting? | `429 Too Many Requests`. |
| What is the Token Bucket algorithm? | A rate-limiting algorithm where a "bucket" holds tokens. Requests consume tokens. Tokens are replenished at a fixed rate. If the bucket is empty, requests are dropped. |
| Why is rate limiting by IP address risky? | Because multiple distinct users might share the same public IP address due to NAT (Network Address Translation), such as employees in an office or students on a campus Wi-Fi. |
| How do you communicate limits to the client? | Using standard HTTP response headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` (time until quota refreshes). |
| What happens to rate limits when you restart your FastAPI server? | If using in-memory rate limiting, all counters are wiped to zero. If using Redis, the counters persist securely. |

### 11. Tricky Questions & Edge Cases

- **"If a user is rate-limited and gets a 429 error, should that 429 request also count against their quota?"**
  Generally, no. Once the limit is reached, further requests are blocked and the counter is not incremented, otherwise the user might permanently lock themselves out by continuing to spam the endpoint before the reset window.

### 12. Real-World Usage

- **Twitter API:** Limits developers to a certain number of tweet reads/writes per month based on their pricing tier.
- **Auth0 / Okta:** Strictly rate limits login endpoints to prevent password brute-forcing and credential stuffing.
- **OpenAI (ChatGPT API):** Limits based on Tokens-Per-Minute (TPM) and Requests-Per-Minute (RPM) to manage massive GPU compute costs.

### 13. Best Practices

| DO | DON'T |
|---|---|
| Apply different limits based on the route's cost. A static `/ping` route can have a high limit, but a heavy `/export-pdf` route needs a strict limit. | Do NOT apply a single blanket rate limit across the entire API. |
| Provide clear error messages indicating exactly *when* the user can try again (using the Retry-After header). | Do NOT return vague 500 Internal Server Errors when a user hits a limit. |

### 14. Production Notes

<Admonition type="caution" title="The Load Balancer Trap">
If your FastAPI application is sitting behind a load balancer or reverse proxy (like Nginx, AWS ALB, or Cloudflare), the `request.client.host` property will often be the IP address of the *load balancer*, not the actual user! If you rate limit by IP in this setup, you will instantly block all traffic to your site. You must configure your proxy to forward the `X-Forwarded-For` header and instruct your rate limiter to read the IP from that header instead.
</Admonition>

### 15. Common Mistakes

| Trap | Code Example | Solution |
|---|---|---|
| **Relying on Local Memory in Multi-Worker Setups** | Running `gunicorn -w 4 ...` with local memory rate limits. | If you have 4 workers, a "5 per minute" limit effectively becomes "20 per minute" because each worker has its own isolated memory bucket. Use Redis. |
| **Silent Failures** | Dropping requests without headers. | Always include `Retry-After` or `X-RateLimit-Reset` headers so well-behaved client apps know exactly how long to sleep before retrying. |
