### 2. Engineering Story

Your team is running a massive e-commerce API. Yesterday, the system crashed during a flash sale. Why? A mobile client bug sent the `price` field as the string `"free"` instead of a float `0.00`. The JSON payload hit your API, bypassed the routing, and crashed deep inside the PostgreSQL database layer when it tried to insert a string into a numeric column.

Worse, another bug allowed users to register with the email `not_an_email_at_all`, breaking your outbound marketing pipelines. Your engineering team is spending half their time writing `if type(price) != float:` and `if "@" not in email:` boilerplate code. It is brittle, unreadable, and error-prone.

You need a bouncer at the front door of your API. A tool that looks at the incoming JSON, verifies that `price` is a float (or can be safely coerced into one), ensures `email` is structurally valid, and confirms `age` is over 18—all before your core business logic ever runs. That tool is Pydantic.

### 4. Concept Explanation

**Pydantic** is a data validation and settings management library using Python type annotations. It enforces type hints at runtime.

Unlike standard Python where `x: int = "hello"` is just ignored at runtime, Pydantic will actively inspect the data and raise a `ValidationError` if it cannot conform to the declared type.

Key concepts:
- **Validation vs. Parsing:** Pydantic is primarily a *parsing* library. It guarantees the types and constraints of the output model, not necessarily the input data. (e.g., If it expects an `int` and receives the string `"123"`, it will parse/coerce it into the integer `123`).
- **Field Constraints:** Beyond basic types, you can enforce ranges (`gt=0`), string lengths (`max_length=50`), and regex patterns.
- **Custom Validators:** You can write custom Python functions that run during parsing to enforce complex business rules (e.g., "end_date must be after start_date").

**When to use Pydantic:** To validate all incoming API payloads, load environment variables, and serialize data to JSON.
**When NOT to use Pydantic:** Inside a tight, high-frequency internal loop where the data is already guaranteed to be clean, as the parsing overhead (though small) will add up.

### 5. Syntax Table

| Feature | Syntax | Description |
|---|---|---|
| **Basic Type** | `age: int` | Enforces integer. |
| **Optional Field** | `bio: Optional[str] = None` | Allows `None` or missing field. |
| **Field Constraints** | `price: float = Field(gt=0)` | Must be greater than 0. |
| **String Constraints**| `username: str = Field(min_length=3)` | Must be at least 3 chars. |
| **Email Validation** | `email: EmailStr` | Validates standard email format. |
| **Custom Validator** | `@field_validator('name')` | Runs a custom Python function. |

### 6. Beginner Example

A simple Pydantic model demonstrating basic types and the `Field` constraint.

```python
from pydantic import BaseModel, Field, EmailStr, ValidationError

class UserSignup(BaseModel):
    username: str = Field(min_length=3, max_length=20)
    email: EmailStr
    age: int = Field(ge=18, description="Must be at least 18")

# --- Valid Data (Coercion in action) ---
# Notice 'age' is passed as a string, Pydantic converts it to int
valid_data = {"username": "rahul99", "email": "rahul@m.com", "age": "25"}
user = UserSignup(**valid_data)
print(user.age) # Outputs: 25 (integer)

# --- Invalid Data ---
invalid_data = {"username": "ab", "email": "not-an-email", "age": 12}
try:
    UserSignup(**invalid_data)
except ValidationError as e:
    print(e.json())
    # Outputs detailed JSON explaining exactly which fields failed and why
```

### 7. Real-World Engineering Example

In production, you often need cross-field validation (e.g., checking two passwords match, or ensuring a date range makes sense). Here is a real-world example of an Event booking model.

```python
from pydantic import BaseModel, Field, model_validator, field_validator
from datetime import datetime

class EventBooking(BaseModel):
    event_id: str = Field(pattern=r'^EVT-\d{4}$') # Regex: EVT-1234
    tickets_requested: int = Field(gt=0, le=10)
    start_time: datetime
    end_time: datetime

    # 1. Single-field Custom Validator
    @field_validator('start_time')
    @classmethod
    def start_time_must_be_future(cls, v: datetime) -> datetime:
        if v <= datetime.utcnow():
            raise ValueError('Event start time must be in the future')
        return v

    # 2. Cross-field (Model) Validator
    @model_validator(mode='after')
    def check_time_range(self) -> 'EventBooking':
        if self.end_time <= self.start_time:
            raise ValueError('end_time must be after start_time')
        return self

# Usage in FastAPI:
# @app.post("/bookings")
# def create_booking(booking: EventBooking):
#     return booking
```
*Note: In FastAPI, if this model fails validation, FastAPI catches the `ValidationError` and automatically returns a 422 HTTP response to the client.*

1. **Initialization:** You pass a dictionary to the Pydantic Model.
2. **Rust Core:** In Pydantic V2, the heavy lifting of parsing base types is passed to `pydantic-core`, an engine written in Rust for extreme speed.
3. **Coercion & Parsing:** It attempts to coerce types (e.g., `"true"` to `True`, `"1"` to `1`).
4. **Custom Logic:** If base parsing succeeds, it hands control back to Python to run your custom `@field_validator` and `@model_validator` methods.
5. **Success or Failure:** If everything passes, you get a clean Python object. If anything fails, it halts and aggregates all errors into a single `ValidationError`.

### 9. Performance Table

| Metric | Pydantic V1 | Pydantic V2 (Rust) | Note |
|---|---|---|---|
| **Simple Model Parsing** | ~5-10 µs | ~1-2 µs | V2 is 5x-50x faster. |
| **JSON Serialization** | Uses Python `json` | Uses Rust `serde_json` | Drastic reduction in CPU load for large API responses. |
| **Custom Validators** | Moderate overhead | Moderate overhead | Dropping into Python execution (custom validators) slows down the Rust engine. Use standard `Field()` constraints when possible. |

### 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is type coercion in Pydantic? | Pydantic tries to be helpful. If a field expects an `int` but receives the string `"42"`, Pydantic will parse it and convert it to the integer `42` rather than failing immediately. (You can disable this in V2 with `Strict` types). |
| What is the difference between `@field_validator` and `@model_validator`? | `@field_validator` runs on a single specific field (e.g., checking an email). `@model_validator` runs on the entire model *after* all fields are parsed, useful for cross-field checks (e.g., password matching password_confirm). |
| How do you validate a list of items in Pydantic? | By using Python's `List` type hint. E.g., `tags: List[str] = Field(max_items=5)`. Pydantic will validate every item in the list. |
| Should you do database lookups inside a Pydantic validator? | Generally, no. Pydantic is for synchronous *structural* and *business* validation. Database lookups require I/O (often async), which should be handled in the FastAPI route or dependency injection layer. |

### 11. Tricky Questions & Edge Cases

**Tricky Question: If you have a Pydantic model with `timestamp: datetime` and you pass it the string `"2023-10-01"`, what happens?**
Pydantic parses the string into a native Python `datetime` object. The edge case is timezone awareness. By default, it might parse it as naive. You must be very careful with datetime configurations in Pydantic if your API requires strict UTC enforcement.

### 13. Best Practices

| DO | DON'T |
|---|---|
| Use `Field(..., min_length=X)` for all strings to prevent users from submitting empty `""` values. | DON'T use Pydantic models for complex ORM logic; keep them focused on data parsing. |
| Use `EmailStr` (requires `email-validator` package) instead of writing your own Regex for emails. | DON'T write custom validators for things `Field` can do (like `gt=0`). |
| Set `extra = "forbid"` in model config if you want to aggressively reject unknown fields. | DON'T silently ignore extra fields if they indicate a major client-side bug. |

### 14. Production Notes

<Admonition type="info" title="Strict Mode">
In highly sensitive applications (like banking APIs), Pydantic's default "coercion" (converting `"1"` to `1`) is dangerous. In Pydantic V2, you can enforce strict mode: `age: StrictInt`. If the client sends `"25"` instead of `25`, it will be rejected. This forces frontend teams to send proper JSON types.
</Admonition>

### 15. Common Mistakes

| Mistake | Bad Code (Conceptual) | Good Code (Conceptual) |
|---|---|---|
| **Mutable Defaults** | `tags: list = []` (Will share the same list instance across all requests!) | `tags: list = Field(default_factory=list)` |
| **Missing Optional** | `bio: str = None` (Mypy will complain) | `bio: Optional[str] = None` |
| **Re-inventing the wheel**| `@field_validator('url')` writing custom URL parsing | `website: HttpUrl` (Pydantic built-in) |
