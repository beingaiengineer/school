# Middleware & CORS: The Gateway Guards

### 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Middleware functions and Cross-Origin Resource Sharing (CORS) |
| Difficulty | Intermediate |
| Used For | Applying global logic (logging, timing, security headers) to every API request |
| Common Mistake | Forgetting to configure CORS, causing the frontend to silently fail on API calls |
| Performance | Heavy middleware (like doing a DB lookup for every request) will throttle the entire application |

### 2. Engineering Story

You've built a beautiful FastAPI backend and deployed it to AWS. A separate frontend team has built a React app hosted on Vercel. They try to call your `/login` endpoint, and nothing happens. The React console is screaming: *"Blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present"*. You frantically Google the error and install a random package that sets `Allow-Origin: *`. The frontend starts working!

A month later, your manager asks for a report on API latency. You realize you have 50 different endpoints and no way to track how long they take to execute without pasting `start = time.time()` into every single one of them.

You've stumbled into the two most common global API problems: browser security (CORS) and global request lifecycle management. By understanding Middleware, you can elegantly solve both. You configure a strict CORS middleware to only allow the Vercel domain, and you write a custom timing middleware that intercepts every request, times it, and logs it automatically. The architecture is clean, and the frontend is secure.

### 4. Concept Explanation

**Middleware** is a function that sits between the web server receiving the raw HTTP request and your actual route handler function processing it.
It intercepts the request before it reaches your code, and intercepts the response before it goes back to the client.

**CORS (Cross-Origin Resource Sharing)** is a browser security feature. By default, a web browser running a script from `https://my-frontend.com` will block API requests to `https://my-backend.com` because the origins (domains) are different. The backend must explicitly return HTTP headers (like `Access-Control-Allow-Origin`) telling the browser that this specific frontend is allowed to make the call. This is almost always handled via a pre-built CORS Middleware.

**When to use Middleware:** For global concerns: adding CORS headers, calculating request execution time, logging IP addresses, or modifying all outgoing responses.
**When NOT to use Middleware:** For business logic that only applies to a few routes (use Dependency Injection instead), or for heavy database operations (it will slow down every single route in your app).

### 5. Syntax Table

How to implement Middleware and CORS in FastAPI.

| Concept | FastAPI Implementation | Description |
|---|---|---|
| **Adding CORS** | `app.add_middleware(CORSMiddleware, ...)` | Adds the built-in CORS handler. |
| **Allowed Origins** | `allow_origins=["https://myapp.com"]` | Tells the browser who can call the API. |
| **Custom Middleware**| `@app.middleware("http")` | Decorator to create custom interceptors. |
| **Call Next** | `response = await call_next(request)` | Passes the request down the chain to the route. |
| **Modifying Headers**| `response.headers["X-Time"] = "1ms"` | Adding custom data to the outbound response. |

### 6. Beginner Example

The absolute minimum required to fix the dreaded CORS error for a React frontend.

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

# 1. Define the specific domains allowed to talk to your API
origins = [
    "http://localhost:3000",        # React local development
    "https://www.my-production-app.com", # Production frontend
]

# 2. Add the CORS middleware to the application
app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,          # Domains allowed
    allow_credentials=True,         # Allow cookies/auth headers
    allow_methods=["GET", "POST"],  # Only allow these HTTP verbs
    allow_headers=["*"],            # Allow any HTTP headers
)

@app.get("/")
def read_root():
    return {"message": "CORS is configured!"}
```

### 7. Real-World Engineering Example

In production, you often need to track how long requests take and attach a unique "Trace ID" to every request so you can track a single user's action through millions of log lines. Here is a custom middleware to do exactly that.

```python
from fastapi import FastAPI, Request
import time
import uuid
import logging

# Setup basic logging
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

app = FastAPI()

@app.middleware("http")
async def add_process_time_and_trace_header(request: Request, call_next):
    # --- BEFORE THE ROUTE HANDLER ---
    start_time = time.time()

    # Generate a unique ID for this specific request
    trace_id = str(uuid.uuid4())

    # Log the incoming request
    logger.info(f"[{trace_id}] Received {request.method} request to {request.url.path}")

    # --- EXECUTE THE ROUTE HANDLER ---
    # call_next passes the request to the actual endpoint
    response = await call_next(request)

    # --- AFTER THE ROUTE HANDLER ---
    process_time = time.time() - start_time

    # Add custom headers to the outgoing response
    response.headers["X-Process-Time-Sec"] = str(round(process_time, 4))
    response.headers["X-Trace-ID"] = trace_id

    # Log completion
    logger.info(f"[{trace_id}] Completed in {process_time:.4f}s with status {response.status_code}")

    return response

@app.get("/heavy-computation")
async def heavy_work():
    time.sleep(0.5) # Simulate work
    return {"result": "Done"}
```

### 9. Performance Table

| Middleware Type | Performance Impact | Note |
|---|---|---|
| **CORS** | Negligible | Purely header string checking and manipulation. |
| **Timing/Logging** | Very Low | `time.time()` is extremely fast. Standard logging might cause minor I/O block if not async. |
| **Auth/DB Checks** | High | Doing a database query inside middleware slows down *every* route, even public ones like `/health`. Use Dependency Injection instead. |

### 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is a CORS Preflight Request? | When a browser attempts a "complex" request (like a POST with JSON), it first sends an `OPTIONS` HTTP request (the preflight) to ask the server if it is allowed. The server's CORS middleware responds with the allowed origins and methods. Only if approved does the browser send the actual `POST`. |
| Why does CORS only affect browsers? | CORS is enforced by the browser, not the server. If you run a `curl` command or a Python script to hit an API, CORS is ignored because those tools don't implement browser security sandboxes. |
| What is the difference between Middleware and Dependency Injection in FastAPI? | Middleware runs globally on *every* request (e.g., logging). Dependencies are injected locally into *specific* routes (e.g., fetching the current user from the DB for protected routes). |
| Why is `allow_origins=["*"]` dangerous? | It tells the browser to allow any website on the internet to make requests to your API on behalf of the user. If the API uses cookies for auth, a malicious site could perform CSRF (Cross-Site Request Forgery) attacks. |

### 11. Tricky Questions & Edge Cases

**Tricky Question: A request fails validation in the CORS middleware. Does your Route Handler still execute?**
No. The middleware stack acts like an onion. If the CORS middleware rejects the request (e.g., unauthorized origin), it immediately returns an HTTP 403 or 400 response. `call_next()` is never invoked, meaning the inner layers (your other middleware and your route handler) are completely protected from the request.

### 12. Real-World Usage

- **Datadog / New Relic APM:** Application Performance Monitoring tools work by installing heavy middleware that intercepts every request, calculates performance metrics, and ships them to a dashboard.
- **Rate Limiting:** APIs like GitHub or Twitter use middleware to track the incoming IP address and instantly reject requests with a `429 Too Many Requests` if the user exceeds their quota.
- **Sentry:** Error tracking tools use middleware to wrap `call_next()` in a massive `try/except` block, catching unhandled exceptions globally and sending the stack trace to Sentry before returning a generic 500 error to the client.

### 13. Best Practices

| DO | DON'T |
|---|---|
| Use exact domains in `allow_origins` for production (e.g., `["https://app.com"]`). | DON'T use `allow_origins=["*"]` in production. |
| Keep middleware as light and fast as possible. | DON'T do heavy synchronous network requests or database writes in middleware. |
| Wrap `call_next(request)` in a try/except block if you want to handle global 500 errors gracefully. | DON'T swallow exceptions in middleware without logging them. |

### 14. Production Notes

<Admonition type="caution" title="Middleware Ordering Matters">
In FastAPI, the order you add middleware is the *reverse* order of execution. The last middleware added is the first one to receive the incoming request. Ensure that CORS middleware is added at the very end of your setup (so it executes first) so it can handle `OPTIONS` requests before your custom logging or auth middleware tries to process them.
</Admonition>

### 15. Common Mistakes

| Mistake | Bad Code (Conceptual) | Good Code (Conceptual) |
|---|---|---|
| **CORS trailing slashes** | `allow_origins=["http://app.com/"]` (Browser sends without slash) | `allow_origins=["http://app.com"]` |
| **Blocking Middleware** | `def middleware(): time.sleep(1)` (Hangs the server) | `async def middleware(): await asyncio.sleep(1)` |
| **Forgetting to return** | `await call_next(request)` but missing `return response` | `return await call_next(request)` |
