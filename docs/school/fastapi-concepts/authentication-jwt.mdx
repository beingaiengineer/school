# Authentication & JWT: The Digital Bouncer

### 1. Quick Summary

| Area | Details |
|---|---|
| Topic | API Authentication using JSON Web Tokens (JWT) |
| Difficulty | Intermediate |
| Used For | Securely proving client identity across stateless API requests |
| Common Mistake | Storing sensitive data (like passwords) inside the JWT payload |
| Performance | Fast verification (no database lookup required), but large tokens increase HTTP header size |

### 2. Engineering Story

Your startup just launched its API. Initially, you used simple session cookies. It worked great when you had one backend server. But as traffic spiked, you scaled to 10 servers behind a load balancer. Suddenly, users are complaining they keep getting logged out.

You investigate and find the problem: User A logs in, and Server 1 stores their session in its memory. The next API request is routed to Server 4, which has no idea who User A is, so it rejects the request. You could fix this by forcing all servers to check a centralized Redis database for every single request, but that introduces a massive bottleneck and latency.

You need a way for the user to prove who they are to *any* server, instantly, without the server needing to look up their session in a database. You need a self-contained digital ID card. You implement JSON Web Tokens (JWT). Now, the server simply looks at the token, mathematically verifies the signature, and instantly knows the user's identity.

### 4. Concept Explanation

**Authentication** is the process of verifying *who* a user is (usually via username and password).
A **JSON Web Token (JWT)** is an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object.

A JWT consists of three parts separated by dots (`xxxxx.yyyyy.zzzzz`):
1. **Header:** Defines the type of token and the hashing algorithm used (e.g., HMAC SHA256).
2. **Payload (Claims):** The actual JSON data (e.g., `\{\"user_id\": 123, \"exp\": 1680000000\}`). *This is Base64 encoded, NOT encrypted. Anyone can read it.*
3. **Signature:** Created by taking the Header, Payload, and a secret key known only to the server, and hashing them.

**How it works:** If a hacker changes the `user_id` in the payload from 123 to 1, the signature will no longer match the payload. When the server recalculates the signature using its secret key, it will detect the tampering and reject the token.

**When to use JWT:** For stateless API authentication, Single Sign-On (SSO), and microservice-to-microservice communication.
**When NOT to use JWT:** For traditional server-rendered web apps (use secure cookies/sessions), or when you need the ability to instantly revoke a user's access globally (JWTs cannot be easily revoked before they expire without stateful workarounds).

### 5. Syntax Table

Key JWT concepts and libraries (using PyJWT and FastAPI).

| Component | Code / Tool | Description |
|---|---|---|
| **Encode Token** | `jwt.encode(payload, SECRET, algo)` | Creates a signed JWT. |
| **Decode Token** | `jwt.decode(token, SECRET, algo)` | Verifies signature and returns payload. |
| **Expiration Claim** | `{"exp": datetime.utcnow() + delta}`| Standard claim telling when token dies. |
| **Auth Header** | `Authorization: Bearer <token>` | Standard HTTP header for sending JWTs. |
| **FastAPI OAuth2** | `OAuth2PasswordBearer(tokenUrl="...")` | FastAPI utility to extract token from header. |

### 6. Beginner Example

Generating and decoding a JWT using the `PyJWT` library.

```python
import jwt
from datetime import datetime, timedelta

SECRET_KEY = "my_super_secret_key_do_not_share"
ALGORITHM = "HS256"

# 1. Create the Payload (Claims)
payload = {
    "sub": "user_123",                    # Subject (who this token is for)
    "role": "admin",                      # Custom claim
    "exp": datetime.utcnow() + timedelta(minutes=15) # Expiration
}

# 2. Encode and Sign the Token
token = jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)
print(f"Generated Token: {token}")

# 3. Decode and Verify the Token
try:
    decoded_payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    print(f"Verified Payload: {decoded_payload}")
except jwt.ExpiredSignatureError:
    print("Token has expired!")
except jwt.InvalidTokenError:
    print("Invalid token signature!")
```

### 7. Real-World Engineering Example

Here is how you protect an endpoint in FastAPI using Dependency Injection to extract and verify a JWT from the HTTP header.

```python
from fastapi import FastAPI, Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer
import jwt

app = FastAPI()

SECRET_KEY = "production_secret_key"
ALGORITHM = "HS256"

# This tells FastAPI to look for 'Authorization: Bearer <token>'
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")

def get_current_user(token: str = Depends(oauth2_scheme)):
    '''Dependency to verify JWT and return the user ID'''
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )

    try:
        # Verify the signature and check expiration automatically
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id: str = payload.get("sub")

        if user_id is None:
            raise credentials_exception

        return user_id

    except jwt.PyJWTError:
        # Catches expired tokens or invalid signatures
        raise credentials_exception

# Protect the route by injecting the get_current_user dependency
@app.get("/users/me")
def read_users_me(current_user_id: str = Depends(get_current_user)):
    # If the code reaches here, the token is 100% valid
    return {"user_id": current_user_id, "secret_data": "Top Secret"}
```

|---|---|---|---|
| **Session (DB)** | Slow (DB query) | Low | Poor (Requires shared DB for all servers) |
| **Session (Redis)** | Fast (Cache query) | Low | Good (Requires managing Redis cluster) |

### 10. Top Interview Questions

| Question | Answer |
|---|---|
| Are JWT payloads encrypted? | **NO.** The payload is just Base64 encoded. Anyone can decode it and read the contents. It is *signed*, not encrypted. Never put passwords or SSNs inside a JWT. |
| How do you log a user out if JWTs are stateless? | You can't easily invalidate a JWT before it expires. The standard practice is to use short expiration times (e.g., 15 minutes) and delete the token from the client. For strict security, you maintain a "blacklist" of revoked tokens in a database, which ruins the "stateless" benefit. |
| What is the difference between Access Tokens and Refresh Tokens? | Access Tokens are short-lived JWTs used to access APIs. Refresh Tokens are long-lived, opaque strings stored in the database, used to request a new Access Token without forcing the user to log in again. |
| What happens if your server's SECRET_KEY is leaked? | It's a catastrophe. A hacker can forge JWTs for any user (including admins) and your server will accept them as valid. You must immediately rotate the key, which invalidates all existing user sessions. |

### 11. Tricky Questions & Edge Cases

**Tricky Question: What is the "None Algorithm" vulnerability in JWTs?**
In early JWT libraries, the header could specify `"alg": "none"`. If a server blindly trusted the header's algorithm choice, a hacker could remove the signature, set `alg` to `none`, modify the payload to be an admin, and the server would accept it. Modern libraries (like PyJWT) require you to explicitly whitelist the algorithms (e.g., `algorithms=["HS256"]`) you accept during decoding to prevent this.

### 12. Real-World Usage

- **Auth0 / Okta:** Massive identity providers whose entire business revolves around generating standardized JWTs for SSO (Single Sign-On).
- **Microservices Architecture:** When Service A calls Service B, it passes a JWT. Service B doesn't need to ask the Auth Service if the token is valid; it just mathematically verifies it, drastically reducing network traffic.
- **Mobile Apps:** Mobile clients heavily favor JWTs with refresh tokens over traditional cookie-based sessions, as handling headers is easier in iOS/Android native networking libraries.

### 13. Best Practices

| DO | DON'T |
|---|---|
| Use short expiration times (`exp`) for Access Tokens (15-60 minutes). | DON'T set a JWT to expire in 10 years. |
| Use a strong, long, cryptographically random string for your `SECRET_KEY`. | DON'T hardcode your secret key in the source code; use Environment Variables. |
| Use standard claims like `sub` (subject/user_id) and `iat` (issued at). | DON'T store massive amounts of data in the payload; it balloons every HTTP request. |

### 14. Production Notes

<Admonition type="caution" title="The Refresh Token Architecture">
Because Access Tokens cannot be revoked, they must expire quickly. But logging users out every 15 minutes is terrible UX.
**Solution:** The login endpoint returns two tokens: a 15-minute JWT (Access Token) and a 30-day random string (Refresh Token, stored in the DB). When the JWT expires, the client sends the Refresh Token to a `/refresh` endpoint. The server checks if the Refresh Token is still valid in the DB, and if so, issues a new 15-minute JWT. If a user's account is compromised, you simply delete the Refresh Token in the DB.
</Admonition>

### 15. Common Mistakes

| Mistake | Bad Code | Good Code |
|---|---|---|
| **Storing Secrets in Payload** | `payload = {"cc_number": "1234..."}` (Visible to anyone who decodes it) | `payload = {"user_id": "123"}` (Look up sensitive data in DB) |
| **Trusting Client Headers** | `alg = jwt.get_unverified_header(token)['alg']`<br/>`jwt.decode(token, key, alg)` | `jwt.decode(token, key, algorithms=["HS256"])` (Force the algorithm) |
| **Missing Expiration** | `payload = {"user_id": 1}` (Token lives forever) | `payload = {"user_id": 1, "exp": time_in_future}` |
