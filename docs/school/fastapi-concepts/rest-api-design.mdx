# REST API Design: The Rules of the Web

### 1. Quick Summary

| Area | Details |
|---|---|
| Topic | REST (Representational State Transfer) Architectural Style |
| Difficulty | Intermediate |
| Used For | Designing scalable, predictable, and stateless HTTP APIs |
| Common Mistake | Using verbs in URLs (e.g., `/getUsers`) instead of nouns (`/users`) |
| Performance | Caching standard HTTP GET requests drastically improves read throughput |

### 2. Engineering Story

You're a backend engineer at a rapidly scaling SaaS company. Your team built an API a year ago, but it grew organically. You have endpoints like `/createUser`, `/updateUserNow`, `/deleteUserById`, and `/fetchUserDataWithPosts`. Mobile engineers are complaining because every endpoint returns a different JSON structure. To get a user's profile and their recent posts, they have to make four different API calls. The API gateway's caching layer is useless because every URL includes unpredictable verbs, and `POST` is being used for everything just to hide parameters.

The system isn't scaling, and developer velocity has ground to a halt because nobody knows how to interact with the system without reading the backend source code. You need a universal grammar. You need to refactor the entire system to use REST (Representational State Transfer). By standardizing on nouns for resources and standard HTTP methods for actions, you can make the API intuitive, leverage network-level caching, and decouple the client from the server's internal chaos.

### 4. Concept Explanation

**REST (Representational State Transfer)** is an architectural style for distributed hypermedia systems, defined by Roy Fielding in 2000. It's not a protocol; it's a set of constraints built on top of HTTP.

The core principle of REST is that everything is a **Resource** (a noun, like `users`, `orders`, `products`). You interact with these resources using standard **HTTP Methods** (verbs, like `GET`, `POST`, `PUT`, `DELETE`).

Key REST Constraints:
1. **Client-Server Architecture:** Separation of concerns. The client handles UI; the server handles data.
2. **Statelessness:** No client context is stored on the server between requests. Every request must contain all the information needed to understand it (e.g., sending the JWT token every time).
3. **Cacheability:** Responses must define themselves as cacheable or not, allowing clients or intermediaries to reuse data.
4. **Uniform Interface:** Consistent URLs, standard methods, and standard media types (like JSON).

**When to use REST:** For almost all standard web and mobile APIs where resource-oriented CRUD operations dominate.
**When NOT to use REST:** When you need real-time bi-directional communication (use WebSockets), or when clients need highly specific, nested, and variable data structures in a single request (use GraphQL).

### 5. Syntax Table

Here is the standard mapping of HTTP methods to CRUD operations on REST resources.

| Method | /users (Collection) | /users/123 (Specific Resource) |
|---|---|---|
| **GET** (Read) | Fetch a list of all users | Fetch user 123 |
| **POST** (Create) | Create a new user | *Usually error (405 Method Not Allowed)* |
| **PUT** (Update) | *Usually error (405)* | Replace user 123 entirely |
| **PATCH** (Modify)| *Usually error (405)* | Partially update user 123 (e.g., just email) |
| **DELETE** (Remove)| Delete all users (Dangerous!) | Delete user 123 |

### 6. Beginner Example

A good vs. bad example of designing a URL endpoint for getting a specific user's orders.

```text
// BAD (RPC-style, verb in URL)
GET /getOrdersForUser?userId=123
POST /createNewOrder

// GOOD (RESTful, nouns and hierarchy)
GET /users/123/orders
POST /users/123/orders
```

### 7. Real-World Engineering Example

Here is a Python FastAPI implementation of a fully RESTful `Product` resource, demonstrating proper use of status codes and paths.

```python
from fastapi import FastAPI, HTTPException, status
from pydantic import BaseModel
from typing import List

app = FastAPI()

class Product(BaseModel):
    id: int
    name: str
    price: float

# Fake database
db_products = {
    1: Product(id=1, name="Laptop", price=999.99),
    2: Product(id=2, name="Mouse", price=49.99)
}

@app.get("/products", response_model=List[Product])
def get_products():
    '''GET /products -> Returns 200 OK and a list'''
    return list(db_products.values())

@app.get("/products/{product_id}", response_model=Product)
def get_product(product_id: int):
    '''GET /products/{id} -> Returns 200 OK or 404'''
    if product_id not in db_products:
        raise HTTPException(status_code=404, detail="Product not found")
    return db_products[product_id]

@app.post("/products", status_code=status.HTTP_201_CREATED, response_model=Product)
def create_product(product: Product):
    '''POST /products -> Returns 201 Created'''
    if product.id in db_products:
        raise HTTPException(status_code=409, detail="Product ID already exists")
    db_products[product.id] = product
    return product

@app.delete("/products/{product_id}", status_code=status.HTTP_204_NO_CONTENT)
def delete_product(product_id: int):
    '''DELETE /products/{id} -> Returns 204 No Content'''
    if product_id not in db_products:
        raise HTTPException(status_code=404, detail="Product not found")
    del db_products[product_id]
    return None
```

1. **Routing:** The web server receives the raw HTTP string. The framework's router uses regular expressions to match the method (`GET`) and path (`/users/123`) to a specific function in your code.
2. **Execution:** The matched controller function executes. It extracts path parameters (like `123`).
3. **Data Retrieval:** The business logic talks to the database.
4. **Serialization:** The raw database object (or ORM model) is serialized into a JSON representation.
5. **Response Construction:** The framework constructs a valid HTTP response with the correct headers (Content-Type) and Status Code (200 OK).

### 9. Performance Table

| REST Constraint | Performance Impact | Why |
|---|---|---|
| **Statelessness** | Extremely High Scale | Servers don't need memory for sessions. Any load balancer can route any request to any server instance. |
| **Cacheability (GET)** | High | CDNs (like Cloudflare) or browsers can intercept `/products` and return it without hitting your server. |
| **JSON Serialization** | Moderate overhead | JSON parsing is fast, but sending massive nested JSON trees consumes network bandwidth. |
| **Over-fetching** | Negative impact | REST often returns the *entire* resource even if the client only needed one field (unlike GraphQL). |

### 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between PUT and PATCH? | `PUT` replaces the entire resource. If a field is omitted in a PUT payload, it should be set to null. `PATCH` applies a partial update, modifying only the fields provided. |
| What makes a REST API "Stateless"? | The server stores no client context (session state) between requests. Every request must contain all info (like an Auth token) needed to process it. |
| How do you handle pagination in REST? | By using query parameters, typically offset/limit (`?limit=10&offset=20`) or cursor-based pagination (`?cursor=xyz123`). |
| Is `POST` idempotent? | No. Making the same `POST /orders` request twice will create two distinct orders. `GET`, `PUT`, and `DELETE` are expected to be idempotent. |
| What is HATEOAS? | Hypermedia As The Engine Of Application State. It's the highest level of REST maturity where API responses include links to related actions, allowing the client to discover the API dynamically. |

### 11. Tricky Questions & Edge Cases

**Tricky Question: How do you design a REST endpoint for an action that doesn't fit neatly into CRUD, like "logging in" or "calculating a complex tax rate"?**
REST purists try to find the hidden resource.
- Instead of `/login`, use `POST /sessions`.
- Instead of `/calculateTax`, use `POST /tax-calculations` (treating the calculation request itself as a resource you create).
If it completely fails to map to a resource, it's acceptable to use a verb in the URL for an RPC-style endpoint as an exception (e.g., `POST /users/123/ban`), but keep it rare.

### 12. Real-World Usage

- **GitHub API v3:** One of the most famous and well-designed REST APIs. It strictly uses nouns (`/repos`, `/pulls`), proper status codes, and hypermedia links for pagination.
- **Stripe API:** Exemplifies REST principles combined with excellent developer experience, using standard HTTP methods and standardizing error response structures globally.
- **Twitter/X API v2:** Heavily REST-oriented, using extensive query parameters (`?expansions=author_id`) to solve the over-fetching problem inherent in standard REST.

### 13. Best Practices

| DO | DON'T |
|---|---|
| Pluralize resource nouns consistently (`/users`, not `/user`). | DON'T mix singular and plural nouns in paths. |
| Nest resources to show hierarchy, up to two levels deep (`/users/123/orders`). | DON'T nest too deeply (`/users/1/orders/2/items/3`). It gets brittle. |
| Use query parameters for filtering, sorting, and pagination (`/users?role=admin&sort=desc`). | DON'T use query parameters for things that identify a unique resource. |
| Return standard error payloads with a message and error code. | DON'T return an HTML error page when a JSON client makes a bad request. |

### 14. Production Notes

<Admonition type="caution" title="Handling Long-Running Processes">
If a `POST` request triggers a 5-minute video encoding job, do not make the HTTP request hang for 5 minutes. The network will time out.
**Solution (The 202 Accepted Pattern):** The `POST` should return immediately with a `202 Accepted` status code and a `Location` header pointing to a status URL (e.g., `/jobs/987`). The client then polls that URL (using `GET`) until the status changes from `pending` to `completed`.
</Admonition>

### 15. Common Mistakes

| Mistake | Bad Code (Conceptual) | Good Code (Conceptual) |
|---|---|---|
| **Verbs in URLs** | `POST /updateEmail?id=123` | `PATCH /users/123` with payload `\{\"email\": \"new@m.com\"\}\` |
| **Wrong Status Code** | Returning `200 OK` when `\{\"error\": \"User not found\"\}\` is in the body. | Returning `404 Not Found`. |
| **Using GET for mutations** | `GET /deleteUser?id=123` (Dangerous: Browsers or CDNs might pre-fetch this URL and accidentally delete data!) | `DELETE /users/123` |
