# Clean Architecture in Production

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Clean Architecture |
| Difficulty | Advanced |
| Used For | Structuring entire codebases so that business logic is completely isolated from frameworks, databases, and UI. |
| Common Mistake | Leaking database entities or HTTP request objects into the inner core domain. |
| Performance | No runtime cost, but increases initial development time due to boilerplate and strict layer boundaries. |

## 2. Engineering Story

You inherit a massive legacy Django application. The business logic is deeply intertwined with Django's ORM and HTTP request parsing inside massive `views.py` files.

The company decides they want to launch a mobile app and need a GraphQL API instead of the existing REST API. Furthermore, they are migrating away from Postgres to MongoDB. Because the core business logic (calculating taxes, applying discounts) is hardcoded against Django's SQL ORM and REST request objects, the team has to rewrite the entire application from scratch. It takes two years.

If the original team had used Clean Architecture, the business logic would have been isolated in the center. The Web Framework and the Database would just be "plugins" on the outside. Swapping REST for GraphQL and Postgres for Mongo would have taken weeks, not years, because the core domain code wouldn't need to change at all.

## 3. Real-World Analogy

Think of a high-security Diamond Vault (The Core Domain).

The diamonds and the rules for organizing them (Business Logic) are at the center. They never change, regardless of what's happening outside.

Surrounding the vault are layers of security (Interfaces/Adapters). The outer wall (Frameworks/UI) is where the delivery trucks arrive. The delivery driver (HTTP Request) cannot walk directly into the vault. They must hand the package to a security guard at the perimeter. The guard inspects it, strips away the cardboard box (JSON parsing), puts the raw diamonds in a standard tray (DTO), and passes the tray through the layers until it reaches the vault.

If you change the delivery company from FedEx (REST) to UPS (GraphQL), the vault doesn't care. The vault only knows about standard trays.

| OOP Concept | Vault Analogy |
|---|---|
| Entities / Core Domain | The Diamonds (Pure business rules) |
| Use Cases / Services | The Vault Manager (Coordinates operations) |
| Interface Adapters | The Security Guards (Translates data formats) |
| Frameworks / DB / UI | The Delivery Trucks (External systems) |
| Dependency Rule | Information only flows inward toward the vault |

## 4. Concept Explanation

**Clean Architecture** (popularized by Robert C. Martin / Uncle Bob) is a software design philosophy that separates the elements of a design into ring levels. The main rule is the **Dependency Rule**: *Source code dependencies must point only inward, toward higher-level policies.*

The inner layers contain business rules and have absolutely zero knowledge of the outer layers (databases, web frameworks, UI). The outer layers act as plugins to the inner layers.

## 5. Syntax Table

| Layer (Inner to Outer) | What lives here | Allowed to import from |
|---|---|---|
| 1. Entities | Pure Domain Objects, Enterprise Rules | Nothing (No external libraries) |
| 2. Use Cases | Application specific business rules (Services) | Entities |
| 3. Interface Adapters | Controllers, Presenters, Gateways | Use Cases, Entities |
| 4. Frameworks & Drivers | Web Framework (Express), DB ORM, UI | Adapters, Use Cases, Entities |

## 6. Beginner Example

Let's look at how directories are structured in a standard tightly-coupled app versus a Clean Architecture app.

**Bad (Framework-Driven Architecture):**
```text
src/
  models/ (Mongoose Schemas mixed with logic)
  controllers/ (Express routers mixed with logic)
  views/ (React components)
```

**Good (Clean Architecture):**
```text
src/
  domain/ (Pure entities, e.g., User.ts. Zero dependencies)
  usecases/ (Services, e.g., CreateUser.ts)
  adapters/
    controllers/ (HTTP handlers mapping JSON to UseCases)
    repositories/ (SQL logic mapping DB to Domain)
  infrastructure/
    express-server.ts (Wires up the web framework)
    postgres-db.ts (Wires up the database connection)
```

## 7. Real-World Engineering Example

Here is how the boundaries are crossed using Dependency Inversion in TypeScript. The UseCase needs to save data, but it cannot depend on the Database layer (outer ring).

```typescript
// 1. INNER RING (Domain)
export class User {
    constructor(public id: string, public email: string) {}
}

// 2. USE CASE RING (Application Logic)
// Defines the contract the outer ring must fulfill
export interface UserRepository {
    save(user: User): Promise<void>;
}

export class RegisterUserUseCase {
    // Injects the interface, pointing dependencies INWARD
    constructor(private repo: UserRepository) {}

    async execute(email: string) {
        if (!email.includes("@")) throw new Error("Invalid email");
        const user = new User(Date.now().toString(), email);
        await this.repo.save(user); // Calls the DB without knowing what the DB is
        return user;
    }
}

// 3. ADAPTER RING (Infrastructure/DB)
// Lives on the outside, depends on the inner ring's Interface
import { Client } from 'pg';
export class PostgresUserRepository implements UserRepository {
    constructor(private db: Client) {}

    async save(user: User) {
        await this.db.query('INSERT INTO users(id, email) VALUES($1, $2)', [user.id, user.email]);
    }
}

// 4. FRAMEWORK RING (Delivery)
// Wires it all together
const dbClient = new Client();
const repo = new PostgresUserRepository(dbClient);
const useCase = new RegisterUserUseCase(repo);
// Pass useCase to Express controller...
```

## 8. Internal Working

Clean Architecture is entirely about controlling the direction of imports (pointers) at compile time and runtime. By forcing the outer rings (DB, Web) to implement interfaces defined by the inner rings (Use Cases), the runtime uses dynamic dispatch to execute the database code, even though the inner ring's source code never imported the database library.

## 9. Performance Table

| Aspect | Impact |
|---|---|
| Developer Velocity | Slower initially due to writing DTOs, Mappers, and Interfaces. Massively faster later when maintaining or refactoring. |
| Testability | Perfect. You can test 100% of the business logic in milliseconds without a DB, Web Server, or UI. |
| Runtime Overhead | Very slight overhead from mapping objects across boundaries (e.g., mapping an ORM Model to a Domain Entity). |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is the Dependency Rule? | Source code dependencies must only point inward, toward higher-level policies (business rules). |
| How do you cross boundaries outwards in Clean Architecture? | Using the Dependency Inversion Principle. The inner layer defines an Interface, and the outer layer implements it. |
| Why shouldn't you pass a database ORM object into the Use Case layer? | Because it violates the Dependency Rule. The Use Case would become coupled to the specific ORM library. If you change ORMs, the Use Case breaks. |
| Are Clean, Hexagonal, and Onion Architecture the same? | They are structurally very similar and share the exact same objective: separation of concerns and isolating the core domain from infrastructure. |
| When should you NOT use Clean Architecture? | For small scripts, MVPs, or simple CRUD applications where the framework (like Django or Ruby on Rails) can do 90% of the work out of the box. |

## 11. Tricky Questions & Edge Cases

**Question:** If the UI (outer layer) needs a specific data format, should the Use Case (inner layer) format the data for the UI?
**Answer:** No. The Use Case should return raw, unformatted domain data. The Presenter/Controller (Adapter layer) is responsible for taking that raw data and formatting it specifically for the UI (e.g., converting a Date object to a localized string). The inner layers must not care about the UI.

## 12. Real-World Usage

- **Enterprise Fintech / Healthcare:** Codebases that are expected to live for 10-20+ years heavily use Clean Architecture so they can survive multiple UI framework changes (e.g., migrating from Angular to React) without rewriting core logic.
- **Android Development:** Google officially recommends a flavor of Clean Architecture for modern Android apps (separating UI, Domain, and Data layers).

## 13. Best Practices

| Do | Don't |
|---|---|
| **DO** create Data Transfer Objects (DTOs) and Mappers to translate data when crossing layer boundaries. | **DON'T** let library imports (like `import express` or `import mongoose`) sneak into your Domain or Use Case folders. |
| **DO** treat the database as a "detail". You should be able to write and test your business rules before you even pick a database. | **DON'T** strictly enforce all 4 rings if it's overkill. You can simplify to 3 rings (Domain, Application, Infrastructure). |

## 14. Production Notes

<Admonition type="caution" title="The Boilerplate Trap">
Clean Architecture requires mapping data between layers (e.g., DB Row -> ORM Object -> Domain Entity -> DTO -> JSON). In massive scale applications, this mapping process can consume CPU cycles and memory. Pragmatic teams sometimes allow slight "leaks" (like returning the ORM object on pure read-only queries) to optimize performance, but you must do this consciously.
</Admonition>

## 15. Common Mistakes

| Mistake | Bad Code | Good Code |
|---|---|---|
| Framework leaking into domain | `import { Request } from 'express'; class UseCase { execute(req: Request) }` | `class UseCase { execute(data: UserDTO) }` |
| DB leaking into domain | `class User extends Sequelize.Model {}` | `class User { id: string, name: string }` |
