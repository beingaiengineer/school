# Code Review Culture

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Code Review Culture & Etiquette |
| Difficulty | Intermediate |
| Used For | Ensuring code quality, spreading knowledge, and mentoring juniors. |
| Common Mistake | Using aggressive language or nitpicking syntax instead of logic. |
| Performance | Human bottleneck. Poor culture leads to weeks of PR stagnation. |

## 2. Real-World Analogy

Bro, think of a Code Review like an architectural inspection before building a house.

| Construction Scenario | Code Review Equivalent |
|---|---|
| Architect draws the blueprint | You write the code |
| City Inspector checks for safety codes | Reviewer checks for security and performance |
| Inspector says: "This beam looks weak, use steel instead." | Reviewer says: "This loop is O(n^2), use a hash map." |
| Inspector approves the permit | Reviewer clicks "Approve" (LGTM) |

If the inspector just screams "THIS IS TERRIBLE" and leaves, nothing gets built. Feedback must be constructive.

## 3. Concept Explanation

Code review is a human process, not a technical one. It happens inside a Pull Request. The goal is not just to find bugs—that is what automated tests are for. The goal of a code review is to ensure the code is maintainable, readable, and aligns with the company's architecture.

More importantly, it is a knowledge-sharing mechanism. It is how junior engineers learn the "senior moves" and how senior engineers discover what the rest of the team is building. A toxic code review culture destroys a team's velocity; a healthy one acts as a force multiplier.

## 4. Syntax Table

*Note: Code review is conceptual, but here is the "syntax" of good communication.*

| Review Comment Style | Impact |
|---|---|
| "Why did you do it this way?" | Aggressive/Defensive. |
| "Can you explain the reasoning here?" | Collaborative. |
| "Change this to an array." | Demanding. |
| "Consider using an array here for better lookup speed." | Mentoring. |
| "Nit: missing comma." | Minor formatting note (use a linter instead). |

## 5. Beginner Example

A good review comment focuses on the code, not the person.

**Bad Comment:**
> "You forgot to check if the user is null again. Fix it."

**Good Comment:**
> "If `user` is null here, this will throw an exception. Should we add a guard clause at the top of the function to return early?"

## 8. Performance Table

| Activity | Metric | Target |
|---|---|---|
| Review Turnaround Time | Hours | Under 24 hours. Don't block your teammates. |
| PR Size | Lines of Code | Under 400 LOC. Larger PRs get poorer reviews. |
| Comment Volume | Count | 1-5 major architectural notes > 50 nitpicks. |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| How do you handle a disagreement during a code review? | Take it offline or to a synchronous call (Zoom/Slack huddle). Text easily escalates. |
| What do you look for when reviewing code? | Logic flaws, edge cases, security risks, test coverage, and readability. I let the linter handle syntax. |
| A senior engineer leaves a review you think is wrong. What do you do? | Ask for clarification respectfully: "Could you help me understand why X is better here? I thought Y because..." |
| Why is self-review important? | You catch 80% of silly mistakes before wasting a reviewer's time. |

## 10. Tricky Questions & Edge Cases

**Should you review code you don't understand?**
Bro, if you are a junior and you don't understand the PR, *ask questions in the review!* "Hey, I'm trying to learn this part of the system, can you explain how this regex works?" A good PR should be readable. If you can't read it, that is valid review feedback.

**The PR has 3,000 lines. How do you review it?**
Reject it gracefully. "This is too large for me to review effectively and guarantee no bugs slip through. Can we split this into 3 smaller PRs?"

## 11. Real-World Usage

Google has one of the strictest code review cultures in the world. They enforce a concept called "Readability". Before you are allowed to approve PRs in a certain language (like C++ or Go), you must earn "Readability" status by having your own code heavily scrutinized by experts to prove you understand Google's exact idioms. Until then, every PR you write must be approved by someone who holds the Readability badge.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Prefix non-blocking comments with `Nit:` (Nitpick) so the author knows it's optional. | Don't hold a PR hostage over variable naming preferences. |
| Automate everything you can (formatting, linting) so humans focus on logic. | Don't argue about spaces vs. tabs in a code review. |
| Praise good code! "This is a really elegant solution." | Don't just point out the negatives. |

## 13. Production Notes

> **Warning: The Rubber Stamp**
> If your team culture is just clicking "Approve" without reading the code because everyone is "too busy," you are accumulating massive technical debt. This is how memory leaks and security vulnerabilities get shipped to production. Reviewing code is not a distraction from your job; it *is* your job.

## 14. Common Mistakes

| Mistake | Fix |
|---|---|
| Reviewing code when you are angry/tired | Step away. Your tone will bleed into your text and damage team trust. |
| Making 20 round-trips of comments | Jump on a 5-minute video call. Summarize the decision in the PR afterward. |
