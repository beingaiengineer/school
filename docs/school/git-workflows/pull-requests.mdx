# Pull Requests

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Pull Requests (Merge Requests) |
| Difficulty | Beginner |
| Used For | Proposing changes, code review, and CI validation before merging to main. |
| Common Mistake | Opening massive PRs with 2,000+ lines of code changes. |
| Performance | Not a Git command, but a platform feature. Network dependent. |

## 2. Engineering Story

In the early days of open-source, contributing to a project meant emailing a patch file to the maintainer, who would then manually apply it. It was slow, insecure, and offered no place for discussion. When GitHub launched, they introduced a revolutionary concept: instead of emailing code, a developer could simply 'request' that the maintainer 'pull' their changes directly from their forked repository. This created a centralized, visual arena where code could be reviewed, discussed, and automatically tested before ever touching the main project.

## 3. Real-World Analogy

Bro, think of a Pull Request (PR) like submitting a draft of an article to your editor.

| Publishing Scenario | PR Equivalent |
|---|---|
| You write the draft on your laptop | Developing on a local feature branch |
| You send the draft to the editor asking them to publish it | Opening the Pull Request |
| The editor adds red notes ("Fix this paragraph") | Code Review comments |
| You update the draft and resend it | Pushing new commits to the PR branch |
| The editor approves and it goes in the magazine | PR is Approved and Merged into `main` |

## 4. Concept Explanation

A Pull Request (known as a Merge Request in GitLab) is NOT a native Git concept. It is a feature provided by hosting platforms like GitHub, GitLab, and Bitbucket.

You use a PR to tell your team: *"Hey, I finished my feature. Can you review it, ensure the automated tests pass, and pull it into the main codebase?"* It acts as a gatekeeper. It is the core collaboration point in modern software engineering where knowledge is shared, bugs are caught, and code quality is enforced.

## 5. Syntax Table

*Note: PRs are primarily UI-driven on GitHub, but you can use the GitHub CLI (`gh`).*

| Command / Action | What it does |
|---|---|
| `git push -u origin <branch>` | Uploads your branch to GitHub so you can open a PR. |
| `gh pr create` | Creates a Pull Request from the CLI. |
| `gh pr checkout <number>` | Checks out someone else's PR branch to your local machine. |
| `gh pr status` | Shows the status of your current PRs. |

## 6. Beginner Example

Let's say you just finished styling the footer. Here is the workflow to get a PR up.

```bash
# 1. You are on your branch and commit your work
git commit -m "style: update footer colors"

# 2. Push your branch to the remote server
git push -u origin feat/footer-styles

# 3. GitHub will print a URL in your terminal.
# Click it to open the PR in your browser!
# Output looks like:
# remote: Create a pull request for 'feat/footer-styles' on GitHub by visiting:
# remote:      https://github.com/org/repo/pull/new/feat/footer-styles
```

## 7. Real-World Engineering Example

In a corporate setup, PRs trigger CI/CD pipelines automatically.

```yaml
# A typical GitHub Actions workflow triggered by a PR
name: CI PR Check
on:
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install dependencies
        run: npm ci
      - name: Run Linter
        run: npm run lint
      - name: Run Unit Tests
        run: npm run test
# If this fails, the PR cannot be merged!
```

## 9. Performance Table

| Action | Performance Bottleneck | Note |
|---|---|---|
| Creating PR | Network Speed | Fast, mostly just creating a database record on GitHub. |
| Loading diff | UI Rendering | GitHub truncates PRs with >3,000 changed files to save memory. |
| CI Pipeline | Server Compute | This is where you wait. Tests can take 5 mins to 1 hour. |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| What is the difference between a PR and a Merge Request? | Nothing. GitHub calls it a Pull Request. GitLab calls it a Merge Request. Same concept. |
| Why did my PR update automatically without me doing anything? | Because you pushed new commits to the branch associated with the PR. The PR auto-updates. |
| Can a PR be merged if there are conflicts? | No. You must resolve the conflicts locally and push the resolution before the merge button unlocks. |
| What is a Draft PR? | A PR marked as "WIP" (Work In Progress). It signals to reviewers not to review it yet, but allows CI tests to run. |
| What does LGTM mean? | "Looks Good To Me". A common approval comment. |

## 11. Tricky Questions & Edge Cases

**What happens to the PR if I force push (`git push -f`) to my branch?**
Bro, the PR updates to match the new history. If you rebase your branch and force push, GitHub will rewrite the PR's commit history. However, be careful: if a reviewer was in the middle of looking at your old code, you just blew away their context.

**Can I open a PR from a fork?**
Yes! This is how Open Source works. You fork the repo, push to your fork, and open a PR pointing from your fork's branch to the original repo's `main` branch.

## 12. Real-World Usage

At companies like Spotify or Airbnb, nobody merges to `main` locally via CLI. Every single change, even a 1-line typo fix, must go through a PR. The repo is configured with "Branch Protection Rules". These rules enforce that a PR *must* have at least 1 approving review from a senior engineer, and all CI tests *must* pass before the green "Merge" button is clickable.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Keep PRs small (under 400 lines of code). Reviewers get fatigued on big PRs. | Don't include 5 different unrelated bug fixes in one PR. |
| Write a descriptive PR body explaining *why* the change was made and *how* to test it. | Don't leave the PR description blank. |
| Review your own PR diff before requesting a review from others. | Don't leave commented-out code (`// console.log`) in your PR. |

## 14. Production Notes

> **Warning: The +1000 Line PR Trap**
> There is a famous saying: "A 10-line PR will get 10 comments. A 500-line PR will get 'LGTM'." When you open massive PRs, reviewers cannot physically comprehend the blast radius of the changes. They will skim it and approve it. This is exactly how critical production bugs slip into the codebase. Always break large features down into multiple, smaller, stackable PRs.

## 15. Common Mistakes

| Mistake | Fix |
|---|---|
| You opened the PR against `main` but meant `develop` | Use the "Edit" button near the PR title on GitHub to change the base branch. |
| Reviewer requested changes, but you opened a *new* PR | Don't! Just commit the fixes to the existing branch and `git push`. The original PR updates automatically. |
