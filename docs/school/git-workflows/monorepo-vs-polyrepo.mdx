# Monorepo vs Polyrepo

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Monorepos vs Polyrepos (Multi-repo) |
| Difficulty | Advanced |
| Used For | Organizing codebases and managing shared dependencies at scale. |
| Common Mistake | Using a monorepo without specialized tooling, turning it into a massive "monolith". |
| Performance | Polyrepos are fast to clone. Monorepos require advanced tooling (Bazel, Nx) to remain fast. |

## 2. Engineering Story

A ride-sharing app started with 50 different microservices, each in its own Git repository (a polyrepo). Whenever a core UI button changed, developers had to open 50 separate pull requests, wait for 50 CI pipelines, and coordinate 50 deployments to ensure all apps had the new button. Version mismatches caused constant production crashes. The engineering director finally said, 'Enough. We are moving all 50 services into a single giant repository.' This shift dramatically simplified dependency management but introduced entirely new challenges in scaling Git itself.

## 3. Real-World Analogy

Bro, think of storing your company's code like storing spices in a kitchen.

| Storage Strategy | Real-World Equivalent |
|---|---|
| **Polyrepo (Multi-repo)** | Every spice has its own individual jar, on its own shelf. It's perfectly isolated. But if you want to update the label style on all jars, you have to open 50 different shelves. |
| **Monorepo** | You use a traditional Indian "Masala Dabba" (a large box containing smaller, separate cups of spices). Everything is in one box. Updating the whole box is easy, but the box can get very heavy. |

## 4. Concept Explanation

A **Polyrepo** (multi-repo) means every project, microservice, or shared library gets its own dedicated Git repository. This is the default way most startups begin. It provides strong isolation, but sharing code requires publishing packages (like to npm or PyPI) and managing complex versioning across 20 different repos.

A **Monorepo** places multiple distinct projects (frontend, backend, shared UI components) into a *single* Git repository. Note: A Monorepo is NOT a Monolith. A Monolith is tightly coupled code that deploys as one unit. A Monorepo is loosely coupled code that lives in one folder but deploys independently.

## 5. Syntax Table

*Note: This relies on tooling, not native Git syntax.*

| Architecture | Key Characteristics | Common Tools |
|---|---|---|
| Polyrepo | 1 repo = 1 project. High isolation. | standard `git`, standard CI/CD. |
| Monorepo | 1 repo = N projects. Shared commits. | Nx, Turborepo, Lerna, Bazel, Rush. |

## 6. Beginner Example

What does a Monorepo actually look like on your hard drive? It's just folders.

```text
my-company-monorepo/
├── packages/
│   ├── shared-ui/      # A library of React buttons
│   └── math-utils/     # A library of math functions
├── apps/
│   ├── web-app/        # Imports from shared-ui and math-utils
│   └── mobile-app/     # Imports from shared-ui
├── package.json        # Root dependencies
└── .git/               # ONE Git history for the whole company
```

## 7. Real-World Engineering Example

In a Polyrepo, if you fix a bug in the `shared-ui` repo, you must:
1. Commit to `shared-ui`
2. Wait for CI to publish v1.0.1 to npm
3. Go to the `web-app` repo, open a PR to update the `package.json` to v1.0.1
4. Go to the `mobile-app` repo, open a PR to update to v1.0.1

In a **Monorepo**, you do this:
```bash
# You change the button in shared-ui, and use it in web-app immediately.
# You commit the ENTIRE atomic change at once.

git commit -am "fix(ui): update button styling across all apps"
# One PR. One CI run. Zero publishing delays.
```

## 8. Internal Working

Git was originally designed for the Linux Kernel, which is essentially a massive Monorepo. However, Git tracks *content*, not folders. When a monorepo gets to Google or Meta size (terabytes of code), standard `git clone` fails because you can't fit the repo on a laptop.

Modern monorepos use Git features like `sparse-checkout` (only pulling the folders you care about) and specialized build tools (like Turborepo) that cache outputs so you don't recompile the `mobile-app` if you only changed the `web-app`.

## 9. Performance Table

| Metric | Polyrepo | Monorepo |
|---|---|---|
| Initial `git clone` | Fast (kilobytes) | Slow (can be gigabytes). |
| Code Sharing | Slow (npm publish required) | Instant (local symlinks). |
| CI/CD Setup | Simple, independent pipelines | Complex, requires dependency graph analysis. |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Is a Monorepo the same as a Monolithic architecture? | No. Monorepo is how code is *stored* (version control). Monolith is how code is *deployed* (architecture). You can store 50 independent microservices in a monorepo. |
| Why do large companies prefer Monorepos? | To avoid "dependency hell". In a monorepo, there is only "One Version of the Truth". You never have microservice A using v1 of a library and microservice B using v2. |
| What is the biggest downside of a Monorepo? | Tooling complexity. Standard Git and standard CI/CD break down at scale. You must invest in build systems like Bazel or Nx. |
| How do you handle CI in a Monorepo? | You use a tool to generate a dependency graph. If a PR only touches the frontend app, the CI only runs frontend tests, skipping backend tests. |

## 11. Tricky Questions & Edge Cases

**Bro, if we use a Monorepo, does that mean everyone sees everyone else's code?**
Yes, usually. Monorepos encourage open visibility across the engineering team. If your company requires strict access control (e.g., contractors can only see Project A but not Project B), a Monorepo is a massive security headache. Git does not support folder-level read permissions.

**What if two teams want different versions of React in the Monorepo?**
In a strict monorepo, they can't. The rule is usually "One Version Rule" (one `package.json` version of React at the root). This forces teams to upgrade together, preventing the company from accumulating legacy tech debt.

## 12. Real-World Usage

Google runs the largest Monorepo in the world (billions of lines of code). To make this work, they invented their own version control system (Piper) and build system (Bazel). Facebook, Twitter, and Uber also use monorepos. However, Netflix uses Polyrepos. The industry trend for frontend/fullstack JS teams is heavily leaning toward Monorepos using tools like **Turborepo** or **Nx** to manage Next.js apps alongside shared React component libraries.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Use a smart build system (Nx, Turborepo) to cache builds and only run tests on changed code. | Don't use a Monorepo without tooling. Running `npm test` on 50 apps simultaneously will crash your CI. |
| Establish clear boundaries and ownership (using `CODEOWNERS` files) so PRs require the right team's approval. | Don't let apps import internal, private files from other apps directly. |

## 14. Production Notes

> **Warning: Git Clone Limits**
> If your monorepo contains thousands of high-res images or heavy binaries, `git clone` will eventually time out or run out of memory on developer laptops. You must implement `Git LFS` (Large File Storage) and use `git sparse-checkout` to ensure developers only download the directories they are actively working on.

## 15. Common Mistakes

| Mistake | Fix |
|---|---|
| Creating a "Monolith" by accident | Enforce strict linting rules so App A cannot import code from App B. They must communicate over APIs or use explicitly shared libraries. |
| Unmanageable CI times | Stop running `npm run build` on everything. Use tools that understand the dependency graph to only build what changed. |
