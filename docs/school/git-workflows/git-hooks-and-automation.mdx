# Git Hooks & Automation

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Git Hooks and Pre-commit Automation |
| Difficulty | Intermediate |
| Used For | Running scripts automatically before or after Git events (commit, push). |
| Common Mistake | Bypassing hooks with `--no-verify` because tests take too long. |
| Performance | Depends entirely on the scripts you run. |

## 2. Engineering Story

An e-commerce company had a strict rule: all code must be linted and tested before being pushed. However, developers being human, often forgot to run `npm run lint` locally. As a result, the CI pipeline would fail 20 minutes later over a missing semicolon. The CI servers became completely backlogged, and deployments ground to a halt just because of basic formatting errors. The DevOps team needed a way to physically stop developers from committing bad code on their local machines *before* it ever reached the remote server. They needed automated triggers.

## 3. Real-World Analogy

Bro, think of a Git Hook like the security scanner at an airport.

| Airport Scenario | Git Hook Equivalent |
|---|---|
| You pack your bags | Writing your code (`git add`) |
| You try to walk to the airplane | You try to commit (`git commit`) |
| Security stops you and scans your bag | The `pre-commit` hook runs a linter/tests |
| The scanner finds a prohibited item (water bottle) | The hook finds a syntax error |
| You throw the bottle away and try again | You fix the error and commit again |

If you don't pass the scanner, you don't get on the plane.

## 4. Concept Explanation

Git hooks are custom scripts that Git executes before or after events such as commit, push, and receive. They are a built-in feature of Git.

In modern engineering, we rely heavily on client-side hooks (specifically `pre-commit` and `pre-push`) to catch silly mistakes *before* they leave your laptop. If everyone runs a linter and unit tests automatically via a hook, the Pull Request review can focus on actual logic instead of missing semicolons or broken build steps.

## 5. Syntax Table

| Hook Name | When it runs | Typical Use Case |
|---|---|---|
| `pre-commit` | Before the commit is generated. | Run code linters (ESLint, Prettier). |
| `prepare-commit-msg` | Before the commit message editor opens. | Auto-insert branch names (e.g., `[JIRA-123]`) into the commit message. |
| `commit-msg` | After you save the message, before the commit is finalized. | Validate Conventional Commits format. |
| `pre-push` | Before `git push` sends data to the remote. | Run the full unit test suite. |

## 6. Beginner Example

Git hooks are just executable shell scripts located in `.git/hooks/`. By default, Git creates sample files there.

```bash
# 1. Navigate to your hooks directory
cd .git/hooks/

# 2. Create a new pre-commit file
echo '#!/bin/sh' > pre-commit
echo 'echo "Bro, I am a Git Hook!"' >> pre-commit
echo 'exit 1' >> pre-commit # Exit 1 blocks the commit

# 3. Make it executable
chmod +x pre-commit

# 4. Try to commit
git commit -m "test"
# Output:
# Bro, I am a Git Hook!
```

## 7. Real-World Engineering Example

Nobody writes bash scripts in `.git/hooks/` manually in production because `.git/hooks/` is not tracked by Git! It stays on your laptop.

Instead, JavaScript teams use a tool called **Husky** to track hooks in version control so the whole team shares them.

```json
// package.json
{
  "scripts": {
    "prepare": "husky install"
  }
}
```

```bash
# Set up Husky pre-commit hook
npx husky add .husky/pre-commit "npm run lint && npm run test"

# Now, whenever ANY developer on the team runs `git commit`:
# 1. ESLint runs.
# 2. Jest runs.
# 3. If either fails, the commit is aborted.
```

## 8. Internal Working

When you execute a Git command, the Git binary explicitly looks for executable files in the configured hooks directory (default `.git/hooks/`). If a hook matching the event name exists, Git pauses, runs the script, and checks the exit code. If the exit code is `0` (success), Git continues. If the exit code is non-zero (failure), Git aborts the operation.

## 9. Performance Table

| Hook Strategy | Performance | Developer Experience |
|---|---|---|
| Linting all files | Slow O(N) | Frustrating. Takes 30s to commit a 1-line change. |
| Linting ONLY staged files (`lint-staged`) | Fast O(1) | Excellent. Only lints what you actually modified. |
| Running all E2E tests | Very Slow (Mins) | Terrible for a `pre-commit` hook. Move this to CI or `pre-push`. |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Can Git hooks be pushed to a remote repository? | No. The `.git/hooks/` directory is strictly local and not tracked. You must use tools like Husky or `pre-commit` framework to share them. |
| How do you bypass a git hook? | By passing the `--no-verify` (or `-n`) flag. e.g., `git commit -m "fix" --no-verify`. |
| What is a server-side hook? | Hooks that run on the remote server (e.g., `pre-receive`). They can reject pushes if the code doesn't meet corporate policies. |
| What is `lint-staged`? | A tool that only runs linters on files currently in the staging area, making pre-commit hooks much faster. |

## 11. Tricky Questions & Edge Cases

**Bro, my hook works, but when my teammate clones the repo, the hook doesn't run for them. Why?**
Because hooks in `.git/hooks` are not cloned! To fix this, you need a package manager (like `npm`) to "install" the hooks on their machine automatically after they run `npm install`.

**I wrote a python script for my pre-commit hook, but Git ignores it!**
Did you make it executable? Run `chmod +x .git/hooks/pre-commit`. Also, ensure the first line is a valid shebang (e.g., `#!/usr/bin/env python3`).

## 12. Real-World Usage

In almost all modern Web and Python stacks, developers do not trust humans to format code. Tools like **Prettier** (JS) or **Black** (Python) are hooked into `pre-commit`. When you type `git commit`, the hook automatically reformats your messy code, stages the formatted file, and then creates the commit. You literally never have to format code manually again.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Use `lint-staged` to only run hooks on files that actually changed. | Don't run a 5-minute test suite on every `git commit`. Devs will start using `--no-verify`. |
| Track your hook configuration in version control (via Husky or `.pre-commit-config.yaml`). | Don't write manual bash scripts in `.git/hooks/` and expect your team to copy-paste them. |

## 14. Production Notes

> **Warning: Server-Side vs Client-Side**
> Client-side hooks (like `pre-commit` on your laptop) are for *developer convenience*. They are easily bypassed using `--no-verify`. You cannot rely on them for security. **Server-side validation** (CI/CD pipelines on GitHub Actions) is the only real source of truth for code quality and security gates before merging to production.

## 15. Common Mistakes

| Mistake | Fix |
|---|---|
| Hook script is tracked, but not executable | On Linux/Mac, files must have the executable bit set. Git tracks this bit! Fix with `git update-index --chmod=+x script.sh`. |
| CI Pipeline fails because a developer bypassed the pre-commit hook locally | This is expected. Ensure your CI runs the exact same lint/test commands as your local hooks. |
