# Git Advanced Operations

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Git Advanced Operations (Cherry-pick, Bisect, Stash, Reflog) |
| Difficulty | Advanced |
| Used For | Surgical history manipulation, finding bugs, and recovering lost work. |
| Common Mistake | Using these tools without understanding the underlying commit graph. |
| Performance | Operations like `bisect` save hours of manual debugging. |

## 2. Real-World Analogy

Bro, think of standard Git like driving a reliable sedan to work. Advanced Git is like having a mechanic's toolset in the trunk when the car breaks down on the highway.

| Mechanic Tool | Git Equivalent |
|---|---|
| Plucking one specific good apple from a rotten basket | `git cherry-pick` (taking a specific commit from another branch) |
| A time machine that takes you to parallel universes to find lost items | `git reflog` (finding lost commits) |
| Shoving your messy desk into a drawer before the boss walks in | `git stash` (saving uncommitted work temporarily) |
| Cutting the phonebook in half repeatedly to find a name | `git bisect` (binary search to find which commit caused a bug) |

## 3. Concept Explanation

As you grow into a senior engineer, you will inevitably destroy your local repository, lose code, or face a production bug where nobody knows when it was introduced.

Advanced operations give you surgical precision over the Git history. You can grab individual commits across branches without merging (`cherry-pick`), you can find commits that you thought you deleted (`reflog`), you can park half-finished work safely (`stash`), and you can mathematically prove exactly which commit broke the app (`bisect`).

## 4. Syntax Table

| Command | What it does |
|---|---|
| `git stash` | Saves your modified, uncommitted files to a temporary stack. |
| `git stash pop` | Applies the most recently stashed changes back to your working directory. |
| `git cherry-pick <hash>` | Applies the changes from a specific commit to your current branch. |
| `git reflog` | Shows a log of *every* movement of the HEAD pointer (even deleted branches). |
| `git bisect start` | Starts a binary search through history to find a bug. |

## 5. Beginner Example

You are working on a complex feature, and a critical bug is reported in production. You need to switch to `main` to fix it, but you have half-written code you don't want to commit yet.

```bash
# 1. Stash your messy work
git stash

# 2. Switch to main and fix the bug
git checkout main
# ... fix bug, commit, push ...

# 3. Go back to your feature and restore your messy work
git checkout feature/complex-logic
git stash pop
```

## 6. Real-World Engineering Example

The `git bisect` command is pure magic for debugging. Assume the app is broken on `main` right now, but you know it was working perfectly last Friday (on tag `v1.0`). There are 500 commits between now and last Friday.

```bash
# Start the search
git bisect start

# Tell Git the current commit is broken
git bisect bad

# Tell Git the older commit (v1.0) was working
git bisect good v1.0

# Git will checkout a commit exactly halfway between them.
# You test the app... "Does it work?"
# If yes:
git bisect good
# If no:
# git bisect bad

# Git will keep cutting the commits in half until it prints:
# "a1b2c3d4 is the first bad commit"
# You just found the exact line of code that caused the bug in 6 steps instead of 500!

# Reset when done
git bisect reset
```

## 7. Internal Working

To understand `reflog`, you must realize that Git rarely actually deletes data immediately. When you "delete" a branch or run `git reset --hard`, Git just moves pointers. The commits themselves still exist on your hard drive until Git's garbage collector (`git gc`) runs (usually 30-90 days later).

The `reflog` is literally a diary of where your `HEAD` pointer has been.

## 8. Performance Table

| Command | Performance | Note |
|---|---|---|
| `git stash` | O(N) | N = number of modified files. Extremely fast. |
| `git cherry-pick` | O(1) | Same speed as a standard commit. |
| `git bisect` | O(log N) | Mathematically the fastest way to search history. 1000 commits = ~10 tests. |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| I accidentally ran `git reset --hard` and lost my commits. How do I get them back? | Use `git reflog` to find the SHA hash of the lost commit, then `git checkout <hash>` or `git reset --hard <hash>` to restore it. |
| What is cherry-picking? | Applying the changes introduced by a specific commit on one branch onto a completely different branch. |
| Can you cherry-pick a merge commit? | Yes, but you must specify which parent branch to use as the mainline using the `-m` flag (e.g., `git cherry-pick -m 1 <hash>`). |
| What happens if you `git stash` but your files are untracked (new files)? | By default, Git ignores untracked files. You must use `git stash -u` (include untracked) to stash new files. |

## 10. Tricky Questions & Edge Cases

**Bro, I used `git stash pop` and I got a merge conflict! Where did my stash go?**
When `stash pop` results in a conflict, Git applies the changes to your working directory but *does not* remove the stash from the stash list. It keeps it safe. You must resolve the conflict, commit, and then manually run `git stash drop` to clean up the stack.

**If I cherry-pick a commit, is it the exact same commit?**
No. It has the same *changes* and the same *message*, but because its parent has changed, the SHA-1 hash is completely different.

## 11. Real-World Usage

**Cherry-picking** is heavily used in Release Management. Imagine `main` is unstable, but you need to release a critical hotfix to production immediately. You branch off the stable production tag, fix the bug, and deploy. But now `main` doesn't have the bugfix! You use `git cherry-pick <bugfix-hash>` while on `main` to pull that exact fix over without merging the old production branch.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Use `git stash` for temporary context switching. | Don't use `git stash` as a long-term backup strategy. Stashes are easy to forget and lose. Just commit to a WIP branch. |
| Use `git bisect run <script>` to fully automate the binary search process using a test script. | Don't manually test 50 commits if you have a unit test that proves the bug. |

## 13. Production Notes

> **Warning: Overusing Cherry-Pick**
> If you find yourself using `cherry-pick` every single day, your branching strategy is broken. Cherry-picking duplicates changes in your Git history (different hashes, same code). When those branches eventually merge later, you will face horrific, confusing merge conflicts.

## 14. Common Mistakes

| Mistake | Fix |
|---|---|
| Running `git bisect` and forgetting to stop | If your branch looks crazy and detached, run `git bisect reset` to return to where you started. |
| Forgetting what is in the stash | Run `git stash list` to see the stack, and `git stash show -p stash@{0}` to see the actual diff. |
