## 2. Engineering Story

An engineering team at a fintech startup set an SLI based on CPU utilization. CPU was consistently at 20%, so they thought everything was fine. However, the application was returning 500 errors to users because the database connection pool was exhausted. CPU was the wrong indicator because it didn't reflect the user's pain. They learned that measuring infrastructure doesn't equate to measuring the service.

## 4. Concept Explanation

SLIs should measure user happiness. They are the foundational metrics that tell you if your service is functioning correctly from the perspective of the people using it. Common types of SLIs include availability, latency, error rate, and throughput. A good SLI is directly correlated with user satisfaction. If the SLI is good, the user should be happy.

## 5. Syntax Table

| SLI Type | Calculation | Ideal Use Case |
|---|---|---|
| Availability | Good Requests / Total Requests | Web servers, APIs |
| Latency | Requests &lt; Threshold / Total | Interactive systems |
| Freshness | Data Updated Recently / Total | Data pipelines |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| Why is CPU a bad SLI? | It measures infrastructure, not user experience. |
| How do you measure an availability SLI? | Successful requests divided by total requests. |
| What is a latency SLI? | The percentage of requests served under a specific time limit. |
| Where should you measure SLIs? | As close to the user as possible (e.g., load balancer). |
| How many SLIs should a service have? | Usually 2-3 key ones (like latency and error rate). |

## 11. Tricky Questions & Edge Cases

How do you measure an SLI for an asynchronous background job? Measure 'freshness' or 'queue depth' instead of direct latency.

## 12. Real-World Usage

Netflix uses stream start latency and video playback error rates as their primary SLIs.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Measure at the load balancer | Don't measure deep inside the app server |
| Use ratio metrics (good/total) | Don't use absolute numbers |

## 14. Production Notes

<Admonition type="caution" title="Measurement Point">
  Measuring latency at the app server misses network latency. Measure at the edge.
</Admonition>

## 15. Common Mistakes

| Mistake | Correction |
|---|---|
| Tracking too many SLIs | Focus on the critical user journeys |
| Not defining 'success' | Clearly define what a successful request looks like |

Defining the right SLIs requires a deep understanding of your product architecture and user behavior. A poorly defined SLI will result in alerts firing when users are fine, or silence when the system is actually broken.

Always ask: 'If this metric degrades, is a user actually noticing?' If the answer is no, it shouldn't be an SLI.

Getting this right is often the hardest part of adopting SRE practices. It forces engineering teams to talk to product teams and actually understand the business value of their code.

Engineers must constantly evaluate the trade-offs between consistency, availability, and partition tolerance (the CAP theorem) when designing their services. No system can be perfect in all three dimensions simultaneously. Therefore, the decisions you make regarding which metrics to track and what targets to set will fundamentally shape the architecture of your application.

Continuous learning and iteration are key. What works for a startup with thousands of users will fail spectacularly for an enterprise with millions. Adapt your strategies as your scale dictates, always keeping the end-user experience as your north star.
