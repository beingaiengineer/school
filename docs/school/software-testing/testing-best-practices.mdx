# Testing Best Practices

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Testing Best Practices & Anti-Patterns |
| Difficulty | All Levels |
| Used For | Writing tests that are maintainable, reliable, and actually catch bugs without slowing down CI/CD. |
| Common Mistake | Writing tests that are tightly coupled to the implementation details, making refactoring impossible. |
| Performance | Good practices lead to blazing fast test suites (< 5 mins in CI). |

## 2. Engineering Story

A team inherited a 5-year-old Django monolith. It had 10,000 tests. Amazing, right? Except the test suite took 2 hours to run. Worse, every time an engineer renamed an internal variable or split a massive function into two, 50 unrelated tests would break. 

The engineers developed a deep fear of the codebase. Instead of refactoring, they layered new hacks on top. When a new tech lead arrived, she realized the tests were asserting against private variables and using the real database for simple math calculations. By enforcing clean testing practices—testing behaviors instead of implementations, and mocking heavy I/O—they reduced the suite time to 8 minutes and restored team velocity. Bro, bad tests are worse than no tests.

## 3. Real-World Analogy

Imagine setting up a security system for an art museum.

| Museum Security | Testing Equivalent |
|---|---|
| Putting an alarm on the diamond (the outcome) | **Testing public interfaces/behavior** |
| Putting an alarm on the specific chair the guard sits on | **Testing implementation details** (fragile!) |
| Testing the alarm once a month | **Running tests in CI/CD pipelines** |
| The alarm ringing randomly when a bird flies by | **Flaky Tests** |

If you wire the alarm to the guard's chair, the alarm goes off every time the guard stands up to stretch. The diamond is safe, but the alarm is useless. Tests must focus on the diamond (the output), not the chair (the internal code structure).

## 4. Concept Explanation

Testing Best Practices are the established guidelines that separate brittle, slow test suites from robust, fast ones. High-quality tests have three characteristics:
1. **Reliability:** They do not fail randomly (flakiness).
2. **Speed:** They provide immediate feedback to developers.
3. **Resilience to Refactoring:** You can completely rewrite the internal code, and the tests will still pass as long as the external output remains correct.

**When to use:** Every single time you write a test.
**When NOT to use:** Never. There is no excuse for writing bad tests.

## 5. Syntax Table

| Principle | Action | Explanation |
|---|---|---|
| **AAA Pattern** | Arrange, Act, Assert | Structure every test into 3 distinct blocks separated by a newline. |
| **DAMP over DRY** | Descriptive And Meaningful Phrases | In tests, code duplication is often better than complex, shared test logic. |
| **Deterministic** | Remove randomness | Hardcode seeds for random numbers or freeze time to prevent flaky tests. |

## 6. Beginner Example

Let's look at a bad test vs a good test using the AAA pattern and testing behavior.

```python
# Bad Test: Tests internal state, no clear structure
def test_user_creation_bad():
    u = User("john", "password123")
    u.save()
    assert u._is_active == True # Checking a private variable
    assert u._pwd_hash != "password123" 

# Good Test: AAA pattern, tests public behavior
def test_user_creation_good():
    # Arrange
    username = "john"
    raw_password = "password123"
    
    # Act
    u = User(username, raw_password)
    u.save()
    
    # Assert
    assert u.is_active() == True  # Checking public method
    assert u.check_password(raw_password) == True # Testing behavior, not the hash structure
```

## 7. Real-World Engineering Example

Handling Time in tests is the #1 cause of flaky tests in production systems.

```python
# The code checking if a token is expired
import datetime

def is_token_expired(created_at: datetime.datetime):
    now = datetime.datetime.now()
    return (now - created_at).total_seconds() > 3600

# BAD TEST: Relies on real time. Might fail if CPU is slow.
def test_token_expiration_bad():
    import time
    created = datetime.datetime.now()
    time.sleep(1) # Slows down test suite!
    assert is_token_expired(created) == False

# GOOD TEST: Freezing time
from freezegun import freeze_time

@freeze_time("2026-07-12 10:00:00")
def test_token_expiration_good():
    # Arrange: Create token EXACTLY at 10:00:00
    created = datetime.datetime.now()
    
    # Act & Assert 1: Exactly 1 hour later (Still valid)
    with freeze_time("2026-07-12 11:00:00"):
        assert is_token_expired(created) == False
        
    # Act & Assert 2: 1 hour and 1 second later (Expired)
    with freeze_time("2026-07-12 11:00:01"):
        assert is_token_expired(created) == True
```

## 10. Top Interview Questions

| Question | Answer | Level |
|---|---|---|
| **What is a "Flaky Test"?** | A test that occasionally fails without any code changes, usually due to network latency, time-zone issues, or asynchronous race conditions. | Intermediate |
| **Why is DAMP preferred over DRY in tests?** | In app code, Don't Repeat Yourself (DRY) is king. In tests, Descriptive And Meaningful Phrases (DAMP) is better. A test should be completely readable from top to bottom without jumping through 5 shared helper functions. | Advanced |
| **What does "Test Behavior, Not Implementation" mean?** | Tests should assert the final output or state change, not the internal variables or the specific algorithm used to achieve that state. | Intermediate |
| **How do you handle testing third-party APIs?** | Mock them in Unit tests to ensure logic works, and use Contract Tests or sandbox APIs in Integration tests to ensure compatibility. | Beginner |

## 11. Tricky Questions & Edge Cases

- **Random Data Generators (Faker):** Generating random data (like random emails or numbers) in tests seems like a good idea until it generates an edge-case string that breaks the test 1 out of 10,000 times. Now you have an unreproducible flaky test. Always use a hardcoded random `seed` in tests so "random" data is identical on every run.
- **The Global State Trap:** If Test A modifies an environment variable (`os.environ`) and forgets to reset it, Test B might fail. Every test must leave the global environment exactly as it found it.

## 12. Real-World Usage

Google uses a monolithic repository with millions of tests. They have strict, automated rules to detect flaky tests. If a test is detected as flaky, it is automatically removed from the blocking CI pipeline and quarantined. Developers are heavily penalized for writing tests that depend on sleep statements (`time.sleep()`) or exact dictionary orderings.

## 13. Best Practices

| DO | DON'T |
|---|---|
| **DO** name tests exactly what they do: `test_returns_404_when_user_not_found`. | **DON'T** name tests vaguely like `test_user_api`. |
| **DO** isolate tests completely. Test A must be able to run before or after Test B. | **DON'T** let tests share state. If Test B relies on Test A creating a DB record, you are doing it wrong. |
| **DO** treat test code as production code. Review it with the same rigor. | **DON'T** use `time.sleep()`. Use mocking or polling with timeouts instead. |

## 14. Production Notes

> **Warning: Code Coverage Illusion**  
> We covered this before, but it bears repeating: A codebase with 100% coverage and bad tests is vastly inferior to a codebase with 60% coverage and incredibly rigorous, behavior-driven tests on critical paths. Do not game the metric.

## 15. Common Mistakes

| Mistake | Bad Practice | The Fix |
|---|---|---|
| **Logic in Tests** | Using `if`, `else`, or `while` loops inside a test function. | Tests should be a straight line of code (AAA). If you have logic, you need multiple separate tests. |
| **Ignoring the Build** | "The tests failed on CI, but it works on my machine. Just force merge it." | The CI server is the source of truth. Fix the test or fix the environment. |
| **Over-mocking** | Mocking the ORM, the DB, the cache, and the auth service for a single integration test. | Use real test infrastructure (like Testcontainers) for integration testing. |
