# Threat Modeling in Software Engineering

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Threat Modeling |
| Difficulty | Intermediate |
| Used For | Identifying structural security flaws during the architecture and design phase before writing code. |
| Common Mistake | Treating it as a post-development checklist instead of a pre-development design requirement. |
| Performance | Zero runtime impact; it is a design-time process that informs architectural decisions. |

## 2. Engineering Story

A major fintech company launched a highly anticipated "quick-loan" feature. The engineering team had spent weeks writing code, achieving 100% test coverage, and extensively load-testing the microservices. Everything looked solid on launch day. However, just two days later, an attacker figured out that the `/api/v1/loan/disburse` endpoint did not verify if the `accountId` in the request body matched the `userId` in the authenticated JWT token. 

The attacker simply logged in with their own legitimate account, intercepted their own API call, changed the `accountId` payload to point to a stranger's account, and initiated loan disbursements. This wasn't a complex buffer overflow or a zero-day exploit. It was a simple business logic flaw—an **Insecure Direct Object Reference (IDOR)** resulting in **Elevation of Privilege**. 

The engineering team had spent weeks on unit tests but zero hours asking, "What if a valid user sends malicious input?" They had no threat model. Fixing this bug took only five lines of authorization code, but the reputational damage and financial loss cost the company millions. If they had spent just two hours whiteboarding the architecture and systematically asking "How can this system be abused?", they would have spotted the missing authorization check instantly.

## 3. Real-World Analogy

Imagine you are setting up a new, high-end jewelry store in a busy market like Zaveri Bazaar or Chandni Chowk. You wouldn't just stock the gold, open the doors, and blindly hope people are honest. Before the grand opening, you stand outside and intentionally think like a thief. You ask: "If someone distracts the cashier, can they grab that necklace?" "If they break the front glass at night, how long until the police arrive?" "What if an employee pockets a diamond?"

Once you brainstorm these threats, you install mitigations: reinforced shatterproof glass, CCTV cameras with no blind spots, a security guard at the door, and a vault with a time lock. Threat modeling in software is exactly this process—but instead of securing a physical store, you are securing your application's architecture against digital thieves.

| Physical World (Jewelry Store) | Software Equivalent |
|---|---|
| Thinking like a thief | Threat Modeling Session |
| The physical store boundary | Trust Boundaries (e.g., Public Internet vs. Private VPC) |
| Breaking the glass | Bypassing Authentication / Injection attacks |
| Installing CCTV and alarms | Audit Logging and Monitoring (SIEM) |
| Hiring a security guard | Implementing Rate Limiting, IAM, and WAF |
| Locking gold in a safe | Encrypting Data at Rest (AES-256) |

## 4. Concept Explanation

Threat modeling is a structured, systematic process of evaluating a software architecture to identify potential security vulnerabilities and defining countermeasures *before* a single line of code is written. It shifts security to the absolute extreme "left" of the Software Development Life Cycle (SDLC).

**Why does it exist?** Because fixing a security flaw in the design phase is incredibly cheap—it costs nothing but time and a whiteboard eraser. Fixing the same flaw in production after a data breach can cost millions of dollars in compliance fines, lost revenue, and lawsuits.

**When to use it?** You must perform threat modeling during the system design or architecture phase. It is strictly required when introducing new features that handle sensitive PII (Personally Identifiable Information), authentication workflows, financial transactions, or external third-party integrations.

**When NOT to use it?** Avoid doing a full, heavy-weight threat model for trivial frontend UI changes, modifying static text, or updating documentation. Threat modeling is a high-leverage activity; apply it where the risk justifies the effort.

## 5. Syntax Table

Threat modeling is a conceptual process, not a programming language. However, the industry standard "syntax" or framework used to categorize threats is **STRIDE**, developed by Microsoft. Here is how it maps to threats:

| STRIDE Element | What it stands for | Example Threat | Mitigation Strategy |
|---|---|---|---|
| **S** | **Spoofing Identity** | An attacker logs in as another user or system. | Strong Authentication (MFA, OAuth2, mutual TLS). |
| **T** | **Tampering with Data** | An attacker modifies a product price while data is in transit. | TLS / SSL, Digital Signatures, HMACs. |
| **R** | **Repudiation** | A malicious user denies transferring funds and you cannot prove they did. | Secure, append-only Audit Logging. |
| **I** | **Information Disclosure** | An attacker reads sensitive database records. | Encryption at rest, Strict RBAC, Data Masking. |
| **D** | **Denial of Service** | A botnet floods your API, causing it to crash for legitimate users. | Rate Limiting, Web Application Firewalls (WAF), Auto-scaling. |
| **E** | **Elevation of Privilege** | A standard user exploits a bug to become an admin. | Strict Authorization checks, Principle of Least Privilege. |

## 6. Beginner Example

Let's perform a mini threat model for a basic login endpoint.

**Scenario**: A web application allows users to log in using an email and password.

**Threats Identified**:
1. **Spoofing**: An attacker tries a billion passwords until they guess the correct one (Brute Force).
2. **Information Disclosure**: An attacker steals the database backup and reads all user passwords.

**Mitigations**:
1. **Mitigation for Spoofing**: Implement rate limiting on the `/login` endpoint (e.g., maximum 5 failed attempts per minute) and require Multi-Factor Authentication (MFA).
2. **Mitigation for Info Disclosure**: Never store plaintext passwords. Hash them using a strong cryptographic algorithm like bcrypt or Argon2 with a unique salt.

## 7. Real-World Engineering Example

Let's look at a real system design scenario: A backend microservice that generates PDF invoices and emails them to enterprise customers.

**Threat Model for the Invoice Generator System:**

1. **Threat (Elevation of Privilege):** The Invoice Service microservice connects directly to the central PostgreSQL database to read customer data. If the service is compromised via a dependency vulnerability, can the attacker drop the database tables?
   - *Mitigation:* The database user credential assigned to the Invoice Service must have `SELECT` privileges only. Absolutely no `UPDATE`, `INSERT`, or `DELETE` permissions should be granted (Principle of Least Privilege).

2. **Threat (Denial of Service):** Generating large PDFs is heavily CPU-bound. A malicious competitor registers a free account and requests 10,000 invoices at once, locking up the server threads and crashing the service for paying customers.
   - *Mitigation:* Decouple the request flow. The public API should immediately place the generation request into an SQS or RabbitMQ message queue and return an HTTP 202 Accepted. Background worker nodes will pull from the queue and generate PDFs asynchronously at their own controlled pace.

3. **Threat (Information Disclosure):** Once generated, the PDF invoices are stored in an Amazon S3 bucket. If the bucket URL is guessed, can anyone download my company's invoice?
   - *Mitigation:* Ensure the S3 bucket is strictly private. When a user clicks "Download", the API must verify their session authorization, and only then generate an AWS Pre-signed URL with a short 5-minute expiration window.

## 8. Internal Working

Threat Modeling is fundamentally a process of analyzing data flows across **Trust Boundaries**. When an engineer constructs a threat model, they evaluate the exact path a request takes from an untrusted zone (like the internet) into the core execution engine (like a database). 

Tools like Microsoft Threat Modeling Tool or Threagile programmatically parse a Data Flow Diagram (DFD). They identify entities, processes, data stores, and the data flows connecting them. The engine then overlays the STRIDE ruleset wherever a data flow crosses a Trust Boundary.

## 9. Performance Table

Threat modeling itself is a design-time activity and has absolute zero runtime performance cost. However, the *mitigations* you decide to implement will introduce overhead. You must balance security with performance.

| Mitigation Strategy | Runtime Impact | Performance Considerations |
|---|---|---|
| **Hashing Passwords** (Bcrypt/Argon2) | High (CPU) | These algorithms are intentionally designed to be slow to deter brute forcing. Do not run them on the main Node.js event loop without offloading. |
| **TLS/SSL Encryption** | Low | Modern CPUs have hardware acceleration (AES-NI) that handles encryption overhead easily. |
| **Audit Logging** | Low to Medium | Avoid synchronous disk I/O. Use asynchronous logging agents or background workers to send logs to your SIEM. |
| **Role-Based Authorization Checks** | Low | Usually implemented via a fast DB index query, an in-memory cache check, or by inspecting a signed JWT claim. |
| **Web Application Firewall (WAF)** | Medium (Latency) | Adds network hops and regex inspection time to every incoming request. |

## 10. Top Interview Questions

| Question | Answer |
|---|---|
| **What is Threat Modeling?** | It is a structured process to identify, quantify, and address security risks in a software architecture before code is written. |
| **Explain the STRIDE methodology.** | It is a framework that categorizes threats into Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege. |
| **At what phase of the SDLC should threat modeling occur?** | It should occur during the Architecture and Design phase, early in the Software Development Life Cycle. |
| **What is a "Trust Boundary"?** | A line in an architecture diagram where data changes its level of trust (e.g., crossing from the public internet into a private API subnet). |
| **How do you prioritize threats once identified?** | Threats are typically prioritized by assessing risk using frameworks like DREAD (Damage, Reproducibility, Exploitability, Affected users, Discoverability) or a standard High/Medium/Low risk matrix based on impact vs. likelihood. |


## 11. Tricky Questions & Edge Cases

**Question:** If we run everything in a managed, highly secure AWS environment (RDS, API Gateway, Fargate), do we still need to threat model our application?
**Answer:** Absolutely. Cloud providers operate on a **Shared Responsibility Model**. AWS secures the physical infrastructure, but *you* are responsible for securing the data, the application business logic, and the IAM permissions. A misconfigured S3 bucket, an IDOR vulnerability, or a lack of rate limiting is entirely your fault, not Amazon's. 

**Question:** We just finished our threat modeling session and found 50 potential threats. Do we have to fix all of them before launching the feature?
**Answer:** No. You are required to *evaluate and document* all of them. Some threats might have an extremely low probability or negligible impact. The business might choose to explicitly "Accept the Risk." Threat modeling is about informed, conscious decision-making, not aiming for an impossible state of perfect security.

## 12. Real-World Usage

Top-tier technology companies integrate threat modeling deeply into their engineering culture. Microsoft mandates threat modeling as a core pillar of their **Security Development Lifecycle (SDL)**. Before a new Azure service, Windows feature, or Xbox update is approved for development, engineering leads must present a comprehensive threat model to a security review board. This presentation must clearly show data flow diagrams, identified trust boundaries, and the planned engineering mitigations. Only after approval can the sprint begin.

## 13. Best Practices

| DO | DON'T |
|---|---|
| **DO** perform threat modeling early in the design phase alongside the RFC or design document. | **DON'T** wait until the code is fully written to start analyzing structural threats. |
| **DO** focus heavily on analyzing data crossing trust boundaries (e.g., external API to internal Database). | **DON'T** waste time exhaustively threat-modeling internal, highly trusted components communicating over secure, isolated channels. |
| **DO** make threat modeling a collaborative whiteboard effort between developers, architects, and product managers. | **DON'T** let the security team do it in isolation and hand over a disconnected PDF report. |
| **DO** create actionable engineering tasks (Jira tickets) for the identified mitigations. | **DON'T** treat the threat model document as a one-time compliance exercise that sits on a shelf. |

## 14. Production Notes

> ⚠️ **Warning on Scope Creep and "Hollywood Scenarios"**
> A common failure mode when teams first start threat modeling is getting bogged down in unlikely "Hollywood" scenarios. Engineers might spend hours debating, "What if an advanced persistent threat from a nation-state compromises the CPU microcode on our database server via a zero-day hardware exploit?" 
> 
> Unless you are building systems for nuclear silos or central banks, **ignore these**. Focus your energy on the highest-probability, most common attack vectors: missing authentication, broken authorization (IDOR), lack of input validation (SQLi/XSS), and overly permissive IAM roles.

## 15. Common Mistakes

| Mistake | Why it's bad | Corrective Action |
|---|---|---|
| **Doing it too late in the SDLC** | Mitigations often require foundational architectural changes. Changing a system's architecture after the code is written is incredibly expensive and frustrating. | Make a lightweight threat model document a required, gating step of every RFC or Technical Design Document. |
| **Ignoring Insider Threats** | Assuming all attacks come from "evil hackers on the outside" leaves you entirely vulnerable to compromised employee accounts or rogue admins. | Apply Zero Trust architecture principles and strict RBAC/Least Privilege even for internal company tooling. |
| **Missing Trust Boundaries** | If you do not draw clear boundaries where data enters or exits secure zones, you will inevitably miss injection flaws and spoofing attacks. | Explicitly draw and label trust boundaries on your Data Flow Diagrams (DFDs) using thick red dashed lines. |
