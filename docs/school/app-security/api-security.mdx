# API Security

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | API Security |
| Difficulty | Intermediate |
| Used For | Protecting the programmatic interfaces that power modern applications |
| Common Mistake | Assuming undocumented APIs are hidden from attackers |
| Performance | Slight latency added by WAFs, rate limiters, and payload inspection |

## 2. Real-World Analogy

| Physical World | Software Equivalent |
|---|---|
| A restaurant menu ordering system | An API Endpoint |
| Ordering 10,000 pizzas to crash the kitchen | Denial of Service (DoS) attack |
| Limiting customers to ordering 5 pizzas at once | Rate Limiting |
| Changing the table number on your ticket to get someone else's food | Broken Object Level Authorization (BOLA) |

Bro, imagine you run a massive cloud kitchen (like a backend API) that serves Swiggy and Zomato. You don't have waiters; you just have an automated counter where delivery guys drop tickets. If you don't check if the ticket is valid (AuthN), if you don't check if the guy is allowed to take that specific order (AuthZ), or if you let one guy drop 50,000 tickets at once (Rate Limiting), your kitchen will collapse. API security is building strict rules for that automated counter.

## 3. Concept Explanation

API Security focuses on securing the Application Programming Interfaces that connect microservices, mobile apps, and single-page applications to backend data.

Why does it exist? Because APIs are designed to be consumed by machines. They expose direct access to business logic and databases. Unlike traditional web pages, if an API is vulnerable, an attacker can write a script to exploit it thousands of times per second.

When to use it? Any time you expose an endpoint over HTTP/gRPC. Internal APIs need security just as much as public APIs (Zero Trust architecture).

## 4. Syntax Table

Here are the primary layers of API security you need to implement:

| Security Layer | Implementation | Purpose |
|---|---|---|
| **Transport Security** | TLS (HTTPS) | Encrypts data in transit to prevent sniffing. |
| **Authentication** | JWT, OAuth 2.0 | Verifies the identity of the API caller. |
| **Authorization** | Scope checking, RBAC, ABAC | Ensures the caller can perform the action. |
| **Rate Limiting** | Redis-based token bucket | Prevents brute-forcing and resource exhaustion. |
| **Input Validation** | Joi, Zod, express-validator | Rejects malformed or malicious payloads (SQLi/XSS). |

## 5. Beginner Example

Let's look at a very common API flaw: Mass Assignment.

```javascript
// DON'T: Mass Assignment Vulnerability
app.put('/api/users/:id', async (req, res) => {
  const user = await User.findByPk(req.params.id);

  // Attacker sends { "email": "hacked@a.com", "isAdmin": true }
  // You blindly update everything they sent!
  await user.update(req.body);
  res.json(user);
});

// DO: Explicit payload extraction
app.put('/api/users/secure/:id', async (req, res) => {
  const user = await User.findByPk(req.params.id);

  // Only extract the fields the user is allowed to change
  const { email, name } = req.body;

  await user.update({ email, name });
  res.json(user);
});
```

## 6. Real-World Engineering Example

Bro, let's look at how you secure an API endpoint using multiple layers of defense: Authentication, Authorization, Input Validation, and Rate Limiting.

```javascript
const express = require('express');
const rateLimit = require('express-rate-limit');
const { z } = require('zod');
const { verifyToken, requireRole } = require('./middlewares/auth');

const app = express();
app.use(express.json());

// 1. Rate Limiting: Max 5 payment requests per minute per IP
const paymentLimiter = rateLimit({
  windowMs: 1 * 60 * 1000,
  max: 5,
  message: { error: 'Too many payment requests, please try again later.' }
});

// 2. Input Validation Schema (Zod)
const paymentSchema = z.object({
  amount: z.number().positive().max(10000), // Prevent sending -500 to steal money
  currency: z.enum(['USD', 'INR', 'EUR']),
  targetAccountId: z.string().uuid()
});

// 3 & 4. Authentication (verifyToken) & Authorization (requireRole) applied as middleware
app.post('/api/payments/transfer', paymentLimiter, verifyToken, requireRole('verified_user'), async (req, res) => {

  try {
    // 5. Strictly validate the incoming body
    const validatedData = paymentSchema.parse(req.body);

    // Process the payment safely using validatedData
    const receipt = await PaymentService.transfer(req.user.id, validatedData);

    res.json(receipt);
  } catch (error) {
    // Fail securely
    if (error instanceof z.ZodError) {
      return res.status(400).json({ error: 'Invalid payload', details: error.errors });
    }
    res.status(500).json({ error: 'Internal server error' });
  }
});
```
This is production-grade. The attacker has to pass the rate limiter, have a valid JWT, have the right role, and provide perfectly formatted data before the business logic even executes.

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What is BOLA (Broken Object Level Authorization)? | When an API allows a user to access an object (like a receipt) simply by guessing its ID in the URL, without checking if the user actually owns that object. |
| Why is "Security through Obscurity" a bad idea for APIs? | Because attackers use fuzzers to find hidden endpoints. If an endpoint exists (e.g., `/api/v1/internal-admin`), they will find it. It must be secured, not just hidden. |
| What is a replay attack and how do you prevent it in APIs? | An attacker intercepts a valid API request and resends it later to duplicate an action (like a payment). Prevent it using short-lived tokens, nonces, or idempotency keys. |
| Why should you implement pagination on list APIs? | To prevent Denial of Service (DoS) and data scraping. If an attacker requests `limit=1000000`, your database will crash trying to fetch all records. |
| What is Mass Assignment? | When an API takes a JSON payload and directly injects it into a database model, allowing an attacker to overwrite fields (like `isAdmin`) they shouldn't have access to. |

## 10. Tricky Questions & Edge Cases

**Question:** If our frontend is a React SPA, do we need to implement CORS (Cross-Origin Resource Sharing) on the API? Does CORS protect our API from hackers?
**Answer:** You must implement CORS so legitimate browsers can make requests. However, CORS is a *browser* security feature, not an API security feature. It protects the *user*, not the server. An attacker using `curl` or Postman bypasses CORS entirely. Don't rely on CORS to secure your API.

**Question:** We use GraphQL instead of REST. Does that solve our API security issues?
**Answer:** It actually introduces new ones, bro! GraphQL allows the client to dictate the query depth. An attacker can write a deeply nested query (e.g., `author -> posts -> comments -> author -> posts`) that causes a catastrophic Denial of Service on your database. You must implement Query Depth Limiting and Query Complexity Analysis.

## 11. Real-World Usage

Companies like Stripe and Twilio, whose entire product is an API, treat API security as their core competency. They use **Idempotency Keys** extensively. If your server makes an API call to charge a customer, but the network drops before you get the response, you might retry the request. The Idempotency Key ensures Stripe recognizes it as a retry and doesn't charge the customer twice, preventing both accidental bugs and malicious replay attacks.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Validate all incoming data against a strict schema (allow-listing). | Rely on the frontend to validate data before sending it. |
| Enforce rate limiting on all endpoints, especially authentication/OTP routes. | Allow unlimited requests assuming your infrastructure will scale. |
| Use UUIDs (v4) for sensitive object IDs (e.g., `/users/d4e5f6...`). | Use sequential integers (`/users/5`), which makes BOLA and scraping trivial. |
| Implement Idempotency Keys for critical state-changing actions (payments, orders). | Assume network requests will only ever happen once. |

## 13. Production Notes

> ⚠️ **Warning on Versioning and Zombie APIs**
> A massive risk in production is "Zombie APIs". You launch `v2` of your API and secure it perfectly. But you leave `v1` running in the background for "legacy support." Attackers will find `v1`, realize it lacks the new rate limiters, and exploit it. Always have a strict deprecation and sunsetting process for old API versions.

## 14. Common Mistakes

| Mistake | Why it's bad | How to fix it |
|---|---|---|
| Exposing Stack Traces | A 500 error that returns a full Java or Node stack trace tells the attacker exactly what libraries and database you are using. | Catch all unhandled exceptions globally and return a generic `{"error": "Internal Server Error"}`. |
| Insecure HTTP Methods | Using `GET` for state-changing operations allows them to be cached, bookmarked, and subjected to CSRF attacks. | Strictly adhere to REST semantics. Use `POST`, `PUT`, `DELETE` for state changes. |
| Lacking Pagination Limits | Allows attackers to exhaust server memory and DB CPU. | Hardcode a maximum limit (e.g., `MAX_LIMIT = 100`) in your API handlers, even if the user requests more. |
