# Security in CI/CD (DevSecOps)

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Security in CI/CD (DevSecOps) |
| Difficulty | Intermediate/Advanced |
| Used For | Automating security checks into the software delivery pipeline |
| Common Mistake | Running security scans only right before a major release |
| Performance | Can significantly slow down build times if scans are not optimized |

## 2. Real-World Analogy

| Physical World | Software Equivalent |
|---|---|
| An airport security metal detector | SAST (Static Application Security Testing) |
| A guard checking the background of a contractor | Dependency/SCA Scanning |
| Running a simulation to test the fire alarms | DAST (Dynamic Application Security Testing) |
| Giving a pilot the keys to the plane only at takeoff | Dynamic Secrets in Deployments |

Bro, think about airport security. You don't just let someone walk from the street straight onto the airplane and then check their bags mid-flight. You check their ticket at the entrance, scan their bags at the checkpoint, and verify their identity at the gate. DevSecOps is exactly this. Instead of a security team doing a massive "penetration test" at the very end of the year (mid-flight), we build metal detectors directly into the GitHub Actions pipeline so bad code can't even get on the plane.

## 3. Concept Explanation

Security in CI/CD, often referred to as DevSecOps or "Shifting Left", is the philosophy of integrating automated security checks throughout the software development lifecycle, rather than treating security as an afterthought.

Why does it exist? If a developer writes a SQL injection vulnerability on Monday, and the security team finds it six months later in staging, it takes weeks of meetings and context-switching to fix. If a CI pipeline catches it on Monday afternoon before the PR is merged, the developer fixes it in 5 minutes.

When to use it? Every modern engineering team must have at least basic security gates (dependency scanning, secret scanning, and static analysis) in their CI pipeline.

## 4. Syntax Table

Here are the primary types of automated security scans you put into a pipeline:

| Scan Type | Stands For | What it does | When it runs |
|---|---|---|---|
| **Secret Scanning** | - | Greps the code for API keys (e.g., `sk_live_...`). | Pre-commit / PR open |
| **SCA** | Software Composition Analysis | Checks npm/pip dependencies for known CVEs. | PR open / Daily |
| **SAST** | Static Application Security Testing | Analyzes source code without running it to find flaws (e.g., SQLi). | PR open |
| **DAST** | Dynamic Application Security Testing | Attacks a running application to find vulnerabilities. | Staging deployment |
| **Container Scan** | - | Checks Docker images for vulnerable OS packages. | Post-build / Registry push |

## 5. Beginner Example

Let's look at the easiest win for a pipeline: preventing hardcoded secrets from being merged.

```yaml
# DON'T: A pipeline that just builds and deploys.
name: Unsafe Pipeline
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm install && npm run build

# DO: A pipeline that blocks secrets
name: Secure Pipeline
on:
  pull_request:
    branches: [ main ]
jobs:
  secret-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0 # Important: fetch history to scan all commits
      - name: TruffleHog Secret Scanner
        uses: trufflesecurity/trufflehog@main
        with:
          path: ./
          base: ${{ github.event.repository.default_branch }}
          head: HEAD
          extra_args: --debug --only-verified
```
If a developer accidentally leaves `AWS_SECRET_KEY="AKIA..."` in a file, this pipeline fails the PR and blocks the merge.

## 6. Real-World Engineering Example

Bro, let's look at a comprehensive DevSecOps pipeline for a Dockerized Node.js application. We will scan dependencies, run SAST, build the image, and scan the image.

```yaml
name: DevSecOps Pipeline
on: [pull_request]

jobs:
  security-gates:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      # 1. SCA: Dependency Scan
      - name: Run npm audit
        run: npm audit --audit-level=high

      # 2. SAST: Static Analysis using Semgrep
      - name: Run Semgrep SAST
        uses: returntocorp/semgrep-action@v1
        with:
          config: "p/owasp-top-ten"

      # 3. Build the Docker Image
      - name: Build Image
        run: docker build -t my-app:${{ github.sha }} .

      # 4. Container Scan using Trivy
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'my-app:${{ github.sha }}'
          format: 'table'
          exit-code: '1' # Fails the build if vulnerabilities are found
          ignore-unfixed: true
          vuln-type: 'os,library'
          severity: 'CRITICAL,HIGH'
```
This is a robust gate. Code cannot reach production unless it is free of severe known vulnerabilities at the code, dependency, and OS levels.

## 8. Performance Table

Integrating security tools increases CI pipeline times. Optimization is crucial.

| Scan Type | Average Time | Optimization Strategy |
|---|---|---|
| Secret Scanning | ~5-30s | Very fast. Run it on every commit. |
| SCA (npm audit) | ~10-60s | Fast. Run on every PR. Cache `node_modules`. |
| SAST (Semgrep/Sonar) | ~2-10 mins | Can be slow on massive monorepos. Run differential scans (only scan changed files). |
| DAST (ZAP) | ~15-60 mins | Very slow. Run nightly or on staging deploys, NOT on every PR. |
| Container Scan (Trivy) | ~1-3 mins | Cache the vulnerability database locally in the CI runner. |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| What does "Shift Left" mean in DevSecOps? | Moving security testing earlier in the Software Development Life Cycle (SDLC), towards the developers (the "left" side of the timeline), rather than waiting until the deployment phase. |
| What is the difference between SAST and DAST? | SAST (White-box) analyzes source code statically without running it. DAST (Black-box) attacks a running, compiled application from the outside to find flaws. |
| Why is Secret Scanning necessary if the repository is private? | Because private repositories are often cloned to dozens of developer laptops. If one laptop is compromised, the attacker gets the repo and all the secrets inside it. |
| What is a false positive in SAST? | When the static analysis tool flags a piece of code as vulnerable, but in the actual context of the application, it cannot be exploited. |
| How do you securely provide deployment credentials (like AWS keys) to a CI pipeline? | Never hardcode them. Use the CI provider's secure Secrets manager (e.g., GitHub Actions Secrets) or use OIDC (OpenID Connect) to assume short-lived cloud roles without storing long-lived keys. |

## 10. Tricky Questions & Edge Cases

**Question:** We run a SAST tool on every PR, and it takes 45 minutes to finish. Developers are complaining and bypassing it. What do we do?
**Answer:** Security must be ergonomic, bro. A 45-minute PR gate ruins developer velocity. You should configure the SAST tool to run in "differential mode" (only scanning the files changed in the PR) for fast feedback, and run the full 45-minute deep scan asynchronously on the `main` branch every night.

**Question:** If we use Docker and scan our images in CI, do we still need SCA (Dependency Scanning) for our Node.js code?
**Answer:** Yes. Container scanners (like Trivy/Clair) are great at finding vulnerabilities in OS-level packages (like `apt-get install curl`), but they sometimes miss application-level dependencies (like `npm` or `pip` packages) buried inside the container layers. You need both.

## 11. Real-World Usage

At companies like GitHub and GitLab, security is completely API-driven. When a developer pushes code, an event webhook triggers asynchronous serverless functions that run SAST, SCA, and Secret Scanning in parallel. If a secret is detected (like an AWS key), GitHub doesn't just fail the build—it uses automated workflows to immediately notify AWS to revoke the key within seconds, preventing any exploitation window.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Fail the build for Critical/High vulnerabilities. | Fail the build for Low/Informational findings, frustrating developers. |
| Scan for secrets as early as possible (e.g., using pre-commit hooks locally). | Wait until the code is merged into `main` to look for secrets. |
| Use OIDC to authenticate your CI runner to your cloud provider (AWS/GCP/Azure). | Store long-lived AWS IAM Access Keys inside GitHub Secrets. |
| Dedicate time in sprints to fix the "technical security debt" found by scanners. | Ignore the scanner output because "we will fix it later." |

## 13. Production Notes

> ⚠️ **Warning on CI/CD Compromise**
> Your CI/CD platform (GitHub Actions, Jenkins, GitLab CI) has god-level access to your production environment. Attackers know this. The SolarWinds hack happened because the build system was compromised. You must secure the CI/CD pipeline itself: enforce MFA for all developers, restrict who can modify pipeline YAML files, and run CI runners in isolated, ephemeral environments.

## 14. Common Mistakes

| Mistake | Why it's bad | How to fix it |
|---|---|---|
| Ignoring False Positives | If a SAST tool generates 90% false positives, developers will get "alert fatigue" and ignore real threats. | Actively tune your SAST rules. Suppress false positives explicitly in code so the tool ignores them in the future. |
| Over-privileged CI Runners | If a pipeline only needs to deploy to an S3 bucket, but the CI runner has `AdministratorAccess`, a malicious PR can take over your AWS account. | Apply the Principle of Least Privilege to CI/CD IAM roles. |
| Testing in Production | Running aggressive DAST scans against a production database can fill it with garbage data or accidentally trigger destructive actions. | Only run DAST against isolated Staging or ephemeral QA environments. |
