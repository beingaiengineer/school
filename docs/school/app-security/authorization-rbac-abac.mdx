Bro, remember the tech park example from Authentication? So, the guard checked your ID and let you into the building (AuthN). Now you are standing in the lobby. Can you walk into the CEO's office? Can you enter the server room? No. Your badge only opens the door to the engineering floor. That is Authorization (AuthZ).

RBAC is like saying: "He has an Engineer badge, he goes to the engineering floor." ABAC is like saying: "He has an Engineer badge, BUT it's Sunday 2 AM, so the door stays locked."

## 3. Concept Explanation

Authorization is the process of verifying that an authenticated entity has the necessary permissions to access a specific resource or perform a specific action.

- **RBAC (Role-Based Access Control):** Permissions are tied to roles (Admin, Editor, Viewer). Users are assigned roles. If the role can do the action, the user can do the action.
- **ABAC (Attribute-Based Access Control):** Permissions are evaluated dynamically based on attributes of the user (department), the resource (classification level), the action, and the environment (time of day, IP address).

Why does it exist? To enforce the Principle of Least Privilege. Users should only have access to exactly what they need to do their jobs.

When to use it? RBAC is the standard for 90% of applications. Use ABAC when rules become highly contextual (e.g., healthcare apps where a doctor can only view a record IF they are the assigned physician AND they are on shift).

## 4. Syntax Table

Here is how different authorization models manifest in code or configuration:

| AuthZ Model | Example Logic | Use Case |
|---|---|---|
| **ACL (Access Control List)** | `if (doc.allowedUsers.includes(user.id))` | Simple file sharing systems. |
| **RBAC** | `if (user.role === 'ADMIN')` | Standard SaaS apps (Owner/Admin/Member). |
| **ABAC** | `if (user.dept === 'HR' && user.ip === '10.0.0.x')` | High-security or highly regulated systems. |
| **PBAC (Policy-Based)** | Evaluated via an engine like OPA (Open Policy Agent) | Large-scale microservice architectures. |

## 5. Beginner Example

A common beginner mistake is scattering role checks randomly throughout the code.

```javascript
// DON'T: Hardcoding roles in the business logic
app.delete('/article/:id', async (req, res) => {
  if (req.user.role !== 'admin' && req.user.role !== 'superadmin') {
    return res.status(403).send('Forbidden');
  }
  // Delete logic...
});

// DO: Use middleware to centralize RBAC
function requireRole(role) {
  return (req, res, next) => {
    if (req.user.role !== role) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  }
}

app.delete('/article/:id', requireRole('admin'), async (req, res) => {
  // We know they are an admin if we reach here. Delete logic...
});
```

## 6. Real-World Engineering Example

Bro, in a real system, RBAC isn't enough. You often need **Resource-Based Authorization** (a simple form of ABAC). An Editor can edit articles, but can they edit *your* article?

```javascript
// A robust authorization middleware pattern
const authorizeArticleEdit = async (req, res, next) => {
  const articleId = req.params.id;
  const userId = req.user.id;
  const userRole = req.user.role;

  // 1. RBAC check: Does this role even have edit rights generally?
  if (!['admin', 'editor'].includes(userRole)) {
    return res.status(403).json({ error: 'Your role cannot edit articles.' });
  }

  // Admins can edit anything, skip the resource check
  if (userRole === 'admin') {
    return next();
  }

  // 2. Resource/Attribute check: Is this editor the OWNER of the article?
  const article = await db.Article.findByPk(articleId);
  if (!article) return res.status(404).send('Not found');

  if (article.authorId !== userId) {
    return res.status(403).json({ error: 'You can only edit your own articles.' });
  }

  // Pass the article down so the route handler doesn't have to fetch it again
  req.article = article;
  next();
};

app.put('/article/:id', authenticateToken, authorizeArticleEdit, async (req, res) => {
  // Safe to update req.article
});
```
This combines RBAC (must be an editor) with Resource Auth (must be *this* article's editor).

## 8. Performance Table

| Authorization Approach | Performance Impact | Note |
|---|---|---|
| JWT Scopes (RBAC) | Very Low (Under 1ms) | Roles are embedded in the token. No DB lookup required. |
| In-App DB Role Check | Low (2-5ms) | Requires a fast indexed query to the DB. |
| ABAC / Policy Engine | Medium (5-20ms) | Requires evaluating complex logic and often fetching external attributes. |
| External AuthZ SaaS (e.g. Auth0) | High (30-100ms) | Requires a network hop over the internet. Cache aggressively. |

## 9. Top Interview Questions

| Question | Answer |
|---|---|
| When would you choose ABAC over RBAC? | When rules depend on dynamic context (time, location, relationship between user and resource) rather than static job titles. |
| What HTTP status code should be returned for an Authorization failure? | `403 Forbidden`. (`401 Unauthorized` is for Authentication failures). |
| What is a "Scope" in OAuth 2.0? | A string (e.g., `read:messages`) that limits what an application can do on behalf of the user. It is a form of delegated authorization. |
| How do you prevent BOLA (Broken Object Level Authorization)? | By explicitly verifying in code that the logged-in user is the legitimate owner of the requested object ID, not just trusting the ID passed in the URL. |
| What is the Principle of Least Privilege? | Granting users or systems only the minimum access necessary to perform their legitimate function. |

## 10. Tricky Questions & Edge Cases

**Question:** A user's role is updated from Admin to Viewer in the database. But they have a JWT that says they are an Admin, and it doesn't expire for an hour. Can they still act as an Admin?
**Answer:** Yes, bro! This is the inherent tradeoff of stateless JWTs. Unless you implement a token revocation list (which makes it stateful again) or check the DB on every sensitive request, the token remains valid until expiration. Keep JWT lifetimes short to mitigate this window of vulnerability.

**Question:** Is it safe to hide UI buttons (like "Delete") if the user doesn't have the role, and skip backend checks?
**Answer:** Absolutely not. Hiding the UI is purely a UX optimization. An attacker can easily use Postman or `curl` to hit the `/delete` endpoint directly. The backend MUST enforce authorization independently.

## 11. Real-World Usage

Modern architectures often decouple authorization entirely. Netflix and Airbnb use policy engines to evaluate access. Instead of the microservice writing `if (user.role == 'X')`, the microservice asks a sidecar proxy (like Open Policy Agent): "Can User A do Action B on Resource C?" The sidecar evaluates centrally managed policies and replies True/False.

## 12. Best Practices

| DO | DON'T |
|---|---|
| Centralize authorization logic in middleware or dedicated policy files. | Scatter `if` statements checking roles deep inside business logic. |
| Prefer Resource-Based authorization (checking ownership) over blind RBAC. | Assume that because someone is an "Admin", they own everything automatically. |
| Default to Deny. If a rule doesn't explicitly allow it, block it. | Default to Allow and try to block bad actions. |
| Audit your authorization logic and log `403` errors for security monitoring. | Ignore `403` spikes; they often indicate an attacker probing your endpoints. |

## 13. Production Notes

> ⚠️ **Warning on ABAC Complexity**
> ABAC is powerful but can become a maintenance nightmare. If you have rules like "User can edit if they are in department X, unless it is a weekend, unless they have the emergency override flag, unless the document is classified", your system becomes incredibly difficult to debug. Start with RBAC. Add Resource ownership checks next. Only adopt true ABAC when absolutely mandated by business/compliance requirements.

## 14. Common Mistakes

| Mistake | Why it's bad | How to fix it |
|---|---|---|
| BOLA (Insecure Direct Object Reference) | Allows a user to view another user's invoice by changing `invoice/1` to `invoice/2`. | Always include the `userId` in the DB lookup: `where: { id: req.params.id, userId: req.user.id }`. |
| Mixing AuthN and AuthZ | Confusing "who the user is" with "what they can do" leads to spaghetti code. | Use separate middlewares. E.g., `app.use(verifyToken)` followed by `app.use(requireAdmin)`. |
| Trusting JWT payloads blindly | An attacker might forge a token if the secret is leaked. | Always verify the cryptographic signature of the JWT before reading its roles or scopes. |
