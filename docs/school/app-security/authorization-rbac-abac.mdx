# Authorization: RBAC and ABAC

## 1. Quick Summary

| Area | Details |
|---|---|
| Topic | Authorization (AuthZ) |
| Difficulty | Intermediate |
| Used For | Determining what an authenticated user is allowed to do within a system |
| Common Mistake | Hardcoding role checks everywhere instead of centralizing logic, leading to privilege escalation |
| Performance | Fast for basic RBAC, but complex ABAC policies can introduce latency and require caching |

## 2. Engineering Story

It was Black Friday 2023. At a mid-sized e-commerce company, a junior developer deployed a highly requested feature: a dashboard for vendors to view their own sales analytics. The authentication (AuthN) was rock solid; every vendor had to log in with 2FA. However, the authorization (AuthZ) was flawed. The developer had added a simple check: `if (user.role === 'VENDOR')`. 

Everything seemed fine until a vendor decided to playfully change the `vendor_id` parameter in the URL from `/dashboard/analytics?vendor_id=402` to `?vendor_id=105`. The API responded with the entire sales history, customer PII, and profit margins of their biggest competitor. The system correctly verified that the user was a vendor (RBAC), but completely failed to verify if the user had the right to access *that specific vendor's data* (Resource-Based Authorization / ABAC). It took less than an hour for screenshots of competitor data to hit Twitter. The platform had to go offline for four hours to patch a classic Broken Object Level Authorization (BOLA) vulnerability. Understanding the difference between roles and contextual attributes is the difference between a secure platform and a massive data breach.

RBAC is like saying: "He has an Engineer badge, he goes to the engineering floor." ABAC is like saying: "He has an Engineer badge, BUT it's Sunday 2 AM and his manager hasn't signed a weekend pass, so the door stays locked."

## 4. Concept Explanation

Authorization (AuthZ) is the process of verifying that an authenticated entity has the necessary permissions to access a specific resource or perform a specific action. It answers the question: "Are you allowed to do this?"

- **RBAC (Role-Based Access Control):** Permissions are tied to roles (Admin, Editor, Viewer). Users are assigned roles. If the role can do the action, the user can do the action. It is static and hierarchical.
- **ABAC (Attribute-Based Access Control):** Permissions are evaluated dynamically based on attributes of the user (department, security clearance), the resource (classification level, owner ID), the action (read, write), and the environment (time of day, IP address).

**Why does it exist?** 
To enforce the **Principle of Least Privilege**. Users and services should only have access to exactly what they need to do their jobs, and absolutely nothing more. This limits the blast radius if an account is compromised.

**When to use it?** 
RBAC is the standard for 90% of applications. Start here. Use ABAC when rules become highly contextual. For example, in healthcare apps, a doctor can only view a patient record IF they are the assigned physician AND they are currently on shift.

## 5. Syntax Table

Here is how different authorization models manifest in code or configuration:

| AuthZ Model | Example Logic | Use Case |
|---|---|---|
| **ACL (Access Control List)** | `if (doc.allowedUsers.includes(user.id))` | Simple file sharing systems (e.g., Google Drive sharing). |
| **RBAC** | `if (user.role === 'ADMIN')` | Standard SaaS apps (Owner / Admin / Member). |
| **ABAC** | `if (user.dept === 'HR' && user.ip === '10.0.0.x')` | High-security, financial, or highly regulated systems. |
| **PBAC (Policy-Based)** | Evaluated via an engine like OPA (Open Policy Agent) | Large-scale microservice architectures where auth is decoupled. |

## 6. Beginner Example

A common beginner mistake is scattering role checks randomly throughout the code, leading to inconsistencies and bypasses.

```javascript
// DON'T: Hardcoding roles in the business logic
app.delete('/article/:id', async (req, res) => {
  // Hard to maintain. What if we add a 'moderator' role later?
  if (req.user.role !== 'admin' && req.user.role !== 'superadmin') {
    return res.status(403).send('Forbidden');
  }
  // Delete logic...
});

// DO: Use middleware to centralize RBAC
function requireRole(role) {
  return (req, res, next) => {
    if (req.user.role !== role) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  }
}

// Clean and declarative route definition
app.delete('/article/:id', requireRole('admin'), async (req, res) => {
  // We confidently know they are an admin if we reach here.
  // Delete logic...
});
```

## 7. Real-World Engineering Example

Bro, in a real system, RBAC isn't enough. You often need **Resource-Based Authorization** (a simple form of ABAC). An Editor can edit articles, but can they edit *your* article?

```javascript
// A robust authorization middleware pattern combining RBAC and Resource Checks
const authorizeArticleEdit = async (req, res, next) => {
  const articleId = req.params.id;
  const userId = req.user.id;
  const userRole = req.user.role;

  // 1. RBAC check: Does this role even have edit rights generally?
  if (!['admin', 'editor'].includes(userRole)) {
    return res.status(403).json({ error: 'Your role cannot edit articles.' });
  }

  // Admins can edit anything, skip the resource check
  if (userRole === 'admin') {
    return next();
  }

  // 2. Resource/Attribute check: Is this editor the OWNER of the article?
  const article = await db.Article.findByPk(articleId);
  if (!article) return res.status(404).send('Not found');

  // This prevents Broken Object Level Authorization (BOLA)
  if (article.authorId !== userId) {
    return res.status(403).json({ error: 'You can only edit your own articles.' });
  }

  // Pass the article down so the route handler doesn't have to fetch it again
  req.article = article;
  next();
};

app.put('/article/:id', authenticateToken, authorizeArticleEdit, async (req, res) => {
  // Safe to update req.article, authorization is fully verified
});
```

## 9. Performance Table

| Authorization Approach | Performance Impact | Note |
|---|---|---|
| **JWT Scopes (RBAC)** | Very Low (< 1ms) | Roles are embedded in the token payload. No DB lookup required, completely stateless. |
| **In-App DB Role Check** | Low (2-5ms) | Requires a fast indexed query to the database for every protected request. |
| **ABAC / Policy Engine** | Medium (5-20ms) | Requires evaluating complex AST logic and often fetching external environment attributes. |
| **External AuthZ SaaS (e.g. Auth0/Okta)** | High (30-100ms) | Requires a network hop over the internet. Cache aggressively or run a local agent. |

## 10. Top Interview Questions

| Difficulty | Question | Answer |
|---|---|---|
| Intermediate | **When would you choose ABAC over RBAC?** | When rules depend on dynamic context (time, location, relationship between user and resource) rather than static job titles. |
| Intermediate | **What HTTP status code should be returned for an Authorization failure?** | `403 Forbidden`. (`401 Unauthorized` is for Authentication failures). |
| Intermediate | **What is a "Scope" in OAuth 2.0?** | A string (e.g., `read:messages`) that limits what an application can do on behalf of the user. It is a form of delegated authorization. |
| Intermediate | **How do you prevent BOLA (Broken Object Level Authorization)?** | By explicitly verifying in code that the logged-in user is the legitimate owner of the requested object ID, not just trusting the ID passed in the URL. |
| Intermediate | **What is the Principle of Least Privilege?** | Granting users or systems only the minimum access necessary to perform their legitimate function, minimizing potential attack vectors. |

## 11. Tricky Questions & Edge Cases

**Question:** A user's role is updated from Admin to Viewer in the database. But they have a stateless JWT that says they are an Admin, and it doesn't expire for an hour. Can they still act as an Admin?
**Answer:** Yes, bro! This is the inherent tradeoff of stateless JWTs. Unless you implement a token revocation list (which makes it stateful again) or check the DB on every sensitive request, the token remains valid until expiration. Keep JWT lifetimes short (e.g., 15 minutes) to mitigate this window of vulnerability.

**Question:** Is it safe to hide UI buttons (like "Delete") if the user doesn't have the role, and skip backend checks?
**Answer:** Absolutely not. Hiding the UI is purely a UX optimization. An attacker can easily use Postman, `curl`, or browser dev tools to hit the `/delete` endpoint directly. The backend MUST enforce authorization independently. Never trust the client.

## 12. Real-World Usage

Modern architectures at scale often decouple authorization entirely. Companies like Netflix, Airbnb, and Pinterest use policy engines to evaluate access. Instead of the microservice writing `if (user.role == 'X')`, the microservice asks a sidecar proxy (like Open Policy Agent): "Can User A do Action B on Resource C?" The sidecar evaluates centrally managed policies (written in languages like Rego) and replies True or False. This allows security teams to update rules globally without redeploying microservices.

## 13. Best Practices

| DO | DON'T |
|---|---|
| Centralize authorization logic in reusable middleware or dedicated policy files. | Scatter `if` statements checking roles deep inside disparate business logic functions. |
| Prefer Resource-Based authorization (checking ownership) over blind RBAC. | Assume that because someone is an "Admin", they own everything automatically. |
| Default to Deny. If a rule doesn't explicitly allow it, block it. | Default to Allow and try to write rules to block bad actions (blacklist approach). |
| Audit your authorization logic and log `403` errors for security monitoring. | Ignore `403` spikes; they often indicate an attacker probing your endpoints or a BOLA attempt. |

## 14. Production Notes

> ⚠️ **Warning on ABAC Complexity & Latency**
> ABAC is extremely powerful but can become a maintenance and performance nightmare. If you have rules like "User can edit if they are in department X, unless it is a weekend, unless they have the emergency override flag, unless the document is classified", your system becomes incredibly difficult to debug. Furthermore, fetching all those attributes on every API call can introduce massive latency. Start with RBAC. Add Resource ownership checks next. Only adopt true ABAC when absolutely mandated by business/compliance requirements, and employ heavy caching (like Redis) for user attributes.

## 15. Common Mistakes

| Mistake | Why it's bad | How to fix it |
|---|---|---|
| **BOLA (Insecure Direct Object Reference)** | Allows a user to view another user's invoice by simply changing `invoice/1` to `invoice/2` in the URL. | Always include the `userId` in the DB lookup: `where: { id: req.params.id, userId: req.user.id }`. |
| **Mixing AuthN and AuthZ** | Confusing "who the user is" with "what they can do" leads to spaghetti code and security gaps. | Use separate middlewares. E.g., `app.use(verifyToken)` followed by `app.use(requireAdmin)`. |
| **Trusting JWT payloads blindly** | An attacker might forge a token if the secret is leaked, granting themselves 'Admin' roles. | Always verify the cryptographic signature of the JWT before reading its roles or scopes. |
| **Frontend-Only Enforcement** | Attackers bypass the UI entirely and send API requests directly. | The frontend UI should reflect permissions for UX, but the backend API is the absolute source of truth. |
